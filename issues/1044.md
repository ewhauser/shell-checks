# [\#1044 Issue](https://github.com/koalaman/shellcheck/issues/1044) `closed`: SC2221/2222 don't take into account bash 4 ;;& terminator

#### <img src="https://avatars.githubusercontent.com/u/19507282?v=4" width="50">[automorphism88](https://github.com/automorphism88) opened issue at [2017-11-09T23:57:25Z](https://github.com/koalaman/shellcheck/issues/1044):

I tried on shellcheck.net and verified that this is still a problem on the latest commit. It doesn't happen with my locally installed version (0.4.6).

Suppose I have a case statement, where I have shared code between cases. To avoid code duplication, bash 4 allows ending case blocks with `;;&` or `;&` instead of `;;`. When using `;;&`, bash will continue through the remaining case blocks looking for matches, instead of jumping straight to `esac` as with `;;`. 
#### Here's a snippet or screenshot that shows the problem:
```
#!/bin/bash
case $1 in
    foo|bar)
        echo foo bar
        ;;&
    foo)
        echo foo
        ;;
    bar)
        echo bar
        ;;
esac
```

The above is perfectly valid in bash 4 (`$1 = foo` will execute the first and second blocks and `$1 = bar` will execute the first and third), but shellcheck will find two errors, SC2221 with the `foo|bar)` line since it overrides the following lines `foo)` and `bar)`, and SC2222 on the latter two lines for the same reason. Shellcheck should notice that the `foo|bar)` block ends with `;;&` and evaluate the case statement accordingly.

#### <img src="https://avatars.githubusercontent.com/u/19507282?v=4" width="50">[automorphism88](https://github.com/automorphism88) commented at [2017-12-23T23:52:12Z](https://github.com/koalaman/shellcheck/issues/1044#issuecomment-353754911):

> Just one thought here, bash version 4 differs much from v3 which is more like dash, wouldn't it make sense to have it as its own variant? If you intend the code to run on v3 it is a legit error.

The problem is that you can't tell from the shebang if it is intended to be a bash 3+ or bash 4+ script. It's possible a bash 3 mode could be enabled with a command line option, but otherwise, if a bash 4 feature is used, in a script with a bash shebang, it seems a reasonable inference that the script is intended to be used with bash 4, especially considering that it came out in 2009, an eternity ago in software terms. Anyone running a bash version that ancient should expect that it will be likely to break modern scripts. The main reason anyone is still using bash 3 is that Apple doesn't like the GPL v3, so Mac OS users have to install a modern version manually.

Also, this warning isn't even triggered by the bash 4 feature itself being used (i.e., the case block ending with `;;&` instead of `;;`), but just when it's used in a specific way. In general shellcheck does not warn about scripts that will run on bash 4 but not bash 3. Furthermore, bash v4 is not the only time shells have introduced significant new features with different versions. It might be useful to have a tool to identify the oldest version of bash that a script could run on, but shellcheck isn't that tool.
