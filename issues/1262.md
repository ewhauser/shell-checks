#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-07-22T18:09:20Z](https://github.com/koalaman/shellcheck/issues/1262#issuecomment-406885764):

I definitely like the idea of building the fix where it's found. I think it's reasonable to have a SuggestedFix and add it to the PositionedComment (or `Maybe SuggestedFix`).

Immediately before, it would probably also be a good idea to make everything in ShellCheck.Interface more opaque, so that we can add fields to PositionedComment without breaking [hadolint](https://github.com/hadolint/hadolint) again. I added #1300 for this.

[error-prone](https://github.com/google/error-prone) is the only auto-fixing tool I'm vaguely familiar with. Their idea of a fix was a list of (StartPos, EndPos, ReplacementString) tuples per suggestion, e.g. `fgrep foo -> [ (0,1, ""), (6,6, "-F ") ]`, and afaik it was up to the consumer to unify multiple fixes affecting the same line. That seems like a pretty reasonable approach, though it's worth looking into how other tools do it.

One fancy possibility once this is done is a `diff` output format to let you do e.g.  `shellcheck -f diff file.sh | patch`

I find it tricky to construct a text span replacement at the point where we are running the checks, e.g. in https://github.com/koalaman/shellcheck/blob/master/src/ShellCheck/Analytics.hs#L1627

- we only have an ID and the token
- to construct the replacement we need the original text
- to get the original text we have to read from source file

I think separating the fix part into two steps:

1. creating a Fix that is bash syntax specific, i.e. surround token at this ID with double quotes
2. taking that Fix and output arbitrary text replacements, like diff

can possibly make things easier.

Maybe 2 can be a step that is common to all output formatters, that converts a bash-specific fix, into a text replacements.

#### <img src="https://avatars.githubusercontent.com/u/1749303?v=4" width="50">[ngzhian](https://github.com/ngzhian) commented at [2018-09-09T05:48:15Z](https://github.com/koalaman/shellcheck/issues/1262#issuecomment-419692733):

Making up an imaginary API:

```
-- low level
replace_start :: Token -> String -> String -> Fix -- replace matching string at the start of token
replace_end :: Token -> String -> String -> Fix -- replace matching string at the end of token
-- OR
replace_index :: Token -> Int -> Int -> String -> Fix -- replace a range within Token by a String

-- helpers
insert_before :: Token -> String -> Fix -- implemented in terms of replace_*
insert_after :: Token -> String -> Fix -- implemented in terms of replace_*
surround :: Token -> String -> Fix -- implemented in terms of insert_before and insert_after
surround2 :: Token -> Token -> String -> Fix -- for surrounding multiple tokens, implemented with insert_before and insert_after
```

4, can be
- ``replace_start token "`" "$("`` and ``replace_end token "`" ")"``, or
- `replace_index token 0 1 "$("` and `replace_index token (end_column-1) end_column ")"`


8, can be `insert_after find_token " . "`, and maybe replace start and end with empty strings to remove the `$(echo ... )`

10, is tricky, we can maybe do an `insert_after source_token "\nsource"`, and in this case we can get the value "source" at warning time because this will be a simple variable name.

> There's also totally the possibility of just adding source level access to let each check generate a fix, or alternatively to have them return a "FixGenerator" function that optionally runs with access to such data if the fix is needed.

This is doable too, instead of generating a new data type `Fix`, we can use some helper methods to create lambdas that can be passed to the formatter.

I guess the most tricky part I haven't thought much about is applying multiple fix that changes the position of the token. Whichever way we go with that makes this easy to do is probably the better way.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-09-22T04:51:06Z](https://github.com/koalaman/shellcheck/issues/1262#issuecomment-423716659):

I haven't had a chance to evaluate the API yet, but the ability to hit "Fix" in an interactive editor is really the least interesting use case, because it's only marginally more work to fix it manually.

I imagine the more useful case will be "Fix All" to auto-fix all instances in the file/dir, and for code review tool integration where you can hit "Suggest fix" to automatically submit a comment complete with a code replacement suggestion.

Merging multiple fixes per line will indeed be annoying when they affect the same ranges, but merge conflicts always are. Using character range replacements rather than line replacements helps with this.
