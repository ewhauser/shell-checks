#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash

set -u

fn1() {
        local foo=foo
        echo "$foo"
}

fn2() {
        echo "$foo"  # foo is undefined here because it is local to fn1 and not called from fn1.
}

fn1
fn2
```

#### Here's what shellcheck currently says:

Nothing. :-(


#### Here's what I wanted or expected to see:
 
`foo` is not defined in `fn2` because it was declared local in `fn1` so ShellCheck should warn about this. I realize scoping is dynamic, not lexical but I think even a lexical analysis with a warning such as:

```
Line 11:
        echo "$foo"
              ^-- SCXXXX: foo is referenced but may not be assigned.
```

Maybe related to #1497 and #1225. Possible dupe of #518.

#### <img src="https://avatars.githubusercontent.com/u/6006766?v=4" width="50">[nexushoratio](https://github.com/nexushoratio) commented at [2025-12-28T06:11:18Z](https://github.com/koalaman/shellcheck/issues/1611#issuecomment-3694490324):

```shell
#!/bin/bash

set -u

a() {
  local x=0
}

b() {
  echo "${x}"
}
```

It has both an unused variable and and undefined variable.

But, what about this:

```shell
#!/bin/bash

set -u

a() {
  local x=2
  c
}

b() {
  echo "${x}"
}

c() {
  b
}

a
c
```

`b` can see `x` when `a` calls `c`.
But when `c` is called directly, no `x` is present.

---

In a larger script, perhaps `b` gets called from a large variety of paths.  So each one would have to be investigated to see if `x` is present in each path.  And chasing each path would get very resource intensive.
