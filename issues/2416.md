# [\#2416 Issue](https://github.com/koalaman/shellcheck/issues/2416) `open`: Add rule for preventing stray "set -x"

#### <img src="https://avatars.githubusercontent.com/u/1106524?u=69edb2008f90b91957877026d69347bf01c666c0&v=4" width="50">[ZeroEcks](https://github.com/ZeroEcks) opened issue at [2021-12-16 00:56](https://github.com/koalaman/shellcheck/issues/2416):

#### For new checks and feature suggestions
- [x] https://www.shellcheck.net/ (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related


#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/sh
set -x

SECRET="secret-here"

use-secret $SECRET
```

#### Here's what shellcheck currently says:

Nothing

#### Here's what I wanted or expected to see:

Do not use debug mode

#### Reasoning

I would like to ensure that developers do not commit `set -x` to repositories, to prevent the accidental leaking of secrets from shell scripts, debug logging such as `set -x` should also not be committed in general imo. There can be a flag to allow it perhaps?

#### <img src="https://avatars.githubusercontent.com/u/1106524?u=69edb2008f90b91957877026d69347bf01c666c0&v=4" width="50">[ZeroEcks](https://github.com/ZeroEcks) commented at [2022-01-11 00:57](https://github.com/koalaman/shellcheck/issues/2416#issuecomment-1009495819):

The issue I am trying to prevent, is a way to fail CI builds if developers accidentally commit `set -x` to a deployment script which may include variables such as `AWS_ACCESS_KEY` or similar secrets as environment variables. This would lead to secrets being logged.

You also generally, would not want to commit debug statements to repositories in my experience?

The argument they could run it as `bash -x script.sh` doesn't apply since CI wouldn't do that (unless on purpose, but then they would be acting maliciously)

I think it would be beneficial to have to acknowledge that you are committing debug statements that could potentially expose environment variables, since you can always ignore the warning for specific lines.
