#### <img src="https://avatars.githubusercontent.com/u/7482557?v=4" width="50">[Gandalf-](https://github.com/Gandalf-) commented at [2019-12-19T04:43:51Z](https://github.com/koalaman/shellcheck/issues/1726#issuecomment-567330184):

You could use a directive https://github.com/koalaman/shellcheck/wiki/Directive

An alternative is to use escaping with double quotes
```shell
#!/bin/bash
foo=( $(compgen -W "\$(ls /)" -- l) )
echo "${foo[@]}"
```
This tells Shellcheck (another way) that you mean a literal `$` here. 

I think the intent behind SC2016 is good, since something like
`var='$( head -c 10 /dev/urandom )'`
and
`var="$( head -c 10 /dev/urandom )"`
could be easy for a beginner to miss.

Trade offs like this are probably necessary, but maybe we don't need this one?

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-09-26T03:12:54Z](https://github.com/koalaman/shellcheck/issues/1726#issuecomment-927221254):

You're right, these are all false positives. ShellCheck whitelists a few commands to try to improve the signal:noise ratio, but there's still an infinite long tail of commands where literal expansion-like arguments are correct.

Like Gandalf- says, it's most definitely intended for beginners who often assume single and double quotes are interchangeable like in Python or JavaScript. It's not at all meant to stop seasoned scripters from using single quotes when they want literal dollar signs in strings, even though that is tragically often what happens.

Unfortunately, this remains one of the most common newbie problems, it's hard for ShellCheck to avoid it (suggestions welcome).

It's obviously never great for tools to complain about correct code, but please consider helping out the newbies by instead perma-banishing it with `echo "disable=SC2016" >> ~/.shellcheckrc` when you're familiar with the difference.
