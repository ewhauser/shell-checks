#### <img src="https://avatars.githubusercontent.com/u/1793802?u=47c38c4b942fe6f0843f247ea685a6c0faf2f158&v=4" width="50">[Dmole](https://github.com/Dmole) commented at [2021-11-27 18:27](https://github.com/koalaman/shellcheck/issues/2389#issuecomment-980781301):

Firstly 

    ls -1

is more likely to be what you want.
Secondly grep supports non-alphanumeric eg;

    grep -P "[\w\W]"

Thirdly shellcheck often wants null terminated lines like this;

    find . -maxdepth 1 -print0 | grep -Z -A9999 "^./$1\$"

so it looks like shellcheck is showing the wrong error vs a false positive.

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-12-12 02:11](https://github.com/koalaman/shellcheck/issues/2389#issuecomment-991821707):

ShellCheck is right in that this construct is causing the function to fail for non-alphanumeric filenames, and that using a for loop instead would fix it. For example, if you `touch file{' 1',' 2',3}.txt` then it will output

```
file 1.txt\nfile3.txt\n
file 2.txt\nfile3.txt\n
```

but fail to realize that `file 1.txt` is also the same as `file 2.txt`.

If you replaced `ls | grep` with a condition in a for loop as suggested, e.g.

```
alldiff(){
 for file1 in *; do
  found=0
  for file2 in *; do
    if [[ $found = 0 ]]
    then
      [[ $file1 = $file2 ]] && found=1
      continue
    fi

    diff -- "$file1" "$file2" &> /dev/null
    if [[ $? == 0 ]]; then
      echo "$file1\n$file2\n"
    fi
  done
 done
}
```

then it would work for all filenames (though an array with two arithmetic for loops would be even better)

You can definitely choose to make the call that this extra complexity is not worth the improved correctness given your system's filenames (in which case you can ignore it with a directive), but since the suggestion is valid and useful, it does not make sense to suppress it generally.
