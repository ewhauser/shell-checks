#### <img src="https://avatars.githubusercontent.com/u/84648683?v=4" width="50">[wileyhy](https://github.com/wileyhy) commented at [2025-07-12 03:43](https://github.com/koalaman/shellcheck/issues/3118#issuecomment-3064614765):

OP's code (sort of):

#!/bin/bash
set -- Aa Bb Cc
[[ "${@:2:1}" ]] \
  && printf '                %s\n' "${@:2}"

CLI output:
  <sh>
                 Bb
                 Cc
  </sh>

But first, a few tangents:

The `[[` token, in bash, is a "reserved word," a "compound command," and
part of a "conditional expression."

The list of conditional expressions is:
  A) system command `/bin/test`
  B) system command `/bin/[`
  C) shell builtins `test` and `[`
  D) keyword `[[`

The builtins are synonymous with each other, but each of A-D behave a
little differently, and only (D) exhibits the behavior described below. The
man page:

  "The words
      between the [[ and ]] do not
      undergo word splitting and pathname
      expansion."

This sentence means that double quoting a Parameter Expansion in a `[[`
compound command is unnecessary.

Within `[[`, '$*' is concatenated into a single string. No such file "Aa
Bb" exists, yet.

  ~ $ set -- Aa Bb
  ~ $ set -x
  ~ $ [[ -e $* ]]; echo $?
  + [[ -e Aa Bb ]]
  + echo 1
  1

Outside of `[[`, to avoid "Word Splitting," '$*' must be double quoted.

  ~ $ touch "$*"
  + touch 'Aa Bb'
  ~ $ ls -l "$*"
  + ls -l 'Aa Bb'
  -rw-------. 1 u0_a151 u0_a151 0 Jul 11 19:49 'Aa Bb'
  ~ $ [[ -e $* ]]; echo $?
  + [[ -e Aa Bb ]]
  + echo 0
  0

'$@' and '$*' behave differently from each other in this scenario. Within
`[[` an unquoted '$@' is implicitly converted to an unquoted '$*', so,
within `[[` an unquoted '$@' expands as a single string.

Resetting tests.

  ~ $ rm -f "$*"
  + rm -f 'Aa Bb'

The file 'Aa Bb' DNE again.

~ $ [[ -e $@ ]]; echo $?
+ [[ -e Aa Bb ]]
+ echo 1
1

Two separate files are created. '$@' and '"$@"' expand identically in this
scenario because none of the array elements contain IFS characters.

~ $ touch "$@"
+ touch Aa Bb
~ $ ls -l "$@"
+ ls -l Aa Bb
-rw-------. 1 u0_a151 u0_a151 0 Jul 11 19:50 Aa
-rw-------. 1 u0_a151 u0_a151 0 Jul 11 19:50 Bb

The test still fails because within `[[`, '$@' is expanded as a single
string.

~ $ [[ -e $@ ]]; echo $?
+ [[ -e Aa Bb ]]
+ echo 1
1

Creating a file with a name reflecting how '$@' expands within `[[` allows
the test to return true.

~ $ touch "Aa Bb"
+ touch 'Aa Bb'
~ $ ls -l "Aa Bb"
+ ls -l 'Aa Bb'
-rw-------. 1 u0_a151 u0_a151 0 Jul 11 19:51 'Aa Bb'
~ $ [[ -e $@ ]]; echo $?
+ [[ -e Aa Bb ]]
+ echo 0
0

So, double quoting is extraneous inside a `[[` ...except in certain cases
of "Pattern Matching," etc, but that's something else.

OP: "Appears to miss the fact that this is a single value and there's no
concatenation possible, i.e., because of the special case of "$@"."

Concatenation is indeed possible as we've seen.

Also, with `printf`, both system command and shell builtin, there is a
format sequence:

  \t    horizontal tab

That makes it easier to read.
