# [\#764 Issue](https://github.com/koalaman/shellcheck/issues/764) `open`: SC2028 false positive for echo -e alias

#### <img src="https://avatars.githubusercontent.com/u/2199511?v=4" width="50">[Nightfirecat](https://github.com/Nightfirecat) opened issue at [2016-10-28T18:21:33Z](https://github.com/koalaman/shellcheck/issues/764):

When running `echo` after aliasing it as `echo -e`, Shellcheck still warns about SC2028. Test case is as follows:

```
cat << EOF > test.sh
#!/bin/bash
shopt -s expand_aliases # make bash respect 'alias' in non-interactive shell
echo '1\n2'
echo -e '1\n2'
alias echo='echo -e'
echo '1\n2'
echo -e '1\n2'
EOF
bash test.sh
shellcheck test.sh
```

The resulting output is as follows:

```
1\n2
1
2
1
2
1
2

In test.sh line 3:
echo '1\n2'
     ^-- SC2028: echo won't expand escape sequences. Consider printf.


In test.sh line 6:
echo '1\n2'
     ^-- SC2028: echo won't expand escape sequences. Consider printf.
```

I would expect it to be as follows:

```
1\n2
1
2
1
2
1
2

In test.sh line 3:
echo '1\n2'
     ^-- SC2028: echo won't expand escape sequences. Consider printf.
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-02-16T00:19:53Z](https://github.com/koalaman/shellcheck/issues/764#issuecomment-280187051):

While using aliases in scripts is iffy, there's nothing inherently wrong about overriding a command with a function in the same way. ShellCheck is just currently not smart enough to recognize this (and I imagine many humans would find it tricky too)
