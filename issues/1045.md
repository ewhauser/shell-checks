# [\#1045 Issue](https://github.com/koalaman/shellcheck/issues/1045) `open`: SC2154 doesn't understand bash brace expansion

#### <img src="https://avatars.githubusercontent.com/u/19507282?v=4" width="50">[automorphism88](https://github.com/automorphism88) opened issue at [2017-11-13T02:20:19Z](https://github.com/koalaman/shellcheck/issues/1045):

#### For bugs
- Rule Id (if any, e.g. SC1000): SC2154
- My shellcheck version (`shellcheck --version` or "online"): This error occurs both on shellcheck.net and with my locally installed version (0.4.6).
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit
- [ ] It's not reproducible on shellcheck.net, but I think that's because it's an OS, configuration or encoding issue

#### Here's a snippet or screenshot that shows the problem:

```
#!/bin/bash
set -u

my_function() {
    local field{1,2}
    while read -r field{1,2} ; do
        echo "$field1" "$field2"
    done < <(echo foo bar)
}

my_function
```

#### Here's what shellcheck currently says:

```
Line 7:
        echo "$field1" "$field2"
              ^-- SC2154: field1 is referenced but not assigned.
                        ^-- SC2154: field2 is referenced but not assigned.
```

#### Here's what I wanted or expected to see:

When the above script is run, bash's brace expansion replaces `field{1,2}` with `field1 field2` before running the `local` and `read` commands, and this brace expansion occurs before word splitting, so `field1` and `field2` are considered separate arguments. Therefore, the variables `field1` and `field2` have in fact been assigned (and the script will complete successfully, with output `foo bar`, despite the `set -u` at the top), but shellcheck falsely reports that they have not been assigned, presumably because it does not itself perform the brace expansion, and thinks that we are specifying a single variable named `field{1,2}` as an argument to `read`.
