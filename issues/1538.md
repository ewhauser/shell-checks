#### <img src="https://avatars.githubusercontent.com/u/14194483?v=4" width="50">[mhoes](https://github.com/mhoes) commented at [2019-04-02T15:45:30Z](https://github.com/koalaman/shellcheck/issues/1538#issuecomment-479060086):

If you want to test if a string is a number, you could also write it as the code below, which doesn't get you a syntax error if the tested string is not an integer :

```
#!/bin/bash

case $1 in
(*[!0-9]*|'')
        echo not a number
        ;;
*)
        echo a number
        ;;
esac
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-04-04T22:26:14Z](https://github.com/koalaman/shellcheck/issues/1538#issuecomment-480085816):

You're right that this was a false positive. Your code was fine.

This is a conscious design decision though.  It helps catch errors like `[ 2>1 ]`, which is also a perfectly valid POSIX example of redirecting stderr inside `[ .. ]`, but no one would guess that this is what's going on.

Putting any actual redirections after the `]` is hopefully a small price to pay for improved diagnostics, and it doesn't change the meaning of the redirection.
