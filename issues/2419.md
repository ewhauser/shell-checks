#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/ksh

set -o xtrace

param='172.31.2.13/24'

: Shellcheck considers this a syntax error
if [[ "${param}" == {3}({1,3}(\d).){1,3}(\d)* ]]; then
    printf '== matched\n'
else
    printf '== did not match\n'
fi

: Shellcheck says this should be correct, but is not.
if [[ "${param}" =~ {3}({1,3}(\d).){1,3}(\d)* ]]; then
    printf '=~ matched\n'
else
    printf '=~ did not match\n'
fi
```

#### Here's the output from running the script:

```shell
+ param=172.31.2.13/24
+ : Shellcheck considers this a syntax error
+ [[ 172.31.2.13/24 == {3}({1,3}(\d).){1,3}(\d)* ]]
+ printf '== matched\n'
== matched
+ : Shellcheck says this should be correct, but is not.
+ [[ 172.31.2.13/24 == ~(E){3}({1,3}(\d).){1,3}(\d)* ]]
+ printf '=~ did not match\n'
=~ did not match
```

You can see that when executed, ksh converts `=~` into `== ~(E)`, which is an extended regular expression with different behavior and the match fails.

`ksh(1)` man page includes the following:

>     string == pattern
>                   True, if string matches pattern.  Any part of pattern can
>                   be quoted to cause it to be matched as a string.  With a
>                   successful match to pattern, the .sh.match array variable
>                   contains the match and sub-pattern matches.

<!-- -->

>     string =~ ere
>                   True if string matches the pattern ~(E)ere where ere is an
>                   extended regular expression.

Shellcheck should be able to understand this correct syntax with ksh.

FWIW, this happens at least on on macOS (`/bin/ksh`) and illumos (`/usr/bin/ksh93`) (and probably Solaris) platforms. I haven't tested this with other ksh variants (e.g., mksh, pdksh).

This is indeed due to ShellCheck supporting `*(..)` and `@(..)` but not ksh specific `{n}(..)`, nor the ksh parenthesis escaping like `\d` for `[[:digit:]]`. It certainly should.
