#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
declare -A arr=( )
arr+=( "value" )

```

When run in bash, this yields an error:

    bash4: arr: "value": must use subscript when assigning associative array

#### Here's what shellcheck currently says:

    No issues detected!

#### Here's what I wanted or expected to see:

    += cannot be used with an associative array

> #### Here's what I wanted or expected to see:

> ```
> += cannot be used with an associative array

But it can:

```
declare -A X=([a]=2 [b]=3 [c]=5)
X+=([d]=6)
declare -p X
```

produces as output:

```
declare -A X='([a]="2" [b]="3" [c]="5" [d]="6" )'
```

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-11-04T13:39:22Z](https://github.com/koalaman/shellcheck/issues/768#issuecomment-258433263):

> the specific code (providing only a value but no key) remains an exemplar of an uncaught error.

Agreed, from a human perspective.

But variables and functions have dynamic scope rather than lexical scope,
which means all sorts of "obvious" stuff isn't half as obvious as you think
it should be.

Usually it's obvious, and shellcheck does do some value-tracking, but in the
most general case you can't tell whether a "declare" statement is
encountered before an array-append statement (unless you actually run (or
emulate) the program), so you cannot tell by inspection whether or not a
variable is an associative array at a given point in the program.
