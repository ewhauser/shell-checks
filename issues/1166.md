#### Here's a snippet or screenshot that shows the problem:

The following snippet works as expected. The script exits if `"${array[zero]}"` is either null or unset. ShellCheck, however, seems to ignore this syntax and keeps reporting [SC2154](https://github.com/koalaman/shellcheck/wiki/SC2154).
```sh
#!/bin/bash
echo "${array[zero]:?'Index is undefined!'}"
```

#### Here's what shellcheck currently says:

```sh
Line 2:
echo "${array[zero]:?'Index is undefined!'}"
      ^-- SC2154: array is referenced but not assigned.
```

I can do as ShellCheck asks and that disables ShellCheck's warning - at the cost of breaking the intended functionality. Everything after `:?` is treated as the error message which is this case will read, `array: Array is undefined![zero]`.
```sh
#!/bin/bash
echo "${array:?'Array is undefined!'[zero]}"
```

#### Here's what I wanted or expected to see:
It'd be nice if ShellCheck allowed silencing this warning for an array index instead of just for an array as a  whole. My current workaround is to use something like `echo "${array:?'Array is undefined!'}"` to silence the warning _*for the array*_, then use `echo "${array[zero]:?'Index is undefined!'}"` where I want it. The second one will still correctly trigger an `exit` if the index is undefined or null, but its syntax is not currently recognized by ShellCheck.

#### <img src="https://avatars.githubusercontent.com/u/978070?v=4" width="50">[Contik](https://github.com/Contik) commented at [2018-03-31T21:20:58Z](https://github.com/koalaman/shellcheck/issues/1166#issuecomment-377724261):

In this case it's created in a function `functionName`. Its last command is `declare -p array` which I capture on the calling side via `eval "$(functionName)"`. Thus `"${array[@]}"` and its indices are typically available in my script but not declared there.

I can use `"${array:?'Array is undefined!'}"` to `exit` when the array is undefined or null, and ShellCheck recognizes that. Something like `"${array[zero]:?'Index is undefined!'}"` would be neat (with ShellCheck recognizing it as an alternative to the SC2154 warning).
