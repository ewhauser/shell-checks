# [\#643 Issue](https://github.com/koalaman/shellcheck/issues/643) `open`: Add a check for checking return code after a `local` or `declare` line

#### <img src="https://avatars.githubusercontent.com/u/17790097?v=4" width="50">[thegranddesign](https://github.com/thegranddesign) opened issue at [2016-04-05T19:09:12Z](https://github.com/koalaman/shellcheck/issues/643):

Given:

``` sh
declare foo="$(some_script_that_fails_with_exit_code_200)"

[ $? == 0 ] && echo 'yes'
```

Will always echo `yes` because the return code is the code from the `declare` command and not from the subshell.  The proper way to write this would be:

``` sh
declare foo

foo="$(some_script_that_fails_with_exit_code_200)"

[ $? == 0 ] && echo 'yes'
```

It would be great if there was a check that could catch this.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-04-10T12:00:33Z](https://github.com/koalaman/shellcheck/issues/643#issuecomment-207971151):

> The proper way to write this would be:
> 
> ``` sh
> declare foo
> 
> foo="$(some_script_that_fails_with_exit_code_200)"
> 
> [ $? == 0 ] && echo 'yes'
> ```

Ahem, I think you will find that Shellcheck recommends AGAINST that
expression, on two counts.

Firstly the '==' test is not POSIX, and we assume you're using [ ] instead
of [[ ]] because you care about POSIX.

Secondly, NEVER test for $? just against zero; that's vacuous line noise for
no good reason.

Instead use either

```
  if foo="$(some_script_that_fails_with_nonzero_exit_code)"
  then
    echo yes
  fi
```

or

```
  foo="$(some_script_that_fails_with_nonzero_exit_code)" &&
  echo yes
```

Only use $? if you need to keep it for later use, or to make a distinction
other than zero/nonzero. (And if you're doing the latter, take care that $?
doesn't change half way through doing your test -- so for safety, treat it
as the first case: stash it for later use, and then do whatever combination
of tests you actually want. And this turns out to be handy when you want to
print out debug or trace messages.)

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2023-07-21T11:37:26Z](https://github.com/koalaman/shellcheck/issues/643#issuecomment-1645446059):

@pikrzysztof depending on what you mean by "bad assignment" there are two responses, but both have the same outcome.

 `x=$5` can never have a non-zero exit status, so there's nothing for `local` to hide.

When `set -u` is in effect and an unset parameter is expanded, the Shell exits before the assignment is done (which is before `local` is invoked), so again, it can't hide anything.
