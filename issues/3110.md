# [\#3110 Issue](https://github.com/koalaman/shellcheck/issues/3110) `open`: False positive on SC2191 when using `+=` assignment operator

#### <img src="https://avatars.githubusercontent.com/u/384987?u=600c3a111e6aea7056880fd9424480316bf0a455&v=4" width="50">[Xophmeister](https://github.com/Xophmeister) opened issue at [2024-12-31 14:06](https://github.com/koalaman/shellcheck/issues/3110):

#### For bugs
- Rule Id (if any, e.g. SC1000): SC2191
- My shellcheck version (`shellcheck --version` or "online"): 0.10.0
- [x] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2086)
- [x] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/bash

x=(foo bar [1]+=quux)
echo "${x[@]}"
```

#### Here's what shellcheck currently says:

```
In - line 3:
x=(foo bar [1]+=quux)
               ^---^ SC2191 (warning): The = here is literal. To assign by index, use ( [index]=value ) with no spaces. To keep as literal, quote it.

Did you mean:
x=(foo bar [1]+"=quux")

For more information:
  https://www.shellcheck.net/wiki/SC2191 -- The = here is literal. To assign ...
```

#### Here's what I wanted or expected to see:

This shouldn't trigger SC2191.

Note: Only the `+=` operator works in this context, even with `typeset -i`; `++` and `-=` trigger syntax errors in Bash, whereas `--`, `*=` and `/=` get interpreted as literals.
