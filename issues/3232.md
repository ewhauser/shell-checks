# [\#3232 Issue](https://github.com/koalaman/shellcheck/issues/3232) `open`: Request: Warn on unquoted/unescaped wildcards

#### <img src="https://avatars.githubusercontent.com/u/5316?u=8763bdb9ba5cbe23084e8249dcbd21bbb23150f2&v=4" width="50">[mcandre](https://github.com/mcandre) opened issue at [2025-06-27 17:43](https://github.com/koalaman/shellcheck/issues/3232):

Raw string literal wildcards in shell scripts unfortunately carry portability problems, version compatibility problems, and generally unintuitive, unsafe behavior.

For example, bash users may confuse single globs vs. double globs. Double globs tend to fail for vintage, pre-bash-v3 environments, as well as minimalistic environments where the interpreter is `sh` but the bash-like implementation is dash, posh, etc. That's a ton of low level detail that nobody should have to discover, remember, and consistently apply from script to script.

Generally recommend configuring `set -f` (or even `set -eufo pipefail`) in most executable shell scripts.

Instead of using pathname expansion wildcards like tilde (`~`), asterisk (`*`), etc., shell script authors should use safer mechanisms, such as:

* Explicitly list all pathnames out in full
* Apply UNIX `find` commands to match pathnames based on meaningful queries
* Operate on higher level abstractions, such as an entire directory tree
* Ensure that CLI tools (shell or otherwise) feature directory recursion and/or hardcoded lists of relevant application specific pathnames

Wildcards risk expanding too early or too late. Often, wildcards intended for pathname expansion are accidentally quoted, and wildcards intended for purposes other than pathname expansion are unquoted/unescaped. Generally recommend explicitly single quoting, double quoting, or backslash escaping wildcards, so that the interpreter and human authors are more likely to spot the difference.
