#### <img src="https://avatars.githubusercontent.com/u/5316?v=4" width="50">[mcandre](https://github.com/mcandre) commented at [2023-12-24T06:59:58Z](https://github.com/koalaman/shellcheck/issues/58#issuecomment-1868447678):

This kind of support is desperately needed to detect subtle, critical flaws in very many scripts. Nobody is linting for quoting problems in the shell code in their makefiles. Or Jenkinsfiles. Or GitHub Actions. Or Kubernetes container entrypoints. Or legacy Vagrantfiles. Or Ansible playbooks. Or ninja scripts. Or CMake. And so on. Realistically it is safe to assume that most any script or configuration file that involve shell commands, is brittle, and next to no one is even aware of this problem.

At present, the best the user can do is to be incredibly disciplined about moving ALL shell code into dedicated .sh files, linting those pure shell scripts with a suite of linters including ShellCHeck, and then executing those files from the outer programming language. Keeping embedded shell code to a very simple, bare minimum.

If you happen to write *and lint* tons of pure shell scripts yourself, ideally POSIX sh without use of POSIX traps, then you will naturally develop strong, healthy instincts for reliable shell code over time, which you feed into makefiles and other formats. But that kind of discipline is terribly rare in the wild, especially in enterprise systems. Too many teams believe installing a SonarQube CI/CD appliance is the answer to all their code quality problems. (Or Copilot, if you're a manager.) When in fact, each SAST/SAC/compiler/linter tool offers a mutually exclusive list of safeguards, and a full quality report must necessarily involve running complimentary suite(s) of tools.

Nobody in large scale projects seems to understand just how vulnerable this ecosystem is. I haven't found a single DevOps team that has ever bothered to lint their Dockerfiles in terms of Docker best practices, let alone lint their Dockerfiles' RUN/ENTRYPOINT commands shell commands. These little glue code snippets often get ignored, with all the safety and security problems that automatically arise from such negligence.

make itself has an incredibly small linter ecosystem, smaller than even shell's linter ecosystem. I have written an example makefile linter (in Rust), which is able to extract most of the shell commands from a POSIX compliant makefile's list of commands, in the form of a parsing AST. Then the individual commands could be plugged into a linter library such as ShellCheck.

https://github.com/mcandre/unmake

In theory, unmake could export the parser library as object files suitable for C FFI loading into ShellCheck's Haskell FFI bindings. Or someone could even rewrite ShellCheck in Rust. Or you could hook into GNU make's parser, which will be more complete than my fledgeling attempt. Regardless, our users deserve more defensive tools to catch the thousands of basic screwups that shell-adjacent coding tends to invite.

I have a lot of respect for the good work that ShellCheck has been doing to shore up shell scripts. I would very much like to see things like deeper zsh support, and makefile, and Dockerfile support as first tier features, without the ShellCheck user having to install yet more tools to accomplish safety. There does need to be some kind of scope line drawn between what ShellCheck can be reasonably expected to handle, and maybe makefiles + a hundred other outer programming languages, is a bridge too far. So someone needs to step up and build a third party tool, perhaps one tool per wrapping programming language, toward a more reliable Internet.

Unfortunately, make in particular provides nasty pitfalls. Shell commands may be spliced together based on make macros. The final generated shell command may not always be knowable through pure static analysis. Again, discipline is required in order to reduce the prevalence of shell faults.

There are similar problems working with commands in Python, C, or other programming languages. Generally, it is safest there to invoke commands in an exec style via standard library process API. No shell variables, globs, or pipes. Just a lone, plain command string. Any bashisms there, etc. are likely to present portability problems, if not bugs. That's the sort of thing that a programming language specific linter would do well to check for, if the process library makes the mistake of allowing such risky features. Currently, the best they do is warn when the command line is accessed at all (bandit), which assuages a few security fears at the expense of crippling many useful programs.

#### <img src="https://avatars.githubusercontent.com/u/668724?v=4" width="50">[ottok](https://github.com/ottok) commented at [2023-12-28T12:35:07Z](https://github.com/koalaman/shellcheck/issues/58#issuecomment-1871126006):

As said, there is https://github.com/mrtazz/checkmake specifically for Makefiles. For Dockerfiles there is https://github.com/hadolint/hadolint. There is also tools like https://perldoc.perl.org/podchecker for podfiles. 

I don't know how much work it would be to extend ShellCheck to understand new file formats, but my intuition would be to prefer that the same project that owns the respective format (e.g. OCI for Containerfiles) would publish their own file specification and a linter that adheres to it. If is probably too much work for a 3rd party tool to catch up on file format changes in other projects unless there is a very strict and stable standard.

Another avenue would be to extend the editor plugins (e.g https://web.pulsar-edit.dev/packages/linter-shellcheck) to parse out shell invocations from other file types and feed only them to Shellcheck. This is how some spell checking (for human languages) plugins work when they extract English from code comments and UI strings from file types that are otherwise code files.
