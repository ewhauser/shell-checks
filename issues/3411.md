#### <img src="https://avatars.githubusercontent.com/u/84648683?v=4" width="50">[wileyhy](https://github.com/wileyhy) commented at [2026-03-25 14:34](https://github.com/koalaman/shellcheck/issues/3411#issuecomment-4127097806):

Okay. For reference: dynamic scoping of variables (global vs. local), and
use of variables within subshells, ie, `(...)`, `$(...)`, pipelines,
background jobs `&`, etc., are two separate things.

Local variables in bash are created with the shell built-in `local`. dash
and BusyBox sh, as well as zsh, yash and oksh also have a `local` built-in.
ksh does not have `local`, nor does POSIX-2024 specify a `local` built-in.
(I just tested these statements at a command line and looked at the Posix
spec.).

If a shell has the `local` built-in, then its subshells can have both
global and local variables.

In bash and in Linux, a subshell is a command execution environment in a
parent/child relationship with another command execution environment. A
child shell, ie, a subshell, inherits (I believe) an exact copy of the
parent shell's environment, but after a subshell has been created one shell
cannot alter the environment of another shell. That's how subshells can
seem to have dynamic scope.

The `local` built-in allows dynamic scope within just one execution
environment. In bash, the `local` built-in can only be used in functions,
and is limited to within the function where it's defined. "$Foo" can have a
global value of "bar", then within a function "$Foo" can have a local value
of "crazy", then after the function returns "$Foo"s previous value is
visible again.

If you unset a local variable (again, in bash) within a function, the
global variable will remain masked until the function returns. oksh and I
don't know which others makes the global scope visible when a local
variable is unset.
