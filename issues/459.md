# [\#459 Issue](https://github.com/koalaman/shellcheck/issues/459) `closed`: Empty test valid?

#### <img src="https://avatars.githubusercontent.com/u/1042176?v=4" width="50">[Herst](https://github.com/Herst) opened issue at [2015-09-13T09:32:30Z](https://github.com/koalaman/shellcheck/issues/459):

I was trying out the tests from http://tldp.org/LDP/abs/html/testconstructs.html and I got a parser error for the following part:

``` bash
echo "Testing \"NULL\""
if [ ]        # NULL (empty condition)
then
  echo "NULL is true."
else
  echo "NULL is false."
fi            # NULL is false.
```

Now it might not make much sense in the real world, but it seems like empty test expression is valid too (returning a non-true value). I don't know what any standards say though.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2015-09-13T23:31:38Z](https://github.com/koalaman/shellcheck/issues/459#issuecomment-139928323):

TLDP is, shall I say politely, _not_ recommended for guidance on writing shell scripts. (Which is problematic, since that's how it bills itself.) It ignores most corner cases and proffers examples that will fail in tricky-to-detect ways when given something as simple as a filename with a space in it.

So in short, an empty test was always a bug, because it means someone has written `[ $var ]` instead of `[ -n "$var" ]`.

Of course, a syntactically invalid test returns a non-zero status, which is treated as false; the difference is it may also emit a diagnostic.

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-09-18T04:48:01Z](https://github.com/koalaman/shellcheck/issues/459#issuecomment-141343381):

Yeah, the ABS is ... poor ... at best.

An empty `test` expression, while pointless, is valid per spec [test](http://pubs.opengroup.org/onlinepubs/9699919799/utilities/test.html):

> The algorithm for determining the precedence of the operators and the return value that shall be generated is based on the number of arguments presented to test. (However, when using the "[...]" form, the <right-square-bracket> final argument shall not be counted in this algorithm.)
> 
> In the following list, $1, $2, $3, and $4 represent the arguments presented to test:
> 
> 0 arguments:
> &nbsp;&nbsp;&nbsp;&nbsp;Exit false (1).

So shellcheck should probably parse it correctly.

This has been fixed in 5c01b6c7f. They should now parse correctly.

I agree that `false` is a way more descriptive way of doing it, so there's a style suggestion for that. An empty condition in an arithmetic loop `for ((;;))` is always true, but an empty `[ ]` is always false. Might as well be explicit.
