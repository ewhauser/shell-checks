# [\#481 Issue](https://github.com/koalaman/shellcheck/issues/481) `closed`: "brace expansion is not supported" false positive

#### <img src="https://avatars.githubusercontent.com/u/2391761?v=4" width="50">[SpecLad](https://github.com/SpecLad) opened issue at [2015-09-30T11:16:13Z](https://github.com/koalaman/shellcheck/issues/481):

For this code:

``` sh
#!/bin/sh
find -exec echo {} ';'
```

ShellCheck says "SC2039 In POSIX sh, brace expansion is not supported.". But `{}` is not affected by brace expansion, since there are no commas in it. So I think this is a false positive.

#### <img src="https://avatars.githubusercontent.com/u/6459309?v=4" width="50">[Artoria2e5](https://github.com/Artoria2e5) commented at [2015-09-30T13:11:28Z](https://github.com/koalaman/shellcheck/issues/481#issuecomment-144390738):

So does `echo {a}`. In bash, brace expansion is performed only if `,` is found, to afford 'the greatest degree of sh compatibility.' ([bach doc/FAQ](http://git.savannah.gnu.org/cgit/bash.git/tree/doc/FAQ#n1432)) 

> ### D2) Why doesn't bash treat brace expansions exactly like csh?
> 
> The only difference between bash and csh brace expansion is that bash requires a brace expression to contain at least one unquoted comma if it is to be expanded.  Any brace-surrounded word not containing an unquoted comma is left unchanged by the brace expansion code.  This affords the greatest degree of sh compatibility. 
> 
> Bash, ksh, zsh, and pd-ksh all implement brace expansion this way.

You are both correct, well spotted! This has been fixed in 5569292. 

The bash faq does not consider `{a..z}` which expands even without an unquoted comma, but this patch does.
