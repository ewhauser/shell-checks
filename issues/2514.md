I believe the following code is correct, yet it triggers SC2260 and SC3021:

```bash
#!/bin/sh

for fd in 1 2; do
    echo Hello >&"$fd" | grep -o H
done
```

```console

For more information:
  https://www.shellcheck.net/wiki/SC2260 -- This redirection overrides the ou...
  https://www.shellcheck.net/wiki/SC3021 -- In POSIX sh, >& is undefined.
```

Your shebang says `sh`, not `bash`, hence the Shellcheck messages. Shellcheck treats `sh` as the original POSIX shell for compatibility purposes, so you need to explicitly state that your script is a BASH script and not a fully POSIX-compliant shell script.

#### <img src="https://avatars.githubusercontent.com/u/88025?u=f1f16bf9fb10e7f4261ad8871300921572b9ac34&v=4" width="50">[alkisg](https://github.com/alkisg) commented at [2022-06-12 22:38](https://github.com/koalaman/shellcheck/issues/2514#issuecomment-1153310926):

Posix defines output file redirection [there](https://pubs.opengroup.org/onlinepubs/9699919799/utilities/V3_chap02.html#tag_18_07_06):

> The redirection operator:
> [n]>&word
> shall duplicate one output file descriptor from another, or shall close one. **If word evaluates to one or more digits**, the file descriptor denoted by n,

I think the part that I made bold means that `word` can be a variable, like in my script, where $fd is 1 or 2.

Shellcheck doesn't complain about redirection in dash when constants are used:

```console
$ cat script.sh
#!/bin/dash
echo Hello >&1 | grep -o H
echo Hello >&2 | grep -o H

$ shellcheck script.sh 
<no output>
```

It only complains when `word` is a variable. I think by mistake it's expecting "a constant number" there instead of "an expression", and when it doesn't see a constant, it applies the last statement of the posix definition, "If word evaluates to something else, the behavior is unspecified". But in my case it evaluates to a digit, not something else.

Zsh defines redirection [there](https://zsh.sourceforge.io/Doc/Release/Redirection.html).
I'm not sure if the correct behavior is to move the fd or dup the fd, but I think it's not a syntax error in any of the shells...
E.g. the following has consistent behavior in all shells while still triggering a shellcheck warning:

```shell
#!/bin/sh

fd=2
echo Hello >&- >&"$fd" | grep -o H
```
