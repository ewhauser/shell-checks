# [\#1805 Issue](https://github.com/koalaman/shellcheck/issues/1805) `open`: Feature request: separate shellcheck tool or operational mode of the main tool for pretty-printing already generated json

This suggestion is a feature request for the pretty-printing functionality to not only be integrated in the main tool's usual form of operation but also be wrapped in a separate tool (or runnable separately by the main tool when given a certain flag) for pretty-printing input previously generated by `shellcheck -f json`. The use-case I have for this is a current project which could have been much easier, smaller, future-proof, and robust if I could use the functionality just described. I wrote a git pre-receive hook for a shared repo, which sniffs shellscripts from all changed & new blobs based on their filename-extensions or on the shebang in their first-lines, and checks them with shellcheck (and some other checkers too for style, unit-tests, etc) before accepting the push. Because there is non-conforming existing code, I want shellcheck (and the style-checker) to have two modes:

1. Provide error-output only for non-conforming _new/changed lines_ being pushed to the repo
1. Provide error-output for entire files, including existing unchanged code, when code is pushed to a specific "strict" branch

The second mode is easy, but for the first mode I had to choose between three techniques when using shellcheck:

1. Output JSON which I can post-process (filter) easily and robustly with e.g. `jq` but it would then just be JSON and not pretty-printed for easy reading by the person pushing the code.
1. As above, but attempt to pretty-print the filtered JSON myself (which is really not a sensible or rewarding use of time).
1. The technique I chose for now - to post-process/filter the pretty-printed output with a giant sed-script, matching line-numbers scraped from git-diff output. This worked but was much harder than what I propose below and is fragile to potential future changes in output format.

The technique I would like to be able to use is:

1. Output JSON from `shellcheck`, filter it using `jq`, then pretty-print the result using a shellcheck-formatting-tool or `shellcheck` with a new flag for that purpose.
1. If this is acceptable then a nice bonus would be to have a mode where `shellcheck` could be invoked with both "file-original-version" & "file-new-version", and it would output the warnings and errors (pretty-printed, json, gcc, whatever) only for the lines which changed between the original and the new versions.
