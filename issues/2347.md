#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bash

==() {
        echo "I'm a legal function"
}
```

#### Here's what shellcheck currently says:

```console

Line 3:
==() {
^-- SC2275: Command name starts with =. Bad line break?
  ^-- SC1036: '(' is invalid here. Did you forget to escape it?
  ^-- SC1088: Parsing stopped here. Invalid use of parentheses?

$ 
```

#### Here's what I wanted or expected to see:

Code passes without factually wrong warnings (meaning the first two; the third one is correct, although it shouldn't be).

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-10-09 17:24](https://github.com/koalaman/shellcheck/issues/2347#issuecomment-939331222):

You're right, this is valid code. ShellCheck intentionally does not recognize `[]*=!` as part of a function name without the `function` keyword because it tends to lead to confusing and misleading suggestions for broken newbie code, like `array= ()` complaining about missing `{` instead of invalid space. 

It's both practically and philosophically frustrating when tooling complains about correct code, but I hope you're willing to bear with it for the sake of the newbies.
