Over on [gitlab.com](https://gitlab.com/mausy5043/uss_cygnus/-/jobs/78539228) I'm using their web-based CI to run checks on my software.  
One of the checks is a `shellcheck` on the various scripts.  
I get this error:
```
$ shellcheck -V
ShellCheck - shell script analysis tool
version: 0.4.4
license: GNU General Public License, version 3
website: http://www.shellcheck.net
$ shellcheck --format=gcc --external-sources bin/*
bin/include: bin/include: hGetContents: invalid argument (invalid byte sequence)
ERROR: Job failed: exit code 1
```

I've tracked this down to the use of escape codes in the failing script. For example:
```
COL_BOLD='[1m'
```
There is a `<0x1b>` between the `[` and the `1`.
From #846 and #1182 I gather that this has something to do with the locale set on the system.
But I don't get how to fix this.

Given the version that I seem to be stuck with, is there a workaround?

Or is there some way I could force a newer version to be installed somehow?

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-06-30T21:09:55Z](https://github.com/koalaman/shellcheck/issues/1277#issuecomment-401566450):

Huh. I have no idea why a literal escape would be a problem since it's valid ASCII, latin1 and UTF-8. I'm unable to reproduce it.

In any case, yes, setting the locale is the way to work around this. Normally this occurs when your locale is UTF-8 but your files have bytes >= 0x80 that aren't, so I'm surprised that `C.UTF-8` helps at all.

Another option is to run a later version via docker or similar, if Gitlab makes that simple:

    docker run -v "$PWD:/mnt" koalaman/shellcheck:v0.5.0 --format=gcc --external-sources bin/*
