#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
a='a'
[[ $1 =~ $a[0-9] ]]

```

#### Here's what shellcheck currently says:
Line 4:
[[ $1 =~ $a[0-9] ]]
         ^-- SC1087: Use braces when expanding arrays, e.g. ${array[idx]} (or ${var}[.. to quiet).


#### Here's what I wanted or expected to see:
Nothing: in regex context, the meaning of this kind of expression is clear. The wiki page advises to "use ${var}[idx] to prevent this warning", because "this does not change how the script works, but clarifies your intent to ShellCheck as well as other programmers", but my feeling is that it is uselessly heavy in this context (among others).

#### <img src="https://avatars.githubusercontent.com/u/2976480?v=4" width="50">[bugfood](https://github.com/bugfood) commented at [2021-12-06T05:36:20Z](https://github.com/koalaman/shellcheck/issues/1772#issuecomment-986455078):

From testing, it seems that quoting the variable itself will quiet this error. In your example:
```
[[ $1 =~ "$a"[0-9] ]]
```
I think this is an ok workaround as long as the variable part should be matched as a string without respect to regex metacharacters.

As of 5.1.8, the bash manual says:
```
Any part of the pattern may be quoted to force the quoted portion to be
matched as a string.  Bracket expressions in regular expressions must
be treated carefully, since  normal  quoting characters  lose their
meanings between brackets.  If the pattern is stored in a shell variable,
quoting the variable expansion forces the entire pattern to be matched
as a string.
```

If the variable part needs to be interpreted as a regex, then an alternative workaround would be to compose the regex into another variable prior to matching:
```
b="$a"'[0-9]'
[[ $1 =~ $b ]]
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-12-12T01:10:20Z](https://github.com/koalaman/shellcheck/issues/1772#issuecomment-991814704):

It's somewhat clear in this case, but you can imagine `[[ $1 =~ $regex[email] ]]` instead, which is structurally the same. This warning is one of those primarily there to help people coming from other languages where `$arr[x]` is fine, and is not nearly as helpful on an ongoing basis for people who already know shell well. Please feel entirely free to banish the warning forever with `echo disable=SC1087 >> ~/.shellcheckrc`, since you're already aware of it.
