# [\#534 Issue](https://github.com/koalaman/shellcheck/issues/534) `open`: Please warn about magic aliases

#### <img src="https://avatars.githubusercontent.com/u/4005180?v=4" width="50">[sitsofe](https://github.com/sitsofe) opened issue at [2015-11-02T19:13:45Z](https://github.com/koalaman/shellcheck/issues/534):

It's hard to predict what the following will print out in bash:

```
alias magicalias="echo old"
type magicalias
x() { type magicalias ; magicalias; }
x
alias magicalias="echo new"
x
type x
```

This behaviour appears to be brought to you by [magic aliases](http://www.chiark.greenend.org.uk/~sgtatham/aliases.html).

(This might be a duplicate of issue #438)

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-11-03T02:20:20Z](https://github.com/koalaman/shellcheck/issues/534#issuecomment-153219112):

The idea here being to warn that the alias is going to be expanded in the function definition and not inserted literally?

Also this only works in an interactive shell by default as scripts don't expand aliases at all unless `expand_aliases` is enabled in non-interactive/script context.

By "the "type" case where it wasn't expanded" do you mean how `type magicalias` stays `magicalias` in the function definition where the `magicalias` command didn't?

That's because of how aliases expand. They don't expand mid-command; only at the start.

Though I agree that this is certainly surprising behavior. I wouldn't have expected it even though I can understand it.
