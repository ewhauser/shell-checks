# [\#689 Issue](https://github.com/koalaman/shellcheck/issues/689) `closed`: Unicode quotes do not end quotes

#### <img src="https://avatars.githubusercontent.com/u/32383?v=4" width="50">[mina86](https://github.com/mina86) opened issue at [2016-06-15T23:26:16Z](https://github.com/koalaman/shellcheck/issues/689):

```
$ shellcheck a.sh 

In a.sh line 2:
echo "The variable foo has value “$foo”."
                                 ^-- SC1015: This is a unicode double quote. Delete and retype it.
                                  ^-- SC2027: The surrounding quotes actually unquote this. Remove or escape them.
                                  ^-- SC2086: Double quote to prevent globbing and word splitting.
                                      ^-- SC1015: This is a unicode double quote. Delete and retype it.

```

None of those warnings make sense. Perhaps SC1015 does if someone copies code samples from LibreOffice, but SC2027 and SC2086 aren’t valid since Unicode quotes do not end quotes.

The fact that shellcheck treats Unicode quotes as terminating strings means I cannot simply exclude SC1015 on command line and have working shellcheck. I will still get false SC2027 and SC2086.

#### <img src="https://avatars.githubusercontent.com/u/1758777?v=4" width="50">[anthony-geoghegan](https://github.com/anthony-geoghegan) commented at [2017-02-13T20:49:29Z](https://github.com/koalaman/shellcheck/issues/689#issuecomment-279517597):

I love Shellcheck but I’ve had similarly frustrating issues:

In my shell scripts, I use typographic quotes – along with other standard typographic characters (en-dash, em-dash, actual ellipsis, etc. – to make output more readable to users, e.g.

    if command -v wakeonlan >/dev/null; then
      alias wake-server="wakeonlan -i 192.168.1.255 00:1a:a1:19:6e:6e"
    else
      printf 'Warning – ‘wakeonlan’ not installed.\n'
    fi

However, this usage generates multiple warnings due to the typographic quotes being treated as typewriter quotes (ASCII 39) by Shellcheck and (apparently) terminating quoted strings.

I tried setting the environing variable to exclude [SC106](https://github.com/koalaman/shellcheck/wiki/SC1016) so that I could use typographic single quotes. However, this did not make any difference.

I eventually found a workaround: Reading the documentation for [SC1015](https://github.com/koalaman/shellcheck/wiki/SC1015) states that

> If you really want literal Unicode double quotes, you can put them in single
> quotes (or unicode single quotes in double quotes) to make shellcheck ignore
> them.

Some experimentation shows that Shellcheck would be happy with either of the following rewrites:

    printf 'Warning – “wakeonlan” not installed.\n'
    printf "Warning – ‘wakeonlan’ not installed.\n"

I’ve updated the Wiki description for [SC106](https://github.com/koalaman/shellcheck/wiki/SC1016) so that in the short term, other users can work around this issue.

In the long term, I’d love it if Shellcheck could be configured to ignore typographic quotes – just the same as every other POSIX-compatible shell (Powershell is the only shell I’ve come across that uses typographic quotes for delimiting strings).

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-02-24T22:44:02Z](https://github.com/koalaman/shellcheck/issues/689#issuecomment-282426019):

The current behavior exists because it's empirically much more likely that the slanted quotes are accidental. Parsing slanted quotes as double quotes results in better error messages on average.

However, I'll fix this shortly because it makes the worst case quite a bit worse. Having to do `"The tool "'“'"$1"'”'" doesn't exist"` is ridiculous.
