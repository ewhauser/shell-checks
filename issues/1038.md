#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash

LOCAL_STRING="foo"
ssh root@remote_host "ls -l
# shellcheck disable=SC2029
echo $LOCAL_STRING"
```

#### Here's what shellcheck currently says:
Line 6:
echo $LOCAL_STRING"
     ^-- SC2029: Note that, unescaped, this expands on the client side.


#### Here's what I wanted or expected to see:
It would be nice if the `# shellcheck disable=SC2029` could be put into the remote side string/script (as I have done above in my example) so that one can indicate which occurrences are to be expected and which are not.  Requiring the disable before the whole remote string may lead to instances inside the string that should be flagged which are ignored, in the case of a many-line remote script.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-11-20T05:09:29Z](https://github.com/koalaman/shellcheck/issues/1038#issuecomment-345593267):

I definitely see your point, and this would indeed be useful.

However, for the immediate shell, the script passed to ssh is just a string, and ShellCheck intentionally makes no attempt to parse strings passed to `ssh`, `eval`, `su`, `sh -c` or any other way of evaluating strings as shell code.

It would be useful to not emit this for any variable expanded with escaping though, such as `${var@Q}`. This is a strong signal that the local expansion was intentional.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-11-20T17:13:18Z](https://github.com/koalaman/shellcheck/issues/1038#issuecomment-345763422):

It's easy to run it over static strings, and this is basically how backticks already work (unlike `$()` they are defined in terms of a string span and not by a token like `)`, which is why they nest so poorly).

The problem is when the string is not static, such as `name="var$RANDOM"; eval "$name=value"` or `functions=$(wget -O - https://example.com/myfunctions.sh); ssh host "$functions; do_stuff \$bar";` . Like a bad IRC/stackoverflow question, it's hard to give useful and accurate feedback on a script when you're missing pieces of it. 

Since the dynamic case is undecidable, ShellCheck also avoids the simple static case to avoid creating the expectation that indirect code will be checked in a useful way.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-07-02T01:55:48Z](https://github.com/koalaman/shellcheck/issues/1038#issuecomment-401651465):

It's not properly quoted here, so it could indeed be considered a bug. If your variable contains `$` or `"` or such, it'll break in spite of the literal double quotes. The correct way to quote it here (assuming both local and remote is `bash`) is:

    ssh  host "sudo test -f $(printf '%q' "${VAGRANT_JENKINS_HOME}")/secrets/initialAdminPassword"

ShellCheck doesn't recognize that this is correct though.
