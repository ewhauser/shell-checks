# [\#587 Issue](https://github.com/koalaman/shellcheck/issues/587) `open`: Specifying variablesWithoutSpaces Without Modifying the Code

#### <img src="https://avatars.githubusercontent.com/u/17551?v=4" width="50">[eatnumber1](https://github.com/eatnumber1) opened issue at [2016-01-28T07:03:44Z](https://github.com/koalaman/shellcheck/issues/587):

We (Google) have more restrictive requirements around variable quoting than ShellCheck currently warns on. We're able to get the desired verbosity by modifying [variablesWithoutSpaces](https://github.com/koalaman/shellcheck/blob/master/ShellCheck/Data.hs#L38), but this is not ideal since we now have to maintain a patchset deviating from upstream. As a feature request, it would be nice if we could specify these via a configuration file, environment variable or command line flag so that we don't have to modify the source.

#### <img src="https://avatars.githubusercontent.com/u/198379?v=4" width="50">[bittorf](https://github.com/bittorf) commented at [2016-01-28T21:02:01Z](https://github.com/koalaman/shellcheck/issues/587#issuecomment-176414903):

@eatnumber1 so these are vars which are guaranteed to be without spaces, so that shellcheck can handle them in another way?


#### <img src="https://avatars.githubusercontent.com/u/17551?v=4" width="50">[eatnumber1](https://github.com/eatnumber1) commented at [2016-01-28T21:33:45Z](https://github.com/koalaman/shellcheck/issues/587#issuecomment-176430268):

They are variables that are assumed to not contain spaces. There's only a few which bash itself guarantees that fact (`$$`, `$!`, `$#` come to mind). The variables listed in variablesWithoutSpaces will prevent [SC2086](https://github.com/koalaman/shellcheck/wiki/SC2086) from triggering on unquoted use.
