#### Here's a snippet or screenshot that shows the problem:
```bash
#!/bin/bash

# adds two numbers ($1 and $2)
add() {
  echo "$(echo "${1}+${2}" | bc)"
}

echo "$(add 1 1)"
```

#### Here's what shellcheck currently says:
```
Line 5:
  echo "$(echo "${1}+${2}" | bc)"
       ^-- SC2005: Useless echo? Instead of 'echo $(cmd)', just use 'cmd'.
 
Line 8:
echo "$(add 1 1)"
     ^-- SC2005: Useless echo? Instead of 'echo $(cmd)', just use 'cmd'.
```


#### Here's what I wanted or expected to see:

Both echos to me seem necessary. and perhaps obviously so? I need the echo in line 5 to use the pipe and in line 8 because I desire to have that command substitution... well, echoed!

Should shellcheck be made to be aware of these legitimate uses?

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-09-23 17:35](https://github.com/koalaman/shellcheck/issues/2336#issuecomment-926018333):

ShellCheck is right, these are both unnecessary and the code could/should be written like this:

```sh
#!/bin/bash
# adds two numbers ($1 and $2)
add() {
  echo "${1}+${2}" | bc
}
add 1 1
```

It doesn't make a significant difference in this particular case since `bc`'s output is small and it doesn't have a useful exit status on bad input, but it's just easier and works the same. If you refresh the [the wiki](https://github.com/koalaman/shellcheck/wiki/SC2005), I included more cases where avoiding the `echo` *does* make a significant difference to speed and correctness.
