# [\#562 Issue](https://github.com/koalaman/shellcheck/issues/562) `closed`: SC2140 triggers for slash on parameter expansion replacement

#### <img src="https://avatars.githubusercontent.com/u/660663?v=4" width="50">[cdown](https://github.com/cdown) opened issue at [2015-12-15T19:25:23Z](https://github.com/koalaman/shellcheck/issues/562):

Shellcheck complains that `/` is not quoted, but it's part of the expansion syntax, it's not something to be quoted.

```
% shellcheck --version
ShellCheck - shell script analysis tool
version: 0.4.1
license: GNU General Public License, version 3
website: http://www.shellcheck.net
% shellcheck -s bash /dev/stdin << 'EOF'
>>> : "${PHOTO_DIR/"'"/"'\\''"}"
>>> EOF                         

In /dev/stdin line 1:
: "${PHOTO_DIR/"'"/"'\\''"}"
                  ^-- SC2140: Word is on the form "A"B"C" (B indicated). Did you mean "ABC" or "A\"B\"C"?

```

#### <img src="https://avatars.githubusercontent.com/u/2511878?v=4" width="50">[sayoder](https://github.com/sayoder) commented at [2016-11-28T14:25:53Z](https://github.com/koalaman/shellcheck/issues/562#issuecomment-263283737):

I think the issue here is that the quoted parts in the expansion are being parsed as `T_NormalWord`, and therefore match the first pattern in `checkInexplicablyUnquoted`. Ideally, a slash in a parameter expansion would be treated as a special character - I'm currently unsure what Token constructor to use, but I hope to have this working on my fork sometime in the next few days.

As a side note, I'm thinking the error message is meant to say "Word is *of* the form" (rather than "on").
