#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
functionName()([[ $(echo true) ]])

```

#### Here's what shellcheck currently says:
```
Line 2:
functionName()([[ $(true) ]])
              ^-- SC2234: Remove superfluous (..) around test command.
```

#### Here's what I wanted or expected to see:
Nothing, it's valid syntax. It's a simple function that just runs a single command and retains the return code. May not be the most optimized, but it's for compatibility with code that isn't mine.
Technically functions should be wrapped in brackets, but according to the stackoverflow post I saw last night, using parenthesis instead of brackets limits variable scope, and I don't want the output of the command to accidentally come with some extra variables.

I just hit this. Here's a real world example:
```
on-mac() ( [[ $OSTYPE == darwin* ]] )
```

Also get the warning on these variants:
```
on-mac() ([[ $OSTYPE == darwin* ]])
on-mac() ( [[ $OSTYPE == darwin* ]]; )
on-mac() (
  [[ $OSTYPE == darwin* ]]
)
```

I prefer to always use parens over brackets for functions.
They limit more side effects than just variable state.
Bracket functions in my Bash code indicate that the function is purposefully setting some state for the caller.

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-08-30 18:41](https://github.com/koalaman/shellcheck/issues/2169#issuecomment-908591248):

All the examples here are true positives for the bad practices that ShellCheck aims to discourage.

I do agree that 1. the message made it seem like a purely stylistic issue, 2. the wiki exception part didn't help, and 3. the wrong advice was emitted for the right reason for function bodies since `f() [ x ]` is not valid, and ShellCheck purposefully doesn't allow `f() [[ x ]]` to improve diagnostics. Additionally, while it was only ever meant to trigger on side effect free code, it did not properly handle the case of mutating ${} and $(()) expansions like `f() ( [[ $((i++)) -eq 0 ]] )`. All these issues have now been fixed.

The original example now results in a better (and intentional) "SC2235: Use { ..; } instead of (..) to avoid subshell overhead", and only on side effect free `test` expressions.
