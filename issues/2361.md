#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/dash

: >> a
: >> b
: >> c

for m in [^a]
  do printf 'match: %s\n' "$m"
done
# output:
# match: a
```

#### Here's what shellcheck currently says:

*nothing*

#### Here's what I wanted or expected to see:

```
Line 7:
for m in [^a]
         ^-- SC3026 (warning): In dash, [^a] is not equivalent to [!a].
```

---

If the shebang is `#!/bin/sh`, shellcheck will report the folowing:
```
Line 7:
for m in [^a]
         ^-- SC3026 (warning): In POSIX sh, ^ in place of ! in glob bracket expressions is undefined.
```

I was going to report that `shellcheck` doesn't warn about this in `dash`, but after reading the source code

https://github.com/koalaman/shellcheck/blob/5669eb22037980c5a6b74b0d420cb452990bcf88/src/ShellCheck/Checks/ShellSupport.hs#L238-L239

I figured this was probably intentionally not being reported since the behaviour of `[^a]` in `dash` is defined and is equivalent to `[a^]` in other shells.

But then I read [the wiki page for SC3026](https://github.com/koalaman/shellcheck/wiki/SC3026) and saw:

> `[^c]` is frequently used in most regular expression variants to mean "any character except `c`". This is so pervasive that bash, ksh, dash, and BusyBox ash, all allow it.

All the mentioned shells, except `dash`, allow `[^a]` as an alternative to `[!a]`.

So I realised that, maybe, `dash` was being ignored because who wrote it incorrectly thought that `dash` supported it.

I am reporting the issue since, at least, the wiki page is incorrect.

If we want to keep not reporting `[^a]` for dash, I think that the wiki page should at least mention that if the intent of the glob was to match "any character that is either `a` or `^`", `[a^]` should be preferred to `[^a]`.

---

Here is a script to check the behaviour of `[!a]` vs `[^a]` vs `[a^]` in different POSIX shells:

```sh
#!/bin/sh

IFS=

for f in a b c '^'
  do : >> "$f"
done

for pat in '[!a]' '[^a]' '[a^]'; do
  for i in $pat
    do printf '%s matches: %s\n' "$pat" "$i"
  done
done
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-10-22 04:08](https://github.com/koalaman/shellcheck/issues/2361#issuecomment-949269941):

The reason why it was this way was because it works on my Dash:

```
[!a] matches: ^
[!a] matches: b
[!a] matches: c
[^a] matches: ^
[^a] matches: b
[^a] matches: c
```

I bisected the Dash history and found that it was due to [this commit](https://git.kernel.org/pub/scm/utils/dash/dash.git/commit/?id=7638476c18f2a8451b05d49f4e767ee1f98cfc17) enabling fnmatch/glob from glibc by default. This means that not only does support differ between shells, it also differs between builds of the same version of the same shell!

The warning now triggers for Dash as well. Nice find, thanks.
