#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-09-12T03:30:39Z](https://github.com/koalaman/shellcheck/issues/999#issuecomment-328724801):

TL;DR - working as intended.

"Works in shell" and "Shellcheck should parse" are two different things.

The `$math` inside (( )) can only be parsed after `math` has been set at run-time.

Shellcheck is a **_static_ analyzer**, and advertised as such. That means that it does _not_ follow the flow of control that the shell would execute.

One consequence of this is that Shellcheck may not know the values of variables, functions and aliases where they are expanded or called; so it rightly takes a pessimistic view, and warns you.

If you really know what you're doing, disable the warning with
```
# shellcheck disable=SC1073
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-09-14T16:31:37Z](https://github.com/koalaman/shellcheck/issues/999#issuecomment-329537611):

ShellCheck could try to determine the span of the `$(( .. ))` expression first, and parse the contents as an arithmetic expression afterwards. This is basically how shells do it, and how ShellCheck already parses `` `..` ``.

It would still result in a parsing error, but instead of aborting it could emit a warning and continue. The `$((..))` would remain as a black box for the purpose of variable usage and analysis.

You can already do this manually as a workaround:

    expr="(${test}+512)$math"
    echo "$(( expr ))"
