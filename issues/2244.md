#### Here's a snippet or screenshot that shows the problem:

```sh
# shellcheck shell=bash

f() {
        local myvar=$1
        local myarr=(${myvar//,/ })
        printf 'Values:\n'
        printf '  %s\n' "${myarr[@]}"
}

f foo,bar,baz
```

#### Here's what shellcheck currently says:

    -- SC2206: Quote to prevent word splitting/globbing, or split robustly with mapfile or read -a.

#### Here's what I wanted or expected to see:

I believe shellcheck should make an exception when a variable assignment to an array explicitly replace a character or group of characters to a space.

To retain proper semantics and give flexibility, perhaps allowing either the first or all characters of `$IFS` rather than space or default `IFS` when the value of `$IFS` cannot be determined.

-----

If you still believe `read` or `mapfile` should be used, how would you write this? Assume you have no control over input to `f()`.

My use case for this it to split into a list any type of delim-separated list (ex delim=, in example above) where elements cannot contain spaces (ex hostnames, ip addresses...) - most frequently from command line parameters.


#### <img src="https://avatars.githubusercontent.com/u/403890?v=4" width="50">[dermoth](https://github.com/dermoth) commented at [2021-06-03 20:04](https://github.com/koalaman/shellcheck/issues/2244#issuecomment-854142930):

Answering myself, at quick glance the best way to achieve this behavior (and also perform similar tasks from command output where I want to map to an array regardless of the separators - newlines or spaces):

```bash
read -rn0 -d '' -a myarr <<<"<command or variable>"
```

I don't find this very clear, and while there are simpler alternatives for the command above it also means having to presume output is either space or newline-delimited, and it would break if that changes.
