#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-11-02T01:07:14Z](https://github.com/koalaman/shellcheck/issues/1037#issuecomment-341288433):

At the moment this is intentional: uppercase is conventionally used for environment variables that will be inherited from the parent.

Typical examples are `PATH`, `LANG`, `TERM`, `TZ`, `EDITOR`, `PAGER`, `JAVA_HOME`, `PYTHONPATH`, `SUDO_USER`, `SSH_CLIENT`, all of which are generally assumed to be set when relevant, rather than being defined by the current script.

There is currently no good story for how to define which uppercase variables this does and doesn't apply to.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2022-03-02T17:15:53Z](https://github.com/koalaman/shellcheck/issues/1037#issuecomment-1057177115):

Since 0.7.0 you can put `enable=check-unassigned-uppercase` in your (or your project's) `.shellcheckrc`, or use `# shellcheck enable=check-unassigned-uppercase` in the script itself.

How you whitelist variables is not thoroughly defined, but currently a variable appearing in `[ -z "$VAR" ]` or `[ -n "$VAR" ]` is assumed handled, so you can pragmatically verify that the variable is set to avoid warnings about it:

```
#!/bin/sh
# shellcheck enable=check-unassigned-uppercase

if [ -z "$FOO" ]
then
  echo >&2 "FOO is unset"
  exit 1
fi
echo "$FOO"  # No warning
echo "$BAR"  # Warning
```
