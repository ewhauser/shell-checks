#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2021-01-23 01:33](https://github.com/koalaman/shellcheck/issues/2054#issuecomment-765811682):

Functions can be nested in un-obvious ways; consider:

```sh
set -e

f() {
    do_something  # is this exempt?
}
g() {
    echo starting "$1"
    "$@"
}

if g f
then echo Happy
else echo Sad
fi
```
or
```sh
set -e

f() {
    do_something  # is this exempt?
}
g() {
    echo starting "$1"
    if "$@"
    then r=$? # 0
    else r=$?
    fi
    echo finished "$1"
    return "$r"
}

g f
```

And functions aren't the only way this can happen, they're just the most obfuscating:
* Any block with a test condition that's a _LIST_ can exhibit this issue:
 ```sh
 # shell equivalent of C's "do-while" loop:
 while
     some_command        # unintentionally exempt from set -e and ERR trap
     ((some==test))      # condition for loop
 do :; done
 ```
* Anything inside a compound statement when the whole statement is tested:
   * a subshell:
    ```sh
    foo=$( some_cmd     # un?intentionally exempt from set -e and ERR trap
           another_cmd  # intentionally exempt from set -e and ERR trap
          ) || return
    ```

My view is that `set -e` may be useful for linting and regression testing, but if you're relying on it in production, you're doing it wrong. Rather, critical operations should be checked explicitly:
```sh
abort() {
    r=$?
    printf >&2 'Unexpected return code %u when %s\n' "$r" "$1"
    exit "$r"
}

something_complicated                          || abort 'complicated thing failed'
```
If you're too busy/lazy/in-a-hurry to do anything else, simply preface any critical operations with a standard checker:
```sh
critical() {
    "$@" || abort "when attempting $( printf '[%s]' "$@" ) at $( caller 1 )"
}

critical cd "$dir"
```

@marc-h38: My overarching point is `set -e` can sometimes be *actively harmful*, and the assumption that it's always benign is simply wrong. The inherent assumption that non-zero status means "critical failure" is simply wrong, and quite irrepairable.

The worst harm done by `set -e` is to the minds of users: it lulls them into a false sense of security.

Advocates for `set -e` never point out that using it can also _cause_ failures, especially in unusual cases that may not show up during regression testing. Nor do they point out that it's not _guaranteed_ to stop dangerous operations after critical prerequisites have failed.

I'm _not_ recommending manually checking everywhere; I'm saying pay attention and _do_ check where the exit status indicates failure, and _don't_ check where the exit status does _not_ indicate a critical failure.

Given a choice between marking the innocuous commands whose exit status can be ignored, and marking the critical ones, I prefer to err on the side of _readability_ rather than _writability_: highlight the critical operations, and leave everything else as simple as possible.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2021-02-07 09:34](https://github.com/koalaman/shellcheck/issues/2054#issuecomment-774642722):

We're veering a long way off the issue, but...

> I believed this harmful myth for a long time. Fortunately for me, I eventually saw evidence of the opposite and then successfully used set -e in production for a couple of decades and it saved me and others a lot of time.

I believe the largest harm is to the mindset of the programmer. Conversely, knowing that one is working without guard rails on sharpens the mind.

I used to believe as you do now, that `set -e` would be more help than hindrance, but stopped using it after I experienced a number of quite catastrophic failures where error recover from some minor problem was broken by `set -e`, leading on one occasion to irreparably deleting over 10 million customer emails.

I've been scripting for a couple of decades since then, and _avoiding_ set -e _and_ applying alternative rigorous approaches has saved me and others a lot of time and money.

The point is not simply to stop using `set -e`.
If the choice is between _"sloppy programs without `set -e`"_ and _"sloppy programs with `set -e`"_ then the later would obviously be an improvement.
My point is to replace `set -e` with something better.

My non-exhaustive list of "something better" would include:
* Use linting tools (such as Shellcheck) often, preferably every editor save, or every git commit;
  * Don't expect Linting tools to solve the halting problem (which is what this feature request amounts to)
* Apply skill testing: _You have to be this tall to play._
  * If you can't explain a script, you shouldn't be modifying it.
  * If you don't understand expansions, assignments, quoting, word-splitting, and globbing, you shouldn't be modifying _any_ shell script.
* Have peers review code;
  * If a script seems to use "quotes" to mean "strings", its author probably failed the previous point. The ability to use quotes selectively shows that they understand the quoting rules, and so they're actually less likely to get them wrong in ways that confuse the linting tools.
* Avoid known anti-patterns, and apply good practices;
  * Include comments explaining why you need to use an exception, to the satisfaction of the code reviewer.
  * Rigorously identify which lines are prerequisite checking, actions, and cleanup, as they have different error-handling characteristics;
  * Use `|| die "explanation"` after critical code - or an equivalent that suits your style - so that users won't be left in the dark when things fail, and so that reading _down_ the code shows the non-failure path.
  * As far as reasonable, avoid mixing code & data, so that linting tools can work better, and so humans can read the code more easily.
* Avoid prescriptive rules based on misunderstandings or overreach, as they will cause resentment and be ignored.
  * For example, _don't_ prohibit `((var=expr))` on the grounds that it will fail under `set -e` when the expression is zero. Instead, recommend that, like any other command whose non-zero status is innocuous, it should be prefaced by `!` to avoid triggering an abort.
* Do use `set -e` on sections of code where you know what "failure" means, and where the consequences of simply aborting the shell are acceptable. Inside certain subshells, for instance. Just don't set it blindly at the top of the script.

All that said, if `set -e` stops people writing code like this:

    somecmd
    if [ $? = 0 ] ; then ...

then maybe I should change my mind after all. :smiley:
