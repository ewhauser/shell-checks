#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2020-06-29 00:36](https://github.com/koalaman/shellcheck/issues/1998#issuecomment-650846046):

I see why this is confusing: the wiki had been incorrectly edited to say that this would work. The intended behavior of `source-path` was just to root constant relative or absolute paths like `./foo` or `/usr/bin/foo`.

It's been discussed a bit before, and now ShellCheck will strip a leading expansion followed by a slash and treat it as `./`. This allows the most common idioms like `source "${this_dir}/file.sh` to work without explicit `source=file.sh` type directives.

Paths with overrides, constant paths, or paths with multiple expansions will be treated like before. 

Thanks!

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2022-03-10 17:41](https://github.com/koalaman/shellcheck/issues/1998#issuecomment-1064327016):

Before 0.7.2, ShellCheck would never be able to find `${this_dir}/menu-common.sh` because it doesn't try to figure out what `${this_dir}` is, and therefore it doesn't have a filename to look for in the source-path (so this option has no effect).

In and after 0.7.2, ShellCheck will strip 0 or 1 leading dynamic element thereby treating `${this_dir}/menu-common.sh` as `/menu-common.sh`, which it can then try to find in the source-path (meaning the option takes effect).

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2022-11-13 18:50](https://github.com/koalaman/shellcheck/issues/1998#issuecomment-1312796060):

Run `shellcheck -x` or specify `external-sources=true` in `.shellcheckrc`.

ShellCheck started life as an online tool that would run on random strangers' code in a time before containerization, so it doesn't give itself permission to read arbitrary files around the filesystem just because an untrusted script asked them to. If it had been designed today as the local tool it has become, it would have had no problems doing that.
