#### <img src="https://avatars.githubusercontent.com/u/243351?v=4" width="50">[brother](https://github.com/brother) commented at [2016-02-12T10:33:50Z](https://github.com/koalaman/shellcheck/issues/598#issuecomment-183264862):

``` shell
apa=foo

case $apa in
  foo)
       echo "first"
       # DROP TO NEXT INTENTIONAL
  bar)
       echo "second"
       ;;
esac
```

This produces [SC1009](https://github.com/koalaman/shellcheck/wiki/SC1009) for the case line. [SC1073](https://github.com/koalaman/shellcheck/wiki/SC1073) for "foo)". [SC1072](https://github.com/koalaman/shellcheck/wiki/SC1072) and [SC1074](https://github.com/koalaman/shellcheck/wiki/SC1074) for "bar)".

I have no opinion on what type of message to produce (warning/error). However, having a "fall through if magic keyword in comment" I am against.

#### <img src="https://avatars.githubusercontent.com/u/243351?v=4" width="50">[brother](https://github.com/brother) commented at [2016-02-12T12:09:07Z](https://github.com/koalaman/shellcheck/issues/598#issuecomment-183297764):

bash added the support "recent". But I am not sure my example above is for a correct implementation, it works but that's all.

The switch statement page at wikipedia has the following quote (without source): "Similarly, bash defaults to not falling through, when terminated with ;; but more recent versions allow fallthrough with ;& instead."

testing it in bash 4.3.42 I see that my example is broken.

``` shell
#!/bin/bash
apa=foo

case $apa in
    foo)
        echo "first"
        # DROP TO NEXT INTENTIONAL
        ;&
    bar)
        echo "second"
        ;;
esac
```

There. And that code does NOT trigger any warnings. Maybe it's a case of broken code that shellcheck actually catches for OP?
