#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-08-17T08:54:31Z](https://github.com/koalaman/shellcheck/issues/970#issuecomment-323010050):

I'm not a fan of putting command paths in variables; it violates the separation of code & data (which is tenuous enough already in the shell, without making it worse).

They really make a script much harder to read, and make it harder for tools like Shellcheck to check that what you're doing is sane.

Moreover, they're not exempt from globbing and word-splitting, but most scripts quietly ignore that (as exemplified by the lack of quoting around `$AWK` in the original report).

I would simply recommend setting PATH to an appropriate value, but failing that, using functions like:

    awk() { /usr/local/bin/awk "$@";}

or if you really feel like you MUST use a variable, then at least wrap it so that it's not cluttering the whole script:

    awk() { "$AWK" "$@"; }

Also note that

    AWK=$(which awk)

is completely pointless, because simply `awk` will necessarily run the same program as `"$AWK"`.

(It's also dubious because `which` is untrustworthy and slow compared with `command -v` or `type -p`, but I use it here to illustrate the most common case.)

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-08-30T11:35:36Z](https://github.com/koalaman/shellcheck/issues/970#issuecomment-325964032):

SC2016 tries to reduce false positives by recognizing common commands where you often single-quote things that look like shell variables (`awk`, `sh`, `perl`, some `sed`) and avoid warning. This is why `echo '$1'` warns but `'awk '$1'` does not. `$AWK` is not on that list.

Naming a function `awk` is a good workaround to get the same logic.

Setting or relying on PATH is arguably better, but that's a different discussion.
