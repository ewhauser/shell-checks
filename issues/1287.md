# [\#1287 Issue](https://github.com/koalaman/shellcheck/issues/1287) `open`: Detect assignment to readonly variable

#### <img src="https://avatars.githubusercontent.com/u/27913?v=4" width="50">[Warbo](https://github.com/Warbo) opened issue at [2018-07-09T15:08:42Z](https://github.com/koalaman/shellcheck/issues/1287):

#### For new checks and feature suggestions
- [x] shellcheck.net (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

I was writing a script which involved a "unique ID", and foolishly assigned it to a variable called `UID`. This causes a bash error when run: `UID: readonly variable`. I run all my scripts through shellcheck, but it didn't catch this. It would be nice if it could.

#### Here's a snippet or screenshot that shows the problem:

The snippet `UID="anything"` should trigger this. Here's the whole script I wrote, for context:

```sh
#!/usr/bin/env bash
set -e

# Open PDFs in mupdf

function isOpen {
    # shellcheck disable=SC2009
    ps auxww | grep mupdf-x11 | grep "$1" > /dev/null
}

function openIfNeeded {
    # Open $2 if it's not already (i.e. if `mupdf-x11 ... $1` appears in `ps`)
    if isOpen "$1"
    then
        echo "Already have reader for $2, skipping" 1>&2
    else
        echo "Opening $2" 1>&2
        mupdf-x11 "$2" &
    fi
}

while read -r LINE
do
    # Skip comments
    echo "$LINE" | grep '^\s*#' > /dev/null && continue
    UID=$(echo "$LINE" | cut -f1)
    LOC=$(echo "$LINE" | cut -f2)
    openIfNeeded "$UID" "$LOC"
done < ~/.pdfs
```

Running a script like this gives:

```sh
$ pdfs
/nix/store/87lwk6i1hjfmswafhvbwsdh9m8rgjamj-warbo-utilities/bin/.pdfs-wrapped: line 26: UID: readonly variable
```

#### Here's what shellcheck currently says:

```
No issues detected!
```

#### Here's what I wanted or expected to see:

Something like:

```
The variable "UID" is readonly and built in to the shell. Trying to assign it to a new value will cause an error.
```

Other readonly variables (built in and perhaps even user-defined) could be spotted too.

#### <img src="https://avatars.githubusercontent.com/u/84648683?v=4" width="50">[wileyhy](https://github.com/wileyhy) commented at [2024-09-26T06:56:32Z](https://github.com/koalaman/shellcheck/issues/1287#issuecomment-2376086312):

  That warning has mostly to do with `set -e`, is my understanding. Here's
how that would be done:

  DELETE_AT_EXIT="$(mktemp -d)"
  readonly DELETE_AT_EXIT

  And here's the existing warning text and SC's wiki page:

  "SC2155 (warning): Declare and assign separately to avoid masking return
values."

https://www.shellcheck.net/wiki/SC2155

  "Declare" and "assign" for this context are defined in the "PARAMETERS"
section of the man page as...

  "A variable may be assigned to by a statement of the form
`name=[value]`....
  "Assignment statements may also appear as arguments to the alias,
declare, typeset, export, readonly, and local builtin commands (declaration
commands)."
https://tiswww.case.edu/php/chet/bash/bash.html

Concerning `set -e` (ie, "errexit'): There are people who love it and there
are people who hate it. The reality is that errexit was written after the
lion's share of bash's grammar (based on Bourne shell) was already in
existence, so it's a bit like tailoring a sweater to fit neatly on a
Christmas tree. Aforethought additions of security features in email
protocols suffer a similar malady.

For my money, errexit is "un"-necessary security theater, and I was a
theater major. Sometimes requiring the extra level of assurance might be
reasonable. As an additional stricture available to help hone one's skills
I find it valuable, but it's far from intuitive and quite full of gotchas.
From what little I have seen (possible sample bias) among shell programers,
the experienced tend to dislike it while the less experienced occasionally
lean on it, which is okay if you ask me.

Shell has so much to do with external binaries' exit codes, which
themselves are not in any way standardized. Usually I can test whether some
error occurred by testing the data output of whatever commands. The notion
of, whether or not some utility's output is intended to remain "stable" or
"--terse" or "machine readable," is a long standing convention in
Unix/Linux. You can see in the man pages where developers will warn users,
'caution, this output format may change in the future, do not rely on it
for scripting purposes.' Those notes lead me to think that predictability /
reliability of output format was something the original developers of
whatever program valued and considered from the beginning. To me, it seems
like a wise choice to favor shell / program features that were written with
reliability in mind. If errexit were labeled 'Caveat Emptor' in the man
page or GNU manual, I'm not sure how I could argue against that.

Some really good examples and explanations:

https://mywiki.wooledge.org/BashFAQ/105
https://mywiki.wooledge.org/BashPitfalls?highlight=%28errexit%29#set_-euo_pipefail
