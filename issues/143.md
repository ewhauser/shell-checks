#### <img src="https://avatars.githubusercontent.com/u/3999221?v=4" width="50">[pawamoy](https://github.com/pawamoy) commented at [2018-02-23T12:08:07Z](https://github.com/koalaman/shellcheck/issues/143#issuecomment-367992333):

@coudenysj, about how to tell if a file is a shell script: I remembered myself using the `file` command:

```bash
for f in $(file path/to/scripts/* | grep 'shell script' | cut -d: -f1); do
  shellcheck "$f"
done
```

I think `file` uses the shebang as well to determine if a file is a shell script, and it might handle it better than a custom regex (`#\!\(/usr/bin/env \|/bin/\)sh`), though I can't confirm.

As a consequence, `file` will not detect shell scripts that do not have shebangs (like the ones we source, therefore not needing a shebang), but the `grep` search will not either. Though, I think it's conventional to name sourced files with the `.sh` extension, or `.bash` depending on the shell being targeted.

So a combination of `find` and `file` would be:

```bash
for f in $({ find path/to/scripts -type f -regex ".*\.\w*sh"
             file path/to/scripts/* | grep 'shell script' | cut -d: -f1
           } | sort -u); do  # note that we remove potential duplicates
  shellcheck "$f"
done
```

Do you think it's worth noting in the Recursiveness page :smile: ?

#### <img src="https://avatars.githubusercontent.com/u/967132?v=4" width="50">[artem-zinnatullin](https://github.com/artem-zinnatullin) commented at [2018-03-02T14:03:54Z](https://github.com/koalaman/shellcheck/issues/143#issuecomment-369928774):

Great addition to the project wiki!

Here is a snippet i use to run it in Docker container using `shellcheck-alpine` image provided by the project:

```console
# shellcheck disable=SC2016
docker run \
--rm \
--volume "$(pwd)":/project:ro \
--entrypoint sh \
koalaman/shellcheck-alpine:v0.4.7 \
-c 'for file in $(find /project/ -type f -name "*.sh"); do
if ! shellcheck --format=gcc $file; then export FAILED=true; fi; done;
if [ "$FAILED" != "" ]; then exit 1; fi'
```

Note that it requires Alpine image to run `sh` (regular `koalaman/shellcheck` image doesn't have `sh`) and `--entrypoint sh` to override [default one which is `shellcheck` itself](https://github.com/koalaman/shellcheck/blob/ee354ffce872a733890b356b63a96bd371c7c26f/.travis.yml#L37)

I've combined it with [wiki snippet](https://github.com/koalaman/shellcheck/wiki/Recursiveness#with-exit-code) that produces exit code, so you can use it for CI and build scripts, but of course you can customize it. 

Hope someone will find it useful! Maybe wiki should have it as well :)

#### <img src="https://avatars.githubusercontent.com/u/4165105?v=4" width="50">[rivertam](https://github.com/rivertam) commented at [2018-05-25T17:35:31Z](https://github.com/koalaman/shellcheck/issues/143#issuecomment-392128931):

@artem-zinnatullin Thank you so much for this. This is exactly what I needed.

It's worth noting, however, that I think if the inner part (the command) were to actually be shellchecked, I think it would complain about using `find` output for a `for` loop ([SC2044](https://github.com/koalaman/shellcheck/wiki/SC2044)).

I modified it once more to be cleaner IMO and to fit my needs personally.

```bash
docker run \
  --rm \
  --volume "$(pwd)/scripts":/mnt \
  --entrypoint sh \
  koalaman/shellcheck-alpine:latest \
  -c "
exitcode=0
find /mnt -type f > /tmp/filestolint
while IFS= read -r file
do
  echo \"Checking \$file\"
  if ! shellcheck --format=gcc \"\$file\"; then
    exitcode=1
  fi
done < /tmp/filestolint
exit \$exitcode"
```

The `find` command will need to be tweaked for your needs (for example, most people use `.sh` to represent shell files, while I decided not to for this project).

#### <img src="https://avatars.githubusercontent.com/u/2211051?v=4" width="50">[HariSekhon](https://github.com/HariSekhon) commented at [2020-02-13T12:19:55Z](https://github.com/koalaman/shellcheck/issues/143#issuecomment-585729002):

My `check_bash_syntax.sh` script does this and a bit more by pushd'ing to each script's src directory so relative `source` directives work too, working around issues https://github.com/koalaman/shellcheck/issues/908 and https://github.com/koalaman/shellcheck/issues/1837:

https://github.com/HariSekhon/DevOps-Bash-tools

recurse all `*.sh` under `$PWD`:
```
check_bash_syntax.sh
```

recurse all `*.sh` under somedir:
```
check_bash_syntax.sh somedir
```

check only these files and recurse only these given directories:
```
check_bash_syntax.sh file1 file2 file3 directory1 directory2
```

The reason it's called check_bash_syntax is that I also warn for using `#!/bin/bash` instead of `#!/usr/bin/env bash`.

This script is easily modified if you wanted to auto-recurse for contents of `#!/bin/sh` for example, but I think it's a good practice to put the `.sh` extension on all scripts as it is more explicit and intuitive at a glance without needing historical project knowledge or running `grep` / `file` on the contents of every file which is slower.

#### <img src="https://avatars.githubusercontent.com/u/33200?v=4" width="50">[jmcantrell](https://github.com/jmcantrell) commented at [2021-08-31T08:19:10Z](https://github.com/koalaman/shellcheck/issues/143#issuecomment-909009632):

This is what I ended up using in my project, and what I consider the correct application of `find`+iteration.
See https://mywiki.wooledge.org/BashFAQ/001 for details.

```sh
#!/usr/bin/env bash

set -eu

is_bash() {
    [[ $1 == *.sh ]] && return 0
    [[ $1 == */bash-completion/* ]] && return 0
    [[ $(file -b --mime-type "$1") == text/x-shellscript ]] && return 0
    return 1
}

while IFS= read -r -d $'' file; do
    if is_bash "$file"; then
        shellcheck -W0 -s bash "$file" || continue
    fi
done < <(find . -type f \! -path "./.git/*" -print0)
```

Nonrecursive scanners invite accidents.

Note that UNIX find for shell identification purposes is a bad joke. I don't think it handles shebangs well. find's notation is arcane. POSIX find has crap exit codes. But file extension alone, basename alone, are still not sufficient to properly identify shell scripts. That's why I prototyped stank.

Frustrated that employers fail to sponsor such vital safety mechanisms as ShellCheck.

Frankly, shell scripting is inherently hazardous. Even short, innocuous scripts trigger numerous bugs. Not to mention, too many commands implement horrible exit code semantics.

I've been steadily rewriting shell glue code as dedicated Go or Rust applications. That way avoids much of the worst glitches arising from shells.

Raku has a lightweight syntax for invoking shell commands, with safe, predictable piping. There's no SCA for Raku dependencies, however. Aim for API's not CLI's.

Well, for what it's worth, there is the POSIX  Standard for Shell and
Utilities. They've codified existing historical practices of early shells
(Bourne, Korn, Almquist, Thompson, Tenex C, etc.) and defined a standard
that is intended to serve as a common (and therefore portable) set of
syntax for Unix `/bin/sh` and a list of ~125 to ~150 common Unix utilities.
The idea being that one can write a script on Fedora 43 in Bash 5.3 today,
and if the syntax meets the, say, POSIX-2008 standard, then there's a
reasonable, high likelihood that the script would also execute on a (still
in service) Oracle Solaris 10 box under Extended Support.

https://pubs.opengroup.org/onlinepubs/9799919799/
