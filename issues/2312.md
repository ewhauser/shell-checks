#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash
declare -A iso
iso[cinnamon]="linuxmint-20.2-cinnamon-64bit.iso"
iso[mate]="linuxmint-20.2-mate-64bit.iso"
iso[xfce]="linuxmint-20.2-xfce-64bit.iso"

if [[ $# -eq 1 && -v "iso[$1]" ]]; then
    eval "$qemu -cdrom ${iso[$1]}"
elif [[ $# -eq 2 && $2 -eq "oem" ]]; then
    eval "$qemu -cdrom ${iso[$1]} -hda $2.img"
else
    echo "This is a start script for testing Linux Mint with QEMU"
    echo "Usage: $0 cinnamon|mate|xfce [oem]"
fi
```

#### Here's what shellcheck currently says:
```
Line 11	SC2154: oem is referenced but not assigned.
```

"oem" is just a string. SC2154 is not an issue here.

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-08-30 18:47](https://github.com/koalaman/shellcheck/issues/2312#issuecomment-908595629):

ShellCheck is right, you are referencing `oem` as a variable:

```
oem=42
[[ oem -eq 42 ]] && echo "True"
```

To compare as a string instead of an integer variable, use `=` instead of `-eq`.

While it's a valid way to reference integer variables, it's also commonly used accidentally like here. Chances are ShellCheck should have a separate warning either for unused variables or in general.

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-08-31 03:22](https://github.com/koalaman/shellcheck/issues/2312#issuecomment-908866798):

This now triggers a new [[SC2309]](https://github.com/koalaman/shellcheck/wiki/SC2309). Thanks!

To see how this bug affects your script, you can run `./yourscript xfce foobar` and see that it's unable to differentiate between `oem` and `foobar` because both evaluate to `0`. Using `=` instead will correctly show the usage info.

Bash treats `=` and `==` the same in `[ .. ]` and `[[ .. ]]`. POSIX only defines `=` and `[ .. ]`.
