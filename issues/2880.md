# [\#2880 Issue](https://github.com/koalaman/shellcheck/issues/2880) `open`: Consider suggesting `&> afile` instead of `> afile 2>&1`

Consider suggesting `&> afile` instead of `> afile 2>&1` to redirect both stdout and stderr to the same file.

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/usr/bin/env bash

function foo()
{
	echo -n "$1 " >&2 # send to stderr
	echo "$2"         # this goes to stdout
}

foo error out &>bla       # here both strings to to file bla
```

Nothing

#### Here's what I wanted or expected to see:

This might be a bit obscure (in the sense few people know about), but helps me a lot to not screw up the order of the redirects.

- info: consider using `&> afile` instead of `> afile 2>&1`

Works on Bash and `sh`, did not test on other shells.

#### <img src="https://avatars.githubusercontent.com/u/16061266?v=4" width="50">[josephcsible](https://github.com/josephcsible) commented at [2023-12-22 05:41](https://github.com/koalaman/shellcheck/issues/2880#issuecomment-1867266359):

That's not valid in POSIX sh, and in fact, we already have a warning to make the exact opposite change ([SC3020](https://www.shellcheck.net/wiki/SC3020)).

See also https://unix.stackexchange.com/a/590707/217726:

> `&>` is a synonym of `>&` followed by a file name in bash (and zsh). It has the advantage of working even if the file name is a sequence of digits or `-` (for example `foo >&"$a"` is equivalent to `foo 1>&3 2>&1` if the value of `a` is the digit `3`, but to `foo >myfile 2>&1` if the value of `a` is `myfile`). Its downside is that it is incompatible with POSIX. In a POSIX-compliant shell, `echo foo &>bar` is parsed as the command `echo foo`, the `&` operator, and the command `>bar`: a background process that prints `foo`, and a foreground process that creates or empties the file `bar`.

And https://unix.stackexchange.com/a/159514/217726:

>  The `&>` variant originates in `bash`, while the `>&` variant comes from csh (decades earlier). They both conflict with other POSIX shell operators and should not be used in portable `sh` scripts.

#### <img src="https://avatars.githubusercontent.com/u/255938?v=4" width="50">[ormaaj](https://github.com/ormaaj) commented at [2024-10-16 19:59](https://github.com/koalaman/shellcheck/issues/2880#issuecomment-2417829027):

@drjasonharrison It's `|&` not `&|`. (`&|` does have several unrelated usages specific to zsh however.). It's supported by bash and zsh. `|&` collides with the `coproc` syntax used by ksh93 and mksh (and gawk). Its behavior is counterintuitive due to the way it implicitly injects the stderr redirect to the end of the previous command after other redirects. It's barely even a shortcut and just adds a pointless syntax inconsistency. 

So it's non-portable, crappy, and clobbers the syntax of a much more useful feature. Bash allows it, but it shouldn't be used.
