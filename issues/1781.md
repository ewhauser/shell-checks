#### <img src="https://avatars.githubusercontent.com/u/215702?v=4" width="50">[gromgit](https://github.com/gromgit) commented at [2020-01-07T01:03:58Z](https://github.com/koalaman/shellcheck/issues/1781#issuecomment-571385208):

Interesting. ShellCheck seems to be triggering off the fact that there's no `{}` in your `-execdir` command line. Simply adding it:
```
find "${dirs[@]}" -name .git -print -execdir "${cmdline[@]}" {} \;
```
makes ShellCheck shut up.

Note that a missing `{}` _is_ a bug in most cases (e.g. your command just prints a blank line for each found entry), so I think it's not so much a false positive as a misclassification.

#### <img src="https://avatars.githubusercontent.com/u/215702?v=4" width="50">[gromgit](https://github.com/gromgit) commented at [2020-01-07T14:27:08Z](https://github.com/koalaman/shellcheck/issues/1781#issuecomment-571608784):

> Since the `.git` directory already contains all the data the command needs,  there's no need for `{}`

That's an _atypical_ use of `find`; most `find` workflows explicitly process the items found, e.g.
```
# compress all *.log larger than 1MB
find /my/log/dir -type f -name \*.log -size +1M -execdir gzip -9 {} +
```
so a missing `{}` placeholder really is a bug in the general case, and a `disable` directive is the most appropriate way to deal with your code chunk:
```
# shellcheck disable=SC2150
find "${dirs[@]}" -name .git -print -execdir "${cmdline[@]}" \;
```
But like I said, SC2150 is a misclassification in this scenario, though I don't know whether it requires a new error code.

Sidenote: The following:
```
#!/bin/bash
cmdline=(printf %q\\n {})
find /bin -type f -exec "${cmdline[@]}" +
```
_also_ triggers SC2150, despite being syntactically equivalent to my first (SC-approved) example. This may be a parsing problem somewhere deep in SC's bowels, or perhaps it requires venturing into "dynamic program analysis" for a general solution.
