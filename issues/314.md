#### <img src="https://avatars.githubusercontent.com/u/7399565?v=4" width="50">[Dirk-Herrmann](https://github.com/Dirk-Herrmann) commented at [2021-06-17T10:55:13Z](https://github.com/koalaman/shellcheck/issues/314#issuecomment-863139960):

Hi and thanks for this great tool!
The "perfect" solution would issue SC2035 (or a related warning) in call cases where expansions could unintentionally produce "arguments looking like options".  However, sometimes variable expansion to arguments is not unintentional, like using some variable like `$verbosity` to configure script verbosity.  In such cases, to avoid too many false positives, variable expansion between other options should not be flagged.  Example: `rm $verbosity -r <args>` should not lead to a warning, also `rm $verbosity -- <args>`, however, `rm $verbosity <args>` should be flagged, because here ShellCheck can not deduce whether expansion to options would be intentional or not.  Also, `rm somefile $var` should be flagged, because `rm` even accepts options after file arguments.
Kind regards

The problem is that to always be relevant that rule would require knowing which external commands effectively support `--`.
Maybe all GNU commands and other GNU-getopt-based ones do, but shellcheck probably cannot be expected to test or know these implementation details.

Also this rule should probably be optional, because in my experience it would trigger a lot of violations on mundane, real-world code (not necessarily a bad thing for security though :smirk:...).
