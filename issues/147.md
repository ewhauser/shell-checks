#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-02-04T00:53:55Z](https://github.com/koalaman/shellcheck/issues/147#issuecomment-72769542):

This doesn't seem to be working (assuming the intention is to warn when a variable with a string value is compared using `-eq`)

The following script passes without comment on shellcheck.net.

```
state=IN_INIT
[[ $state -eq 0 ]]
echo $?
[ $state -eq 0 ]
echo $?
```

This is particularly unfortunate with `[[` since that, apparently, expands the variable and then evaluates the resulting word in arithmetic context and ends up using `0` as the value (for undefined variables).


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-02-04T01:15:32Z](https://github.com/koalaman/shellcheck/issues/147#issuecomment-72771962):

This is true -- the fix only triggers when comparing against literal non-integers. 

It's tricky since it's hard to determine whether an integer contains a number in a way that isn't intrusive, especially considering things like `state=$(grep state file | cut -d ':' -f 2)`. 

Reopening to look into whether easily detectable literal string examples like this one are frequent, and whether there is more that could be done to differentiate int from non-int variables in a way that doesn't flood with false positives.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-10-22T07:31:57Z](https://github.com/koalaman/shellcheck/issues/147#issuecomment-338458041):


My suggestion would be to recommend reserving [[ ]] for strings and related
operations, and instead recommend (( )) for all numeric operations including
comparisons.

(That is, treating comparators as just more arithmetic operators, as the C
language does, but which many Shell guides do not.)

Some shell guides already recommend using (( )) for arithmetic, but fall
short of explaining that, as far as C (and C++ and Perl and Python and PHP
and JS etc) are concerned, comparisons ARE arithmetic, along with bitwise
ops and bitshifts; that leaves the reader presuming that [[ ]] should be
used for comparisons, even when numbers are intended. So -eq and -ne are
strangely popular.

Which would mean effectively treating the -eq, -ne, -gt, -ge, -lt & -le
operators as if they were deprecated, and issuing warnings that recommend
replacing them with the corresponding C-like operators within (( )).

If that seems a step too far, a less intrusive possibility would be to raise
a warning along the lines of "quoted string is not literal in numeric
context". (There's no globbing or word-splitting there, so it doesn't matter
whether $ or * are present.)
