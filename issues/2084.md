#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/sh

printf "test $1\n"
printf "%s $1 \n" "test"
printf "%s ""$1 \n" "test"

```

#### Here's what shellcheck currently says:

```
Line 3:
printf "test $1\n"
       ^-- SC2059: Don't use variables in the printf format string. Use printf '..%s..' "$foo".
```



#### Here's what I wanted or expected to see:

```
Line 3:
printf "test $1\n"
       ^-- SC2059: Don't use variables in the printf format string. Use printf '..%s..' "$foo".
Line 4:
printf "%s $1 \n" "test"
       ^-- SC2059: Don't use variables in the printf format string. Use printf '..%s..' "$foo".
Line 5:
printf "%s ""$1 \n" "test"
       ^-- SC2059: Don't use variables in the printf format string. Use printf '..%s..' "$foo".
```



#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2020-12-31 05:30](https://github.com/koalaman/shellcheck/issues/2084#issuecomment-752849945):

This is currently intentional: ShellCheck does not emit this warning when the string already contains `%`s. The rationale for this is that if the command currently contains format specifiers, then the user is aware that they exist and therefore presumably chose to expand some of the values directly.
