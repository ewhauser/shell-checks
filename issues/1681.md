#### <img src="https://avatars.githubusercontent.com/u/2199511?v=4" width="50">[Nightfirecat](https://github.com/Nightfirecat) commented at [2019-09-01T15:17:38Z](https://github.com/koalaman/shellcheck/issues/1681#issuecomment-526927504):

`local` is allowed within bash, you just need to specify it in the shebang.

```bash
#!/bin/bash
foo() {
  local foo="bar"
  echo "$foo"
}
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-09-09T03:44:53Z](https://github.com/koalaman/shellcheck/issues/1681#issuecomment-529288694):

@vuxyp The job of `bash --posix` is to run POSIX compatible scripts, not to fail POSIX-incompatible ones.

POSIX explicitly says the behavior of `local` is undefined, so Bash can do whatever it wants with it and still be compatible. It's up to the script to not depend on undefined behavior.

You're right that strictly POSIX shell scripts can't define local variables, so you can either pick a naming scheme where global variables used only in single functions won't collide, or try to control the scope with subshells.

PS: `local` is a very common extension that works on bash, zsh, ksh, dash and BusyBox, so it wouldn't be totally unreasonable for you to just choose to depend on it anyways
