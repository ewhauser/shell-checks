# [\#2687 Issue](https://github.com/koalaman/shellcheck/issues/2687) `open`: New warnings regarding "set +H", "history expansion", and the dreaded "!"

#### <img src="https://avatars.githubusercontent.com/u/110406?v=4" width="50">[ramses0](https://github.com/ramses0) opened issue at [2023-02-06 22:50](https://github.com/koalaman/shellcheck/issues/2687):

Shellcheck is blissfully silent w.r.t. history-expansion exclamation points within double-quoted strings, which is a totally non-intuitive mis-feature of bash.

https://mywiki.wooledge.org/BashGuide/SpecialCharacters

https://www.gnu.org/software/bash/manual/html_node/Quoting.html

https://www.gnu.org/software/bash/manual/html_node/Double-Quotes.html

"""A double quote may be quoted within double quotes by preceding it with a backslash. If enabled, history expansion will be performed unless an ‘!’ appearing in double quotes is escaped using a backslash. The backslash preceding the ‘!’ is not removed."""

#### For new checks and feature suggestions
- [x] shellcheck.net (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

Related issue: "prefer single quotes": https://github.com/koalaman/shellcheck/issues/2346

Stack Exchange post that helped diagnose my https://unix.stackexchange.com/questions/33339/cant-use-exclamation-mark-in-bash

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash
echo "WARN: This message appears normal! But bash does funny things" \
&& echo "WARN: when there are random !123 exclamation points in strings :-("
&& echo "DONE."

$ echo "hello"
hello

$ echo "hello!-1"
echo "helloecho "hello""
helloecho hello

$ echo 'hello!-1'
hello!-1
```

#### Here's what shellcheck currently says:
```
Nothing, zip zero, nada...
```
#### Here's what I wanted or expected to see:

SC1234: Exclamation point detected in double-quoted string... are you _SURE_ you know what you're doing?

SC1235: Double-quoted string used WITHOUT an obvious variable expansion inside (eg: want "foo $BAR baz" vs. "foo BAR baz"), prefer single quoted strings when no variables are in use to reduce cognitive load and future bugs because bash's quoting rules about variable expansion and special characters are byzantine.

...basically, I am a student of "wooledge" since long back and I've been randomly bitten by `!` quite often (eg: `echo "DONE!"`).

```
$ echo "DONE!"
-bash: !": event not found
```

...I am a hard-agree w/ the issue #2346 that single quotes should be universally preferred unless there is actual variable expansion.

There's basically three warnings:

1) Exclamation point found in double-quoted string

2) No $VARIABLES found in double-quoted string (prefer single-quoting)

3) Single quoted string with '$VARIABLE' found (did you mean double-quotes?)

...anything being _mentally_ processed by a human as a single-quoted string is "easier", therefore a pedantic shellcheck should guide users to use single quotes in bash as much as possible.  All of them are related, and at least the `!` one can be devastatingly surprising if you're not careful.

Please, annoy me more frequently and make me use single quotes as much as possible!
