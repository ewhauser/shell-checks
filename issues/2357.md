# [\#2357 Issue](https://github.com/koalaman/shellcheck/issues/2357) `open`: Use JSON as format of `.shellcheckrc`

#### <img src="https://avatars.githubusercontent.com/u/29582865?u=0c57eebb21e1922b4a0345f7a830924cc295998b&v=4" width="50">[felipecrs](https://github.com/felipecrs) opened issue at [2021-10-17 17:32](https://github.com/koalaman/shellcheck/issues/2357):

The list of supported options and fields in `.shellcheckrc` tends to grow, naturally. There are many benefits of adopting an existing format, as JSON, for the config file:

1. Easier and better editor integration: any editor will already know how to handle JSON (and JSONC, if comments are desired).
2. Support for JSONSchema: allows to easily provide intellisense such as (advanced) auto-complete, and validation. Not to mention [tools that auto-generates documentation](https://json-schema.org/implementations.html#documentation-generators) based on a JSONSchema.
3. As it's a stablished format, it will handle any future corner case which `.shellcheckrc` would also have to deal, such as escaping spaces in a path, for example.

As migration plan, I can suggest:


## The old format

- Support for it can still remain for simpler use-cases
- Or, support for it can be deprecated in the next minor release (as shellcheck is in 0.x, breaking changes are allowed in minor releases)
- Or, support for it can be supported until a subsequent minor release (e.g. supported in 0.8 but deprecated in 0.9 to give people time to migrate)

## The file name

- It can remain the same. It's reasonable easy to distinguish a JSON format from the old format looking at the contents of the file (look for `{}`). There are examples of tools which uses this pattern.
- Or, the new JSON-based file name can have a `.json` suffix, `.shellcheckrc.json`. There are examples of tools which uses this pattern too.

## The JSON schema

While not mandatory, the JSON Schema would help editors to integrate better with the configuration file. The `shellcheck` cli could also use it to validate the config file internally.
