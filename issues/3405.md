#### Here's a snippet or screenshot that shows the problem:

```bash
#!/usr/bin/env bash
# shellcheck enable=all
foo(){
  declare -i cd_result
  cd "$1" || {
    cd_result=$?;
    echo "foo no such directory";
    return ${cd_result};
  }
}
```

#### Here's what shellcheck currently says:
shellcheck --enable=all truc.sh 

In truc.sh line 7:
    return ${cd_result};
           ^----------^ SC2248 (style): Prefer double quoting even when variables don't contain special characters.

Did you mean:
    return "${cd_result}";

For more information:
  https://www.shellcheck.net/wiki/SC2248 -- Prefer double quoting even when v...

#### Here's what I wanted or expected to see:

I think the only valid exception to rule SC2248 is when the variable is declared as integer.
Script above is such a case,
and in my opinion there should be no warning in that case.



#### <img src="https://avatars.githubusercontent.com/u/4527928?v=4" width="50">[LLyaudet](https://github.com/LLyaudet) commented at [2026-03-03 15:24](https://github.com/koalaman/shellcheck/issues/3405#issuecomment-3991783678):

Note that there is a slightly distinct case, where we return an integer via an indirection.
```
#!/usr/bin/env bash
# shellcheck enable=all
foo(){
  declare -r result_name="cd_result"
  declare -i "${result_name}"
  cd "$1" || {
    printf -v "${result_name}" "%d" $?
    echo "foo no such directory"
    return ${!result_name}
  }
}
```
In that case shellcheck doesn't return SC2248, but SC2086.
I don't know if both cases can be solved in one fix.
Otherwise if this first step is corrected one day, then maybe it will be ok to open another issue to solve the false positive for SC2086.
