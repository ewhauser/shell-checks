# [\#548 Issue](https://github.com/koalaman/shellcheck/issues/548) `closed`: SC2035 should suggest using --

#### <img src="https://avatars.githubusercontent.com/u/937876?v=4" width="50">[dfong](https://github.com/dfong) opened issue at [2015-11-16T22:27:04Z](https://github.com/koalaman/shellcheck/issues/548):

the SC2035 warning suggests using ./\* instead of *, but in many cases i believe a better solution would be to insert "--".

consider a command like `tar cf whatever.tar *.conf` .  if we change the glob pattern to `./*.conf`, it affects the filenames stored in the tarball.  in a tar listing, the ./ would be present.

similarly, with a command like "ls *.conf", adding ./ would clutter up the listing, and possibly confuse commands downstream that many not expect ./ in the output.

i noticed that adding -- to the command line does silence the shellcheck warning.  i feel that this can be a better solution in the cases where the command in question honors "--".

The problem with `--` is that it isn't universal (across commands), or universally portable (between the same command from different sources). The `--` is, I believe, a GNU-ism and so isn't available with BSD-derived (or other) tools.


#### <img src="https://avatars.githubusercontent.com/u/937876?v=4" width="50">[dfong](https://github.com/dfong) commented at [2015-11-16T23:09:12Z](https://github.com/koalaman/shellcheck/issues/548#issuecomment-157203898):

although i agree that support for "--" is not universal, i believe it is a posix standard (in the posix getopts).

plus, when i added "--" to the command line that shellcheck had been complaining about, the SC2035 warning went away.  this leads me to believe that shellcheck already has some special-casing for "--".

i would also note that there are other shellcheck warnings that make suggestions that are not universal.  for example, the one about using bash's `${var//}` notation instead of sed if possible.

in the case where it is possible to use "--", i feel it is a better solution.  because, unlike adding ./ to the glob pattern, adding "--" doesn't change the command's output in the usual case where there aren't any filenames starting with "-".

The suggestion to use `${var//}` happens in the context of a bash script where it is known that it will work. No such knowledge can be had about whether `--` will work for any given command (without a list and some attempts at guessing based on observed options).

Yes, `--` is POSIX for [`getopts`](http://pubs.opengroup.org/onlinepubs/9699919799/utilities/getopts.html) but `getopts` is a specific mechanism.

Neither of those things changes the fact that it isn't universal across commands or sources and that shellcheck can't know which do and which don't. It could certainly include `--` as an option as well as `./` though (which it does on the [SC2035 wiki page](https://github.com/koalaman/shellcheck/wiki/SC2035)).

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-11-17T03:52:42Z](https://github.com/koalaman/shellcheck/issues/548#issuecomment-157260859):

I believe the "See if" in SC2001 is because the `//` pattern support is _not_ a regular expression and so _cannot_ do everything that `sed`/etc. can and without evaluating the specific pattern (and even then) shellcheck can't know whether you can replace one with the other. I do _not_ believe it has anything to do with not knowing that `//` is valid. (And, in fact, in `#!/bin/sh` mode SC2001 does **not** fire for the example on the SC2001 wiki page.)

The "hesitation" in possible misspelling warnings about variable names are also **not** about whether the environment of the script itself can support the correction but about whether shellcheck correctly guessed the author's intent (something it can't know if it got right). Similarly with quoting variables. It isn't that shellcheck doesn't know whether quoting the variable is **valid** or not. It is that it doesn't know whether, for some reason, you want **this** specific variable usage unquoted (because you need globbing or word splitting). 

I cannot think of a situation in which adding `./` to a path would cause any programs to do anything incorrect with a file in the local directory but I can't say that none ever will either. You are certainly correct that in cases where `--` is possible it can be preferable (and is my personal preference as well). That doesn't change the fact that it simply isn't universally suitable and cannot be depended on to work for any given situation. Whereas prepending `./` _can_ (barring ill-behaved programs that treat `file` and `./file` differently in some meaningful way).
