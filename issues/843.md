#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-03-05T07:28:39Z](https://github.com/koalaman/shellcheck/issues/843#issuecomment-284211085):

On the contrary, anything that involves indirect variable naming should be
warned unless there's an explicit "don't warn" override in the code.
Symbolic indirection is fraught with dangers, special cases, etc.

What about if $var is not actually a valid name? What if it's readable but
not writable? What if it refers to a special var?

For example, what happens if $var happens to be IFS or PATH or TZ or
LD_LIBRARY_PATH or EUID or ...?

Yes you could put in guards, but did you check for PATH[anything] ?
Did you check for anything[IFS++] ?

Really it's worth avoiding just on principle.

There are almost always better ways to achieve the same effect -- and 99
times out of 100, that's by using an associative array.

What are you actually trying to do with that code?


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-03-18T22:07:32Z](https://github.com/koalaman/shellcheck/issues/843#issuecomment-287578319):

This warning is purely a bug caused by reusing the lenient assignment parser for `define`, `export` and friends. It has other unintentional effects like not warning about quoting for $var/$val, and this thing:

```
declare n$1=foo
        ^-- SC1067: For indirection, use (associative) arrays or 'read "var$n" <<< "value"'
```

Indirect assignment might be bad, but a warning about an unrelated issue doesn't help that.

This has been fixed in 2688a81
