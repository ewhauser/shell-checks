# [\#3429 Issue](https://github.com/koalaman/shellcheck/issues/3429) `open`: dangerous SC2164 recommendation

#### <img src="https://avatars.githubusercontent.com/u/5316?u=8763bdb9ba5cbe23084e8249dcbd21bbb23150f2&v=4" width="50">[mcandre](https://github.com/mcandre) opened issue at [2026-03-25 19:21](https://github.com/koalaman/shellcheck/issues/3429):

ShellCheck provides toxic advice for `cd` customization lines, for example in .zshrc configuration files.

SC2164 insists on triggering `exit` or `return` when change directory fails. Wouldn't `exit` force the entire shell session to terminate?

Frankly, `cd` has its own exit codes. Why not simply allow those to propagate normally, instead of trying to wrap cd's exit codes in inherently worse error handling code?

Unfortunately, SC2164 is one of those cases where shellcheck may do more harm than good. Giving overly cautious, impractical advice "just to be sure" is not for free but has a price: It wears out mental awareness more than sharpens it, and thereby easily erodes the security it aims to establish.

> Frankly, `cd` has its own exit codes. Why not simply allow those to propagate normally, instead of trying to wrap cd's exit codes in inherently worse error handling code?

Without following SC2164—`set -e`, or `exit` or `return`—how do you propagate its exit code?

Personally I don&rsquo;t believe it is good practice to store the cd failure as a variable and have branches:

```sh
cd /foo
fail=$?

blabla

if [ $fail -eq 0 ]; then
  echo 'do stuffs when cd is successful'
else
  echo 'do stuffs when cd failed'
fi

blabla
```

Whilst the [wiki page on SC2164](https://github.com/koalaman/shellcheck/wiki/SC2164) does a good job explaining the problem and presenting a variety of viable solutions, the actual message IMO suggests too strongly two specific solutions that still break, rather than safeguard, supposedly intended behavior.
