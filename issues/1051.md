#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bash

example() {
    echo stdout
    echo stderr > /dev/stderr
}

example |& > /dev/null
```

#### Here's what shellcheck currently says:

```
$ shellcheck myscript
 
Line 8:
example |& > /dev/null
           ^-- SC2189: You can't have | between this redirection and the command it should apply to.
```


#### Here's what I wanted or expected to see:

Nothing. This is from the bash man page:

```
If |& is used, command's standard error, in addition to its standard
output, is connected to command2's standard input through the pipe; it
is shorthand for 2>&1 |. This implicit redirection of the standard
error to the standard output is performed after any redirections
specified by the command.
```

#### <img src="https://avatars.githubusercontent.com/u/8651654?v=4" width="50">[jleffler](https://github.com/jleffler) commented at [2017-11-15T16:56:14Z](https://github.com/koalaman/shellcheck/issues/1051#issuecomment-344656829):

The `|&` is a variation on `|`.  It needs to appear where `|` would appear, between commands.  `command1 |& command2` is a shorthand for `command1 2>&1 | command2`.  Your example use is bogus; the `>/dev/null` isn't associated with any command.  If you had `cat` or `:` or almost anything associated with that redirection, you'd be in with a chance: `example |& > /dev/null :` (I/O redirection before the command is legal, if confusing).
