#### <img src="https://avatars.githubusercontent.com/u/131439?v=4" width="50">[Jemi](https://github.com/Jemi) opened issue at [2020-02-29 06:22](https://github.com/koalaman/shellcheck/issues/1852):

#### For bugs
- Rule Id (if any, e.g. SC1000): 
SC2012
- My shellcheck version (`shellcheck --version` or "online"): 
Any
- [*] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2012)

Specifically, SC2012 wiki should cover the case where ls is being used just to get the number of files matching a certain literal or glob pattern, rather than propagating the actual name of any particular file outside of that line of code.

example:
`NUMGZ="$(ls -l *.gz | wc -l)"`

Either this is a viable use case and it should go in the exceptions (or even detected from within shellcheck), or it is not and the examples should explain what's wrong with it.

#### <img src="https://avatars.githubusercontent.com/u/1885701?v=4" width="50">[dimo414](https://github.com/dimo414) commented at [2020-03-02 08:54](https://github.com/koalaman/shellcheck/issues/1852#issuecomment-593291339):

You don't need to use `ls` to trigger glob expansion. Any unquoted string is subject to globbing regardless of context. If you want to expand a glob and count the number of matches an array is probably easiest:

```
gz_files=(*.gz)
numgz=${#gz_files[@]}
```

I don't think there should be an expectation that SC pages must _exhaustively_ list all possible circumstances. Just because your particular use case isn't mentioned doesn't mean it's a false positive.

Those "several suggestions" seem to have the same filename characteristics as `ls -tr` which BTW detects that it's not writing to a tty and automatically switches to a plain single column format. 

In addition they use an intermediate text file to sort the files by date which may break permanently if a file with a newline exists in the directory. The `ls -tr` actually sorts by the date, so once the logfile gets written to again any junk file is ignored because it's older.

I suppose I could limit it to GNU tools and use the `-z` options to make those suggestions work; but if I do that I might as well use `-q` on `ls` because nuls and newlines are not going to be in valid filenames anyway.

#### <img src="https://avatars.githubusercontent.com/u/3522333?u=dbfb7b259e735c00d66d8b8aaa812e68bc7e52ff&v=4" width="50">[kankaristo](https://github.com/kankaristo) commented at [2022-01-28 15:22](https://github.com/koalaman/shellcheck/issues/1852#issuecomment-1024323571):

Is the [rationale for SC2012](https://github.com/koalaman/shellcheck/wiki/SC2012) relevant when using a "plain" `ls` instead of `ls -l`? I'm genuinely asking, because the current rationale really only explains problematic usage of `ls -l`.

The current rationale also doesn't give the real filenames for the three "seemingly identical" filenames, nor does it show the equivalent output for `find` for those filenames. So, I'm not sure what kind of filenames would be problematic with `ls` vs. `find`.

The other examples for replacing `ls` with `find` don't actually seem to contain any really problematic output for `ls`, and the `ls` commands are more succinct and readable.

-----

As another example, I'm using this to find the version number of the latest installed Android build tools:
```bash
ls "$sdk_root/build-tools" | sort -r | head -n1
```

ShellCheck would want me to change this to:
```bash
find "$sdk_root/build-tools" -mindepth 1 -maxdepth 1 -printf '%P\n' | sort -r | head -n1
```

... which is equivalent, but the code is much less readable. The `find` command could likely use a comment explaining what it does, whereas the `ls` equivalent could go without (in my opinion at least).

Another issue with the above is that `-printf` is not available in all versions of `find` (e.g. on macOS), requiring more changes to the script, possibly making it even less readable.

P.S.
I know I could rely on `ls` for sorting (or use `tail` instead of `head`), but I wanted to make the examples as similar as possible. I could also be using `sdkmanager` to check the installed build tools, but that also queries external servers (making it much slower, etc.).
