# [\#410 Issue](https://github.com/koalaman/shellcheck/issues/410) `open`: Warn about using && without || when set -e is enabled

#### <img src="https://avatars.githubusercontent.com/u/84020?v=4" width="50">[sinelaw](https://github.com/sinelaw) opened issue at [2015-07-09T12:12:37Z](https://github.com/koalaman/shellcheck/issues/410):

`set -e` has a surprising behaviour.

```
#!/bin/bash
set -e

cat notexist && echo ok

echo "This will run regardless of whether 'cat' succeeds. set -e ignores failures before &&"
```

From the bash manpage section about `set -e`:

> Exit immediately if a pipeline (which may consist of a single simple command), a list, or a compound command (see SHELL GRAMMAR above),  exits with a non-zero status.  **The shell does not exit if the command that fails is** part of the command list immediately following a while or until keyword, part of the test following  the if or elif reserved words, **part of any command executed in a && or || list except the command following the final && or ||, any command in a pipeline but the last**, or if the command's return value is being inverted with !.  If a compound command other than a subshell returns a non-zero status because a command failed while -e was being ignored, the shell does not exit.  A trap on ERR, if set, is executed before the shell exits.  This option applies to the shell environment and each sub-shell environment separately (see COMMAND EXECUTION ENVIRONMENT above), and may cause subshells to exit before executing all the commands in the subshell.

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-07-10T17:40:02Z](https://github.com/koalaman/shellcheck/issues/410#issuecomment-120474210):

Warning about using `&&` and `||` when `set -e` is enabled isn't really reasonable and neither is suggesting to use `X && Y || Z` since that's not a valid ternary expression (shellcheck already warns about that in fact).

This exact problem (and worse corner cases) are why many people suggest _not_ using `set -e` (and `set -u`). See [this comment](https://github.com/koalaman/shellcheck/issues/409#issuecomment-120438149) on https://github.com/koalaman/shellcheck/issues/409 for some discussion about that in the context of a request to have shellcheck recommend `set -e`/`set -u`.

So what do you actually want shellcheck to do here?

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-07-12T17:03:42Z](https://github.com/koalaman/shellcheck/issues/410#issuecomment-120740836):

When you have `set -e` in effect using `&&` is an entirely reasonable thing to want to do though. Since you have to handle things that can optionally fail somehow and that works just fine. (Using `X && :` is as valid an avoidance of the error as `X || :` is. Though the logic of the latter is certainly clearer.)

I suppose it would, theoretically, be useful (in a pedantic mode or something) to have shellcheck warn about the possibility of unintended consequences on `X && Y` though I don't expect that to be all that commonly useful.
