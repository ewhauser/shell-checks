# [\#556 Issue](https://github.com/koalaman/shellcheck/issues/556) `closed`: parsing issue with test expression

#### <img src="https://avatars.githubusercontent.com/u/1700395?v=4" width="50">[JoshLitt](https://github.com/JoshLitt) opened issue at [2015-12-09T00:44:32Z](https://github.com/koalaman/shellcheck/issues/556):

Input file:

``` shell
#!/bin/bash
file=$1
[[ ${file::1} = [-.\|/\\] ]] && return 1
```

Result (shellcheck 0.4.1):

```
In /tmp/foo.sh line 4:
[[ ${file::1} = [-.\|/\\] ]] && return 1
^-- SC1073: Couldn't parse this test expression.
                         ^-- SC1020: You need a space before the ]].
                         ^-- SC1072: Missing space before ]. Fix any mentioned problems and try again.
```

If I change the = to =~ the error goes away. But this is not a regex test, it's a glob expression test.
If I add a "*" after the rhs, the error goes away, but since this is testing a single character, that's not needed. It's not bad to work around this, but not clear why it can't be parsed.

#### <img src="https://avatars.githubusercontent.com/u/1700395?v=4" width="50">[JoshLitt](https://github.com/JoshLitt) commented at [2015-12-09T01:09:29Z](https://github.com/koalaman/shellcheck/issues/556#issuecomment-163074826):

No, it's pulling off the first character of $file. It's a form of
`${parameter:offset:length}`


#### <img src="https://avatars.githubusercontent.com/u/1700395?v=4" width="50">[JoshLitt](https://github.com/JoshLitt) commented at [2015-12-09T01:13:02Z](https://github.com/koalaman/shellcheck/issues/556#issuecomment-163075291):

I guess using `${file:0:1}` would be more clear, but generates the same parsing issue.


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-12-13T18:24:13Z](https://github.com/koalaman/shellcheck/issues/556#issuecomment-164285695):

This was due to shortcuts in parsing `[..]` which didn't account for escaped characters. This has been fixed in 5283817. Thanks!
