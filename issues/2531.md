#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
true && { echo "true"; false; } || echo "false"
```

#### Here's what shellcheck currently says:
No issues detected!


#### Here's what I wanted or expected to see:
Don't use && ... || in place of if...then...else...fi as they do not operate equally in all cases.

Here is the case where the above syntax is troublesome:

```sh
$ true && { echo "true"; false; } || echo "false"
true
false
```

Which is not likely what the writer expected.  I suspect they expect the same handling as:

```sh
$ if true; then echo "true"; false; else echo "false"; fi
true
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2022-07-12 20:56](https://github.com/koalaman/shellcheck/issues/2531#issuecomment-1182493595):

There's already a warning for `A && B || C` (https://www.shellcheck.net/wiki/SC2015), it just happens to be suppressed when C is `echo/printf/exit/return` as these tend to be benign.

If you change it to e.g. `true && { echo "true"; false; } || rm -rf foo` you'll see `SC2015 (info): Note that A && B || C is not if-then-else. C may run when A is true.`

Honestly, IMHO, all constructs of `A && B || C` should be flagged and avoided.  In the least, it's symbol-soup (i.e. perl, blech) and at best it's a logic bomb just waiting to blow up on those that don't really understand that `A && B || C` is actually **not** `if A then B; else C; fi`
