# [\#1583 Issue](https://github.com/koalaman/shellcheck/issues/1583) `open`: Don't trigger SC2154 (unassigned var) in `-n`/`-z` expressions

#### <img src="https://avatars.githubusercontent.com/u/594866?v=4" width="50">[trespasserw](https://github.com/trespasserw) opened issue at [2019-05-17T09:45:13Z](https://github.com/koalaman/shellcheck/issues/1583):

#### For bugs
- Rule Id: SC2154
- My shellcheck version: 0.6.0
- [x] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2086)
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/sh

if [ -n "$__wtf__" ]; then
  echo "there is some $__wtf__"
fi

```

#### Here's what shellcheck currently says:

```
Line 3:
if [ -n "$__wtf__" ]; then
         ^-- SC2154: __wtf__ is referenced but not assigned.
```

#### Here's what I wanted or expected to see:

No warnings.

#### Rationale:

The `-n` and `-z` checks are idiomatic ways of probing a global variable before it's use.

#### <img src="https://avatars.githubusercontent.com/u/1111474?v=4" width="50">[leagris](https://github.com/leagris) commented at [2019-07-09T11:30:19Z](https://github.com/koalaman/shellcheck/issues/1583#issuecomment-509604947):

If you use this, ShellCheck does not complain because it has a default empty value
```sh
if [ -n "${__wtf__:-}" ]; then
  echo "there is some $__wtf__"
fi
```

#### <img src="https://avatars.githubusercontent.com/u/1111474?v=4" width="50">[leagris](https://github.com/leagris) commented at [2019-08-26T22:16:39Z](https://github.com/koalaman/shellcheck/issues/1583#issuecomment-525054732):

I agree with @matthewpersico that the default empty value makes it explicit. When you review the code 6 months later, you know that `$__wtf__` may come from an external source, and if it is undefined you want to consider it as an empty string.

Now I also think that shellcheck is a bit too zealously complaining on `"$__wtf__"`, because it is also clear this will result in an empty string when variable is not defined.

This is probably a check to move to some strict switch when/if this is implemented in the future like the `"\\"` backslash escaping in double quote strings.
