# [\#2777 Issue](https://github.com/koalaman/shellcheck/issues/2777) `open`: Emit warning for standalone pre/post increment/decrement operations with 'set -e'

#### <img src="https://avatars.githubusercontent.com/u/93103176?u=45e96e99b151ca242f0ae0920f56bf6eae033de5&v=4" width="50">[varungandhi-src](https://github.com/varungandhi-src) opened issue at [2023-06-12 10:32](https://github.com/koalaman/shellcheck/issues/2777):

#### For new checks and feature suggestions
- [x] https://www.shellcheck.net/ (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bash

set -e

f() {
        local a=0
        ((a++))
}

f
```

#### Here's what shellcheck currently says:

No issues detected!

#### Here's what I wanted or expected to see:

```
(warning): ((a++)) may fail if a is 0, which may be counter-intuitive 
```

There could be a hint to optionally add a `|| :`. There should be a similar warning for a standalone pre-increment, as well as corresponding decrement operations. I'm not picky about the exact wording, but I think there should be a warning here.

I think this is worth warning about, as in most other PLs, arithmetic operations like increment and decrement don't ever fail.

#### <img src="https://avatars.githubusercontent.com/u/11050776?u=ffd74c82e5dba21167975435396bf0d39e654640&v=4" width="50">[auspicacious](https://github.com/auspicacious) commented at [2026-01-02 12:31](https://github.com/koalaman/shellcheck/issues/2777#issuecomment-3705234041):

I ran into this recently when someone gave me a script that ran fine on their MacOS machine, but wouldn't work on my Linux machine. As far as I can tell, the ancient version of Bash that ships with MacOS doesn't have the "correct" behavior here and doesn't fail. (Worse, AI agents are apparently writing code that includes `((var++))`, since I know that one wrote much of this script).

It might be better to suggest using `((var+=1))` instead, since this doesn't distract someone reading the code with syntax that isn't related to the operation being performed.
