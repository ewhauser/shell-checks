# [\#1730 Issue](https://github.com/koalaman/shellcheck/issues/1730) `closed`: SC2086 false positive on builtin command exit.

#### <img src="https://avatars.githubusercontent.com/u/319907?v=4" width="50">[nnathan](https://github.com/nnathan) opened issue at [2019-10-23T10:53:38Z](https://github.com/koalaman/shellcheck/issues/1730):

SC2086 (quoting to avoid word splitting) should be disabled when a parameter is supplied to the `exit` command since it is a built-in and only requires an integer parameter. It kind of looks yuck to quote it.

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/usr/bin/env bash
bad() {
  ret=$1
  shift
  echo -e failed: "$@"
  exit $ret
}
```

#### Here's what shellcheck currently says:

On the `exit $ret` line:
SC2086: Double quote to prevent globbing and word splitting.

#### Here's what I wanted or expected to see:

I'm thinking shellcheck should detect the exit command is being used and not apply the SC2086 rule.

#### <img src="https://avatars.githubusercontent.com/u/319907?v=4" width="50">[nnathan](https://github.com/nnathan) commented at [2019-10-23T12:49:33Z](https://github.com/koalaman/shellcheck/issues/1730#issuecomment-545427477):

Exactly, you can't lint against an invalid parameter to exit in this case. Well you might be able to - but that would be incredibly tricky.

I should point out that using bad is used carefully:

```
good() {
  echo -e success: "$@"
}

bad() {
  ret=$1
  shift
  echo -e failed: "$@"
  exit $ret
}

try() {
  "$@" || bad $? "$@" && good "$@" 
}

try ls /nonexistentfile
```

And you can then imagine bad and good spitting out appropriate colours/statuses/unicode checkmarks/etc. which I've omitted for brevity.

So assuming a valid input to exit, it then must be an integer. An integer doesn't need quotes with standard IFS. Ergo no need for SC2086.

#289 argues for a similar thing, namely they want shellcheck to get hints by declaring a variable as integer. I don't even want to do that - I think it's unpleasing. I just want SC2086 to be disabled when it detects the exit command builtin.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-10-09T17:00:19Z](https://github.com/koalaman/shellcheck/issues/1730#issuecomment-939327882):

I would tend to agree that it's the variable's value that determines whether the expansion needs quotes, and not the other way around. With better dataflow analysis, ShellCheck could conceivably determine that `$1` is only ever `$?`, but it's not there yet.

`declare -i ret=$?` is not yet recognized but would have the additional advantage of making `ret` local.
