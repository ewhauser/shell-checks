SC2029 gives an error on intentional client-side-expanded variables used in remote SSH commands.  The wording implies the error should actually be a Warning.  The wording of the page on SC2029 even says
`If you do want your string expanded on the client side, you can safely ignore this message.`

- Rule Id (if any, e.g. SC1000): SC2029
- My shellcheck version (`shellcheck --version` or "online"): 0.4.5
- [X] I tried on shellcheck.net and verified that this is still a problem on the latest commit
- [ ] It's not reproducible on shellcheck.net, but I think that's because it's an OS, configuration or encoding issue


#### Here's a snippet or screenshot that shows the problem:

```sh

#!/usr/bin/env bash
source "variables.sh"
ssh "$deploy_user"@"$server" "rm -rf \"$final_root_directory/feeds/feeds\""

```

#### Here's what shellcheck currently says:

Warning: SC2029: Note that, unescaped, this expands on the client side.
