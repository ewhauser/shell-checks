# [\#518 Issue](https://github.com/koalaman/shellcheck/issues/518) `open`: SC2153 confuses local variables from other functions

#### <img src="https://avatars.githubusercontent.com/u/937876?v=4" width="50">[dfong](https://github.com/dfong) opened issue at [2015-10-23T19:01:07Z](https://github.com/koalaman/shellcheck/issues/518):

this is in shellcheck-0.4.1.  i did not see it in shellcheck-0.3.5 which i had been using before.
example:

```
#! /bin/bash
# test script for shellcheck SC2153

# defines local $myvar
F()
{
        local myvar=1
        echo "in F: myvar=$myvar"
}

# uses global $MYVAR.  note that $myvar should be out of scope
G()
{
        echo "in G: MYVAR=$MYVAR"
        echo "in G: myvar=$myvar"
}

# ----- start of mainline code
F
G
```

when i run shellcheck on the above script, i get:

```
In t.sh line 14:
    echo "in G: MYVAR=$MYVAR"
                          ^-- SC2153: Possible misspelling: MYVAR may not be assigned, but myvar is.
```

note that at line 14, "myvar" from F is out of scope.

On Fri, 29 Jul 2016, Adam Romanek wrote:

> OK I understand Shellcheck limitations.

That's a tad disingenuous; deciding whether some code gets executed is not
generally soluble -- it's equivalent to The Halting Problem.

The Halting Problem has itself been formally proven: it is impossible to
prove or disprove whether every program will halt or not. (For some simple
programs yes, but for any given proving method, some programs will be
undecidable.)

In the face of exported functions, it is basically impossible for
Shellcheck to know exactly what your program will do, without actually
running it (which would rather defeat the point).

So Shellcheck takes the approach of issuing warnings unless it can prove
that the warning is not necessary. Failing that, if the human can prove it,
they're welcome to add a "disable" annotation.

> This is still relevant on version 0.8.0 :(

Yes :( I wish I could take a month off my day job and get this done, but that has not been an option so far.

> BTW it seems that if the local variable (and the env variable) don't have underscores this doesnt happen

It's actually the length. ShellCheck's heuristic is "the variable might be misspelled if it's 4+ characters long and 1 char apart, or 7+ chars long and up to 2 chars apart". The point of this is to not trigger on `$bar` vs `$baz`, and do trigger on `$referer` and `$referrer`. For the purpose of this check, two strings that only differ in case is considered 1 char apart, and thus it somewhat unintentionally ends up only triggering on 4+ character strings.

>aside: shouldn't ShellCheck warn when both function and () are used together

It's valid in Bash so there's no warning against it, but if you do it in Ksh you get https://github.com/koalaman/shellcheck/wiki/SC2111

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2021-12-28T02:39:37Z](https://github.com/koalaman/shellcheck/issues/518#issuecomment-1001838755):

> It's valid in Bash so there's no warning against it

I know it's allowed, but it seems to me like a gratuitous incompatibility that should be discouraged as it's a difference without any purpose, so I'd like to see [SC2111](../wiki/SC2111) included in the warnings for Bash as well, at least when in "pedantic" mode.

One reason I dislike this difference is because it means that we can't simply translate
```bash
foo() ( echo bar )
```
to
```bash
function foo ( echo bar )
```
To be fair, we can't yet write that in Ksh either, where the grammar says <tt>function <i>NAME</i> { <i>LIST</i> ; }</tt> (ie, a brace-block), but Ksh could relax that rule without breaking existing code, whereas Bash would have yet another messy "look 2 tokens ahead" to maintain compatibility both ways.
