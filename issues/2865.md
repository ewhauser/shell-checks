#### Here's a snippet or screenshot that shows the problem:

Busybox SH is an ash derivative with a few features from ksh.
Running some tests, I noticed the following SCxxxx errors do not apply to busybox sh

- SC3010: [[ ]] in place of [ ]
- SC3014:  == in place of =
- SC3020: &> 
- SC3046/SC3051: 'source' in place of '.'
- SC3048: prefixing signal names with 'SIG'
- SC3057: string indexing 
- SC3060: string replacement

```sh
#!/bin/busybox sh

 X="thisisastring"

stop() {
    echo stopping
    exit 0
}
# SC3048
trap stop EXIT SIGTERM SIGINT

# SC3010 and SC3014
[[ $X == "thisisastring" ]] && echo "this shell supports == and [[ ]]"

# SC3020
echo "busybox sh supports &>" &> /dev/null

# SC3046/SC3051
source /dev/null

# SC3057
echo "${X:0:4} shell supports ${X:7} indexing" # "this shell supports string indexing"

# SC3060
echo "${X/string/magic}"

# for SC3048
sleep 3600
```

and the output is:

#### <img src="https://avatars.githubusercontent.com/u/68940237?v=4" width="50">[slycordinator](https://github.com/slycordinator) commented at [2023-11-26 15:31](https://github.com/koalaman/shellcheck/issues/2865#issuecomment-1826814290):

IMO, this isn't valid.

busybox supports these only if it was built with bash-compatible extensions (ASH_BASH_COMPAT) enabled.

If busybox were built with CONFIG_ASH_BASH_COMPAT=y, then these options will be supported/included.
If busybox were built with CONFIG_ASH_BASH_COMPAT=n, then they will not.

#### <img src="https://avatars.githubusercontent.com/u/68940237?v=4" width="50">[slycordinator](https://github.com/slycordinator) commented at [2023-12-06 07:27](https://github.com/koalaman/shellcheck/issues/2865#issuecomment-1842243067):

If you have a look at:
https://git.busybox.net/busybox/tree/shell/ash.c#n193

From lines 210-256, you will see that all of these options are enabled when ENABLE_ASH_BASH_COMPAT and IF_ASH_BASH_COMPAT are set to true.

The options seem to be enabled by default when building busybox. But then a user could build it themselves with those disabled or run a system that disables them.

Perhaps this should be added to a "busybox bash" dialect or something for shellcheck?
