# [\#219 Issue](https://github.com/koalaman/shellcheck/issues/219) `open`: SC2123 seems to do pattern matching

#### <img src="https://avatars.githubusercontent.com/u/1823914?v=4" width="50">[arth1](https://github.com/arth1) opened issue at [2014-08-20T16:13:26Z](https://github.com/koalaman/shellcheck/issues/219):

consider this:

```
case ":${PATH:-}:" in
  *:/usr/libexec:*)
    ;;
  *)
    PATH="/usr/libexec${PATH:+:$PATH}"
    ^-- SC2123: PATH is the shell search path. Use another name.
    ;;
esac
```

However, the same works fine with /usr/local/bin or $HOME/bin, but not with /usr/libexec or /usr/lib64/ccache.  I suspect there is pattern matching going on, which seems like the Wrong Thing to do.

Is it even worth it to warn about setting the PATH?  One might as well warn about setting DISPLAY, MAIL, TERM, LANG and a bunch of other variables that I would presume that a user would be familiar with.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2014-09-04T15:06:26Z](https://github.com/koalaman/shellcheck/issues/219#issuecomment-54491071):

It does pattern match, but this is actually the first false positive I've seen. 

The philosophy here isn't to avoid overriding system variables by the way. Scripts often deal with files and paths, and PATH is the obvious variable name in which to store one. I've seen this happen a couple of times in real life, but I've never seen anyone having problems due to calling their variables DISPLAY.


#### <img src="https://avatars.githubusercontent.com/u/447615?v=4" width="50">[rwe](https://github.com/rwe) commented at [2022-02-16T19:26:50Z](https://github.com/koalaman/shellcheck/issues/219#issuecomment-1042078837):

This issue could perhaps be renamed more clearly, since the problem is mainly that SC2123 doesn't recognize that the `"/some/path${PATH:+":${PATH}"}"` or `"${PATH:+"${PATH}:"}/some/path"` idioms include the separator.

(_"If `PATH` is non-empty, then a colon and its value, otherwise nothing"_). That idiom is useful and pretty common, especially in early setup scripts where `PATH` may not actually be set up: for example Perl's CPAN tooling uses it in the generated shell profile additions.

Suggested name: "SC2123 does not recognize separator in conditional expansion idiom" or similar?
