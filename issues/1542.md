#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-04-11T20:55:36Z](https://github.com/koalaman/shellcheck/issues/1542#issuecomment-482310454):

I'm glad you're finding it useful!

It's rare not to care which directory you're in after a `cd`. You've clearly thought it through though, so it's fine. When you know something ShellCheck doesn't, you should feel free to ignore the issue with [a directive](https://github.com/koalaman/shellcheck/wiki/Directive) or similar.

In this case you can also rewrite it to show your intent to ShellCheck and other humans using `cd ... || true`, the same way you otherwise ignore failures in certain commands in exit status and `set -e`:

```
if git rev-parse
then
  # Try to go to the top-level git directory, but don't care
  # if it fails because then we'll fall back on the current directory
  cd "$(git rev-parse --show-toplevel)" || true
fi
```

The reason why you only occasionally get warnings for `echo` flags is probably the shebang. You don't get warnings in a Bash script, because Bash always supports flags for echo:

```
#!/bin/bash
echo -e -n "hello world"  # No warning, supported in bash
echo ...
```

However, you do get them in `sh` scripts:

```
#!/bin/sh
echo -e -n "hello world"  # Warning, not always supported by sh
echo ...
```

The bash script would run the same on all platforms, whereas the `sh` script may behave differently. Here's the output of the `sh` script on Debian Linux whose `sh` doesn't support `-e`:

```
debian$ ./myscript
-e -n hello world
...
```

and here's the output on macOS whose `sh` is actually Bash:

```
hello world...
```

Hope that answers your questions.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-04-12T18:11:36Z](https://github.com/koalaman/shellcheck/issues/1542#issuecomment-482671192):

I don't have any particularly strong opinions on either, since they both have their place. 

I would lean towards `[[` when writing Bash/Ksh, if only because it additionally supports glob, extglob and regex matching. The downside is that it may [execute arbitrary code](https://www.vidarholen.net/contents/blog/?p=716) when comparing integers. 

I don't consider less strict quoting or the ability to do `&&/||` with canonical precedence a particular benefit myself, but I know other people appreciate that.

As for `echo`/`printf`, I would definitely use `printf` if I needed specific format options like padding, or to ensure that a string is exactly the way I want it, especially across shells.

Otherwise, `echo` is quick and simple, and while no one ever localizes their messages, `echo $"..."` is better than `printf $"..." ..` since translators can change the parameter order to fit the language.

So I would say tl;dr: Be aware of the differences and use whatever makes sense in that context
