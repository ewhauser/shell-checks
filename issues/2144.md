# [\#2144 Issue](https://github.com/koalaman/shellcheck/issues/2144) `open`: Feature: Warn about masked return value in `true "$(false)"`

#### <img src="https://avatars.githubusercontent.com/u/640328?u=6d280aad048eae5736131adab01f53c2e8dc9f44&v=4" width="50">[bukzor](https://github.com/bukzor) opened issue at [2021-02-12 00:17](https://github.com/koalaman/shellcheck/issues/2144):

#### For new checks and feature suggestions
- [x] https://www.shellcheck.net/ (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related


#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
set -o errexit

echo "$(echo FAILURE; exit 1)"

echo "errexit means there's been no failures before this line"
```

#### Here's what shellcheck currently says:

No issues detected!

#### Here's what I wanted or expected to see:

There should be a warning similar to [SC2155](https://github.com/koalaman/shellcheck/wiki/SC2155), and for the same reason -- the exit code is gobbled.


#### <img src="https://avatars.githubusercontent.com/u/640328?u=6d280aad048eae5736131adab01f53c2e8dc9f44&v=4" width="50">[bukzor](https://github.com/bukzor) commented at [2021-02-12 00:20](https://github.com/koalaman/shellcheck/issues/2144#issuecomment-777883972):

If implemented, this check should warn about masked rreturn value for this example too:

```sh
#!/bin/bash
set -e
X="$(echo "$(exit 1)")"
```

This is just a special case of the same problem -- the `echo` is successful due to ignoring the return code of its command-substituion argument.

```sh
#!/bin/sh
set -e
(
  false 
  true

```
 
This will not trigger the error message, unlike what the author intended.
