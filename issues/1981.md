```sh

#!/bin/sh
: "${foo:=oneword}"
echo $foo

```

#### Here's what shellcheck currently says:

nothing

#### Here's what I wanted or expected to see:

*tl;dr:* For default assignments, shellcheck should not assume that a variable contains no spaces.

*long:* For unsplittable strings, shellcheck does not issue a warning. The above snippet uses a [default assignment](https://pubs.opengroup.org/onlinepubs/9699919799/utilities/V3_chap02.html#tag_18_06_02), meaning if the variable `$foo` gets passed in through the environment, it is not overwritten with `oneword`. Then, splitting is again a problem.

Both default assignments (`:=` and `=`) are affected.

```
~ % ./example
oneword
~ % foo="two     words" ./example
two words
~ % # notice that the multiple spaces have been eaten by the word splitting
