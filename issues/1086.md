# [\#1086 Issue](https://github.com/koalaman/shellcheck/issues/1086) `closed`: SC2010 should not warn against a negated matcher (grep -v)

#### <img src="https://avatars.githubusercontent.com/u/1844664?v=4" width="50">[ecbrodie](https://github.com/ecbrodie) opened issue at [2018-01-05T05:11:57Z](https://github.com/koalaman/shellcheck/issues/1086):

#### For bugs
- Rule Id: SC2010
- My shellcheck version: 0.4.7
- [ ] I tried on shellcheck.net and verified that this is still a problem on the latest commit
- [ ] It's not reproducible on shellcheck.net, but I think that's because it's an OS, configuration or encoding issue

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash
ls | grep -v IGNORE_ME | while read -r line; do
  cd "$line" || exit
  bundle
  cd ..
done
```

#### Here's what shellcheck currently says:

```
^-- SC2010: Don't use ls | grep. Use a glob or a for loop with a condition to allow non-alphanumeric filenames.
```

#### Here's what I wanted or expected to see:

This warning should not appear. [According to the SC2010 wiki page](https://github.com/koalaman/shellcheck/wiki/SC2010), the exception to this rule is a negated matcher via `grep -v`, which is indeed what my code does.

#### <img src="https://avatars.githubusercontent.com/u/2199511?v=4" width="50">[Nightfirecat](https://github.com/Nightfirecat) commented at [2018-01-05T08:51:08Z](https://github.com/koalaman/shellcheck/issues/1086#issuecomment-355502826):

Assuming extglob is set, the loop condition could be more simply written as `for line in !(*IGNORE_ME*); do`. May help in your case but given that dependency (and given that globs don't match grep's full regex capabilities), I don't know that this is a suitable replacement in the general case.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-01-05T21:21:30Z](https://github.com/koalaman/shellcheck/issues/1086#issuecomment-355668590):

The wiki page was wrong, I've fixed it. Thanks for bringing it up!

In this case (bash), an extglob like @Nightfirecat suggests would be the way to go.
