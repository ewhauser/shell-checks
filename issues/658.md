# [\#658 Issue](https://github.com/koalaman/shellcheck/issues/658) `closed`: false positive of SC1035 for `!!`

#### <img src="https://avatars.githubusercontent.com/u/927609?v=4" width="50">[despairblue](https://github.com/despairblue) opened issue at [2016-04-22T08:57:40Z](https://github.com/koalaman/shellcheck/issues/658):

``` shell
while [ $? -ne 0 ]; do !!; echo retry; sleep 5; done
```

This will retry the last command every 5 seconds until it succeeds. Adding a whitespace between the two exclamation marks does alter the meaning of this, to quote the rationale behind the rule:

> Bourne shells are very whitespace sensitive. Adding or removing spaces can drastically alter the meaning of a script. In these cases, ShellCheck has noticed that you're missing a space at the position indicated.

Thanks for making this, btw :+1: 

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-04-22T12:03:42Z](https://github.com/koalaman/shellcheck/issues/658#issuecomment-213396512):

> ``` shell
> while [ $? -ne 0 ]; do !!; echo retry; sleep 5; done
> ```
> 
> This will retry the last command every 5 seconds until it succeeds.

In an interactive shell, that will test the previous command, and if it failed, will repeat it indefinitely without regard for whether it succeeds or fails, until the sleep gets interrupted.

That is not a scripting fragment as such, because history expansion is only done in interactive shells by default, and even if you enable it inside a script, it would read from the history created by an interactive shell outside the script.

Comparing $? against zero should never be necessary, and generally indicates poorly thought out program flow.

Instead just do this:

```
repeat() { while ! "$@" ; do sleep 5 ; done ; }
repeat try_some_command_here "and don't" forget the\ args
```

Or interactively
    repeat !!


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-04-23T22:38:58Z](https://github.com/koalaman/shellcheck/issues/658#issuecomment-213849585):

You're right, of course. ShellCheck doesn't understand history expansion. It's not really a feature used in scripts where it's disabled by default, but I can definitely see this coming up when checking your `.bashrc`.

Supporting history expansion would probably squash more true than false positives though: `!true` is valid history expansion but probably not what the user wanted. Please ignore the warning.
