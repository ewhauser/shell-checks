#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-05-22T23:42:34Z](https://github.com/koalaman/shellcheck/issues/675#issuecomment-220863900):

You can not put redirections in variables, so neither works. 

This produces no errors because shellcheck doesn't warn when using a single unquoted variable as the command name: 

```
cmd="iptables -L INPUT"
if ${cmd}
then
echo "OK"
fi
```

The better alternative, when using bash or ksh, is to use an array:

```
cmd=(iptables -L INPUT)
if "${cmd[@]}"
then
echo "OK"
fi
```

The problem is not that shellcheck is wrong, but that its diagnostic message
could be more helpful.

As others have noted, that style of code is doubly discouraged:
1. Redirections are done before variable expansions, which means that what
   look like redirections inside variables won't act as redirections; instead
   you'll just get a diagnostic from iptables that it does not understand
   '2>&1' and/or that 'INPUT>/dev/null' is not a valid table name.
2. The remainder relies on word-splitting, which might not go how you expect
   if IFS has been reassigned.

This is pretty much the archetypal case where shell functions would be
recommended:

```
nullout() { "$@" > /dev/null 2>&1 ; }

iptlist() { iptables -L INPUT "$@" ; }

if nullout iptlist
then
  echo OK
fi
```

(Though I'm not really clear why you would want to list a table, only to
discard it; if you simply want to test whether the xtables kernel module is
installed, you could just inspect the output of lsmod, or /proc/net.)
