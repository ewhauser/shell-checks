# [\#213 Issue](https://github.com/koalaman/shellcheck/issues/213) `open`: Add warning for variables set / changed in a function which are not declared as local.

#### <img src="https://avatars.githubusercontent.com/u/1261222?v=4" width="50">[dougborg](https://github.com/dougborg) opened issue at [2014-08-14T22:50:37Z](https://github.com/koalaman/shellcheck/issues/213):

This is similar to issue #115, but my take is slightly different. I feel that any variable modified inside a function should trigger a ShellCheck warning if it is not declared `local`. This includes explicit modifications using `=`, `let`, and `for` loops.

Here is an example that illustrates why I think this is a problem:

``` bash
#!/bin/bash

MY_SERVERS='server1 server2 server3'

trust_my_server() {
  my_server="${1}"
  shift
  servers="${@}"

  for server in ${servers}; do
    ssh ${server} 'trust ${my_server}'
  done
}

do_stuff() {
  for server in $MY_SERVERS; do
    trust_my_server ${server} ${MY_SERVERS//${server}/}
    do_something_else ${server} # This will not do what you expect; ${server} has been changed on you!
  done
} 

do_stuff
```

In the example above, the `do_something_else` function does not get called with the element of `$MY_SERVERS` you expect it to from the main `for` loop. As side effect of running `trust_my_server` (because server is not declared `local`), `server` will get set to the last element of what was passed to `trust_my_server` by its `for` loop by the time `do_something_else` is called, leading to much woe.

#### <img src="https://avatars.githubusercontent.com/u/1261222?v=4" width="50">[dougborg](https://github.com/dougborg) commented at [2014-08-16T23:35:39Z](https://github.com/koalaman/shellcheck/issues/213#issuecomment-52408973):

It may be good to treat `ALL_CAPS` variables differently. I could be wrong, but my understanding is that the general convention in shell programming is truly global vars should be `ALL_CAPS`. I also don't think of this as a serious error, just a warning that should be brought to the programmer's attention. It is indeed likely in some cases, the intention is to change global variables (or at least variables with a greater scope than the current function) as a result of a function's execution. I think this is a bad practice though and should be avoided wherever possible - not that I haven't done it myself.
