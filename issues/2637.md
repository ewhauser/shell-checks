#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bash

generic()
{
  CURRENT_INSTALLATION_KEYNAME=baseVariable
  
  baseVariable_pipinstallations=("python-dotenv" "two_or_more")  # With this we would have an error
  baseVariable_pythoncommands=("one_no_error")   # This does not fail because the expanded array has a single element
  
  local -r pipinstallations="${CURRENT_INSTALLATION_KEYNAME}_pipinstallations[@]"
  local -r pythoncommands="${CURRENT_INSTALLATION_KEYNAME}_pythoncommands[@]"
  # This line suggests a useless echo, but if we remove the echo the code will fail with script.sh: line 1434: [: too many 
  # arguments or will fail with script.sh: line 1434: [: pythondotenv: binary operator expected
  if [ -z "$(echo "${!pipinstallations}")" ] && [ -z "$(echo "${!pythoncommands}")" ]; then  
    return
  fi
}
```

#### Here's what shellcheck currently says:
Useless echo? Instead of 'cmd $(echo foo)', just use 'cmd foo'


#### Here's what I wanted or expected to see:
Nothing, it is an exception of this rule. When using `echo` around indirect expanded variables that are an array with more than one element, shellcheck should not show this error because when you actually remove the error the code stops working as expected. The suggested code will make my new code look like this:
```sh
  if [ -z "${!pipinstallations}" ] && [ -z "${!pythoncommands}" ]; then
    return
  fi
```

But it will fail with `script.sh: line 1434: [: too many arguments` or will fail with `script.sh: line 1434: [: pythondotenv: binary operator expected` if the array `pipinstallations` or the array `pythoncommands` contains more than one element, or it has one element with spaces in it. 

I hope that this is actually a request that makes sense, since from my point of view this shellcheck is wrong in the special circumstances described. 

If this issue is accepted, I am willing to rise a PR to apply the changes.
