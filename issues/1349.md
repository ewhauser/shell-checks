# [\#1349 Issue](https://github.com/koalaman/shellcheck/issues/1349) `open`: shellcheck parses error if parentheses in let expression

#### <img src="https://avatars.githubusercontent.com/u/14345853?v=4" width="50">[Kontar-pjc](https://github.com/Kontar-pjc) opened issue at [2018-09-25T12:39:36Z](https://github.com/koalaman/shellcheck/issues/1349):

#### For bugs
- Rule Id: SC1036
- My shellcheck version: 0.5.0 


#### Here's a snippet or screenshot that shows the problem:

```sh

#! /bin/bash
file_num=10
all_files=20
let per=(file_num*100)/all_files
echo $per

```

#### Here's what shellcheck currently says:
let per=(file_num*100)/all_files
^-- SC2219: Instead of 'let expr', prefer (( expr )) .
    ^-- SC2034: per appears unused. Verify use (or export if used externally).
        ^-- SC1036: '(' is invalid here. Did you forget to escape it?
        ^-- SC1088: Parsing stopped here. Invalid use of parentheses?


#### Here's what I wanted or expected to see:
Everything runs ok. But it parses failed when parentheses in let expression.
I'm using shellcheck in my Gitlab CI, and I can't modify the code writen by the others.
Be appreciate if you can see this.


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-12-02T20:21:32Z](https://github.com/koalaman/shellcheck/issues/1349#issuecomment-443538637):

Huh, this is indeed another of Bash's special cases where assignment-like words containing parentheses are parsed after a whitelist of commands. 

A workaround for now would be to use the POSIX standard `per=$(( (file_num*100)/all_files ))` or at least quote it with `let per="(file_num*100)/all_files"`, but I guess that's not much help if you can't modify it.
