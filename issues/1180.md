# [\#1180 Issue](https://github.com/koalaman/shellcheck/issues/1180) `closed`: SC2093 false positive (when exec used for redirection)

#### <img src="https://avatars.githubusercontent.com/u/1590301?v=4" width="50">[sqweek](https://github.com/sqweek) opened issue at [2018-04-15T19:01:51Z](https://github.com/koalaman/shellcheck/issues/1180):

This looks similar to #697 except I can still reproduce it online.

- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash
exec {origout}>&1- >"tmp.log" 2>&1

echo $origout

pwd
```

#### Here's what shellcheck currently says:

```
Line 2:
exec {origout}>&1- >"tmp.log" 2>&1
^-- SC2093: Remove "exec " if script should continue after this command.
```

Interestingly if you comment out the `pwd` or replace it with an `echo` (even `echo $(pwd)`) then the SC2093 warning disappears. Somehow it depends on the commands coming after the exec...

#### Here's what I wanted or expected to see:

No output.

#### <img src="https://avatars.githubusercontent.com/u/1590301?v=4" width="50">[sqweek](https://github.com/sqweek) commented at [2018-04-18T08:54:16Z](https://github.com/koalaman/shellcheck/issues/1180#issuecomment-382314815):

The contents of `origout` don't matter. The `{variable}>&1-` redirection *assigns* the allocated FD to the variable, no expansion takes place. From the man page:

>       Each redirection that may be preceded by a file descriptor number may instead be
>       preceded by a word of the form {varname}.  In this case,  for  each  redirection
>       operator  except  >&- and <&-, the shell will allocate a file descriptor greater
>       than or equal to 10 and assign it to varname.  If >&-  or  <&-  is  preceded  by
>       {varname}, the value of varname defines the file descriptor to close.

Note that the `>&-` operator mentioned here is different to the `>&DIGIT-` operator I'm using so the exception doesn't apply (see also "Moving File Descriptors" in the man page).

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-05-13T01:40:11Z](https://github.com/koalaman/shellcheck/issues/1180#issuecomment-388594947):

Nice catch! 

The parser didn't recognize the FD move operations and stopped parsing it after `{origout}>&1`. The `-` then became a regular shell word that was interpreted as a command name, which is why ShellCheck thought it would stop executing.

This has now been fixed, thanks!
