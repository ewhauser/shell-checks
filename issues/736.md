# [\#736 Issue](https://github.com/koalaman/shellcheck/issues/736) `closed`: "command >& filename"-redirect not recognized as error for POSIX shells

#### <img src="https://avatars.githubusercontent.com/u/7539154?v=4" width="50">[johnLate](https://github.com/johnLate) opened issue at [2016-09-13T20:36:44Z](https://github.com/koalaman/shellcheck/issues/736):

```
$ cat bad.sh 
#!/bin/sh
echo foo >& bar

$ shellcheck bad.sh; echo $?
0

$ dash bad.sh 
bad.sh: 3: bad.sh: Syntax error: Bad fd number
```

If I understand [POSIX](http://pubs.opengroup.org/onlinepubs/9699919799/utilities/V3_chap02.html#tag_18_07_06) correctly, this syntax is only for duplicating output file descriptors — unlike bash, which uses `>&` and `&>` for redirection of stdout and stderr at the same time.

#### <img src="https://avatars.githubusercontent.com/u/5104323?v=4" width="50">[hwdsl2](https://github.com/hwdsl2) commented at [2016-10-10T21:58:22Z](https://github.com/koalaman/shellcheck/issues/736#issuecomment-252757243):

Hello! The fix for this issue has unintended side effects. For example:

```
exiterr()  { echo "Error: ${1}" >&2; exit 1; }
                                ^-- SC2039: In POSIX sh, >& is undefined.
```

I think the above is valid POSIX for redirection to stderr. The warning is incorrect.
