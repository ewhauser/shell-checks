# [\#2213 Issue](https://github.com/koalaman/shellcheck/issues/2213) `closed`: Proposal: Warn about using commas with printf

#### <img src="https://avatars.githubusercontent.com/u/7087596?u=b288cd251914bbbd4595b7deddc22f1478939393&v=4" width="50">[kmmiles](https://github.com/kmmiles) opened issue at [2021-04-20 19:36](https://github.com/koalaman/shellcheck/issues/2213):

I make this mistake constantly. Is it possible for `shellcheck` to warn about it?

```sh
#!/bin/bash
printf "%s\n", "$PATH"
```




#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-05-02 03:43](https://github.com/koalaman/shellcheck/issues/2213#issuecomment-830734538):

I really like this idea. I ran some experiments though, and it doesn't appear to be a systemic issue. The majority of shell words with a trailing, unquoted comma are things like `cut -d,`  `tr , ' '`, `echo Foo, bar, etc`. When only looking at quoted text + unquoted literal `,` the accuracy better, but true positives were mostly things like `foo=("bar", "baz")` already covered by SC2054, plus several false positives like `echo "$foo", "$bar", etc`. I was not able to find any differentiator that resulted in more true than false positives.

Thanks for the suggestion!
