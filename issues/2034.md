#### Here's a snippet or screenshot that shows the problem:

#!/bin/ksh
input_row="one|two|Three,Four,FIVE"

#note the missing second colon for the "upper" param to tr
third_field=$(echo ${input_row} | awk -F'|' '{print $3}' | tr '[:upper]' '[:lower:]')

echo "$third_field"

#### Here's what shellcheck currently says:
Nothing, there is no finding on this

#### Here's what I wanted or expected to see:
No idea how complicated this is, and I realize that technically almost anything is valid there, but would be really great if shellcheck could detect this and give a warning.  I know shellcheck does already detect some incorrect tr usage.

#### <img src="https://avatars.githubusercontent.com/u/66952049?v=4" width="50">[scottdcoulter](https://github.com/scottdcoulter) commented at [2020-08-07 12:22](https://github.com/koalaman/shellcheck/issues/2034#issuecomment-670489958):

Sorry if I was unclear, it's not an awk error, it's a tr error.  The awk was in the original line where this issue occurred, but I could have removed it for this example.  My request is the same if the source line was just:

`  $(echo ${input_row} | tr '[:upper]' '[:lower:]')`

I thought my requested addition was pretty similar to SC2018 through SC2021, which give commentary on the values being sent to tr.  It is true that tr does (sometimes, depending on which : is missed), give an error on this ("tr: misaligned [:upper:] and/or [:lower:] construct"; only if it's the first param that's missing a colon, in the second param tr goes ahead and replaces characters, almost certainly not how one intended), but it would have saved me and three co-workers from being online until midnight last night if shellcheck had spotted it first.

Similarly, if any of the square brackets in that statement are missed, tr goes ahead with the unintended replacement pattern and does not give an error.  SC2021 knows that the brackets are wrong for some usage patterns in tr, and SC2018/SC2019 know about :upper: and :lower: so I feel like this is right in the space that shellcheck is already correctly commenting on.

Shellcheck is a pretty amazing tool and I've come to rely on it.  No complaints, and it's fine if this enhancement request isn't a good fit.
