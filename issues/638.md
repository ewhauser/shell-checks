# [\#638 Issue](https://github.com/koalaman/shellcheck/issues/638) `open`: Feature request: traps don't apply to interpolated strings

#### <img src="https://avatars.githubusercontent.com/u/27913?v=4" width="50">[Warbo](https://github.com/Warbo) opened issue at [2016-03-30T14:55:44Z](https://github.com/koalaman/shellcheck/issues/638):

I recently hit a tricky scripting issue which ShellCheck didn't spot, but I would imagine would be relatively easy to add.

Specifically, I was working on a script with the following trap, inherited by sub-shells via `set -E`:

```
set -E
trap '[ "$?" -ne 77 ] || exit 77' ERR
```

This code is taken from http://unix.stackexchange.com/a/48550/63735 and is meant to exit the whole script, if the magic exit code 77 is returned from any sub-shell. I'm using this in a test script, where any test failure should abort the whole script:

```
function fail {
    echo -e "FAIL: $*" 1>&2
    exit 77
}
```

This works from the top-level:

```
false || fail "FOO"
# Never reach here
```

It works from sub-shells, thanks to the trap:

```
DATA=$(false || fail "BAR")
# Never reach here
```

However, traps _aren't_ triggered by commands which are spliced/interpolated inside strings:

```
echo "$(false || fail "BAZ")" | grep "QUUX"
# Shouldn't reach here, but do

echo "`false || fail "BAZ"`" | grep "QUUX"
# Shouldn't reach here, but do
```

It would be nice if ShellCheck could warn about such splices when it sees that the script specifies an `ERR` trap, e.g. `Interpolated commands like 'echo "$(foo)"' are not handled by traps. Use an intermediate variable like 'TEMP=$(foo); echo "$TEMP"`

In this case using an intermediate variable would look like:

```
TEMP=$(false || fail "BAZ")
# Never reach here
echo "$TEMP" | grep "QUUX"

TEMP=`false || fail "BAZ"`
# Never reach here
echo "$TEMP" | grep "QUUX"
```
