# [\#461 Issue](https://github.com/koalaman/shellcheck/issues/461) `closed`: SC2028, SC2039 and dash as /bin/sh

#### <img src="https://avatars.githubusercontent.com/u/6459309?v=4" width="50">[Artoria2e5](https://github.com/Artoria2e5) opened issue at [2015-09-17T18:52:13Z](https://github.com/koalaman/shellcheck/issues/461):

`dash` is a popular and the default option for a minimal `sh` for Debian systems, including Ubuntu. However, its echo, by default, interprets all backslash options (like in [XCU](http://pubs.opengroup.org/onlinepubs/7908799/xcu/echo.html)) and it can't be overridden by flags, e.g. by GNU echo's `-E`. And unlike XCU, it interprets the `-n` option.

Since Debian is quite popular, I think the description of `SC2028` should be ‘In POSIX sh, echo with operands containing backslashes causes undefined results’, and `SC2039` should be ‘with flags causes..’.

POSIX.1:2008 [echo](http://pubs.opengroup.org/onlinepubs/9699919799/utilities/echo.html): 

> **string**: A string to be written to standard output. If the first operand is -n, or if any of the operands contain a <backslash> character, the results are implementation-defined.
> **_APPLICATION USAGE**_ It is not possible to use echo portably across all POSIX systems unless both -n (as the first argument) and escape sequences are omitted. [...] New applications are encouraged to use printf instead of echo.

SUSv2 XCU [echo](http://pubs.opengroup.org/onlinepubs/7908799/xcu/echo.html):

#### <img src="https://avatars.githubusercontent.com/u/6459309?v=4" width="50">[Artoria2e5](https://github.com/Artoria2e5) commented at [2015-09-17T18:55:47Z](https://github.com/koalaman/shellcheck/issues/461#issuecomment-141187771):

The wiki items should be changed accordingly too, but I think I can do that myself. And maybe we should include this GNU `echo` function substitute in the wiki, as a substitute to XPG/BSD ones mentioned in POSIX:

``` sh
# Variant of http://www.etalabs.net/sh_tricks.html, mimics GNU & bash echo.
ntax.
echo() (
  fmt=%s end='\n' IFS=' '
  while [ $# -gt 1 ] ; do
    case "$1" in ([!-]*|-*[!neE]*) break;; esac # not a flag
    case "$1" in (*n*) end='';; esac # no newline
    case "$1" in (*e*) fmt=%b;; esac # interpret backslash escapes
    case "$1" in (*E*) fmt=%s;; esac # don't interpret backslashes
    shift
  done
  printf "$fmt$end" "$*"
)
```

I have yet to find a use for `env` that I actually trust, since `~/bin` or even `~attacker/rootkit/bin` might be early in `PATH`.

The usual suggestion is to use `printf` instead, which is unambiguous, or if it's too much hassle to edit existing code, to define a shell function as suggested above, or perhaps a simpler one like `echo() { printf '%b\n' "$*"; }` (XPG) or `echo() { if [[ $1 = -n ]] ; then shift ; printf '%s' "$*" ; else printf '%s\n' "$*" ; fi ; }` (BSD).
