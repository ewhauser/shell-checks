# [\#1309 Issue](https://github.com/koalaman/shellcheck/issues/1309) `open`: inconsistent flagging of SC2178/SC2128 based on `local` usage

#### <img src="https://avatars.githubusercontent.com/u/176950?v=4" width="50">[vapier](https://github.com/vapier) opened issue at [2018-08-01T21:22:57Z](https://github.com/koalaman/shellcheck/issues/1309):

#### For bugs
- Rule Id: SC2178/SC2128
- My shellcheck version: online

#### Here's a snippet or screenshot that shows the problem:

if we start with this code:
```sh
#!/bin/bash
get_vars() {
  local vars=( 1 2 3 )
  echo "${vars[@]}"
}
main() {
  local vars="$(get_vars)"
  echo "${vars}"
}
```

we get:
```sh
$ shellcheck myscript
 
Line 7:
  local vars="$(get_vars)"
        ^-- SC2155: Declare and assign separately to avoid masking return values.
        ^-- SC2178: Variable was used as an array but is now assigned a string.
 
Line 8:
  echo "${vars}"
        ^-- SC2128: Expanding an array without an index only gives the first element.
```

but if we fix SC2155, all the other warnings disappear:
```sh
#!/bin/bash
get_vars() {
  local vars=( 1 2 3 )
  echo "${vars[@]}"
}
main() {
  local vars
  vars="$(get_vars)"
  echo "${vars}"
}
```

```sh
$ shellcheck myscript
No issues detected!
```

if we remove the `local` decl, they come back:
```sh
#!/bin/bash
get_vars() {
  local vars=( 1 2 3 )
  echo "${vars[@]}"
}
main() {
  vars="$(get_vars)"
  echo "${vars}"
}
```
```sh
$ shellcheck myscript
 
Line 7:
  vars="$(get_vars)"
  ^-- SC2178: Variable was used as an array but is now assigned a string.
 
Line 8:
  echo "${vars}"
        ^-- SC2128: Expanding an array without an index only gives the first element.
```

this is weird because the scope of var from `main` to `get_vars` hasn't changed ... the `local var` in `main` still cascades down into `get_var`.

the SC2178/SC2128 should always be flagged even if the parent func declared it `local` for its own usage.

#### <img src="https://avatars.githubusercontent.com/u/4522509?v=4" width="50">[kolyshkin](https://github.com/kolyshkin) commented at [2022-02-09T01:14:52Z](https://github.com/koalaman/shellcheck/issues/1309#issuecomment-1033228042):

Similarly, there are cases when SC2128 gets misdetected because a variable comes from a caller's context. For example:

```bash
#!/bin/bash

x() {
	# Here we have args that came from y, i.e. args is an array.
	# This is clearly SC2128 ("Expanding an array without an index ..."),
	# but it is not detected.
	echo x: "$args"

	# Redeclare args as a string, shadowing the parent one.
	local args="three four"
	echo x: "$args"
}

y() {
	# Here we have args that came from global context, i.e. args is an array.
        # This is clearly SC2128 ("Expanding an array without an index ..."),
	# but it is not detected.
	echo y: "$args"

	# Redeclare args.
	local args=(one two)
	echo y: "${args[@]}"
	x
	echo y: "${args[@]}"
}

args=(global args)
y
```
