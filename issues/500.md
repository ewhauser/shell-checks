#### <img src="https://avatars.githubusercontent.com/u/7633626?v=4" width="50">[aw-was-here](https://github.com/aw-was-here) opened issue at [2015-10-10T17:03:55Z](https://github.com/koalaman/shellcheck/issues/500):

For Apache Yetus (http://yetus.apache.org), we use bash sourcing as a plug-in mechanism.  This often means doing things that are the equivalent of:

for x in *; do
  . $x
done

There are a few things that I'd like to do (or, if it already does this and I just missed it, some hand holding!):

a) Specify a regex or wildcard for shellcheck source.  e.g.:

``` bash
# shellcheck source=somedir/*
```

b) Specify a shellcheck source directive even when there isn't one in the source.  For example, the top of a plug-in might be able to:

``` bash
# shellcheck source=foo
function bar {}
```

So that function bar has all the definitions from foo

c) shellcheck source can take a relative path to the location of the file being processed.  e.g., 

``` bash
# shellcheck source=foo.sh
```

if I run shellcheck -x some/long/path/bar.sh , file.sh would expect foo.sh to be in some/long/path.

#### <img src="https://avatars.githubusercontent.com/u/730244?v=4" width="50">[dierobotsdie](https://github.com/dierobotsdie) commented at [2015-10-17T19:08:15Z](https://github.com/koalaman/shellcheck/issues/500#issuecomment-148944713):

These are/maybe incomplete examples, but it will hopefully give you an idea.

b) 

https://github.com/apache/yetus/blob/master/dev-support/personality/tajo.sh  defines a few constants that are specific to the Apache Tajo project.  Those constants are used by https://github.com/apache/yetus/blob/master/dev-support/core.d/01-common.sh and other files that are imported during runtime. My hunch is that it would be significantly preferable to have source lines rather than disable lines when shellcheck analyzes the tajo.sh file.

c) 

Apache Yetus has plug-in support for shellcheck (specifically via https://github.com/apache/yetus/blob/master/dev-support/test-patch.d/shellcheck.sh).  When we are running shellcheck, we are running it on a build tree that has been checked out/cloned from it's repository and not in an installed state.  This means that absolute paths in shellcheck source= aren't going to work because we have no idea where that location will actually be.  Additionally, we run shellcheck from the root of the repository.  So shellcheck source=./foo won't work if the file we're analyzing is actually in a subdirectory since ./ will be the root.  But if shellcheck is aware that ./ really means directory of the file that is being analyzed, then relative paths will work great since the source line will pull in the appropriate content.

Hopefully this makes sense without digging too deep into Yetus. :)
