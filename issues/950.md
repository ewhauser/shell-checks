# [\#950 Issue](https://github.com/koalaman/shellcheck/issues/950) `closed`: Incorrect parsing of default value in variable expansion

#### <img src="https://avatars.githubusercontent.com/u/15023129?v=4" width="50">[martin-schwenke](https://github.com/martin-schwenke) opened issue at [2017-07-12T11:24:37Z](https://github.com/koalaman/shellcheck/issues/950):

#### For bugs
- Results in SC2031, but the problem isn't really with the rule
- 0.4.6
- I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/sh
b=$(d=$(dirname "$0") ; cd -P "$d" || exit ; dirname "$PWD")
bar="${FOOBAR:-${b}/foobar.d}"
echo "$bar"

```

#### Here's what shellcheck currently says:

In foo.sh line 2:
b=$(d=$(dirname "$0") ; cd -P "$d" || exit ; dirname "$PWD")
    ^-- SC2030: Modification of d is local (to subshell caused by $(..) expansion).


In foo.sh line 3:
bar="${FOOBAR:-${b}/foobar.d}"
       ^-- SC2031: d was modified in a subshell. That change might be lost.

#### Here's what I wanted or expected to see:

Nothing!

"d" is erroneously parsed as a variable name when it is part of a default value string.  This also happens if 'd' is preceded by '-' or '+' but not '_'.  So it looks like the default value is not treated at a string.  It looks like the tail end of it is treated as an identifier.

Thanks...

Note that, the substring indexing feature does not exist in POSIX shell.  It is an extension, in bash and others. I'm not sure how the parsing and shell dialects interact in ShellCheck but it seems a shame that a bug in that code is causing a false positive in POSIX shell.
