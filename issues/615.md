# [\#615 Issue](https://github.com/koalaman/shellcheck/issues/615) `closed`: Incorrectly warns of if ((....)) construct

#### <img src="https://avatars.githubusercontent.com/u/351123?v=4" width="50">[cvmiller](https://github.com/cvmiller) opened issue at [2016-02-25T02:41:23Z](https://github.com/koalaman/shellcheck/issues/615):

When running the web version of shellcheck, it reports:
94  if (( $RESTORE == 1 )); then
            ^––SC2004 $/${} is unnecessary on arithmetic variables.

When using double parens in a conditional, it is doing an arithmetic comparison, not an arithmetic operation.

Therefore it should not warn this code.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-02-25T04:31:23Z](https://github.com/koalaman/shellcheck/issues/615#issuecomment-188608966):

I'm not sure I understand. Why is this not better written as `if (( RESTORE == 1 )); then`?


#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-02-25T06:01:08Z](https://github.com/koalaman/shellcheck/issues/615#issuecomment-188628846):

Of course it should warn, because the $ sign is misleading.

In arithmetic context, comparisons ARE arithmetic operations; `a==b` is an
operation returning 0 or 1.

The arithmetic statement ((...)) behaves exactly the same regardless of
context; if by "conditional" you mean it is subordinate to "if", that is
irrelevant.

If RESTORE contains an arithmetic combination of bits, say, "2 | 1",
then using or not using $ changes the behaviour. The statement
(($RESTORE==1)) succeeds, while ((RESTORE==1)) fails.
