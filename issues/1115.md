# [\#1115 Issue](https://github.com/koalaman/shellcheck/issues/1115) `closed`: SC2120 $@ false positive

#### Labels: wontfix

#### <img src="https://avatars.githubusercontent.com/u/6315187?v=4" width="50">[masavini](https://github.com/masavini) opened issue at [2018-02-12T13:57:10Z](https://github.com/koalaman/shellcheck/issues/1115):

```sh
function __wiki_raw_link_remover () {
  sed \
    -e 's/\[\[[^|]\+|\([^]]\+\)/[[\1/g' \
    -e 's/\[\[//g' \
    -e 's/\]\]//g' \
    -- "${@}"
}

cat wiki-raw-file.txt | __wiki_raw_link_remover
```
running shellcheck on the above script results in:
```
In wiki_helpers.sh line 1:
__wiki_raw_link_remover() {
^-- SC2120: __wiki_raw_link_remover references arguments, but none are ever passed.
```
`${@}` as argument for sed, grep, awk (and friends) is great, as it allows to write wrapper functions that can process both input file(s) or standard input.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-02-25T22:06:23Z](https://github.com/koalaman/shellcheck/issues/1115#issuecomment-368349451):

You're right of course. Unfortunately I don't see any good way of detecting that this was intentional. Someone could equally well have written similar code expecting `"$@"` to be the script's arguments.
