#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-06-23T19:49:47Z](https://github.com/koalaman/shellcheck/issues/1606#issuecomment-504781520):

This is indeed a bug. Here's an easier way to reproduce the issue:

```
let x=(1)
```

Bash accepts this just fine, while shellcheck says:
```
In foo line 1:
let x=(1)
^----^ SC2148: Tips depend on target shell and yours is unknown. Add a shebang.
^----^ SC2219: Instead of 'let expr', prefer (( expr )) .
    ^-- SC2034: x appears unused. Verify use (or export if used externally).
      ^-- SC1036: '(' is invalid here. Did you forget to escape it?$ shellcheck foo

      ^-- SC1088: Parsing stopped
```

Bash appears to use its assignment parsing mode for `let`, just like it does for `eval`, `export` and `declare`, thereby allowing parentheses 

Additionally -- and surprisingly -- it allows a suffix after what would have been an array assignment, e.g. `let x=(1)/1`. This has the weird side effect that bash accepts `let x=(1)/1` but not `let x=1/(1)`.
