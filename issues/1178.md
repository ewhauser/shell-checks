# [\#1178 Issue](https://github.com/koalaman/shellcheck/issues/1178) `closed`: `readonly` flagged invalid when preceded by `local`

#### For bugs
- SC2034 
- v0.4.7 
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash

function foo() {
  local readonly bar=baz
  echo "$bar"
}

```

#### Here's what shellcheck currently says:
```
In foo.sh line 4:
  local readonly bar=baz
        ^-- SC2034: readonly appears unused. Verify it or export it.
```


#### Here's what I wanted or expected to see:
Nothing.

#### <img src="https://avatars.githubusercontent.com/u/2199511?v=4" width="50">[Nightfirecat](https://github.com/Nightfirecat) commented at [2018-04-12T20:16:28Z](https://github.com/koalaman/shellcheck/issues/1178#issuecomment-380931156):

`local readonly baz=bar` simply sets a local variable `baz` with value `"bar"`; the readonly word doesn't actually make the variable read-only.

That said, this warning isn't correct; no `readonly` variable is set as a result of this command.

#### <img src="https://avatars.githubusercontent.com/u/6675894?v=4" width="50">[charlescanato](https://github.com/charlescanato) commented at [2018-04-13T00:47:11Z](https://github.com/koalaman/shellcheck/issues/1178#issuecomment-380986855):

Just perfect, Jordan.

This is one of the Bash mysteries (or bugs?). I don't understand why `readonly` is accepted after `local` and yet, it is ignored. You can't even find a reference to this possibility inside Bash documentation (at least, in the one I have locally).

And that would be good: it'd make scope and usage of the variable beautifully clear.

AFAIK, the only correct way to have a *local* and *readonly* variable is by using `declare -r baz=bar`. Yep, not so descriptive, but functional.

`readonly` isn't "accepted but ignored" after `local`. You just created a local variable named `readonly` and shellcheck correctly flagged it as unused.
