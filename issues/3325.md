# [\#3325 Issue](https://github.com/koalaman/shellcheck/issues/3325) `open`: Disable SC1017 (error: Literal carriage return) for referenced .shellcheckrc files

#### <img src="https://avatars.githubusercontent.com/u/99302669?u=bd244efe00cefcb26fc08a6a632a330148d755dd&v=4" width="50">[gabldotink](https://github.com/gabldotink) opened issue at [2025-10-15 06:20](https://github.com/koalaman/shellcheck/issues/3325):

#### For bugs with existing features

- Rule Id (if any, e.g. SC1000): SC1017
- My shellcheck version (`shellcheck --version` or "online"):
- [x] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2086) (https://shellcheck.net/wiki/SC1017)
- [x] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit (`.shellcheckrc` is not available on shellcheck.net, but I did build from source with cabal to check with the latest commit)

#### Here's a snippet or screenshot that shows the problem:

In a directory:

`.shellcheckrc`:
```ShellCheck-Config
external-sources=true\r\n
ignore=SC2034\r\n
shell=sh\r\n
```
(`\r\n` is not literal, but represents a carriage return and a newline.)

`script.sh`:
```sh
#!/bin/sh
unused_var=1
. ./source.sh
```

`source.sh`:
```sh
true # no-op
```

#### Here's what shellcheck currently says:

When checking `script.sh` or `source.sh`:

```text
In […]/.shellcheckrc line 1:
external-sources=true
                     ^-- SC1017 (error): Literal carriage return. Run script through tr -d '\r' .


In […]/.shellcheckrc line 2:
disable=SC2034
              ^-- SC1017 (error): Literal carriage return. Run script through tr -d '\r' .

For more information:
  https://www.shellcheck.net/wiki/SC1017 -- Literal carriage return. Run scri...
```

#### Here's what I wanted or expected to see:

Nothing. While this should be an error for actual <em>scripts</em>, SC1017 should not apply to referenced `.shellcheckrc` files.

This is because on neither Windows or Linux (which I’m using for these demonstrations) does ShellCheck have any trouble following a `.shellcheckrc` file with `\r\n` line endings. For example:

```text
$ shellcheck --norc script.sh

In script.sh line 2:
unused_var=1
^--------^ SC2034 (warning): unused_var appears unused. Verify use (or export if used externally).


In script.sh line 3:
. ./source.sh
  ^---------^ SC1091 (info): Not following: ./source.sh was not specified as input (see shellcheck -x).

For more information:
  https://www.shellcheck.net/wiki/SC2034 -- unused_var appears unused. Verify...
  https://www.shellcheck.net/wiki/SC1091 -- Not following: ./source.sh was no...
$ shellcheck --norc source.sh

In source.sh line 1:
true # no-op
^-- SC2148 (error): Tips depend on target shell and yours is unknown. Add a shebang or a 'shell' directive.

For more information:
  https://www.shellcheck.net/wiki/SC2148 -- Tips depend on target shell and y...
```

These are the expected errors. However, when we remove `--norc`, these messages are correctly skipped, and instead only SC1017 for `.shellcheckrc` is shown:

```text
$ shellcheck script.sh

In […]/.shellcheckrc line 1:
external-sources=true
                     ^-- SC1017 (error): Literal carriage return. Run script through tr -d '\r' .


In […]/.shellcheckrc line 2:
disable=SC2034
              ^-- SC1017 (error): Literal carriage return. Run script through tr -d '\r' .

For more information:
  https://www.shellcheck.net/wiki/SC1017 -- Literal carriage return. Run scri...
$ shellcheck source.sh

In […]/.shellcheckrc line 1:
external-sources=true
                     ^-- SC1017 (error): Literal carriage return. Run script through tr -d '\r' .


In […]/.shellcheckrc line 2:
disable=SC2034
              ^-- SC1017 (error): Literal carriage return. Run script through tr -d '\r' .

For more information:
  https://www.shellcheck.net/wiki/SC1017 -- Literal carriage return. Run scri...
```

`external-sources=true` fixes SC1091, `ignore=SC2034` fixes SC2034, and `shell=sh` fixes SC2148. Note that if `.shellcheckrc` has `\n` line endings, there is, as expected, no output.

It’s quite silly for this to cause an error when no problem actually exists here. It might be minorly <em>inconsistent</em> to have a `.shellcheckrc` file with `\r\n` and scripts with `\n`, but that has nothing to do with the content of the scripts themselves and ShellCheck shouldn’t care, at the very least not as a fatal error.
