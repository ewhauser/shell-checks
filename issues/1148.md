#### <img src="https://avatars.githubusercontent.com/u/19507282?v=4" width="50">[automorphism88](https://github.com/automorphism88) commented at [2018-03-19T22:27:28Z](https://github.com/koalaman/shellcheck/issues/1148#issuecomment-374404779):

Shellcheck's SC2124 warning is correct here. You're trying to assign an array to a regular variable. `$@` is an array, and `REPOS` is not an array. You should either do `declare -a REPOS=("$@")` in order to make `REPOS` an array (or just `REPOS=("$@")`, since the parentheses will tell bash that `REPOS` is an array, and generally `declare` isn't necessary, unless you're declaring an associative array with the `-A` option), or, if `REPOS` not being an array is intentional, use `declare REPOS="$*"` instead. `$*` is a single string, with each element of the array `$@` separated by the first character of `IFS` (by default, the space character). Unless there's a good reason why `REPOS` isn't an array, you probably want the former, since using an array will preserve the original word splitting, while `$*` will not, potentially creating problems depending on how the `REPOS` variable is going to be used.

Protip: You can do the same thing with any array, by using `${foo[*]}` instead of `${foo[@]}`.

> maybe with quotes (let’s see what shellcheck says)

It's generally a good idea to always double quote variables unless you know what you're doing and have a good reason not to. Without the double quotes, bash will perform word splitting on the variable (and glob expansion, where applicable), so the only reason not to use double quotes is if that's what you want to happen.

In the case of an array, as in your for loop, double quotes will ensure that each element of the array is treated as a single word, even if it contains whitespace.
