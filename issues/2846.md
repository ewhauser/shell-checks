```sh

my_array=("20" "30")

$(( 1 + my_array[1] ))
EOF

```

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2023-12-11 03:14](https://github.com/koalaman/shellcheck/issues/2846#issuecomment-1849258735):

Very nice catch! This happened because of the combination of two peculiarities of Bash parsing: 1. Parsing bash array indices is [known to be undecidable](https://www.oilshell.org/blog/2016/10/20.html), because the interpretation depends on whether or not the array has previously been declared associative, and 2. Here documents are not normal language syntax, but rather parsed after the linefeed following commands that reference them.

ShellCheck deals with the first issue by delaying the parsing of array indices until the whole script is parsed, and then fairly naively looking for `declare -A` statements. It deals with the second by adding any pending here documents to a queue, and parsing them after an `\n`, and then associating them with the relevant command at the end.

Unfortunately ShellCheck did it in that order, meaning that indices were reparsed before the here docs were re-associated with the commands, thereby causing array references inside here docs to remain unprocessed.

The bug must have been around forever, but it only became symptomatic in 0.9.0 because of the new support for building Control Flow Graphs, which failed to handle the unprocessed array references. This particular bug has been fixed by flipping the order, and the CFG builder should avoid failing for any such issues in the future.

Thanks!
