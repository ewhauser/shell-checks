# [\#127 Issue](https://github.com/koalaman/shellcheck/issues/127) `closed`: SC2048 is overly zealous

#### <img src="https://avatars.githubusercontent.com/u/1823914?v=4" width="50">[arth1](https://github.com/arth1) opened issue at [2014-04-03T16:50:06Z](https://github.com/koalaman/shellcheck/issues/127):

If you have an assignment like

MYARGS=$*

SC2048 should not apply.  It does the exact same as

MYARGS="$@"

No matter whether you pass whitespace or  globbable characters as parameters.
I have not been able to find a single instance where it differs, as long as there is an assignment.

#### <img src="https://avatars.githubusercontent.com/u/442919?v=4" width="50">[dualbus](https://github.com/dualbus) commented at [2014-04-03T17:22:01Z](https://github.com/koalaman/shellcheck/issues/127#issuecomment-39479100):

Yes, the error message is a bit misleading. Though they're not equivalent.

The first one shouldn't be an error, because $\* is precisely a way of converting the parameter vector into a single string by joining each element using the first character of IFS as the glue. Though that's maybe not what the user expects...

The second one should be marked as an error, because $@ is an array, and assigning an array to a string is usually not what you want. If you want to preserve the 'arrayness' of the arguments, you should be using an array to store it, i.e. myargs=("$@")

And regarding the equivalence, no, they're _not_ equivalent. That's the whole point of having two expansions. See:

```
dualbus@debian:~$ set -- 1 2 3; IFS=:; a=$* b="$*" c=$@ d="$@"; args(){ printf '<%s>' "$@"; printf \\n; }; args "$a"; args "$b"; args "$c"; args "$d"
<1:2:3>
<1:2:3>
<1 2 3>
<1 2 3>
```
