#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env sh
set -o noglob
read -r REPLY
# Explicitly intended word splitting
set -- $REPLY
for arg; do
  echo "$arg"
done
```

#### Here's what shellcheck currently says:
```
Line 5	SC2086: Double quote to prevent globbing and word splitting.
```
#### Here's what I wanted or expected to see:

In POSIX shell, SC2086 shall ignore unquoted variables in `set -- $varname` because it is the only way to explicitly split words to iterate as the arguments array without forking sub-shell to use external tools like `xargs` to convert a string into arguments.


#### <img src="https://avatars.githubusercontent.com/u/215702?v=4" width="50">[gromgit](https://github.com/gromgit) commented at [2020-01-06T12:11:48Z](https://github.com/koalaman/shellcheck/issues/1763#issuecomment-571116541):

This is exactly what the [`disable` directive](https://github.com/koalaman/shellcheck/wiki/Directive#disable) is for:
```
# Explicitly intended word splitting
# shellcheck disable=SC2086
set -- $REPLY
```
Disabling SC2086 by default, even if only for POSIX shell, and even if only for `set -- XXX`, means it'll  never catch the cases when it actually **is** a mistake.

#### <img src="https://avatars.githubusercontent.com/u/119972?v=4" width="50">[jasonkarns](https://github.com/jasonkarns) commented at [2020-03-21T18:24:41Z](https://github.com/koalaman/shellcheck/issues/1763#issuecomment-602083341):

This is a dupe of #538

If an author is using `set -- XXX`, then there is virtually zero chance they _don't_ want word splitting. That is an exceeding common pattern meant specifically to leverage splitting.

I agree that SC2086 should be ignored for this conventional pattern.

Related to #1149 and #1725 

Both `set --` and `IFS=` are two widely-used patterns that explicitly indicate intentional word splitting. Adding the disable directive for these cases should be considered redundant.
