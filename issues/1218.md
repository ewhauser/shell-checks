# [\#1218 Issue](https://github.com/koalaman/shellcheck/issues/1218) `closed`: Conflicting advice about quoting variables

#### <img src="https://avatars.githubusercontent.com/u/4327840?v=4" width="50">[bje-](https://github.com/bje-) opened issue at [2018-05-12T08:48:25Z](https://github.com/koalaman/shellcheck/issues/1218):

#### For bugs

- Rule SC2076/SC2086
- Shellcheck version 0.4.7
- [X] I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash

re=$1
echo $re
[[ foo =~ "$re" ]] && echo y
```

#### Here's what shellcheck currently says:

```
Line 4:
echo $re
     ^-- SC2086: Double quote to prevent globbing and word splitting.
 
Line 5:
[[ foo =~ "$re" ]] && echo y
          ^-- SC2076: Don't quote rhs of =~, it'll match literally rather than as a regex.
```

#### Here's what I wanted or expected to see:

I don't understand why I should not quote the variable containing the regular expression to protect against the variable being unset. If `$re` is unset, then the shell will complain that there is no right-hand argument to the `=~` binary operator. Why isn't the advice consistent between echoing `$re` and using it with a regex operator?

#### <img src="https://avatars.githubusercontent.com/u/8651654?v=4" width="50">[jleffler](https://github.com/jleffler) commented at [2018-05-12T08:52:04Z](https://github.com/koalaman/shellcheck/issues/1218#issuecomment-388540804):

See the Bash manual on the [`[[`](https://www.gnu.org/software/bash/manual/bash.html#index-_005b_005b) operator.  If any part of the RHS of `=~` is quoted, that part isn't treated as a regular expression (_any part of the pattern may be quoted to force the quoted portion to be matched as a string_).  It's weird, but it is the way Bash is defined to work — for worse or worse.
