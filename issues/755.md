#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-10-30T03:27:46Z](https://github.com/koalaman/shellcheck/issues/755#issuecomment-257129834):

`# shellcheck disable=SC2016` will normally only apply to the next command.

It only applies to the entire file if it's immediately after the script's `#!` shebang, because it will be considered a file-wide directive. You may have accidentally triggered this by trying to suppress something on the first line in the file:

```
#!/bin/bash
# shellcheck disable=SC2016
sed '$n' # intentionally no warning
sed '$n' # accidentally no warning
```

If there's anything at all before the first line, so that the directive does not follow the shebang, this will not happen:

```
#!/bin/bash
# some command or comment here
# shellcheck disable=SC2016
sed '$n' # intentionally no warning
sed '$n' # now has warning as intended
```

If you want more advanced scoping, `{ ..; }` is considered a command and can be used for that:

```
#!/bin/bash
# some command or comment here
# shellcheck disable=SC2016
{
  sed '$n' # intentionally no warning
  sed '$n' # intentionally no warning
}
sed '$n' # has warning
```

There's no concept of "the rest of the file" when it comes to annotations, so an enable directive can't really work the way you describe it.
