# [\#306 Issue](https://github.com/koalaman/shellcheck/issues/306) `open`: Accepts $(echo ./*), and wider discussion about the use of $()

#### <img src="https://avatars.githubusercontent.com/u/7931883?v=4" width="50">[calimeroteknik](https://github.com/calimeroteknik) opened issue at [2015-02-13T18:37:48Z](https://github.com/koalaman/shellcheck/issues/306):

Out of curiosity I entered this (SH code):

``` sh
touch 'a a' b
for i in $(echo ./*); do echo "$i"; done
```

I don't know if it's a good thing or not that this be accepted, but in the current example I'm leaning towards 'not'. Anyway, this given example can be fixed by replacing $(echo./*) by ./*

But that's the tip of the iceberg.

I tend to always consider a $() that's not a "$()" bogus (dangerous) by definition.
Of course, when the code is for foo in $(); do… then it would be stupid to make that one only word.
But still, that's dangerous.

If one wants to iterate on the _lines_ output by a command, this is OK in SH:

``` sh
command | while read line; do foo "$line"; done
```

It's **not** OK however, if the command is find. (filenames can contain newlines).
This is OK in SH:

``` sh
find -print0 | xargs -0 foo
```

But if foo is a shell function, that won't work, obviously.

So, **in BASH only**, make IFS empty and use read without escaping or delimiter:

``` bash
find -print0 | while IFS= read -r -d '' file; do foo "$file"; done
```

Is it possible to make shellcheck detect the dangerous problems described here?

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-02-14T02:53:37Z](https://github.com/koalaman/shellcheck/issues/306#issuecomment-74358497):

The reason there is currently no warning for this is that it's a local minimum of badness -- there's nothing you can easily do to make it better. Case in point, your page long description of multiple cases each with caveats.

ShellCheck recognizes a few special cases where the path to a fix is more straight forward, such as `for f in $(ls *.mp3)` or `for f in $(find dir)` (the wiki page for just the latter deals with issues similar to your post, and it's [awful and complicated](https://github.com/koalaman/shellcheck/wiki/Sc2044)).

Ideas so far have included:
- Inform/warn always
- Warn people unless they've `set -f` and set IFS, or otherwise seem to know what they're doing
- Have a pedantic mode that warns about it
- Leave it

I'm still not entirely sure how to handle it and input is appreciated.
