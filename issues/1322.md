# [\#1322 Issue](https://github.com/koalaman/shellcheck/issues/1322) `open`: Shellcheck doesn't always warn of unsafe array subscript expansions in arithmetic context

#### <img src="https://avatars.githubusercontent.com/u/10137?v=4" width="50">[ghost](https://github.com/ghost) opened issue at [2018-08-25T18:27:43Z](https://github.com/koalaman/shellcheck/issues/1322):

As tested in shellcheck 0.5.0 and the online version, the following sample does not warn of the unsafe expansion of $bar:-

```shcipherfox mozilla addons
#!/bin/bash
declare -A foo
bar='$(echo "evil code" >&2)a'
(( foo[$bar]++ ))
declare -p foo
```

Here's what the program itself prints:

```
evil code
evil code
declare -A foo=([a]="1" )
```

Here's what shellcheck reports:-

```
bar='$(echo "evil code" >&2)a'
    ^-- SC2016: Expressions don't expand in single quotes, use double quotes for that.
```

What would be preferable is for shellcheck to also report that the expansion of the subscript is unsafe due to the arithmetic context. For reference, this is the text from greybot's current `$subscript` entry:-

> In an arithmetic context, never reference an array subscript dynamically as foo[$bar]. Instead, write foo[bar] for indexed arrays, and any of foo[\\$bar], foo['$bar'] or ${foo[$bar]} for associative arrays. Otherwise, the expansion of bar is repeated and is thus vulnerable to code injection.

Perhaps the scope of SC2004 could be expanded to cover this.
