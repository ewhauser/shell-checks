# [\#3067 Issue](https://github.com/koalaman/shellcheck/issues/3067) `open`: Feature suggestion: Warnings regarding (automatic) recursive indirection/eval within [[ (( and $(( blocks

#### <img src="https://avatars.githubusercontent.com/u/1054399?v=4" width="50">[fedy-cz](https://github.com/fedy-cz) opened issue at [2024-10-05 12:11](https://github.com/koalaman/shellcheck/issues/3067):

Came across this (poorly documented) feature of bash arithmetic expansion within the [[ , (( and $(( blocks (my quick interpretation):

_Within the arithmetic expansion in these blocks all variables are recursively expanded (using indirection) until the content can be parsed as an integer or no further indirection is possible. This includes evaluating arithmetic expressions (contained as a string within the variable)._

**Example:**
```bash
#!/bin/bash
a=b
b=c
c=10
if [[ $a -gt 5 ]] ; then echo Greater ; else "Not greater"  ; fi

```
Outputs:
```
Greater
```

_**Side note:** Interestingly shellcheck complains about b and c being unused here, which in fact is not true :)_

This "feature" is discussed (for example) in this bash mailing list thread:
https://lists.gnu.org/archive/html/help-bash/2018-10/msg00011.html

Definitely an unexpected behavior (in my book), and very poorly documented. This could unearth a whole new class of bugs. 

#### The instance I personally came across (simplified):
```bash
#!/bin/bash

default=10

use_input() {
  return 0
}

read_config() {
  local input
  input=$( < filename )
  echo "Read: $input"
  if use_input ; then
    # Notice the missing $ there
    global_setting=input
  else
    global_setting=$default
  fi

  # Final check
  if [[ $global_setting -gt 0 ]] ; then
    echo "global setting is numeric"
  else
    exit 1
  fi
}
read_config

echo "global setting: $global_setting"
```

When I put value 5 into filename this outputs:
```
Read: 5
global setting is numeric
global setting: input
```

#### Few other simple examples I can think off:

```bash
#!/bin/bash
# Somewhere:
x=5
var=x
# ...
# Later, attempt to make sure that a variable is a positive numeric value
if [[ var -gt 0 ]] ; then
  echo "var is numeric"
  # But in fact it's just x
fi
```

Or:
```bash
#!/bin/bash
internal_val=5
var=$( < file )
# file contains the string internal_val
if [[ var -gt 0 ]] ; then
  # expecting to be processing value from file here but instead working with the internal_val
  x = $((  var * 5 ))
fi
```
You can even put whole expressions in there as mentioned in the referred mailing list thread:
```bash
#!/bin/bash
one=1
two=2
x='one + two'
code='4 > 2 ? x : y'
# This  just evals
echo $(( code ))
```

### The main pain points I see right now are:
- unintended aliasing / recursive indirection & it's effects
- incorrect input validation issues

_Wouldn't be surprised if this led to a few security incidents as well._

#### Generally:
- Can hide an accidental "variable name" string assignment instead of a variable value (missing $).
- Anything that comes from user input should avoid being used within the (( , $(( and arithmetic part of [[  blocks unless properly validated / pattern matched first. 
- **Never use these block for input/value format validation** (one exception being [[ containing no arithmetic expression that uses the input).

_Not sure how well this could translate into actual shellcheck rules. But kicking off the discussion._

I've just found out this [can easily be abused for ACE/RCE](https://mywiki.wooledge.org/BashProgramming/05#Arithmetic_Expansion):

I wrote this simple script:
```bash
#!/usr/bin/env bash
a="$1"
echo "$(( a ))"
```
run it as `./main 'a[$(date >&2) 0]'` and it **infinitely calls `date`** until the Bash instance crashes. This is literally **denial-of-service** potential.

Imagine a similar script runs in a server, and a user-input turns into `'a[$(sudo rm -rf --no-preserve-root / >&2) 0]'`

I'm now trying to remember all the scripts where I used `$(())` to "validate" input. I've even saw other projects doing that, and now I'm concerned there's a thousand [0-days](https://en.wikipedia.org/wiki/Zero-day_vulnerability) already happening ([shellshock](https://en.wikipedia.org/wiki/Shellshock_(software_bug)) Electric Boogaloo?).

#### <img src="https://avatars.githubusercontent.com/u/76864299?u=30937bbe043d9d087762d03013f3f9353a4fbc2a&v=4" width="50">[Rudxain](https://github.com/Rudxain) commented at [2025-10-02 06:09](https://github.com/koalaman/shellcheck/issues/3067#issuecomment-3359293981):

In the meantime, here's [how to `assert` that a string can be parsed as an `int`eger](https://github.com/dylanaraps/pure-sh-bible/blob/26bc7e8d6cfb8716e9ccb97f31c2da80468c5737/README.md?plain=1#L914-L933):
```bash
# Bash
builtin printf %d "$1" &>/dev/null || exit
# POSIX
builtin printf %d "$1" >/dev/null 2>&1 || return $?
```
The POSIX one is only useful if the script might be executed by Bash, which is the case for scripts that are distributed for multiple platforms

[More info](https://mywiki.wooledge.org/BashFAQ/054#How_can_I_tell_whether_a_variable_contains_a_valid_number.3F)
