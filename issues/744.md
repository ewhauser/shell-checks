#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-10-03T22:38:50Z](https://github.com/koalaman/shellcheck/issues/744#issuecomment-251247758):

That is indeed tricky. It happens because shellcheck directives are per complete command/structure, and not per line. Since `elif` is not a (complete) command, you can't put directives on it. 

Perhaps the best way forward would be to fix the issue that shellcheck complains about. 

In [the function in question](https://github.com/Winetricks/winetricks/blob/master/tests/winetricks-test#L261) there's a line `command="$*"` that defines the variable that is then used unquoted. This would join and split the string, thereby ruining the original spacing (causing it to fail on filenames with spaces), and may additionally fail on anything containing glob characters.

This would not happen if you used `"$@"` instead of `$command`, but I don't know if any of the other code depends on the current buggy behavior.

Anyways, if you just want to ignore this issue, what you can do is to put it after the elif and before the condition, which unfortunately makes formatting a bit weird:

```
if [ "$EXPECT_FAIL" = "yes" ]
then
echo blah

elif #shellcheck disable=SC2086
      ! w_time sh winetricks --no-isolate -q nocrashdialog $command
then
echo do stuff
else
echo stuff
fi
```

Or you can disable it for the entire `if` statement:

```
#shellcheck disable=SC2086
if [ "$EXPECT_FAIL" = "yes" ]
then
echo blah
elif ! w_time sh winetricks --no-isolate -q nocrashdialog $command
then
echo do stuff
else
echo stuff
fi
```
