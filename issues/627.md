I thought the path parameter in find was optional and if left out the current path (".") was be implied but apparently that is not everywhere the case and not according to the POSIX standard (and apparently also not e.g. in a version of `bash` for OS X from what I heard from someone using my script).

``` sh

```

or

``` bash

```

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-03-14T00:02:27Z](https://github.com/koalaman/shellcheck/issues/627#issuecomment-196085958):

No no no noooo! Do _NOT_ recommend `which`; it's a bad solution looking for
a question that already has a better answer.

The `which` command will report where _it_ can find commands, which means
you will have unreliable results for (or possibly always completely missing)
shell keywords, shell built-ins, shell functions, dynamic loaders, and shell
aliases. Even if your version of `which` can read `~/.bashrc` it may still
give wrong answers, because defining aliases and functions may be dependent
on context that it can't compute, or use a syntax it doesn't understand.

The `type` command is built into the shell and will tell you what the shell
will actually use, rather than what some external utility will second-guess
about the shell.

The only reliable version of `which` is to define it a shell function:
`which() { type "$@"; }` but if you're going to do that, you might as well
just type `type` to begin with.
