On Sat, 25 Feb 2017, Tom Hale wrote:
> I've since learned that this particular `bash` idiom is not required as
> with `set -e`, the exit status of the erroring command is automatically
> returned.

It's not the `set -e` that's important; rather, it's the fact that it's the EXIT trap, which does not override the exit status of the whole script _unless_ you include "exit" inside the trap.

I generally prefer to use:

    trap myexittrap EXIT
    myexittrap() {
        do_stuff_here
    }

That way I don't have to wrangle the extra quoting levels. And it'll be easier for Shellcheck to see what's happening too.
