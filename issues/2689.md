# [\#2689 Issue](https://github.com/koalaman/shellcheck/issues/2689) `closed`: Inconsistent triggering of SC2268 (x-prefix comparisons)

#### <img src="https://avatars.githubusercontent.com/u/68940237?v=4" width="50">[slycordinator](https://github.com/slycordinator) opened issue at [2023-02-08 06:45](https://github.com/koalaman/shellcheck/issues/2689):

SC2268 recommends that one not use x-prefix comparisons, but it allows them in many cases.

SC2268 appears to only get triggered if:
1) The comparison is = or ==. Nothing happens when the comparison is '!="
[ ! "x$var" = "x" ] and [ ! "x$var" == "x" ] trigger the warning, but [ "x$var" != "x" ] and [ ! "x$var" != "x" ] do not trigger it.

2) The prefix is solely the lowercase 'x'. Any other prefix doesn't trigger the issue. Recently came across scripts that occasionally have  "X" as the prefix and it doesn't get triggered. Also doesn't get triggered when any other letter is used.
[ "x$var" = "x" ] triggers this, but [ "X$var" = "X" ] does not.

Thank you

What is the preferred syntax here when $var could be undefined or the empty string? I ask because it is relevant for the documentation and one of the reasons I still use `[ "x$1" = "x--my-only-cmdline-option" ]` (yes I know I could use getopt).

#### <img src="https://avatars.githubusercontent.com/u/68940237?v=4" width="50">[slycordinator](https://github.com/slycordinator) commented at [2023-09-17 04:12](https://github.com/koalaman/shellcheck/issues/2689#issuecomment-1722386720):

Very old shells were confused when the first argument began with "-" as well as when it was simply "!" or "(".

So, people would put x at the front of the argument/variable.

POSIX makes that unnecessary.

Unless you're on a shell that predates POSIX, you can replace
[ "x$1" = "x--my-only-cmdline-option" ]

with
[ "$1" = "--my-only-cmdline-option" ]
