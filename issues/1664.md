# [\#1664 Issue](https://github.com/koalaman/shellcheck/issues/1664) `open`: shellcheck should encourage explicit `-z` & `-n` string tests

#### <img src="https://avatars.githubusercontent.com/u/176950?v=4" width="50">[vapier](https://github.com/vapier) opened issue at [2019-08-07T17:40:49Z](https://github.com/koalaman/shellcheck/issues/1664):

#### For new checks and feature suggestions

in my experience, people omit `-z` & `-n` with string tests not because they want to, but because they think they're testing the exit status of a tool and not its stdout.

we should discourage these implicit forms in favor of explicit `-z` & `-n` operators.  the difference between `!` and `-z` is a single byte.  `-n` is an additional 3 bytes.  i think people can live with that in order to catch thinkos in code.

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash

# People implicitly testing strings.
foo="bar"
[[ "${foo}" ]]
[[ ! "${foo}" ]]

# People testing stdout -- did they mean to test the exit status instead?
[[ "$(foo)" ]]
[[ ! "$(foo)" ]]
```

#### Here's what shellcheck currently says:

shellcheck issues no guidance here.

#### Here's what I wanted or expected to see:

shellcheck should have suggested these changes:
* use `[[ -n "${foo}" ]]` instead of `[[ "${foo}" ]]`
* use `[[ -z "${foo}" ]]` instead of `[[ ! "${foo}" ]]`

bonus points to also suggest `[[ "$(...)" ]]` forms either use `...` directly in order to test the exit status (instead of stdout output) or `[[ -n "$(...)" ]]` if they really wanted to test stdout.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-08-07T20:05:50Z](https://github.com/koalaman/shellcheck/issues/1664#issuecomment-519249091):

There's already an optional check `avoid-nullary-conditions`:

```
name:    avoid-nullary-conditions
desc:    Suggest explicitly using -n in `[ $var ]`
example: [ "$var" ]
fix:     [ -n "$var" ]
```

The support for `-z` is currently an annoying two-step process though. By fixing `[ ! "$stuff" ]` into `[ ! -n "$stuff" ]`, then SC2236 will trigger and suggest `-z`
