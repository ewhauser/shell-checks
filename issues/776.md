The question is: how could this even be interpreted as an option?

As for a background: the issue is about constructing a command based on information that is only known at run time. My use case involves generating a list of `name=value` pairs that preserve spaces in a POSIX compliant way.
Specifically for `curl`:
```sh
curl -fsSL -F "$formname1=$content1" -F "$formname2=$content2" ... "$URL"
```
UPD. Related to #380. As someone commented there, using quoted strings to represent possibly space containing values is a bad practice. POSIX `sh` doesn't have arrays, so we have to use functions and positional parameters. But is this really the only viable solution? (sorry if the actual issue became offtopic for this project).

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-11-19T20:34:05Z](https://github.com/koalaman/shellcheck/issues/776#issuecomment-261737925):

You're right, this is most definitely a bug. No warning should be given for flags after `--`. 

From my reading of POSIX, `--` shouldn't even be accepted, but I didn't find a shell where this was problematic.

Functions and positional parameters are a good solution because it's the one array `sh` supports, but it's also possible to e.g. pick a delimiter you won't support in values and use word splitting (while disabling globs), or to very carefully escape the string in a shell independent way and eval it.
