#### Here's a snippet or screenshot that shows the problem:

1st example:
```sh
#!/bin/sh
IFS=0
true
if [ $? = 0 ]; then
    echo "foo"
else
    echo "bar"
fi
```

2nd example:
```sh
#!/bin/sh
IFS=0
true
echo returned code was $?
```

#### Here's what shellcheck currently says:

For the 1st example:
```
$ shellcheck myscript
 
Line 4:
if [ $? = 0 ]; then
     ^-- SC2181: Check exit code directly with e.g. 'if mycmd;', not indirectly with $?.
```

For the 2nd example:
```
$ shellcheck myscript
No issues detected!
```

#### Here's what I wanted or expected to see:

Sometimes scripts are run in a way where the IFS is not set to an expected value or the developer has no control over it

Lots of people ignore SC2181 as a style choice for the 1st example. In both examples, shellcheck should issue SC2086 because the only way to make that script work is to double quote the "$?".

The wiki page for SC2181 or SC2086 does not cover this case.

Possibly similar issues exist for $$ and other vars like that.

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-10-07 02:06](https://github.com/koalaman/shellcheck/issues/2341#issuecomment-937384967):

You're right, this is one of the long tail of things that can go wrong with a script. It could also be that `sh` is `bash` and that it picks up definitions of `[` or `echo` from the environment, or that `if` is an alias, and even if all those are carefully unset,(taking care not to invoke a potentially rigged `unalias` command) there could be a DEBUG trap that resets them immediately. And what if the shell has custom builtins loaded?

It would be fun to try to make scripts bulletproof from all of the above, but it's out of scope for ShellCheck which instead tries to pragmatically find likely sources of realistic, unintentional issues like `IFS='\0'`.
