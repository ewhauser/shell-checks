# [\#2041 Issue](https://github.com/koalaman/shellcheck/issues/2041) `open`: bats: bogus SC2034 (no support for bats' load?)

#### <img src="https://avatars.githubusercontent.com/u/4522509?v=4" width="50">[kolyshkin](https://github.com/kolyshkin) opened issue at [2020-08-20 07:22](https://github.com/koalaman/shellcheck/issues/2041):

#### For bugs
- Rule Id: SC2034
- My shellcheck version (`shellcheck --version` or "online"): 0.7.1
- [x] The rule's wiki page does not already cover this
- [ ] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit
(I can't try since my repro consists of two files)

#### Here's a snippet or screenshot that shows the problem:

helpers.bash:
```sh
#!/bin/bash

FOO=bar

function prepare() {
	echo $FOO
}
```

a.bats:
```sh
#!/usr/bin/env bats

load helpers

@test default {
	prepare | grep -w bar
}

@test override {
	FOO=whatever
	prepare | grep -w whatever
}
```

#### Here's what shellcheck currently says:

```console
$ shellcheck -x a.bats
In a.bats line 10:
	FOO=whatever
        ^-^ SC2034: FOO appears unused. Verify use (or export if used externally).

For more information:
  https://www.shellcheck.net/wiki/SC2034 -- FOO appears unused. Verify use (o...
```

#### Here's what I wanted or expected to see:

No errors, like in the following example using pure bash:

helpers.bash:
```bash
#!/bin/bash

FOO=bar

function prepare() {
	echo $FOO
}
```

a.sh:
```bash
#!/bin/bash

set -e
. helpers.bash

function test_default() {
	prepare | grep -w bar
}

function test_override() {
	FOO=whatever
	prepare | grep -w whatever
}

test_default
test_override
```

Test execution to see if it works:
```console
$ bash ./a.sh 
bar
whatever
$ echo $?
0
```

shellcheck run (shows no warnings):
```console
$ shellcheck -x ./a.sh 
$ echo $?
0
```

#### Additional information

In case I move `function prepare()` from `helpers.bash` to the main a.bats file, the warning goes away.

The very same repro using pure bash (see above) does not result in a warning.

This probably means that bats' `load` statement is not recognized in a way similar to bash' `source`.
