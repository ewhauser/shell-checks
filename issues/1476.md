#### <img src="https://avatars.githubusercontent.com/u/1323808?v=4" width="50">[danielhoherd](https://github.com/danielhoherd) opened issue at [2019-01-24T22:26:34Z](https://github.com/koalaman/shellcheck/issues/1476):

#### For new checks and feature suggestions
- [X] shellcheck.net (i.e. the latest commit) currently gives no useful warnings about this
- [X] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related


#### Here's a snippet or screenshot that shows the problem:

```sh
# From within a git repo

files=()

if git config --get remote.origin.url > /dev/null ; then
  if [ "${#@}" -eq 0 ] ; then
    mapfile -t files < <(git ls-files)
  else
    for item in "$@" ; do
      if [ -d "$item" ] ; then
        files+=( $(git ls-files "$item") )
      else
        files+=( "$item" )
      fi
    done
  fi
else
  echo "Not in a git repo, can't continue."
  exit 1
fi
```

#### Here's what shellcheck currently says:

```
In git-show-repo-experts.sh line 40:
        files+=( $(git ls-files "$item") )
                 ^---------------------^ SC2207: Prefer mapfile or read -a to split command output (or quote to avoid splitting).

For more information:
  https://www.shellcheck.net/wiki/SC2207 -- Prefer mapfile or read -a to spli...
```


#### Here's what I wanted or expected to see:

I want the URL <https://www.shellcheck.net/wiki/SC2207> to explain the append syntax of mapfile. `files+=( item )` is an array append, but the syntax described on the linked article `mapfile -t array < <(mycommand)` does not work when appending is required.

#### <img src="https://avatars.githubusercontent.com/u/1323808?v=4" width="50">[danielhoherd](https://github.com/danielhoherd) commented at [2019-01-25T03:45:02Z](https://github.com/koalaman/shellcheck/issues/1476#issuecomment-457445607):

I found the solution to this problem.

```
files+=( $(git ls-files "$item") )
```
should be changed to
```
mapfile -O "${#files[@]}" -t files < <(git ls-files "$item")
```

```
-O origin    Begin writing lines to array array at index number origin.
```

#### <img src="https://avatars.githubusercontent.com/u/6032558?v=4" width="50">[Mr-Tao](https://github.com/Mr-Tao) commented at [2020-07-08T22:21:26Z](https://github.com/koalaman/shellcheck/issues/1476#issuecomment-655787946):

@danielhoherd that could result in overwriting of array elements when indices are not contiguous. Also, the syntax is much more opaque. I think that no workaround beats clarity and elegance of `+=` operator and the warning should not be issued for the case of appending array.
I wonder if you can get the maximum index with something less abominable than
```
echo "${!array[@]}" | cut -d" " -f ${#array[@]}
```
