#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bash

# Source Bash utilities, if possible
[[ -z "${HOME}" ]] \
    && { "${HOME:?}"; exit 1; }
export bash_utils="${HOME}/bin/bash_utils.sh"
# shellcheck source=${HOME}/bin/bash_utils.sh
if [[ ! -e "${bash_utils}" ]] || ! . "${bash_utils}"; then
    printf '[ERROR] Could not source %s\n' "${bash_utils}" 1>&2
    exit 1
fi
```

#### Here's what shellcheck currently says:
Terminal:
```sh
$ shellcheck -x "$(which rip_up)"
if [[ ! -e "${bash_utils}" ]] || ! . "${bash_utils}"; then
                                   ^-- SC1090: Can't follow non-constant source. Use a directive to specify location.
```
VS Code:
```sh
Not following: ${HOME}/bin/bash_utils.sh: openBinaryFile: does not exist (No such file or directory) [SC1091]
```

#### Here's what I wanted or expected to see:
Support for substituting environment variables should be present.
When someone clones my repo, my home path cannot be expected to match theirs.
Merge conflicts arise for an entirely avoidable reason when the other person updates the shellcheck directive to suit their sourced locations.

Please consider, and thanks for this wonderful tool.

#### <img src="https://avatars.githubusercontent.com/u/1776970?v=4" width="50">[slaiyer](https://github.com/slaiyer) commented at [2018-06-05T19:53:44Z](https://github.com/koalaman/shellcheck/issues/1250#issuecomment-394838822):

[The POSIX specification](http://pubs.opengroup.org/onlinepubs/009695399/basedefs/xbd_chap08.html) requires HOME to be set by the OS (ref. [this](https://unix.stackexchange.com/a/123859/80676)).

Disabling the check results in dozens of errors about using variables defined in the source script, which is expected, but too much noise to filter through.

What about merge conflicts when all parties have their own source paths hardcoded in?
Now everyone involved must strip scripts of shellcheck directives before committing AND add them back after pulling, lest everyone else have a major case of shellcheck noise.
Absolute paths are very restrictive in terms of collaboration.

I have also tried relative paths, but to no avail.
There is a reason even symlinks allow relative paths - you can't expect filesystem paths to mirror each other all the way down from the root.

#### <img src="https://avatars.githubusercontent.com/u/1776970?v=4" width="50">[slaiyer](https://github.com/slaiyer) commented at [2018-06-12T11:50:28Z](https://github.com/koalaman/shellcheck/issues/1250#issuecomment-396562007):

Fixed with this:
```
# Source Bash utilities, if possible
declare script_path
script_path="$(dirname -- "${0}")"
declare -r script_path
declare -rx bash_utils="${script_path}/bash_utils.sh"
if [[ ! -f "${bash_utils}" ]]; then
    printf '[FATAL] Could not find %s\n' "${bash_utils}" 1>&2
    exit 1
fi
# shellcheck source=./bash_utils.sh
if ! . "${bash_utils}"; then
    printf '[FATAL] Could not source %s\n' "${bash_utils}" 1>&2
    exit 1
fi
```
