# [\#2324 Issue](https://github.com/koalaman/shellcheck/issues/2324) `open`: Detect pipe output from cd or umask (creates a subprocess)

#### <img src="https://avatars.githubusercontent.com/u/388671?v=4" width="50">[stevecj](https://github.com/stevecj) opened issue at [2021-09-09 18:06](https://github.com/koalaman/shellcheck/issues/2324):

I was recently bitten by some problems in my code that it would be nice if shellcheck could identify.

Piping output from a command causes the command to run in a subshell, so piping the output of commands like `cd`, `pushd`, `popd`, or `umask` causes them to have no effect.

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-09-19 01:24](https://github.com/koalaman/shellcheck/issues/2324#issuecomment-922398288):

Do you have an example?

ShellCheck warns about variable modifications in subshells including pipelines, but not about `cd`. There's actually [SC2103](https://github.com/koalaman/shellcheck/wiki/SC2103) that explicitly suggests using a subshell to simplify `cd foo; bar; cd ..` type logic. It could be a useful warning if it can differentiate between intentional and unintentional subshells.

#### <img src="https://avatars.githubusercontent.com/u/26471?u=487b2dd94c891819f0b37be1e0f193daef31f226&v=4" width="50">[andersk](https://github.com/andersk) commented at [2021-09-22 03:38](https://github.com/koalaman/shellcheck/issues/2324#issuecomment-924555491):

Perhaps warn if the *last* command in a subshell is `cd` or similar, on the grounds that there are no commands for it to affect?

#### <img src="https://avatars.githubusercontent.com/u/388671?v=4" width="50">[stevecj](https://github.com/stevecj) commented at [2021-10-13 19:03](https://github.com/koalaman/shellcheck/issues/2324#issuecomment-942627672):

What I'm talking about are cases where the command will run in a subshell implicitly, even though neither back-tic nor `$(` `)` was used. An example of when this can happen is when trying to pipe through `tee` to capture output to a log file.

```bash
cd ./foo 2>&1 | tee -a $INSTALL_LOG
```
