#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/sh

un\alias -a

```

#### Here's what shellcheck currently says:

```
un\alias -a
  ^-- SC1001: This \a will be a regular 'a' in this context.
```

#### Here's what I wanted or expected to see:

If someone wanted to remove all aliases they would use `unalias -a`, however its best practice to also escape `unalias` to ensure its not an alias itself. The posix spec for `command` offers some examples along this line. 

http://pubs.opengroup.org/onlinepubs/9699919799/utilities/command.html

If the `\` is embedded shellcheck will print useless and erroneous error. It can be simply silenced prefacing the `\` like `\unalias -a` instead of `un\alias -a`, but it does not matter where the slash is placed to ensure its not an alias so it should not print a warning regardless.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-09-14T19:54:55Z](https://github.com/koalaman/shellcheck/issues/996#issuecomment-329591110):

This is working as intended.

`\a` is a bell character in several other contexts, and in this example it does not help to disable aliases (because aliases are not enabled in scripts by default). Unlike a leading `\`, this is also not an established convention to avoid aliases when they *are* enabled.

Either write it in a canonical way (`command unalias` or `\unalias`) or disable the warning.
