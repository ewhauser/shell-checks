# [\#289 Issue](https://github.com/koalaman/shellcheck/issues/289) `open`: SC2086 ("Double quote to prevent globbing and word splitting.") should not apply to variables declared to be integers

#### <img src="https://avatars.githubusercontent.com/u/179397?v=4" width="50">[srcshelton](https://github.com/srcshelton) opened issue at [2015-01-20T20:58:07Z](https://github.com/koalaman/shellcheck/issues/289):

If a variable is declared to be numeric (with declare/local -i) then SC2086 should not apply - the code:

``` sh
function foo() {
    local -i rc=${?}

    ...

    exit ${rc}
}
```

... should not generate a warning.

#### <img src="https://avatars.githubusercontent.com/u/1396056?v=4" width="50">[Speeddymon](https://github.com/Speeddymon) commented at [2016-12-07T00:08:28Z](https://github.com/koalaman/shellcheck/issues/289#issuecomment-265314564):

I agree with the sentiment, however the example does not go far enough to demonstrate the issue.

```
declare -i RC
function handle_rc()
{
        local -i rc
        rc=$1

        if [ $rc -ne 0 ]
        then
                RC=$rc
        fi
        return $RC
}
handle_rc 1
```

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2021-10-05T01:36:02Z](https://github.com/koalaman/shellcheck/issues/289#issuecomment-933987999):

This seems like the sort of stylistic choice that cannot be made to satisfy everyone.

(I've seen code in the wild that does `IFS=89` in an attempt to avoid complaints about broken octal numbers.)

Maybe we need a option `--assume-sane-ifs` that assumes that IFS won't contain `[-+0-9_a-zA-Z]`, and therefore it is safe to leave unquoted the `$((...))` expansion, the built-in numeric variables, and any variables that are marked with `-i`.
