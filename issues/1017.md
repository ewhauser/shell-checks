#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash

# It knows that `read` sets a variable
read -r -a var1 <<<'foo bar'
printf ' -> %s\n' "${var1[@]}"

# But not if we add a layer of indirection
set_array() {
	read -r -a "$1" <<<$"2"
}
set_array var2 'foo bar'
printf ' -> %s\n' "${var2[@]}"
```

#### Here's what shellcheck currently says:

```
 
Line 12:

                   ^-- SC2154: var2 is referenced but not assigned (did you mean 'var1'?).

```

Nothing, it should come back clean.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-09-29T17:27:45Z](https://github.com/koalaman/shellcheck/issues/1017#issuecomment-333187757):

You're right of course. It's a ShellCheck's design decision not to try to figure out whether variables are used through indirection. Doing that is an infinite rabbit hole that in the general case is equivalent to the halting problem. 

While trying to add this simple case might work and provide some value, you'd quickly find trivial additions that break it again (like `read -r -a "${1%?}"`). Instead, ShellCheck just categorically stops at the indirection boundary.

Sorry about that. Consider explicitly assigning the variables you know will be used or disabling this warning script-wide instead.
