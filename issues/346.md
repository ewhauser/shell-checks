# [\#346 Issue](https://github.com/koalaman/shellcheck/issues/346) `closed`: Encourage ${VAR} for adjacent concatenations

#### <img src="https://avatars.githubusercontent.com/u/5316?v=4" width="50">[mcandre](https://github.com/mcandre) opened issue at [2015-04-09T18:33:49Z](https://github.com/koalaman/shellcheck/issues/346):

I've run into problems multiple times with forgetting to use curly brackets (`${VAR}`) instead of regular `$VAR`, when concatenating variables with other adjacent text.

E.g., `pgrep "test*$DATABASE*logger"` may fail, whereas `pgrep "test*${DATABASE}*logger"` works.

Could ShellCheck add a new rule to warn when curly braces should be used?

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-04-09T20:01:04Z](https://github.com/koalaman/shellcheck/issues/346#issuecomment-91341961):

This would definitely be a good thing, but always requiring curly braces seems like overkill.

In your particular example it doesn't matter, but in e.g. `mv file.jpg "$prefix_$number.jpg"` it does. 

For lowercase variables, assumed to be shell local, that have not been assigned, you get something like `SC2154: prefix_ is referenced but not assigned (did you mean 'prefix'?).`

For uppercase variables, assumed to be mostly environment variables, where a (very) similar name has been assigned, you get `SC2153: Possible misspelling: PREFIX_ may not be assigned, but PREFIX is.` 

Do you have some real world examples not caught by those that could help determine when to warn?


#### <img src="https://avatars.githubusercontent.com/u/5316?v=4" width="50">[mcandre](https://github.com/mcandre) commented at [2015-04-10T00:50:46Z](https://github.com/koalaman/shellcheck/issues/346#issuecomment-91390952):

Agreed, curly braces aren't needed everywhere.

Whenever there's a non-quote, non-whitespace character adjacent (left or right) of a shell variable name, it would be nice to warn the user to use curly braces.

#### <img src="https://avatars.githubusercontent.com/u/10757176?v=4" width="50">[mgutt](https://github.com/mgutt) commented at [2023-04-05T12:20:06Z](https://github.com/koalaman/shellcheck/issues/346#issuecomment-1497395265):

> E.g., `pgrep "test*$DATABASE*logger"` may fail, whereas `pgrep "test*${DATABASE}*logger"` works.

Only a comment as I can't leave this statement: This is false. There is no "may". It will never fail, as "*" is not a valid character for a variable name. Only alphanumeric and underscore are allowed as characters. So the only situation, which can be a problem is the following:

```
foo="bar"
echo "$foo_baz"
```

But in this case Shellsheck will warn you as $foo_baz is not set (SC2154).
