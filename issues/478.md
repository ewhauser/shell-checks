#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-09-27T21:38:34Z](https://github.com/koalaman/shellcheck/issues/478#issuecomment-143596696):

shellcheck is wrong there but only because the quoting inside backticks is odd. If that used `$(...)` substitution instead then shellcheck would be right that the escaped double quotes would not operate as word quotes and the warning would be correct. The difference is that backticks unescape the inner double quotes. Though I can't actually find the spec language that says that it should do that. A quick read makes it look like it shouldn't to me (or I'm misreading things).


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-09-28T21:29:07Z](https://github.com/koalaman/shellcheck/issues/478#issuecomment-143879606):

I see what happened. I read this part:

> backslash shall retain its literal meaning, except when followed by: '$', '`', or '\' (dollar sign, backquote, backslash). The search for the matching backquote shall be satisfied by the first backquote found without a preceding backslash

Which says that `\"` should be treated literally like shellcheck does.

However, it is in the context of this rule of Token Recognition:

> 1. If the current character is backslash, single-quote, or double-quote ( '\', '", or ' )' and it is not quoted, it shall affect quoting for subsequent characters up to the end of the quoted text.

So it is indeed a valid parsing bug.
