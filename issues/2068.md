#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/env bats

@test xexe {
	ECHO_BIN=/bin/echo
	ECHO_FLAGS=-e
	ECHO="$ECHO_BIN $ECHO_FLAGS"

	run $ECHO '\t\twow\t\towo'
	false
}
```

#### Here's what shellcheck currently says:

```
In a.bats line 8:
	run $ECHO '\t\twow\t\towo'
            ^---^ SC2086: Double quote to prevent globbing and word splitting.

Did you mean: 
	run "$ECHO" '\t\twow\t\towo'

For more information:
  https://www.shellcheck.net/wiki/SC2086 -- Double quote to prevent globbing ...
```

#### Here's what I wanted or expected to see:

No warning.

If I run the same code in pure bash:

```sh
#!/bin/env bash

ECHO_BIN=/bin/echo
ECHO_FLAGS=-e
ECHO="$ECHO_BIN $ECHO_FLAGS"

$ECHO '\t\twow\t\towo'
```

I have no warnings. Although if I add `exec` it appears:

```
In ./a.bash line 7:
exec $ECHO '\t\twow\t\towo'
     ^---^ SC2086: Double quote to prevent globbing and word splitting.
```

It looks like `run` (which is a built-in bats feature to execute a command, placing its stdout+stderr to `$output` and its exit status to `$status`) is not recognized.

Maybe I'm wrong here but it seems beneficial to support some reserved words (`exec` for bash and bats, and `run` for bats) to decrease the number of bogus warnings.
