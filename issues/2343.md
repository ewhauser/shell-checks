It looks like the variable is not being expanded before trying to open

```sh
~/tmp» pwd
/home/alex/tmp
~/tmp» <include.sh
#!/bin/sh

. "${HOME}/tmp/common.sh"
~/tmp» <common.sh
#!/bin/sh

printf 'hello, I am sourced\n'
~/tmp» shellcheck -x include.sh

In include.sh line 3:
. "${HOME}/tmp/common.sh"
  ^---------------------^ SC1091: Not following: ./tmp/common.sh: openBinaryFile: does not exist (No such file or directory)

For more information:
  https://www.shellcheck.net/wiki/SC1091 -- Not following: ./tmp/common.sh: o...
~/tmp» ./include.sh # This definitely works
hello, I am sourced
~/tmp»
```

Same thing happens with bash and using `source` keyword

I suspect some extra checks would need to be added [here](https://github.com/koalaman/shellcheck/blob/fac97a5301b39306e6f26cb6d4fadbbaa4e6f8b0/shellcheck.hs#L504)?

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-10-07 21:26](https://github.com/koalaman/shellcheck/issues/2343#issuecomment-938167836):

It is indeed not being expanded, and ShellCheck intentionally does not support this. You should have gotten a more helpful [SC1090](https://github.com/koalaman/shellcheck/wiki/SC1090) stating this instead.

However, ShellCheck tried to be helpful and strip one leading dynamic path element before looking up the rest in the current dir and `source-path`, which gave this less useful suggestion.

This should have an explicit mention or separate warning explaining this. In the mean time I updated the wiki page.

`~` and `$HOME` have been discussed as a possible special case, but in general ShellCheck avoids dependencies on the environment so that check results don't diverge between machines. It's also hard when variables are defined or modified in the script, such as `foo=$(ssh config.example.com 'cat /etc/scriptdir'); source "$foo/config"`.
