#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2021-10-04 10:43](https://github.com/koalaman/shellcheck/issues/2345#issuecomment-933361413):

ShellCheck is correct to issue a warning, but I agree that the choice of warning is quite misleading.

What it's really complaining about is that the `[indexes]` glob contains the same letter multiple times, which is a common error when people think that `[word]` will optionally a whole word, like `?(word)`.

There is a common misconception that commands such as `read` and `unset` are exempt from the usual handling of command arguments. This is simply not true; they are indeed subject to globbing. The effect is normally hidden from view because the default behaviour is to leave unchanged any glob that does not match any files.

This can affect you in three ways:

1. If the subscripted variable taken as a glob actually matches one or more files, then those filenames _will_ be substituted:
   ```
   touch foo0
   foo=( abc def )
   read -r foo[0] <<<xyz
   declare -p ${!foo@}
   ```
   outputs:
   > `declare -a foo='([0]="abc" [1]="def")'`
   > `declare -- foo0="xyz"`
2. If `nullglob` is active, then additionally, any glob that _doesn't_ match a file will simply be ignored:
   ```
   rm -f foo1
   shopt -s nullglob
   foo=( abc def )
   unset foo[1]
   declare -p ${!foo@}
   ```
   outputs:
   > `declare -a foo='([0]="abc" [1]="def")'`
3. Even if you can be certain that `nullglob` is off _and_ that there are no matching files, there's still a less obvious effect: it makes your script slower:
   ```lang=bash
   $ time for ((i=0;i<10000;++i)) do unset "x[0]" ; done
   ```
   > `real    0m0.179s`
   > `user    0m0.120s`
   > `sys     0m0.052s`
   ```lang=bash
   $ time for ((i=0;i<10000;++i)) do unset x[0] ; done
   ```
   > `real    0m0.756s`
   > `user    0m0.248s`
   > `sys     0m0.492s`
