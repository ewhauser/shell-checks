#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2022-10-14 04:57](https://github.com/koalaman/shellcheck/issues/2584#issuecomment-1278483581):

ShellCheck generally tries to avoid supporting especially restricted cases because it tends to overpromise on what it can do, and it's often better to be consistent than helpful but unpredictable. This is e.g. why `eval 'x=1'` is treated as a black box like all other `eval` statements, even though it would be easy to treat it the same as `x=1`. 

However, I really like this idea, because traps are usually less dynamic, bad `trap`s might not be hit during testing (for signals), and they will not be found by `bash -n` or similar.

I implemented it and ran it on a corpus, but only found 1 case in 20,000 (most checks trigger on 1 in 1000 or so). This combined with the odd behavior that you get parsing warnings only, e.g. missing one quote but not two, and that it stops emitting warnings entirely if any part of the string is dynamic makes it more odd than useful, so I won't push it. 

Great suggestion though, thanks!

PS: A function `cleanup() { ..; }` with a `trap cleanup ..` will get you syntax highlighting and checking and everything today. Maybe ShellCheck should suggest that for overly complicated traps.
