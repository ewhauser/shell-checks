```sh
#!/bin/bash

# $1: message, $2: allowed characters
userinput_char() {
  local char
  while read -e -t 0.01; do :; done
  while [[ $char != ["$2"] ]]; do
    read -p "$1 " -n 1 char
  done
  echo "$char"
}

char=$(userinput_char "Are you sure? [Y]es/[N]o" yn)
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2022-01-10 00:57](https://github.com/koalaman/shellcheck/issues/2165#issuecomment-1008464512):

This was caused by ShellCheck's overzealous detection of newbie mistakes like `if [[ $var == foo]]`. It now avoids doing this when the word contains a `[`, which will hopefully avoid this egregious error while also remaining helpful. Thanks!

In affected version, one possible (if ugly) workaround is `[[ $char != ["$2"]'' ]]`. The trailing empty quoted string causes the word not to syntactically end with an unquoted `]` and therefore avoids triggering.
