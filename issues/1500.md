# [\#1500 Issue](https://github.com/koalaman/shellcheck/issues/1500) `closed`: SC2154 triggers incorrectly on ${!prefix@} expansions

#### <img src="https://avatars.githubusercontent.com/u/961548?v=4" width="50">[ribbons](https://github.com/ribbons) opened issue at [2019-02-24T14:01:31Z](https://github.com/koalaman/shellcheck/issues/1500):

- Rule Id: SC2154
- My shellcheck version: 0.6.0
- [x] The rule's wiki page does not already cover this at https://shellcheck.net/wiki/SC2154
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit

This is a very similar issue to #443, just with `@` rather than `*`

#### Here's a snippet that shows the problem:

```sh
#!/bin/bash

# shellcheck disable=SC2034
osfoo=osfoov osbar=osbarv osblah=osblahv

echo "${!os@}"
```

#### Here's what shellcheck currently says:

```
Line 6:
echo "${!os@}"
      ^-- SC2154: os is referenced but not assigned.
```

#### Here's what I wanted or expected to see:

No warnings

#### <img src="https://avatars.githubusercontent.com/u/2210852?v=4" width="50">[balanco](https://github.com/balanco) commented at [2021-07-03T14:52:21Z](https://github.com/koalaman/shellcheck/issues/1500#issuecomment-873418723):

This is a kind reminder that the flagged `${!variablename_prefix@}` issue persists as of shellcheck v0.7.1 (as opposed to the fixed `${!variablename_prefix*}`  #443).
A different use case is the ensuring that any variables starting with a common name prefix are unset
(say, temporary variables with names starting with 'tmp_', i.e.: 'tmp_bla1', 'tmp_bla2', etc.):

`unset "${!tmp_@}"        # note: the builtin 'unset' does not fail if no argument is supplied`

is likewise incorrectly diagnosed (if "just" as a warning): 

`SC2154 tmp_ is referenced but not assigned.`

(the workaround of explicitly shellcheck-commenting the code line of course works, but.)
