#### <img src="https://avatars.githubusercontent.com/u/8521043?u=ea3b2ef3d02e9eef79e0c7141a47664c5e2d8fcd&v=4" width="50">[bewuethr](https://github.com/bewuethr) commented at [2020-02-16 20:15](https://github.com/koalaman/shellcheck/issues/1799#issuecomment-586747812):

Sorry, didn't give any context. [Ale](https://github.com/dense-analysis/ale) is a Vim plugin that lets you asynchronously run linters. For shell scripts, it defaults to shellcheck and `bash -n` for linting. `-n` is described like this in the manual:

> Read commands but do not execute them. This may be used to check a script for syntax errors. This option is ignored by interactive shells.

So, for a minimal script with an extended "except" glob

```console
$ cat poc
#!/usr/bin/env bash

shopt -s extglob
echo !(foo)
```

shellcheck doesn't complain, but `bash -n` does, with the exact error message you're mentioning above:

```console
$ shellcheck poc
$ bash -n poc
poc: line 4: syntax error near unexpected token `('
poc: line 4: `echo !(foo)'
```

That's why I'm thinking that this isn't a shellcheck message. The shellcheck codebase doesn't include "unexpected token" anywhere, as far as I can tell.
