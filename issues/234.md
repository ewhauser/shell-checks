# [\#234 Issue](https://github.com/koalaman/shellcheck/issues/234) `closed`: Don't warn for unprotected array appending

#### <img src="https://avatars.githubusercontent.com/u/217543?v=4" width="50">[DamienCassou](https://github.com/DamienCassou) opened issue at [2014-10-16T13:20:33Z](https://github.com/koalaman/shellcheck/issues/234):

``` bash
declare -a files
files+=($(basename "$file"))
files+=( $file )
```

The 2 last lines are problematic but shellcheck won't tell me.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2014-10-19T03:20:27Z](https://github.com/koalaman/shellcheck/issues/234#issuecomment-59638004):

I definitely agree, both for assignment and appending.

Unfortunately it's common and not excessively wrong to do things like 

```
a=( $(awk -F, '$1 == "foo" { print $2 }' file.csv) )
```

I feel like having to add a directive for this is a hassle. Are there any pragmatic solutions that will allow people to split contents naturally while still warning for bad uses? Maybe ensuring that `IFS` and/or `set -f` is explicitly set first?
