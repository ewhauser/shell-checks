#### Problem description
It seems that when using `--format=gcc` `style` related feedback is marked as `note` whereas using `--format=json` `style` related feedback is correctly marked with the appropriate severity level `style`. Looking at [GCC.hs](https://github.com/koalaman/shellcheck/blob/master/src/ShellCheck/Formatter/GCC.hs#L58-L61) it seems that there is no matching for `style` related messages and everything that is not an `error` or `warning` is categorized as `note`. Not sure though if this is intentional or a bug.

In this specific scenario i'm hooking up shellcheck with `efm-langserver` that AFAIK only understands `gcc` formatting and therefore cannot identify `style` related feedback properly. I have explicitly configured `diagnostic-languageserver` for shellcheck as it does understand `json` formatting and therefore properly identifies `style` related feedback. I'm only using `diagnostic-languageserver` as a workaround for this issue but would get rid of that workaround if `gcc` style formatting would mark `style` messages properly.

It was intentional since these were the only three levels gcc 4 used back then, and ShellCheck's four levels had to be mapped onto them somehow. I know GCC output has changed quite a bit since then. Is there a better way to map it today?

Hmm, not sure how to map a more fitting output to current days GCC standards to be honest.

Perhaps it would make more sense introducing a new (GCC-like) formatting output (i.e. gcc1, or a completely different name) that outputs all 4 levels (error, warning, information, hint, matching up with Neovim LSP category naming) ? This way current GCC formatting stays put not breaking existing workflows while opening up the possibility of using linters that are not capable of processing Shellcheck's JSON output.
