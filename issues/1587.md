# [\#1587 Issue](https://github.com/koalaman/shellcheck/issues/1587) `open`: SC2080: Don't throw on single quotes (literal value)

#### For bugs
- Rule Id: [SC2080](https://github.com/koalaman/shellcheck/wiki/SC2088)
- My shellcheck version:  0.6.0
- [X] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2088)
- [X] I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash
# Use diff-so-fancy (if available)
if [ -x "$(command -v diff-so-fancy)" ]; then
  git config --global include.path '~/.dotfiles/git/git_diffsofancy'
else # Unset gitconfig change (if exists)
  git config --global --unset include.path "~/.dotfiles/git/git_diffsofancy"
fi
```

#### Here's what shellcheck currently says:

> Line 4:
>   git config --global include.path '~/.dotfiles/git/git_diffsofancy'
>                                    ^-- SC2088: Tilde does not expand in quotes. Use $HOME.
>  
> Line 6:
>   git config --global --unset include.path "~/.dotfiles/git/git_diffsofancy"
>                                             ^-- SC2088: Tilde does not expand in quotes. Use $HOME.


#### Here's what I wanted or expected to see:
No errors, tilde is within single quotes as a literal value.

_Why I want to do this (my case):_  I reference the literal string later on (in the else section) - the expanded value of `$HOME` would be needlessly complex to reference. 

If I replace `~` with `$HOME` in single quotes, [SC2016](https://github.com/koalaman/shellcheck/wiki/SC2016) fires, which is the expected behavior if I attempted to use an expression within single quotes.

_Why I want to do this (conceptually):_ It seems SC2088 should respect literal string designation ala single quotes, I would presume to see SC2088 only fire on double quotes. e.g. `'~/pony/isawesome$!@^%dj' > file` should be completely valid, no? 

With safeguards (such as SC2016), single quoted strings containing tilde seem perfectly legal, since appropriate alert fires when accidentally attempting to pass expression in literal string.

I ran into this issue with something like the following:

```bash
#!/usr/bin/env bash
echo '~/ or $HOME/ can be used instead of typing /home/user/.'
```
With `echo`, I can see a use case where '~' would be preferable.  Take a bug report script where you don't want to leak any personal identifiable information that might be in `$USER`.
