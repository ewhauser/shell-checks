#### Here's a snippet or screenshot that shows the problem:

```bash

$ shellcheck -s bash -f diff - <<'EOF'
files='file1 file2'
combined_file=`cat ${files}`
echo "${combined_file}"
EOF
--- a/-
+++ b/-
@@ -1,3 +1,3 @@
 files='file1 file2'
-combined_file=`cat ${files}`
+combined_file=`cat "${files}"`
 echo "${combined_file}"

```

#### Here's what shellcheck currently says:

Currently the diff format does not give any indication of which shellcheck codes
are being fixed. The diff format also does not fix all autofix issues in a single pass when 
there are multiple shellcheck issues on a single line.

#### Here's what I wanted or expected to see:

I am very excited about the new autofix shellcheck capability, but I find the ergonomics of using the feature a bit rough. My preferred workflow would be:

1. Shellcheck provides a list of issues which can be autofixed
2. I run shellcheck against a repo for a single issue, say SC2006
3. I review the diff and pull request the change

This can be accomplished now with a bit of work:

1. `shellcheck -f json test.sh | jq '.[]|.code'`
2. `shellcheck -f diff -i SC2006 | git apply`
3. `git commit -a -m 'Fixes SC2006'`

I am not sure of the best way to make the ergonomics of the diff format better, one possibility would be to add a `codes` format, which would provide an easy way to know which codes can be autofixed in a repo, something like:

```bash
$ shellcheck -s bash -f codes - <<'EOF'
files='file1 file2'
combined_file=`cat ${files}`
echo "${combined_file}"
EOF                    
SC2006 (autofix)
SC2086 (autofix)
```

I could then use that output format to understand which issues could be autofixed in a repo. Then I could use shellcheck to autofix each issue separately and pull request each related change.
