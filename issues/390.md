# [\#390 Issue](https://github.com/koalaman/shellcheck/issues/390) `closed`: No warning for unescaped newlines in $() calls

#### Labels: wontfix

#### <img src="https://avatars.githubusercontent.com/u/8690113?v=4" width="50">[refola](https://github.com/refola) opened issue at [2015-06-15T08:20:08Z](https://github.com/koalaman/shellcheck/issues/390):

This code is invalid.

``` sh
#!/bin/sh
f() { echo "$@"; }
x=$(f arg1
    arg2)
echo "$x"
```

It looks like it's meant to set `x` to the output of `f arg1 arg2`, but the newline between `arg1` and `arg2` causes a `command not found` error for `arg2`. Shellcheck doesn't catch this.

The fix is simple: Add a `\` (space, backslash) to the end of `arg1` (generally, every line before the closing `)`), allowing the multi-line `$()` call to continue as intended.

The fix then becomes requiring either a `\` or a `;` at the end of each line in a `$()` expression. But at that point you might as well do the same everywhere else, which would probably cause more annoyance than it would solve.

Bash already gives a usable error message when this happens, and "fixing" it in Shellcheck would require more annoyance than it's worth. Can you close this as "invalid"/"wontfix" or similar?
