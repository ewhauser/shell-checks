# [\#302 Issue](https://github.com/koalaman/shellcheck/issues/302) `closed`: SC2012: `find` is not an adequate replacement for `ls -ld` piped to `cut`

#### <img src="https://avatars.githubusercontent.com/u/572196?v=4" width="50">[stuartpb](https://github.com/stuartpb) opened issue at [2015-02-06T02:12:45Z](https://github.com/koalaman/shellcheck/issues/302):

See https://github.com/plushu/plushu/blob/master/bin/plushu#L20:

``` bash
    PLUSHU_USER=`ls -ld "$PLUSHU_ROOT" | cut -d' ' -f3`
```

This is to determine the owner of a directory - it cannot be determined using `find`, and will have no problem with paths with filenames.

On Thu, Feb 05, 2015 at 06:12:46PM -0800, Stuart P. Bentley wrote:

> See https://github.com/plushu/plushu/blob/master/bin/plushu#L20:
> 
> ``` bash
>     PLUSHU_USER=`ls -ld "$PLUSHU_ROOT" | cut -d' ' -f3`
> ```
> 
> This is to determine the owner of a directory - it cannot be determined using `find`, and will have no problem with paths with filenames.
> 'ls' use is to be avoided, in this case not because of the file name, but
> because its output isn't stable across multiple UNIX variants:

openbsd
 $ ls -ld
 drwxr-xr-x  3 dualbus  dualbus  512 Jan 24 14:35 .
 $ ls -ld|cut -d' ' -f3
 3

netbsd
 $ ls -ld
 drwx------  9 dualbus  arpa  512 Feb  6 06:36 .
 $ ls -ld|cut -d' ' -f3
 9

linux:
 dualbus@hp ~ % ls -ld
 drwxr-xr-x 57 dualbus dualbus 20480 Feb  6 00:37 .
 dualbus@hp ~ % ls -ld|cut -d' ' -f3
 dualbus

In this case it'd seem that you can get away with it if you change cut with
awk '{print $3}'

But in general, ls output is to be used interactively only. It's not stable
enough for use in scripts, and it's better if shellcheck keeps recommending
that.

I'd say this rule should generally be amended to recommend `ls -l | awk '{print $3}'` (or whatever field number) over piping to `cut` or using `stat`, since `ls -l` output field order is more stable cross-platform than `stat` options.
