# [\#590 Issue](https://github.com/koalaman/shellcheck/issues/590) `closed`: ((expression)) at the start of a $(expression) is miss-parsed

#### <img src="https://avatars.githubusercontent.com/u/1885701?v=4" width="50">[dimo414](https://github.com/dimo414) opened issue at [2016-01-31T23:30:05Z](https://github.com/koalaman/shellcheck/issues/590):

For example:

``` shell
$((( $exit_code == 0 )) && echo GREEN || echo RED)
```

This reports a "SC1073 Couldn't parse this $((..)) expression." error, but this evaluates just fine in Bash:

``` shell
$ exit_code=0
$ echo $((( $exit_code == 0 )) && echo GREEN || echo RED)
GREEN
$ exit_code=1
$ echo $((( $exit_code == 0 )) && echo GREEN || echo RED)
RED
```

If that works then it's a bug in bash; it should use "longest match wins"

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-02-01T00:47:42Z](https://github.com/koalaman/shellcheck/issues/590#issuecomment-177677270):

OK, apparently it's not that simple.

Later versions of Bash have some "smart matching" logic where it counts the
open brackets in the whole structure and checks for a matching closing ))
before deciding whether it's an expression or sub-shell.

Personally I don't think it's a good idea to rely on that;
- it's not portable to older versions of Bash
- it's tantamount to experimental, as there are still corner cases that may break

I reported it as a [bug](https://lists.gnu.org/archive/html/bug-bash/2016-01/msg00173.html), and it sounds like POSIX allows this behavior, though it's undefined. It might not be _good_ syntax, but it seems like it shouldn't fail the ShellCheck parser.


#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2016-02-01T14:23:46Z](https://github.com/koalaman/shellcheck/issues/590#issuecomment-177991912):

Yeah, this is an unfortunate ugliness inherent to shell syntax but it is valid by spec (bottom of [section 2.6.3 Command Substitution](http://pubs.opengroup.org/onlinepubs/9699919799/utilities/V3_chap02.html#tag_18_06_03) though the specific quoted text is new to the 2013 edition of the spec the ambiguity is noted in the 2004 edition as well).

That said counting on it working for you is a bit of a gamble since you are playing with what the shell thinks it can parse as an arithmetic expansion versus what it can't. So this should probably be a warning.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-02-07T06:35:13Z](https://github.com/koalaman/shellcheck/issues/590#issuecomment-180962306):

Any `(((` was a bit noisy and most shells would probably choose to interpret it as `(( (`, so 825c1b5 adds a warning when it figures that it was supposed to be `( ((`. 

Optimally this should also have handled e.g. `$((foo) )`, but frequent missing `)`s caused it to mistrigger.
