#### <img src="https://avatars.githubusercontent.com/u/54114043?v=4" width="50">[dirk-seynhaeve](https://github.com/dirk-seynhaeve) commented at [2019-10-01T12:32:16Z](https://github.com/koalaman/shellcheck/issues/1710#issuecomment-537014413):

It looks like Bash itself is confused about this array:
When not declaring it as an (associative) array, we get:
```
$ numbers[one]=1
$ echo "${!numbers[@]}"
0
```
... the key of numbers is not the string, but an index?!

When trying to declare it as an associative array, we get a telling error message:
```
$ declare -A numbers
-bash: declare: numbers: cannot convert indexed to associative array
```
...indeed indicating that the numbers array is an INDEXED array...

So working around that:
```
$ unset numbers
$ declare -A numbers
$ numbers[one]=1
$ echo "${!numbers[@]}"
one
```
... the key is now correctly identified as the string; So let's try the final experiment:
```
#!/bin/bash
unset numbers
declare -A numbers
numbers[one]=1
echo "${!numbers[@]}"
echo $((numbers[one]+1))
```
...shellcheck now declares:
No issues detected!

Interesting, but since I always declare all my variables (just like I use 'strict' in Perl programming), I had not bumped into this!

#### <img src="https://avatars.githubusercontent.com/u/54114043?v=4" width="50">[dirk-seynhaeve](https://github.com/dirk-seynhaeve) commented at [2019-10-01T12:38:14Z](https://github.com/koalaman/shellcheck/issues/1710#issuecomment-537016641):

I think it is related to how you initialize an associative array; The following script (without any declaration), also reports no issues in shellcheck:
```
#!/bin/bash
numbers=([one]=1)
echo "${!numbers[@]}"
echo $((numbers[one]+1))
```

The Bash reference is a bit vague on this:
https://www.gnu.org/software/bash/manual/html_node/Arrays.html

>  Any variable may be used as an indexed array
> ...
> Associative arrays are created using
>   declare -A name

You could read this as:

> Any variable is not just usable as an associative array
> ...
> You only get an associative array if you declare it

But we have already shown that you don't need to declare it, if only you initialize it differently:
```
numbers=([one]=1)
```
And with that in mind, the message reported in the initial issue, shellcheck is not wrong? But maybe we should figure out what POSIX says about all of this, and have a rule that detects undeclared or wrongly initialized associative arrays...

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-10-03T05:41:46Z](https://github.com/koalaman/shellcheck/issues/1710#issuecomment-537796171):

ShellCheck is right: indices in arrays not declared associative are evaluated in an arithmetic context and not as strings. You can see that `one` *is* being treated as a variable by bash itself by setting it:

```
#!/bin/bash
one=42
numbers[one]=1
declare -p numbers
```

will output

```
declare -a numbers=([42]="1")
```

The fact that `numbers=([one]=1)` doesn't show a warning is an oversight in ShellCheck. It has the same problem. If you want `one` to be treated as a string, you have to `declare -A numbers` first.
