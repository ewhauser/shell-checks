# [\#2786 Issue](https://github.com/koalaman/shellcheck/issues/2786) `open`: Request: Error for ${} around a file path with multiple variables

- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

In Bash, having `${}` around a file path with multiple variables and an expression (like `version/dir`) can cause massive consequences, as mentioned here (tldr: every Blender addon was deleted for every version of Blender): https://github.com/TheDuckCow/MCprep/pull/422#issuecomment-1606525786

#### Here's a snippet or screenshot that shows the problem:

```sh

rm -rf "${i/$NAME:?}/"

```

#### Here's what shellcheck currently says:
Nothing. Ironically the example line was to fix another shellcheck error related to not checking if a variable is null


#### Here's what I wanted or expected to see:
An error saying something like "`${}` should be around each individual variable"  



#### <img src="https://avatars.githubusercontent.com/u/84648683?v=4" width="50">[wileyhy](https://github.com/wileyhy) commented at [2023-06-26 15:01](https://github.com/koalaman/shellcheck/issues/2786#issuecomment-1607670742):

If I've read this report correctly, if the original string was, "${i/$NAME:?}/", then to be a valid parameter expansion, it was just missing a second set of inner curly braces: 

$ `i=12345; NAME=45; echo "${i/$NAME:?}/"`
12345/
$ `i=12345; NAME=45; echo "${i/${NAME:?}}/"`
123/

How was it intended to be expanded, though?  In bash, every parameter expansion requires its own set of curly braces. 

From `man bash`, Parameter Expansions on ${parameter/pattern/string}, first paragraph: "If string is null, matches of pattern are deleted and the / following pattern may be omitted."

#### <img src="https://avatars.githubusercontent.com/u/84648683?v=4" width="50">[wileyhy](https://github.com/wileyhy) commented at [2023-06-26 20:31](https://github.com/koalaman/shellcheck/issues/2786#issuecomment-1608202047):

Nah, parameter expansions are tricky.

"${i}/${NAME}"

...is what you were looking for, then. Alternatively, you could use...

"${i:?}/${NAME:?}"/

...with a trailing backslash and parameter expansions that will halt the
script if a variable is unset or null. ...however in POSIX mode, the shell
will not exit. A custom error message can be included at 'word' with this
syntax:

"${parameter:?word}"
