# [\#2007 Issue](https://github.com/koalaman/shellcheck/issues/2007) `closed`: SC2183 only considers placeholders before the first real newline in format string

#### <img src="https://avatars.githubusercontent.com/u/29466521?v=4" width="50">[nwwdles](https://github.com/nwwdles) opened issue at [2020-07-03 22:48](https://github.com/koalaman/shellcheck/issues/2007):

Hey! Thanks for this great piece of software, it really helps me a lot.

I had quite a long format string so I was using actual newlines instead of `\n`. I noticed that shellcheck issued a warning after I simply removed a newline from a format string.

#### For bugs
- Rule Id: `SC2183`
- My shellcheck version: `0.7.1` and online
- [x] The rule's wiki page does not already cover this (https://shellcheck.net/wiki/SC2183)
- [x] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

#### Here's a snippet or screenshot that shows the problem:

Case 1:

```sh
#!/bin/sh

printf '%s
%s' abc
```

Case 2:

```sh
#!/bin/sh

printf 'this way too
%s%s
%s' abc
```

#### Here's what shellcheck currently says:

Case 1: nothing
Case 2: `SC2183: This format string has 2 variables, but is passed 1 arguments.`

#### Here's what I wanted or expected to see:

Case 1: `SC2183: This format string has 2 variables, but is passed 1 arguments.`
Case 2 : `SC2183: This format string has 3 variables, but is passed 1 arguments.`



#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2020-07-26 00:41](https://github.com/koalaman/shellcheck/issues/2007#issuecomment-663921442):

Nice catch! This was due to ShellCheck's usage of regex to match formats like `%3.2s`, which included a `.*` at the end to get the rest of the string for further processing. However, the regex engine (Text.Regex.TDFA) stopped matching `.*` at line boundaries, effectively ignoring anything after the first literal linefeed. This has been fixed by explicitly matching `\n` along with `.`. Thanks!
