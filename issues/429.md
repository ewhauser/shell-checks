# [\#429 Issue](https://github.com/koalaman/shellcheck/issues/429) `open`: warning against piping to head

#### <img src="https://avatars.githubusercontent.com/u/7633626?v=4" width="50">[aw-was-here](https://github.com/aw-was-here) opened issue at [2015-08-07T16:22:05Z](https://github.com/koalaman/shellcheck/issues/429):

On some OSes/certain situations, head may exit early if they have enough data to fulfill the request. This can result in spurious errors from whatever program is feeding it through a pipe.  While not fatal/harmful, it can be confusing.

#### <img src="https://avatars.githubusercontent.com/u/7633626?v=4" width="50">[aw-was-here](https://github.com/aw-was-here) commented at [2015-08-07T17:54:57Z](https://github.com/koalaman/shellcheck/issues/429#issuecomment-128778122):

This stack overflow question has a much better answer about this particular issue than I could give:
http://superuser.com/questions/554855/how-can-i-fix-a-broken-pipe-error

As you can see, it's hard building a reproducible case.  It's also worth noting that it's the process that feeds the pipe that gets the error, not the head or tail that is reading from the pipe.

In my own code, I've hit it twice in the past year, the last one was a relatively simple construction:

``` bash
tail -n "+${startline}" "${file}"  | head -n ${numlines} > "${outfile}"
```

where startline is generally a large number and numlines is a small number.  On OS X w/bash 3.x, it never throws an error (or, at least, I've never hit it).  On Ubuntu w/bash 4.x, I've seen tail throw out the broken pipe write error a few times.

We fixed it in sort of a bad way:

``` bash
tail -n "+${startline}" "${file}" 2>/dev/null | head -n ${numlines} > "${outfile}"
```

since throwing errors away is generally bad.  (But given the code surrounding this, the risk is primarily if the system is falling over already.... out of mem, etc.)

(The one I hit today which reminded me to file this issue is a little more complex. hah)
