In the examples you provide, disabling SC1064 just leaves SC1073, which says it can't parse the function.  If I also disable SC1073 (which seems overkill), then I end up with only SC1072, which is not helpful.

It seems to me that Shellcheck doesn't just check whether a function as curly braces, it simply cannot process them if they don't.  That seems wrong to me.  It should be able to parse correct, albeit non-recommended syntax.

The man page describes a function definition as such:

    fname () compound-command [redirection]

Where `compound-command` is any of:

  - `(list)`
  - `{ list; }`
  - `((expression))`
  - `[[ expression ]]`
  - `for name [ [ in [ word ... ] ] ; ] do list ; done`
  - `for (( expr1 ; expr2 ; expr3 )) ; do list ; done`
  - `select name [ in word ] ; do list ; done`
  - `case word in [ [(] pattern [ | pattern ] ... ) list ;; ] ... esac`
  - `if list; then list; [ elif list; then list; ] ... [ else list; ] fi`
  - `while list-1; do list-2; done`
  - `until list-1; do list-2; done`

Restricting the Shellcheck parser to only two possibilies, `{ list; }` and `(list)`, is not acceptable IMHO.

https://github.com/koalaman/shellcheck/blob/2a95bc6be3c9fe411e1ddb96e78b467ef2347bdb/src/ShellCheck/Parser.hs#L2753

SC1064 is fine as a shellcheck "error", but it shouldn't prevent parsing.
