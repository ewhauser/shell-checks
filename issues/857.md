#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-03-26T15:54:18Z](https://github.com/koalaman/shellcheck/issues/857#issuecomment-289293006):

Directives actually apply to the command or structure they're placed in front of, and not the command list they're a part of.

In this case you can apply it to the whole function:

```
# shellcheck disable=SC2154
removeFile()
{
        local file="$1"
        ${rm_bin} "$file"
}
```

Or just the particular command:
```
removeFile()
{
        local file="$1"
        # shellcheck disable=SC2154
        ${rm_bin} "$file"
}
```

Directives at the start of the file are the exception, and will indeed apply to the whole script and not just the next command. I see how that's inconsistent and confusing.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-03-29T02:52:05Z](https://github.com/koalaman/shellcheck/issues/857#issuecomment-289966442):


Have more trust in Shellcheck; it tells you these things for a reason.

> removeFile()
> {
>         # shellcheck disable=SC2154
>         local file="$1"
>         ${rm_bin} "$file"
> }

One of the nice uses for shell functions is to *entirely avoid* using variables in contexts like this; rather, simple define the function in different ways depending on what behaviour you want to select, for example:

    if [[ -x /path/to/MySpecialRm ]] ; then
        removeFile() { /path/to/MySpecialRm "$@" ; }
    else
        removeFile() { /bin/rm -f "$@" ; }  # POSIX says `rm` has to be here
    fi

(I can't tell from the given code, but there is no point writing `rm_bin=$(which rm) ; ${rm_bin}` because that has the same effect as just writing `rm`, only slower.)

> checkTrailingSlash()
> {
>         #shellcheck disable=SC1001
>
>         local path="${1:-}"
>
>         if [[ "${path}" =~ ^(.*)\/$ ]] ;then
>                 return 1
>         else
>                 return 0
>         fi
> }

Slash has never been special to regex; and while it is special to some of the languages which embed regex, Bash is not one of those languages. So rather than disabling SC1001, I would recommend simply following its suggestion and removing the unnecessary backslash before the slash, thus:

    checkTrailingSlash() { [[ ${1:-} !~ ^(.*)/$ ]]
        local path="${1:-}"

        if [[ "${path}" =~ ^(.*)/$ ]] ;then
                return 1
        else
                return 0
        fi
    }

And then I would simplify that to:

    checkTrailingSlash() {
        [[ ${1:-} !~ ^(.*)/$ ]]
    }

And then (assuming you're not examining REMATCH) I'd simplify that even further, to:

    checkTrailingSlash() {
        [[ ${1:-} !~ /$ ]]
    }

But I'd do away with regex entirely and just use a glob:

    checkTrailingSlash() {
        [[ ${1:-} != */ ]]
    }

At which point I'd wonder why I was writing a function at all?
