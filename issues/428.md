# [\#428 Issue](https://github.com/koalaman/shellcheck/issues/428) `open`: SC2098 not always "technically" correct

#### <img src="https://avatars.githubusercontent.com/u/4005180?v=4" width="50">[sitsofe](https://github.com/sitsofe) opened issue at [2015-08-07T14:52:41Z](https://github.com/koalaman/shellcheck/issues/428):

With
```
# !/bin/bash
VAR=aaa VAR=${VAR}:bbb bash -c 'echo ${VAR}'
```

Your result will depend on the version of bash you are running:
bash3:
```
:bbb
```
(Sometimes. Try doing
```
echo $VAR; VAR=aaa VAR=${VAR}:bbb bash -c 'echo ${VAR}'
```
to see a different result on OSX)

bash4:
```
aaa:bbb
```

Either way it's great shellcheck warns about this unstable behaviour!

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-08-17T18:39:16Z](https://github.com/koalaman/shellcheck/issues/428#issuecomment-131920652):

This is very interesting. Run that line in a new bash `3.2.25(1)-release` instance (CentOS 5) and it "works" the first time and "fails" every time after that (see caveats at the end).

Put that line in an executable shell script and it fails every time.

Source that same shell script and it works every time.

Caveats for interactive use. Running anything that accesses the variable and the next run works again. So `VAR=aaa VAR=${VAR}:bbb /usr/bin/printenv VAR; : $VAR; VAR=aaa VAR=${VAR}:bbb /usr/bin/printenv VAR` in a new shell will show both lines "working" and a second run of that will show the first attempt "fail" and the second attempt "work".
