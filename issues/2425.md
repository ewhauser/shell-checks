I use shellcheck primarily with pre-commit; this works well for files I can find by pattern `*.sh` or whatever.

With executable scripts that don't have a file extension though, the only way I could glob them with pre-commit to pass to shellcheck would be as say not containing `.` - but then this may well include other files (`Dockerfile` comes to mind) that are not shell scripts.

I think a good solution to this would be if shellcheck had a flag to ignore the file (not error) if the file doesn't have (or has an unsupported) shebang line.

#### <img src="https://avatars.githubusercontent.com/u/3946014?u=8ffea083b3fa3c3d7d6436f587df5b2e8e2999eb&v=4" width="50">[OJFord](https://github.com/OJFord) commented at [2022-01-18 10:37](https://github.com/koalaman/shellcheck/issues/2425#issuecomment-1015282222):

For the pre-commit case, I stumbled across [its ability to filter files by 'type'](https://pre-commit.com/index.html#filtering-files-with-types) instead of filename regex, and specifically the relatively recent addition of `types_or` to combine them in disjunction.

This means I can use:
```yaml
- id: shell-lint
  args:
    - -x
  types_or:
    - bash
    - sh
    - shell
```

and `pre-commit` takes care of finding anything that ends `.sh`, `.bash` (the `shell` element), or has `sh` or `bash` shebang lines (the first two elements).

Pretty nice, maybe `overcommit` has something similar (or would be interested in adding it)? I'll leave this open though, since although that pretty much solves it for me, it is more generally applicable and could be nice just to run `shellcheck` over an entire directory (possibly not even under SVC) and pick everything up.
