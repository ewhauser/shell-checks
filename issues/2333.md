#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
cat $foo | wc -l

```

#### Here's what shellcheck currently says:

    ^-- SC2002: Useless cat. Consider 'cmd < file | ..' or 'cmd file | ..' instead.


#### Here's what I wanted or expected to see:
Calling cat with multiple files is valid and doesn't get flagged by SC2002.
Calling cat with a variable containing two whitespace delimited filenames could be intended.

In this case, one suggestion would change output and the other would result in an error:
```
#!/bin/bash
files="a b"
cat $files | wc -l
```
Or if possible... disabling SC2086 could disable this warning for cat-with-variable, because if you want to use word splitting after variable evaluation, then you don't want this check?

Awesome tool, btw!

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-09-26 02:52](https://github.com/koalaman/shellcheck/issues/2333#issuecomment-927218999):

You're right, it should offer SC2086 or SC2002 but not both. This is now fixed. Thanks!

There is a common and unfortunate regression in `x=foo.txt; cat $x | ..` where SC2086 is smart enough to realize the variable won't split, but SC2002 is not, thus causing both suggestions to be suppressed. This will require more of a rewrite, but is not a huge concern.
