#### Here's a snippet or screenshot that shows the problem:

```sh

#!/usr/bin/env bash

foo=foo

#shellcheck disable=SC1117
cat >>/tmp/foo <<-EOF
hello\tworld
EOF

echo foo=${foo:-}

```
If you comment out the `foo=foo` then the disable directive applies file wide and shellcheck then ignores the `\t`.

#### Here's what shellcheck currently says:

```
Line 7:
hello\tworld
     ^-- SC1117: Backslash is literal in "\t". Prefer explicit escaping: "\\t".
```

no output

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-01-18T03:29:44Z](https://github.com/koalaman/shellcheck/issues/1071#issuecomment-358526903):

This was an interesting case. Well caught!

Directives are kept on a stack, while here documents are not parsed until the next `\n`. This meant that the directives were already popped by the time it was read. The fix was to save the context from the original command, and reusing it later when reading the here docs.

The same fix improved parser errors since it now correctly remembers which command and here document it's working on, and can point that out on failure.
