# [\#151 Issue](https://github.com/koalaman/shellcheck/issues/151) `closed`: Feature request: bash pitfall: local will overwrite $?

#### <img src="https://avatars.githubusercontent.com/u/29916?v=4" width="50">[pihentagy](https://github.com/pihentagy) opened issue at [2014-05-12T07:58:59Z](https://github.com/koalaman/shellcheck/issues/151):

Shellcheck should warn, if you try to use $? followed by a local assignment.

```
#!/bin/bash

fu() {
  local res=$(false)
  exit $? # will return 0
}

print "$(fu)"
```

#### <img src="https://avatars.githubusercontent.com/u/355889?v=4" width="50">[egor-tensin](https://github.com/egor-tensin) commented at [2017-04-11T01:30:01Z](https://github.com/koalaman/shellcheck/issues/151#issuecomment-293123807):

@koalaman I would like to return to your last comment. I don't think `readonly` should be an exception, because you can in fact assign to a variable, and *then* mark it as `readonly`. Both of these would work:

```
readonly x="$( false )"
```

```
x="$( false )"
readonly x
```

but the exit code of `false` is masked by `readonly` in the first example. Therefore I think that it should work the same way it does for `local`s, only that the `readonly` statement should follow the assignment and not precede it.

I'm not sure this is a proper place to raise this concern; I'll open an issue if I notice this isn't addressed for a long time.
