Dear,

I have a script with a dynamic operator and in such case shellcheck is not able to parse it.
Moreover, I do not find a way to tell to shellcheck to skip this statement and continue.

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bash
foo(){
        local -i val1="${1}"
        local -r sign="${2}"
        local -i val2="${3}"
        # shellcheck disable=SC1009,SC1072,SC1073
        if (( val1 "${sign}" val2 )); then
                echo 'The expression: "'"${val1}"' '"${sign}"' '"${val2}"'" is correct' 
        else
                echo 'The expression: "'"${val1}"' '"${sign}"' '"${val2}"'" is not correct'
        fi
}

foo 1 '>' 2
```

### Here's the shellcheck output

```sh
shellcheck --shell bash --exclude=SC1009,SC1072,SC1073 test_dynamic_sign.sh

In test_dynamic_sign.sh line 7:
        if (( val1 "${sign}" val2 )); then
        ^-- SC1009: The mentioned parser error was in this if expression.
           ^-- SC1073: Couldn't parse this ((..)) command.
                   ^-- SC1072: Expected "))". Fix any mentioned problems and try again.
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-05-04T18:02:45Z](https://github.com/koalaman/shellcheck/issues/1568#issuecomment-489350322):

This particular case is not a problem on later versions, because ShellCheck will successfully parse it as `( ( val1 "${sign}" val2 ) )` when it's unable to treat it as `(( .. ))`.

However, there's still no good solution for the more general problem of what to do when ShellCheck fails to parse valid code.
