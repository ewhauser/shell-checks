I have following sample scripts and shellcheck warns me:

cat config.bash

```
# cat config.bash 
#!/bin/bash -e

export list=(
    'abc'
    '123'
)
```

cat test.bash

```
#!/bin/bash -e

function walkArray()
{
    local -r array=("${@}")

    for element in "${array[@]}"
    do
        echo "${element}"
    done
}

function main()
{
    source "$(dirname "${BASH_SOURCE[0]}")/config.bash"
    walkArray "${list[@]}"
}

main "${@}"

```

```
# shellcheck *.bash

In test.bash line 16:
    walkArray "${list[@]}"
               ^-- SC2154: list is referenced but not assigned.
```

I have tried to resolve the warning by adding `:?` to `${list[@]:?}` but did not work. Any thought how to resolve the warning for array? Is disabling the message with a directive the only way?

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-05-19T00:36:25Z](https://github.com/koalaman/shellcheck/issues/376#issuecomment-103282456):

`source`d includes in are an unsolved problem in ShellCheck. 

I'm guessing the `export` here wasn't necessary, but you could treat the variable as an environment variable and give it an uppercase name. ShellCheck doesn't warn if `LIST` is undefined, since it assumes it comes from the environment. This is annoying if the variable isn't intended for export.

You can also explicitly declare it with `declare list` (before or after the source statement). This will not change the variable, and lets ShellCheck know that the variable is indeed defined. This is annoying because you have to manually declare the variables you use.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-05-19T00:48:55Z](https://github.com/koalaman/shellcheck/issues/376#issuecomment-103284158):

For variables that should be exported, or if you don't mind polluting the environment with them, use `export` and uppercase names.

For variables that should be assigned but not referenced or exported, there's no good way apart from a directive (potentially on a block if there are multiple variables).
