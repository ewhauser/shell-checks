Both UTF-8 and Latin-1 have gaps, so that some bytes or byte-sequences are invalid. (Latin-1 does not use 128-159; UTF-8 requires each lead by to be followed by the correct number of follow bytes.)

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-03-10T17:38:01Z](https://github.com/koalaman/shellcheck/issues/846#issuecomment-285734032):

All byte sequences are valid Latin1. Whether the encoded code points are used for anything doesn't matter. 

ShellCheck also won't reject valid UTF-8 encoded code points that are currently undefined in Unicode.

#### <img src="https://avatars.githubusercontent.com/u/176950?v=4" width="50">[vapier](https://github.com/vapier) commented at [2017-03-10T18:00:37Z](https://github.com/koalaman/shellcheck/issues/846#issuecomment-285739979):

sorry, you're correct of course.  the 0x80 to 0x9F range, while being unprintable, are valid codepoints -- they're the [Latin-1 Supplement](https://en.wikipedia.org/wiki/Latin-1_Supplement_(Unicode_block)) block.

so if shellcheck falls back to that on decode errors, it should be fine.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-03-10T18:21:54Z](https://github.com/koalaman/shellcheck/issues/846#issuecomment-285745330):

Good call with the binary data! Turns out that ShellCheck failed for structurally valid UTF-8 in undefined planes (i.e. characters beyond 0x10FFFF).

This is fixed in 22c8625. ShellCheck will now happily chug through megabytes of binary garbage.
