- version 0.5.0 from repo on WSL 
- [x] The rule's wiki page does not already cover this https://shellcheck.net/wiki/SC2215
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/usr/bin/env bash
find . \                # look in dir
        -name '*.txt'   # for this extension

```

#### Here's what shellcheck currently says:

```
Line 3:
        -name '*.txt'   # for this extension
        ^-- SC2215: This flag is used as a command name. Bad line break or missing [ .. ]?
```

#### Here's what I wanted or expected to see:

I break down long command lines to document individual options. Bash ignores the comment and correctly continues the line from the backslash. Shellcheck should, too.

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2020-07-26 00:50](https://github.com/koalaman/shellcheck/issues/1971#issuecomment-663921964):

ShellCheck is right. Bash does not allow `#` after a `\`. 

Your command is equivalent to `find . " "` which finds all files in the current dir plus an error message, then a second error message `line 3: -name: command not found`
