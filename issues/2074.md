# [\#2074 Issue](https://github.com/koalaman/shellcheck/issues/2074) `open`: SC2005: echo is not useless when reading from a file and overwriting it with the output

#### For bugs
- Rule Id (if any, e.g. SC1000): SC2005
- My shellcheck version (`shellcheck --version` or "online"): 0.7.0
- [x] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2086)
- [x] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

`cat file.txt > file.txt` always results in an empty file. `echo $(cat file.txt) > file.txt` results in the file keeping its contents. But shellcheck says that this is a useless `echo`.

```sh
#!/usr/bin/env bash
$ cd /tmp
$ seq 100 > /tmp/file.txt
$ grep 1 /tmp/file.txt | head -n 3 > /tmp/file.txt 
$ cat /tmp/file.txt 
$ 
$ seq 100 > /tmp/file.txt
$ echo "$(grep 1 /tmp/file.txt | head -n 3)" > /tmp/file.txt 
$ cat /tmp/file.txt 
1
10
11
$
```

#### Here's what shellcheck currently says:

A file with these contents

```sh
#!/usr/bin/env bash
echo "$(grep 1 /tmp/file.txt | head -n 3)" > /tmp/file.txt

```

gives this error

```sh
Line 2:
echo "$(grep 1 /tmp/file.txt | head -n 3)" > /tmp/file.txt
     ^-- SC2005: Useless echo? Instead of 'echo $(cmd)', just use 'cmd'.
```

#### Here's what I wanted or expected to see:

No error, because `echo` has an effect. If you do `cat file.txt > file.txt` the `>` will clear the file before `cat` has a chance to read it. One way to overcome this is to wrapt the part before `>` in `echo`, like this: `echo $(cat file.txt) > file.txt`. But shellcheck will tell you this is a useless echo.

The pattern you are suggesting is problematic in that you still lose all the contents of the file if there is an unexpected process termination or an error in the subshell (which can happen for a variety of reasons). 

But SC2005 is still not the right error for calling out the pitfalls associated with editing files in-place.
