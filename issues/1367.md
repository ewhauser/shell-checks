# [\#1367 Issue](https://github.com/koalaman/shellcheck/issues/1367) `closed`: Can't use `*` in regex

#### <img src="https://avatars.githubusercontent.com/u/21826?v=4" width="50">[ingydotnet](https://github.com/ingydotnet) opened issue at [2018-10-19T19:45:19Z](https://github.com/koalaman/shellcheck/issues/1367):

#### For bugs
- Rule Id 2049
- My shellcheck version (`shellcheck --version` or "online"): 0.4.6 
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit


#### Here's a snippet or screenshot that shows the problem:

```sh
echo 'x="foo*"; [[ $x =~ (o*) ]]' | shellcheck -s bash -
```

#### Here's what shellcheck currently says:
```
In - line 1:
x="foo*"; [[ $x =~ (o*) ]]
                   ^-- SC2049: =~ is for regex. Use == for globs.
```


#### Here's what I wanted or expected to see:

Should be fine to use `*` in a regex. `+` and `?` both work here.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-10-21T22:28:26Z](https://github.com/koalaman/shellcheck/issues/1367#issuecomment-431709767):

The problem with this regex is that it always matches. ShellCheck thought you may have intended to use `o.*` instead (or to treat it as a glob).

This was compounded by some parsing bugs that dropped the `()` , and a missing wiki page, which are now fixed. `(o*)` will not be producing this warning the way just `o*` does.
