#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-02-06T02:30:27Z](https://github.com/koalaman/shellcheck/issues/595#issuecomment-180660668):

Where the shell recognizes a variable assignment as part of syntax, it's
treated a little specially; in particular the RHS is not subject to globbing
or word-splitting.

The issue is that some special built-ins can be followed by assignments that
get the special treatment, whereas "ordinary" built-ins are just text and so
what are intended as assignments get subject to wordsplitting and globbing
before they're recognized as assignments.

And which builtins are "special" in this way varies from shell to shell.
POSIX mandates a particular list, and unfortunately both omitting specified
words and adding extra words would make an implementation non-conforming.

#### <img src="https://avatars.githubusercontent.com/u/6459309?v=4" width="50">[Artoria2e5](https://github.com/Artoria2e5) commented at [2016-02-08T12:10:10Z](https://github.com/koalaman/shellcheck/issues/595#issuecomment-181338781):

This is related to a to-be-published POSIX update. See https://lists.gnu.org/archive/html/bug-bash/2015-12/msg00019.html (points to http://austingroupbugs.net/view.php?id=351).

In short, the new approved POSIX interpretation adds something called an assignment context. Basically, with this update,

``` sh
foo=$bar
export foo
```

is strictly equal to

``` sh
export foo=$bar
# oh, strict? Let's pad the line number
```

.
