# [\#2700 Issue](https://github.com/koalaman/shellcheck/issues/2700) `open`: Error message for each file when shellcheck attempts to write to stdout connected to closed pipe

#### <img src="https://avatars.githubusercontent.com/u/9334?v=4" width="50">[tobbez](https://github.com/tobbez) opened issue at [2023-02-24 19:12](https://github.com/koalaman/shellcheck/issues/2700):

- [X] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related
- shellcheck version: 0.9.0

I would expect shellcheck to exit silently when it fails to write to stdout (because it's connected to a pipe that is now closed). However:

```sh
# shellcheck's output needs to be large enough for the problem to trigger
for i in {1..200}; do echo 'echo $0' >> file.sh; done
# -0 for demonstration  purposes only. normally, this would be non-zero
shellcheck file.sh | head -0
```

prints (on stderr):

```
file.sh: <stdout>: commitBuffer: resource vanished (Broken pipe)
```

The problem is exacerbated by the message being repeated for each input file (that is not shellcheck-clean) processed after the pipe has been closed.

The actual use case is when running shellcheck on a code base with many issues, for example:

```sh
shellcheck --color always *.sh | head -n $((LINES-2))
```

where the error messages would consume a significant number of lines.

There are a couple workarounds (`shellcheck ... 2>/dev/null | ...` or `shellcheck ... | { head ...; cat > /dev/null; }`), but it would be nice if that wasn't needed.
