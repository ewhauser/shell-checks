# [\#698 Issue](https://github.com/koalaman/shellcheck/issues/698) `closed`: SC1073 when capturing stderr in variable

#### <img src="https://avatars.githubusercontent.com/u/354741?v=4" width="50">[austinbutler](https://github.com/austinbutler) opened issue at [2016-06-25T09:51:43Z](https://github.com/koalaman/shellcheck/issues/698):

I have a working Bash script with a line that captures the stderr to a variable, but ShellCheck seems to have a really tough time with this method. See below.

For context, I didn't want stderr printing to screen, but I also didn't want to throw it away (dev/null) nor write out to a file if I didn't need to. This method puts the stderror into the variable without printing to the terminal.

```
$ shellcheck myscript

Line 1:
RESULT=$((mysql -u "$DB_USER" -p"$DB_PASS" -h "$db" -e"quit") 2>&1)
       ^-- SC1073: Couldn't parse this $((..)) expression.
       ^-- SC1009: The mentioned parser error was in this $((..)) expression.
                   ^-- SC1072: Fix any mentioned problems and try again.
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-06-25T22:54:21Z](https://github.com/koalaman/shellcheck/issues/698#issuecomment-228574599):

ShellCheck does indeed have some issues disambiguating cases like this.

As a temporary workaround, you can add a space and write `$( (` so ShellCheck realizes this is command substitution with a nested subshell and not a `$((..))` arithmetic expression:

```
RESULT=$( (mysql -u "$DB_USER" -p"$DB_PASS" -h "$db" -e"quit") 2>&1)
```

Also note that the subshell isn't actually necessary, it's equivalent to just this:

```
RESULT=$(mysql -u "$DB_USER" -p"$DB_PASS" -h "$db" -e"quit" 2>&1)
```
