# [\#874 Issue](https://github.com/koalaman/shellcheck/issues/874) `open`: Whitespace in variable assignment not detected

#### <img src="https://avatars.githubusercontent.com/u/25167560?v=4" width="50">[inianv](https://github.com/inianv) opened issue at [2017-04-12T15:19:36Z](https://github.com/koalaman/shellcheck/issues/874):

Expected rule-id: [SC1068]( https://github.com/koalaman/shellcheck/wiki/SC1068)
checked on the http://www.shellcheck.net/ web version


#### Here's a snippet or screenshot that shows the problem:

```
#!/usr/bin/bash
filename="$1"
count=0
file=0
br=$(wc -l $filename | awk '{print $1}')
let br = $br/10
while read -r line
do
let count = count + 1
    name="$line"
echo $name >> file$file.csv
if [ $count = $br ];then
let count=0
let file+=1
fi
done < "$filename"
```

The line 

`let count = count + 1`

(or)

`let br = $br/10`

should have thrown error(s).

Reproducible by a even more simple snippet,

```
#!/bin/bash
let a = a + 1
```

On Wed, 12 Apr 2017, inianv wrote:
> The line
>     let count = count + 1
> should have thrown the error.

It's not the fact that it's an assignment with spaces that matters.

Rather, each word presented to "let" is taken a separate expression.

I suppose one could check that an expression does not start or end with "=",
but something like `let a=b +c++` is quite valid -- though unlikely to be
what was intended.

Conversely `let "count = count + 1"` is valid and does what you'd expect.

Perhaps we should just say "let is deprecated, use ((...)) instead"?
