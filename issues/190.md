# [\#190 Issue](https://github.com/koalaman/shellcheck/issues/190) `closed`: SC2086 (quote to prevent globing) false positive for an integer variable

#### <img src="https://avatars.githubusercontent.com/u/129536?v=4" width="50">[dolmen](https://github.com/dolmen) opened issue at [2014-07-09T21:26:33Z](https://github.com/koalaman/shellcheck/issues/190):

Here is some code that triggers a false positive for SC2086

```
    local -i load=${tmp:-0}/$_lp_CPUNUM

    if (( load > LP_LOAD_THRESHOLD )); then
        local ret="$(_lp_color_map $load 200)${LP_MARK_LOAD}"
                                 # ^^^^^ SC2086 Double quote to prevent globbing and word splitting.
```

Variable `load` is declared as an integer (`local -i`), so it can not contain content subject to globing or word splitting. 

#### <img src="https://avatars.githubusercontent.com/u/255938?v=4" width="50">[ormaaj](https://github.com/ormaaj) commented at [2014-07-11T06:07:52Z](https://github.com/koalaman/shellcheck/issues/190#issuecomment-48697340):

It is still subject to globbing and field splitting. You would need a non-default IFS to see the splitting, but it is in many cases not a good assumption that IFS will be default -- e.g. within a function that may be called from a context with non-default IFS.

As for globbing, Bash doesn't actually store integers as a real integer type. It stores a string and simply modifies the way assignments work. Therefore it is possible to  store arbitrary strings in a variable with the integer attribute set.

```
  $ ( i=* >foo>bar; typeset -i i; echo $i )
bar foo
```

If nothing else, quoting has a theoretical performance benefit, and makes it clear you don't intend to split/glob the expansion.
