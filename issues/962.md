# [\#962 Issue](https://github.com/koalaman/shellcheck/issues/962) `open`: Using wildcards when subdirectories exist, results in "openBinaryFile: permission denied (Permission denied)"

#### <img src="https://avatars.githubusercontent.com/u/501702?v=4" width="50">[edmorley](https://github.com/edmorley) opened issue at [2017-08-04T09:43:45Z](https://github.com/koalaman/shellcheck/issues/962):

**STR:**
```bash
$ mkdir -p foo/bar
$ echo '#!/usr/bin/env bash' > foo/script1
$ shellcheck foo/*
```

**Expected:**
No error.

**Actual:**
`foo/bar: foo/bar: openBinaryFile: permission denied (Permission denied)`

It seems like shellcheck should be skipping any directories passed as arguments.

This is using shellcheck 0.4.6 on Windows 10, under MSYS2.

#### <img src="https://avatars.githubusercontent.com/u/7482557?v=4" width="50">[Gandalf-](https://github.com/Gandalf-) commented at [2019-01-01T03:06:56Z](https://github.com/koalaman/shellcheck/issues/962#issuecomment-450706932):

This is still the case on 0.6.0.  I'm not sure ignoring directories is the right way to go, because it could be confusing if a user provides only a directory on accident and nothing happens.

There could be a nicer error message though.
eg: `shellcheck: foo/bar is not a file`

```
leaf@aspen ~> shellcheck --version
ShellCheck - shell script analysis tool
version: 0.6.0
license: GNU General Public License, version 3
website: https://www.shellcheck.net

leaf@aspen ~> mkdir -p /tmp/apple/bar/

leaf@aspen ~> echo '#!/usr/bin/env bash' > /tmp/apple/script.sh

leaf@aspen ~> shellcheck /tmp/apple/*
/tmp/apple/bar: /tmp/apple/bar: openBinaryFile: inappropriate type (is a directory)
```

This still exists on 0.9.0. I would say it might be more helpful to define this kind of error similar to `SC1071` so that we could exclude it explicitly by the `-e` option.

```
$ shellcheck path_to_dir

$ shellcheck path_to_dir -e SCXXXX
$ # no error here
```
