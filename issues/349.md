# [\#349 Issue](https://github.com/koalaman/shellcheck/issues/349) `closed`: should catch exporting quoted variablename

#### <img src="https://avatars.githubusercontent.com/u/6670329?v=4" width="50">[codrguy](https://github.com/codrguy) opened issue at [2015-04-15T00:54:32Z](https://github.com/koalaman/shellcheck/issues/349):

It'd be nice if this tool can catch cases like below:

TYPE="foo bar"

bad export :
export "${TYPE}"

correct export :
export TYPE

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-04-23T19:56:50Z](https://github.com/koalaman/shellcheck/issues/349#issuecomment-95700263):

This doesn't really have anything to do with quoting or variables.
This is more "check that the arguments to `export` are valid shell identifiers" and that means tracking the contents of variables for cases like this.


#### <img src="https://avatars.githubusercontent.com/u/20518?v=4" width="50">[alexm](https://github.com/alexm) commented at [2015-05-13T12:22:12Z](https://github.com/koalaman/shellcheck/issues/349#issuecomment-101646465):

We just got hit by `export $VAR` at work. It would be a nice feature if shellcheck would catch those.


#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-05-13T14:08:30Z](https://github.com/koalaman/shellcheck/issues/349#issuecomment-101675501):

#### <img src="https://avatars.githubusercontent.com/u/20518?v=4" width="50">[alexm](https://github.com/alexm) commented at [2015-05-13T14:25:46Z](https://github.com/koalaman/shellcheck/issues/349#issuecomment-101685051):

@deryni no it wasn't intentional. The fact is that the value of `$VAR` sometimes started with a letter and sometimes not, so when it didn't the script failed because it wasn't a valid variable name,  e.g. 

```
$ sh -x foo
+ VAR=1234
+ export 1234
foo: 2: export: 1234: bad variable name
```


#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-05-13T14:48:16Z](https://github.com/koalaman/shellcheck/issues/349#issuecomment-101696771):

I know it wasn't intentional there. The question was _could_ it have been. The simple fact that someone wrote `export $VAR` isn't necessarily an error (it could be a warning though) because that might be intentional. I was wondering if you hit an error (you did) or just an erroneous export (you didn't but would have in the "started with a letter" case).


#### <img src="https://avatars.githubusercontent.com/u/20518?v=4" width="50">[alexm](https://github.com/alexm) commented at [2015-05-13T15:34:11Z](https://github.com/koalaman/shellcheck/issues/349#issuecomment-101717112):

Actually we suffered both: the variable in question is PAM_AUTHTOK which
contains the password provided by the user. When the password started with
a non letter the PAM script failed, but it worked otherwise because the
export was in fact not needed at all.

I agree that `export $VAR` has valid use cases, but a warning or notice
would help a lot in chasing the unexpected behaviour most of the time.
