# [\#558 Issue](https://github.com/koalaman/shellcheck/issues/558) `open`: 'Declare and assign separately to avoid masking return values. [SC2155]' isn't triggered for declare/local with arguments

#### <img src="https://avatars.githubusercontent.com/u/7666511?v=4" width="50">[andrey-starodubtsev](https://github.com/andrey-starodubtsev) opened issue at [2015-12-11T07:13:18Z](https://github.com/koalaman/shellcheck/issues/558):

``` bash
    #!/bin/bash

    a="a  b"
    declare b=$(echo "$a" | grep e) # has SC2155 warning

    declare -a c=($(echo "$a" | grep c)) # has no SC2155 warning
    echo "$?" # prints 0

    declare -a d
    d=($(echo "$a" | grep c))
    echo "$?" # prints 1

    declare -r e=($(echo "$a" | grep c)) # has no SC2155 warning
    echo "$?" # prints 0

    f=($(echo "$a" | grep c))
    echo "$?" # prints 1
    declare -r f
    f=("$f{@}" 1) # ./1.sh: line 22: f: readonly variable

    export b c d e # supress shellcheck's warning about unused variables
```

I've read note about readonly variables in https://github.com/koalaman/shellcheck/wiki/SC2155, but it's possible to declare variable as readonly (right) _after_ its assignment, so I think that it would be useful to be able to find `readonly`/`local -r`/`declare -r` coupled with assignments, at least optionally.
And anyway, bash arrays are still an issue.

#### <img src="https://avatars.githubusercontent.com/u/694826?v=4" width="50">[timoreimann](https://github.com/timoreimann) commented at [2016-09-21T23:27:17Z](https://github.com/koalaman/shellcheck/issues/558#issuecomment-248772599):

Though a rather old issue, I find this quite important, especially with regards to `readonly`. Putting `readonly`, assignments, and command substitutions on the same line, i.e.,

`readonly foo=$(this_could_fail)`

is a common mistake easily and perfectly solved by breaking things down into two lines:

```
foo=$(this_could_fail)
readonly foo
```

I see no reason why this shouldn't be marked as [SC2155](https://github.com/koalaman/shellcheck/wiki/SC2155) (or a new rule). The note says that both declaration and assignment must be in the same command; I fail to see why this is supposed to be true given the two-line solution.
