#### For bugs
- Rule Id: SC2086 
- My shellcheck version: 0.9.0
- [X] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2086)
- [X] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash
touch /var/mail/$USER
```

#### Here's what shellcheck currently says:

`SC2086 Double quote to prevent globbing and word splitting.`

#### Here's what I wanted or expected to see:

Should this check be lenient on standard environment variables that won't have problematic characters in any sane environment? (`HOME UID LOGNAME RANDOM COLUMNS LINES`, etc.). `touch /var/mail/"$USER"` seems a bit redundant.

#### <img src="https://avatars.githubusercontent.com/u/84648683?v=4" width="50">[wileyhy](https://github.com/wileyhy) commented at [2025-11-03 10:23](https://github.com/koalaman/shellcheck/issues/2706#issuecomment-3479823646):

A lot of people use Shellcheck as a sort of tutoring program, for better or
worse. What you're suggesting is perhaps appropriate for advanced users,
but a newbie might find it more confusing than less confusing, in my
opinion.

Bash doesn't enforce parameter types, beyond basic attributes like
"readonly." It might be unorthodox, but...

~ $ UID=' WTH '
bash: UID: readonly variable
~ $ HOME=' WTH '
.../files/home $ echo $?
0
.../files/home $ declare -p HOME
declare -x HOME=" WTH "
.../files/home $

...repurposing environment variables for "off-label" uses can be valid code.
