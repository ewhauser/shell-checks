#### <img src="https://avatars.githubusercontent.com/u/71984?v=4" width="50">[acook](https://github.com/acook) opened issue at [2016-09-25T21:24:08Z](https://github.com/koalaman/shellcheck/issues/737):

ShellCheck validates scripts in a couple of different ways, but pivotally uses [`goodShells`](https://github.com/koalaman/shellcheck/blob/13a2070a323eab98ef54b78b22ae4c33d66d4084/ShellCheck/Parser.hs#L2593) to check against script types.

We've duplicated this list in [CodeClimate's ShellCheck engine](https://github.com/filib/codeclimate-shellcheck) in [this file](https://github.com/filib/codeclimate-shellcheck/blob/master/src/CC/ShellCheck/ShellScript.hs#L27) but we'd prefer to use the upstream version so we can remain in sync.

Purpose: We're using this list to check file extensions (as is best-practice for non-portable shell scripts) so we can make sure to send ShellCheck only receives a subset of files instead of having to vet every file in an entire repo.

On Sun, 25 Sep 2016, Anthony M. Cook wrote:

> Purpose: We're using this list to check file extensions (as is
> best-practice for non-portable shell scripts) [...]

If anything this seems _contrary_ to best practice ...

Best practice is that suffixes should be entirely avoided for any program --
script or otherwise -- that is installed in a "bin" directory, and thus is
reachable via PATH rather than its full pathname. To do otherwise breaks the
principle of substitutability.

Which means that the appropriate check is to look at the basename of the
shebang line, and only fall back to the extension if there is none.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-10-25T18:27:58Z](https://github.com/koalaman/shellcheck/issues/737#issuecomment-256127339):

For me the braces are a signal that something _other_ than a simple variable
expansion is involved. Consequently code that _always_ use them is just that
bit harder to read as I can't skim-read based on that signal.

When I read

```
 somecmd  "$foo.bar"
```

it is completely obvious that "foo" is the variable name, whereas if I read

```
 somecmd "${some_long_variable_name}.bar"
```

then the initial "${" primes me to pay more attention because it will be
something more complex than a simple variable. (It is vaguely irritating my
effort to pay extra attention is wasted.)

(And if you think "what difference does it make to skim reading", consider
when you go looking for places where a variable is _modified_; one way that
can happen is with the ${var:=default} expansion; so looking closely at
brace-enclosed expansions is important.)

The notion that braces should always be used "for consistency" is IMO
flawed. Simple expansions usually outnumber complex ones by an considerable
margin, so you're changing the majority of the code to fit with the
minority; that's just backwards.

It makes the code longer and noisier.

It does not "resolve ambiguity" because there is no ambiguity. (In the
vanishingly small proportion of cases where you do need to delimit the
variable name from following alphanumeric characters, braces are OK, but
there are other options including quotes.)

It kills the utility of the "here be dragons" signal.

Readers will always have to cope with both, and if they deal with any other
programming language, the "pay extra attention" signal is lost if it's
over-used.

-Martin
