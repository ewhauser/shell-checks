# [\#3214 Issue](https://github.com/koalaman/shellcheck/issues/3214) `open`: The use of '\e' isn't POSIX compliant.

#### <img src="https://avatars.githubusercontent.com/u/5372642?v=4" width="50">[kkdd](https://github.com/kkdd) opened issue at [2025-05-29 17:10](https://github.com/koalaman/shellcheck/issues/3214):

Hello,
I think that the use of '\e' should be warned by shellcheck to be non-POSIX compliant. 

```
$ dash -c "printf '\e'" 
\e$ 
$ bash -c "printf '\e'" 
$ 
$ echo "printf '\e'" | shellcheck -s sh -
$ 
$ shellcheck -V
ShellCheck - shell script analysis tool
version: 0.10.0
license: GNU General Public License, version 3
website: https://www.shellcheck.net
```


#### <img src="https://avatars.githubusercontent.com/u/135267736?u=1a3342b4d9d40c552d807e2c7bbf2fcda185d891&v=4" width="50">[marilene38](https://github.com/marilene38) commented at [2025-05-30 03:42](https://github.com/koalaman/shellcheck/issues/3214#issuecomment-2921143009):

Thank you for raising this issue!

I agree that ShellCheck should emit a warning when \e is used with printf in POSIX mode. Since \e is not specified by POSIX, its behavior can vary between shells—for example, dash prints it literally, while bash interprets it as the escape character. This can lead to non-portable scripts and unexpected results.

Suggestion:
ShellCheck could display a warning such as:

Use of \e in printf is not POSIX-compliant and may behave differently across shells. Consider using \033 or \x1b for portability.

Flagging this usage would help users write more portable scripts and avoid subtle bugs. If maintainers are in agreement, updating the rule set to detect this in POSIX mode would be a valuable enhancement.
