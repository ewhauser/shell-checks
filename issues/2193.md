# [\#2193 Issue](https://github.com/koalaman/shellcheck/issues/2193) `open`: Suggest `readlink -f` for reading symbolic links

#### <img src="https://avatars.githubusercontent.com/u/23638587?u=6d1f4ed9b2481fd4af374eb927f9a73602c53ba5&v=4" width="50">[jyn514](https://github.com/jyn514) opened issue at [2021-03-29 22:41](https://github.com/koalaman/shellcheck/issues/2193):

#### For new checks and feature suggestions
- [x] https://www.shellcheck.net/ (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/sh
# Obtain the symbolic link (if any)
ls -l "$shared_object" | awk '{print $NF}'
```

#### Here's what shellcheck currently says:

Line 3:
ls -l "$shared_object" | awk '{print $NF}'
^-- SC2012: Use find instead of ls to better handle non-alphanumeric filenames.
       ^-- SC2154: shared_object is referenced but not assigned.

#### Here's what I wanted or expected to see:

Line 3:
ls -l "$shared_object" | awk '{print $NF}'
^-- SCXXXX: Use `readlink -f` to canonicalize symbolic links

This is a bit of a "stretch" feature, I wouldn't have known what the awk bit was trying to do without reading the comment.
