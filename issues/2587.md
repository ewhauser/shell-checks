#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bats

@test test_name() {
    echo test
}
```

#### Here's what shellcheck currently says:
```
Line 3	SC1073: Couldn't parse this bats @test. Fix to allow more checks.
Line 3	SC1036: '(' is invalid here. Did you forget to escape it?
Line 3	SC1072:  Fix any mentioned problems and try again.
```
#### Here's what I wanted or expected to see:

bats-core support definitions via `@test`  and parentheses. The `()` part seems to be unsupported by ShellCheck.

As far as I understand, bats-core's regex to find test function is here: https://github.com/bats-core/bats-core/blob/008e8ce922938c5ba4ae1ca7c3ed0a26119c3452/libexec/bats-core/bats-preprocess#L34

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2022-10-14 03:25](https://github.com/koalaman/shellcheck/issues/2587#issuecomment-1278427769):

It apparently supports arbitrary gibberish as long as it's terminated with ` {`:

```
@test )!(*%^*$ { &@!$   {
  true
}
```

ShellCheck now does the same. Thanks! Bats happens to put the contents unescaped in double quotes in the test body, so `@test #@%*@$foo%!@ {` will end up expanding `$foo` when the test is run, but as a result of this change, ShellCheck now treats it as a literal string instead.
