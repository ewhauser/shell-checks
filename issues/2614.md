#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash
foo() { bar; baz; bang; biff; bow; }
exit 0
```

#### Here's what shellcheck currently says:

```
$ shellcheck myscript
 
Line 2:
foo() { bar; baz; bang; biff; bow; }
        ^-- SC2317 (info): Command appears to be unreachable. Check usage (or ignore if invoked indirectly).
             ^-- SC2317 (info): Command appears to be unreachable. Check usage (or ignore if invoked indirectly).
                  ^-- SC2317 (info): Command appears to be unreachable. Check usage (or ignore if invoked indirectly).
                        ^-- SC2317 (info): Command appears to be unreachable. Check usage (or ignore if invoked indirectly).
                              ^-- SC2317 (info): Command appears to be unreachable. Check usage (or ignore if invoked indirectly).
```

#### Here's what I wanted or expected to see:

It's [debatable](https://github.com/koalaman/shellcheck/issues/2542) whether this should trigger at all, but clearly there's no reason for this finding to trigger on _every_ command in an unreachable code path. This generates a lot of noise and makes it hard to tell which findings are actionable.

In this case `foo` is what's actually not being reached, not (just) the individual commands _within_ `foo`. So it would be even better to flag a case like this as "`foo` appears unused" rather than "the contents of `foo` appear unreachable".

#### <img src="https://avatars.githubusercontent.com/u/1245341?u=92bc3bae0adb38941124e2f7b3d15e1648fab20a&v=4" width="50">[oliv3r](https://github.com/oliv3r) commented at [2023-03-26 09:02](https://github.com/koalaman/shellcheck/issues/2614#issuecomment-1484037246):

It's not even correct, is it? It's not an unreachable function, it's an unused function. Which is fine, isn't it?

e.g. if we have a template, that always has a 'pr_info', 'pr_error' function, and in a script, it doesn't have one of those, is that an error? a warning? gcc treats unreachable functions as a warning, this is true, but then we also have the `__maybe_unused`.

I'm all for telling the user of _unreachable_ codepaths (you did something not so smart, this code can not ever be executed), vs 'you have an unused function'. One is (potentially) an error, the other at the most a warning.

I'd be in favor in splitting these two cases, and probably not even care about unused functions, but at least unused functions (or supporting the '__may_be_unused' flag).
