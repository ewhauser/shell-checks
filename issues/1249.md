# [\#1249 Issue](https://github.com/koalaman/shellcheck/issues/1249) `closed`: Do not warn about mixing a string and an array in some cases

#### <img src="https://avatars.githubusercontent.com/u/79761?v=4" width="50">[jzinn](https://github.com/jzinn) opened issue at [2018-06-05T05:05:28Z](https://github.com/koalaman/shellcheck/issues/1249):

- SC2145 Argument mixes string and array
- version: 0.5.0
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/sh

subcommand_foo() {
  echo "In foo"
  echo "\$1 == $1 == aaa"
  echo "\$2 == $2 == bbb"
}

subcommand_bar() {
  echo "In bar"
}

subcommand_baz() {
  echo "In baz"
}

main() {
  case "$1" in
    foo | bar | baz )
      subcommand_"$@"
    ;;
    * )
      echo "Error: Unrecognized command"
    ;;
  esac
}

main foo aaa bbb
```

#### Here's what shellcheck currently says:

```
$ shellcheck myscript
 
Line 20:
      subcommand_"$@"
                  ^-- SC2145: Argument mixes string and array. Use * or separate argument.

$ 
```

#### Here's what I wanted or expected to see:

IMO, this is a legitimate use of mixing a string and an array.  Perhaps a special case can be made for instances like this so that shellcheck does not warn about it.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-06-11T02:28:11Z](https://github.com/koalaman/shellcheck/issues/1249#issuecomment-396105897):

You're right in that this is a working and intentional mixing of strings and arrays. You've checked that there is at least one argument, and you know how `$@` behaves in the middle of words.

How can ShellCheck differentiate this from the problematic cases like `--var="$@"` or `printf "You wrote $@\\n"`?

#### <img src="https://avatars.githubusercontent.com/u/79761?v=4" width="50">[jzinn](https://github.com/jzinn) commented at [2018-06-11T08:19:10Z](https://github.com/koalaman/shellcheck/issues/1249#issuecomment-396161078):

Differentiating might be tricky, and I haven't given it serious thought.  In the absence of a clever idea for that, always warning is reasonable.  Using a ShellCheck disable comment for the few cases where it's intentional is not a problem.

One condition to look for might be when the string + first element of the array would expand to a command (i.e. there's no other string before the string in question).  Maybe this test is sufficient, but I'm not certain.  For your `--var="$@"` example, in the wild there would likely be some string before it, such as `foo` in `foo --var="$@"`, that would become the command.

If there is text between the `$@` and the close quote, then that should be warned about.  For instance, warn about `"blah${@}foo" "bar"` because of the foo, but not because of the `bar`.
