Maybe shellcheck can warn for potential double slashes like here:

Problematic code
```sh
# Set directorie from argument, or use default.
targetdir=${1-.}

 # Check for rejects...
 if [ "$(find "$targetdir/" '(' -name '*.rej' -o -name '.*.rej' ')' -print)" ] ; then
     echo "Aborting.  Reject files found."
     exit 1
 fi
 
 # Remove backup files
 find "$targetdir/" '(' -name '*.orig' -o -name '.*.orig' ')' -exec rm -f {} \;

```
correct code:
```sh
# Set directorie from argument, or use default.
targetdir=${1-.}

 # Check for rejects...
 if [ "$(find "${targetdir%/}/" '(' -name '*.rej' -o -name '.*.rej' ')' -print)" ] ; then
     echo "Aborting.  Reject files found."
     exit 1
 fi
 
 # Remove backup files
 find "${targetdir%/}/" '(' -name '*.orig' -o -name '.*.orig' ')' -exec rm -f {} \;

```

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2016-12-25T20:52:24Z](https://github.com/koalaman/shellcheck/issues/808#issuecomment-269137073):

The "find" command does not need to have trailing slashes on filenames at
all.

Aggregating the entire output of a command before checking "is it
(non-)empty" [ -n "$(...)" ] is a rather "heavy" approach. Instead, stop as
soon as you know that you've found at least one item.

Find is not likely to produce URLs, but even there it does not matter if
you have multiple slashes, except immediately after the scheme tag.
