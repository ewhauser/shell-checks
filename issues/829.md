#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-01-29T00:59:43Z](https://github.com/koalaman/shellcheck/issues/829#issuecomment-275886321):

The combination of "env" and "sh" is useless; "sh" is *always* /bin/sh on a
POSIX system.

And using "alias" in a POSIX shell script is broken. If "sh" isn't bash, it
probably won't work.

In fact, aliases inside scripts don't work how you think they should
either; the script may be parsed in its entirety before execution, and
aliases only take effect after they are executed.

Functions are also subject to the same "only after executed" rule, but they
don't affect how the script is parsed, only how the commands are
interpreted as they are executed.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-04-02T22:26:41Z](https://github.com/koalaman/shellcheck/issues/829#issuecomment-291019665):

You're right, this would not be an issue in practice because `$HOME` is generally a benign variable.

There's a good reason for still quoting variables that will never change though: you avoid double evaluation. I would seriously question this line, even though it's pretty much the same as yours:

    alias tree_my_dir='eval tree $HOME/my_dir/' 

On principle, it would probably be better to avoid this by default. I've added a note about this to the wiki. Thanks for reporting!
