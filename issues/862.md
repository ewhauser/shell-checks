#### <img src="https://avatars.githubusercontent.com/u/90273?v=4" width="50">[LukeShu](https://github.com/LukeShu) commented at [2017-05-22T21:46:18Z](https://github.com/koalaman/shellcheck/issues/862#issuecomment-303228790):

No, the body of the bug report says exactly what I meant for it to say.  (I have, however, changed the title to be more descriptive of the problem).

  If I had written what @nihilus suggested,

    exec ${fd}>&-

the shell would interpret the value of `${fd}` as a simple command, rather than as the file descriptor number to operate on.  To achieve the desired effect with "normal" variable expansion, we'd have to use eval:

    eval "exec $fd>&-"

However, this isn't necessary, because Bash 4.1 introduced the simpler syntax that this bug is about.  To quote the Bash 4.1 release notes:

    p.  If the optional left-hand-side of a redirection is of the form {var}, the
        shell assigns the file descriptor used to $var or uses $var as the file
        descriptor to move or close, depending on the redirection operator.

The feature is explained in the Bash manual:

> Each redirection that may be preceded by a file descriptor number may instead be preceded by a word of the form {varname}.  In this case, for each redirection operator except >&- and <&-, the shell will allocate a file descriptor greater than or equal to 10 and assign it to varname.  If >&- or <&- is preceded by {varname}, the value of varname defines the file descriptor to close.

I began looking at implementing this.  This bashism is already handled by shellcheck--a fact that I should have realized if I looked at the warning more closely.

The problem is that usually, `{var}` is being set, and it being read is a special case for `>&-` and `<&-`.

To use the quote from the Bash manual again, shellcheck's understanding of the syntax missed the bolded bits:

> Each redirection that may be preceded by a file descriptor number may instead be preceded by a word of the form {varname}. In this case, for each redirection operator **except >&- and <&-,** the shell will allocate a file descriptor greater than or equal to 10 and assign it to varname. **If >&- or <&- is preceded by {varname}, the value of varname defines the file descriptor to close.**
