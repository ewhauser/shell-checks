I'm adjusting the VS Code extension to contribute the correct syntax for the `.shellcheckrc` file. It's important, because since VS Code 1.60 it tries to infer the syntax of a file automatically when it doesn't know and often [it detects it as a `shellscript`, causing a lot of mess](https://github.com/vscode-shellcheck/vscode-shellcheck/issues/350).

The safest option is of course `plaintext`:

`.properties` also does:

#### <img src="https://avatars.githubusercontent.com/u/29582865?u=0c57eebb21e1922b4a0345f7a830924cc295998b&v=4" width="50">[felipecrs](https://github.com/felipecrs) commented at [2021-10-11 17:36](https://github.com/koalaman/shellcheck/issues/2355#issuecomment-940221509):

Not without an [extension](https://marketplace.visualstudio.com/items?itemName=mikestead.dotenv), and I'm not sure if it's reasonable to add this extension as dependency of the ShellCheck extension just because of that.

Both `.ini` and `.properties` accepts comments as `#` though.

`.properties` supports some weird things, such as:

```properties
key\:with\=colonAndEqualsSign = This is the value for the key "key:with=colonAndEqualsSign"
```

Ref https://en.wikipedia.org/wiki/.properties

And `.ini` supports sections, such as:

```ini
[section]
a=a
b=b
```

However, `ini` supports multiple entries of the same key such as:

```ini
enable=require-variable-braces
enable=quote-safe-variables
```

But both does also accept spaces between the equal sign.

#### <img src="https://avatars.githubusercontent.com/u/29582865?u=0c57eebb21e1922b4a0345f7a830924cc295998b&v=4" width="50">[felipecrs](https://github.com/felipecrs) commented at [2021-10-11 22:31](https://github.com/koalaman/shellcheck/issues/2355#issuecomment-940491983):

Providing syntax highlight is not complicated. Pretty straightforward. I'm thinking on doing it, actually.

Now for providing intellisense, such as auto-completion of keys and directives, descriptions on hover and such, a Language Server would need to be implemented.

Ideally, the LS should be implemented in `shellcheck`'s CLI in the same way as `deno` does (see `deno lsp` command (the communication is made through `stdin` and `stdout`), or at least have a shellcheck LS released with the same cadency of the main shellcheck, so that new additions or changes to shellcheck could be reflected to the LS and so.

This is because the Language Server Protocol is adopted by [many other editors other than VS Code.](https://microsoft.github.io/language-server-protocol/implementors/tools/) The VS Code ShellCheck extension would be just one of the consumers of it.
