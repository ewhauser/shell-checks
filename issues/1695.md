#### <img src="https://avatars.githubusercontent.com/u/5931253?v=4" width="50">[olorin37](https://github.com/olorin37) commented at [2020-01-20T10:22:42Z](https://github.com/koalaman/shellcheck/issues/1695#issuecomment-576206784):

Problem is even more demonic, because using `${1:=default}` makes shellcheck happy, but it breaks bash code, because bash do not allow for assigning to the arguments (`$1`, `$2` ... `$@`). So, now shellcheck encourages to use not valid bash code.

```
#!/usr/bin/env bash

# Problematic case #

fun_ok() {  # <- shellcheck warns here 2120
  local var=${1:-default_value}
  echo "$var"
}

fun_ok # <- and here 2119

# Even more problematic #

fun_bash_runtime_error() {
  local var=${1:=default_value}  #  <- now shellcheck is quiet
              # ^ cannot assign to $1 or $@
  echo "$var"
}

fun_bash_runtime_error

# Current solution #

# shellcheck disable=SC2120
fun_and_shellcheck_ok() {
  local var=${1:-default_value}
  echo "$var"
}

fun_and_shellcheck_ok
```

So, in my opinion it is important to eliminate second case, and then do not warn in first case. By the way shellcheck should warn in each case when someone try to assign argument with parameter expansion (but this should be probably another issue).
