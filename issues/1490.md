# [\#1490 Issue](https://github.com/koalaman/shellcheck/issues/1490) `open`: False Positive for SC1003

#### <img src="https://avatars.githubusercontent.com/u/904614?v=4" width="50">[gnzlbg](https://github.com/gnzlbg) opened issue at [2019-02-13T13:17:15Z](https://github.com/koalaman/shellcheck/issues/1490):

#### For bugs
- Rule Id: SC1003 
- My shellcheck version (`shellcheck --version` or "online"):  version: 0.6.0
- [x] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC1003)
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env sh
touch foo
grep '[\d|\w|-]* \\' foo
```

#### Here's what shellcheck currently says:

```
Line 3:
grep '[\d|\w|-]* \\' foo
                  ^-- SC1003: Want to escape a single quote? echo 'This is how it'\''s done'.
```

#### Here's what I wanted or expected to see:

Nothing probably, I want to match a `\` in the grep. The wiki page states:

> ### Exceptions
>
> If you want your single quoted string to end in a backslash, you can rewrite as 'string'\\ or ignore this warning.

But writing `grep '[\d|\w|-]* '\\ foo` does not work.

#### <img src="https://avatars.githubusercontent.com/u/325899?v=4" width="50">[zackw](https://github.com/zackw) commented at [2023-12-01T15:42:20Z](https://github.com/koalaman/shellcheck/issues/1490#issuecomment-1836328565):

The equivalent of

```
grep '[\d|\w|-]* \\' foo
```

with the trailing backslash outside the quotes, is

```
grep '[\d|\w|-]* '\\\\ foo
```

because you need to supply two backslashes in a row to grep.

I think this is another argument for suppressing the warning when the character after `\'` is a space (as suggested in <https://github.com/koalaman/shellcheck/issues/1548#issuecomment-483767340>).
