#### Here's what shellcheck currently says:

If it outputs multiple lines, each of which should be an element:
``` sh
# For bash 4.x
mapfile -t array < <(mycommand)

# For bash 3.x+
while IFS=$'\n' read -r line; do array+=("$line"); done < <(mycommand)

# For ksh
mycommand | while IFS="" read -r line; do array+=("$line"); done
```

#### Here's what I wanted or expected to see:

If it outputs multiple lines, each of which should be an element:
``` sh
# For bash 4.x
mapfile -t array < <(mycommand)

# For bash 3.x+
IFS=$'\n' read -d '' -r -a array < <(mycommand)
# or
while IFS=$'\n' read -r line; do array+=("$line"); done < <(mycommand)

# For ksh
mycommand | while IFS="" read -r line; do array+=("$line"); done
```

#### Description

The above excerpts are from wiki.

I found `read` has `-d` option:
> **-d delim**
>        The first character of **delim** is used to terminate the input line, rather than newline.

This can be used for constructing an array from lines. I think the command using `read -d '' -r -a` is quite simpler than using `while`, how do you think?

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-02-23T21:44:55Z](https://github.com/koalaman/shellcheck/issues/1496#issuecomment-466699676):

It's very nifty, though it does have some caveats: it skips blank lines, and it always returns failure.  (It also stops on the first `\0` whereas the loop skips it, but that's less of an issue.)

This is not necessarily a bad thing, especially since `var=( $(foo) )` also strips empty lines, but to avoid surprises regarding that and also when running with `set -e`, the loop is probably a better drop-in replacement. 

Thanks for the suggestion!
