# [\#35 Issue](https://github.com/koalaman/shellcheck/issues/35) `closed`: test with implicit "-n"

#### <img src="https://avatars.githubusercontent.com/u/1123418?v=4" width="50">[knweiss](https://github.com/knweiss) opened issue at [2013-11-08T16:06:58Z](https://github.com/koalaman/shellcheck/issues/35):

The following test case was created from `/etc/cron.daily/prelink` on RHEL6:

<pre>
#!/bin/sh
[ "`find /var/lib/prelink/quick -mtime -${PRELINK_NONRPM_CHECK_INTERVAL:-7} 2>/dev/null`" \
   -a -f /var/lib/rpm/Packages \
   -a /var/lib/rpm/Packages -ot /var/lib/prelink/quick ] && exit 0
</pre>


<pre>
# shellcheck testcase.sh 

In testcase.sh line 2:
[ "`find /var/lib/prelink/quick -mtime -${PRELINK_NONRPM_CHECK_INTERVAL:-7} 2>/dev/null`" \
^-- Couldn't parse this test expression.


In testcase.sh line 3:
   -a -f /var/lib/rpm/Packages \
         ^-- Unexpected "/". Fix any mentioned problems and try again.
</pre>


Here's a slightly modified version with an explicit "-n" which doesn't show the parsing problem:

<pre>
#!/bin/sh
[ -n "`find /var/lib/prelink/quick -mtime -${PRELINK_NONRPM_CHECK_INTERVAL:-7} 2>/dev/null`" \
   -a -f /var/lib/rpm/Packages \
   -a /var/lib/rpm/Packages -ot /var/lib/prelink/quick ] && exit 0
</pre>


<pre>
# shellcheck testcase2.sh 

In testcase2.sh line 2:
[ -n "`find /var/lib/prelink/quick -mtime -${PRELINK_NONRPM_CHECK_INTERVAL:-7} 2>/dev/null`" \
     ^-- Use $(..) instead of deprecated `..`
</pre>


I.e. there is a parsing problem when the explicit "-n" is missing. According to the man page  test(1) both variants should be equivalent:

<pre>
      -n STRING
             the length of STRING is nonzero

      STRING equivalent to -n STRING
</pre>

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2013-11-08T18:45:30Z](https://github.com/koalaman/shellcheck/issues/35#issuecomment-28086942):

Nice catch! The shortest test case for this is `[ foo -a -f bar ]`

Shellcheck accepts arbitrary operators for compatibility between different shells and versions, and in this case, it parses `-a` as a binary operator rather than a low precedence logic operator. It therefore fails just like it would for `[ foo -nt -f bar ]`.
