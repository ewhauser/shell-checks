
#### <img src="https://avatars.githubusercontent.com/u/1823914?v=4" width="50">[arth1](https://github.com/arth1) commented at [2016-01-25T21:15:23Z](https://github.com/koalaman/shellcheck/issues/583#issuecomment-174665415):

Or, try length=9999999999999999 or so, and you'll get an xmalloc error and the eval exits the shell, which is not good.

#### <img src="https://avatars.githubusercontent.com/u/10137?v=4" width="50">[ghost](https://github.com/ghost) commented at [2016-01-25T22:39:48Z](https://github.com/koalaman/shellcheck/issues/583#issuecomment-174704757):

Neither of those issues were caused by the use of a variable. If you remove the variable from the command and run it with a negative number or a massive number you get the same result as if you were storing that number in a variable. 

Also eval is a bash builtin, so what does it mean to say that bash doesn't support it but eval does? 

Lastly, I can make shellcheck stop complaining about the variable inside the brace expansion simply by double quoting the brace expansion. While one might want to double quote the expansion, it doesn't make any sense for that to resolve SC2051.

I feel this is a bug in shellcheck because even if one shouldn't be using the command above, though I have found it useful, the warning given by shellcheck does not apply.
