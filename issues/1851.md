#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/sh
set -eu

# shellcheck disable=SC2039
: "${HOSTNAME:?}"

echo "$HOSTNAME"
```

#### Here's what shellcheck currently says:

```
+:( $ shellcheck x

In x line 7:
echo "$HOSTNAME"
      ^-------^ SC2039: In POSIX sh, HOSTNAME is undefined.

For more information:
  https://www.shellcheck.net/wiki/SC2039 -- In POSIX sh, HOSTNAME is undefined.

```

#### Here's what I wanted or expected to see:

Nothing since the `disable=` should silence the first warning and from there on the HOSTNAME should be considered as present.

Use case: I'm getting the HOSTNAME set by thing running me (kubernetes), not by bash. So it is always present since it comes from outside. I've tried supressing it but as you can see it's not completely working. My current workaround is `hostname="${HOSTNAME:?}"` and using `$hostname` from there on but I feel like it should not be necessary and the same solution like for other variables (`${xx:?}`) should work.

#### <img src="https://avatars.githubusercontent.com/u/43144838?v=4" width="50">[graywolf-at-work](https://github.com/graywolf-at-work) commented at [2020-02-28 23:00](https://github.com/koalaman/shellcheck/issues/1851#issuecomment-592768332):

Well yeah but after the `: "${HOSTNAME:?}"` shellcheck should understand that the variable is defined. At least that is my understanding by comparing

```
$ cat x
#!/bin/sh
set -eu

echo "$Z"
$ shellcheck -o check-unassigned-uppercase x

In x line 4:
echo "$Z"
      ^-- SC2154: Z is referenced but not assigned.

For more information:
  https://www.shellcheck.net/wiki/SC2154 -- Z is referenced but not assigned.
```

and

```
$ cat x
#!/bin/sh
set -eu

: "${Z:?}"

echo "$Z"
$ shellcheck -o check-unassigned-uppercase x
```

Shellcheck learned that the `Z` will be set so it no longer warns about it. I think it should behave the same for `HOSTNAME`. I mean, sure, in POSIX it is not guaranteed to be present but I'm making explicit check to make sure it is. So after that point the warning does not make sense.

Thinking about it I guess this ticket is about extending "smartness" of SC2154 to SC2039.

#### <img src="https://avatars.githubusercontent.com/u/2199511?u=6cc22d0767078b50e47021699ffbb376251628ad&v=4" width="50">[Nightfirecat](https://github.com/Nightfirecat) commented at [2020-02-29 01:16](https://github.com/koalaman/shellcheck/issues/1851#issuecomment-592801426):

I'm not sure why the second is not giving the same error as the first. [Substring expansion](https://wiki-dev.bash-hackers.org/syntax/pe#substring_expansion) (the `"${var:default}"` notation you're using) does not define a value on an unset or empty variable.

#### <img src="https://avatars.githubusercontent.com/u/790463?v=4" width="50">[graywolf](https://github.com/graywolf) commented at [2020-02-29 16:44](https://github.com/koalaman/shellcheck/issues/1851#issuecomment-592964371):

It's not substring expansion. See https://pubs.opengroup.org/onlinepubs/009695399/utilities/xcu_chap02.html#tag_02_06_02 , it is `Indicate Error if Null or Unset.`. So after it passes that line, shellcheck knows that the variable must be set (well, set and not null).

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2020-03-17 05:42](https://github.com/koalaman/shellcheck/issues/1851#issuecomment-599888334):

It's a shame to ruin the illusion, but ShellCheck's not actually being clever in that example. There's no warning because ShellCheck only warns once for each unique variable name on the first use, and this first used happened to be guarded by the `${..?}`, thus suppressing the warning. This is a bit of a bug-as-a-feature, since it lets you do exactly what you did.

ShellCheck doesn't warn if the name is explicitly assigned though, so you can do a 
`HOSTNAME="$HOSTNAME"` with a suitably snarky comment.
