#### Here's what I wanted or expected to see:

Hello,

I found your website and I must say it is a very nice tool.
But while testing it with some bash scripts I have made, your website propose to use the command mapfile instead of the command line: IFS=$':' read -r -a paths <<<"${PATH}" to transport a file into an array.
But while testing if the result is the same as with my preferred method, I came up with a difference behaviour.
The last array entry is always ended with an '/n' when using the command: mapfile -d : -t paths < <( echo "${PATH}" )
When I use the command line: IFS=$':' read -r -a paths <<<"${PATH}", this behaviour is not shown.

Do I something wrong?

Regards,

Rammy61

p.s.: below you find some BASH code that shows this behaviour.

```sh
#!/bin/bash

#	readarray -d : -t paths <<<"${PATH}"										#splitting occurs, last parameter ends with '/n'
#	readarray -d : -t paths < <( echo "${PATH}" )						#splitting occurs, last parameter ends with '/n'
#	mapfile -d : -t paths <<<"${PATH}"											#splitting occurs, last parameter ends with '/n'
#	mapfile -d : -t paths < <( echo "${PATH}" )							#splitting occurs, last parameter ends with '/n'
#	mapfile -t paths < <( sed -r 's?:?\n?g' <<<"${PATH}" )	#splitting occurs, without '/n'
	IFS=$':' read -r -a paths <<<"${PATH}"									#splitting occurs, without '/n', my preferred method

	echo "N: '${#paths[@]}'"
```

#### <img src="https://avatars.githubusercontent.com/u/243351?v=4" width="50">[brother](https://github.com/brother) commented at [2024-06-04 08:50](https://github.com/koalaman/shellcheck/issues/2994#issuecomment-2146966433):

`mapfile -d : -t paths < <( echo "${PATH}" )`

echo will add a newline by default. Consider using the flag -n to omit that.

`mapfile -d : -t paths < <( echo -n "${PATH}" )`
