#### <img src="https://avatars.githubusercontent.com/u/75445?v=4" width="50">[lloeki](https://github.com/lloeki) commented at [2023-07-07T10:00:51Z](https://github.com/koalaman/shellcheck/issues/760#issuecomment-1625169424):

An example:

```
#!/usr/bin/env bash

# read data starting from the provided section marker up to the next one or EOF
function section() {
    awk '/^__[A-Z0-9]+__$/{f=0} f{print} /^'"$1"'$/{f=1}' "${BASH_SOURCE[0]}"
}

section __JSON__ | jq
section __YAML__ | ruby -ryaml -e 'p YAML.load(STDIN.read)'

exit

# shellcheck should stop here

__JSON__
{ "a": 1 }
__YAML__
b:
  - 1
  - 2
  - 3
```

Thinking out loud, how to make `shellcheck` stop could be achieved in multiple ways:

- detect unconditional `exit` or `exec` with unreachable code afterwards, then:
  - warn about unreachable code after the unconditional `exit` (catering "I have forgotten random debug exits" in scripts)
  - if `exit` is top-level, disable everything else _after_ the unconditional `exit` (that's the magic bit which allows the next point to happen)
  - have a directive that disable the unreachable code check on the `exit` line _but keeps everything disabled too_ past the top-level `exit` line (this would in turn allow anything to be embedded without errors)
- have a heavy handed specific directive that says "disable everything from there on" (`shellcheck:stop`?)
- ...?
