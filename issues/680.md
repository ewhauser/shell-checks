# [\#680 Issue](https://github.com/koalaman/shellcheck/issues/680) `open`: SC2178/SC2128 false positives on local variables

#### <img src="https://avatars.githubusercontent.com/u/1693206?v=4" width="50">[knirch](https://github.com/knirch) opened issue at [2016-05-29T11:15:57Z](https://github.com/koalaman/shellcheck/issues/680):

``` bash
#!/bin/bash

foo () {
   local baz=()
   echo "${baz[@]}"
}

bar () {
   local baz=""
   # Variable was used as an array but is now assigned a string. [SC2178]
   echo "$baz"
   # Expanding an array without an index only gives the first element. [SC2128]
}
```

#### <img src="https://avatars.githubusercontent.com/u/1693206?v=4" width="50">[knirch](https://github.com/knirch) commented at [2018-10-15T12:51:30Z](https://github.com/koalaman/shellcheck/issues/680#issuecomment-429840900):

Unsure if things have changed, but whilst clearing up old warnings I noticed that the above code can be fixed by simply changing local baz="" to local baz

```sh
#!/bin/bash

foo () {
   local baz=()
   echo "${baz[@]}"
}

bar () {
   local baz
   echo "$baz"
}
```

#### <img src="https://avatars.githubusercontent.com/u/21826?v=4" width="50">[ingydotnet](https://github.com/ingydotnet) commented at [2022-08-12T15:12:26Z](https://github.com/koalaman/shellcheck/issues/680#issuecomment-1213219961):

Also functions defined using outer parens rather than braces like:
```
do-it() (
  foo=(xyz)
)

do-also() (
  foo=xyz
)
```
should not trigger these warnings.

I use parens for defining functions as much as possible as it limits global side-effects. 
Vars defined within are essentially "local".
