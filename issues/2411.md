Greetings!

In case of [Visual Studio Code](https://code.visualstudio.com/) ("VSCode"; as of version `1.83.1`, `2023-10-10`) and [Shellcheck extension](https://marketplace.visualstudio.com/items?itemName=timonwong.shellcheck), it's possible to configure the extension to ignore specific locations due to the approach the extension is implemented, i.e. exposed class constructor parameter [`ignorePatterns`](https://github.com/vscode-shellcheck/vscode-shellcheck/blob/8f8f68461069e311ad9b5bb843796293b1fbfec7/src/utils/filematcher.ts#L11).

---

From the extension documentation (as of version `0.34.0`, `2023-08-27`):

> The `shellcheck.ignorePatterns` works exactly the same as `search.exclude`, read more about glob patterns [here](https://code.visualstudio.com/docs/editor/codebasics#_advanced-search-options)
> 
> For example:
> ```json
> {
>   "shellcheck.ignorePatterns": {
>     "**/*.zsh": true,
>     "**/*.zsh*": true,
>     "**/.git/*.sh": true,
>     "**/folder/**/*.sh": true
>   }
> }
> ```
> To add additional ignore patterns atop the default patterns, you have to copy the default ignore patterns and then add yours to the end of the list (#1196).

[Source](https://github.com/vscode-shellcheck/vscode-shellcheck/tree/34842378b058bd11e9689a1415053bf8e983c8cf#shellcheckignorepatterns)

---

Just in case, the above JSON excerpt may be also stated in VSCode [User and Workspace Settings](https://code.visualstudio.com/docs/getstarted/settings).

---

Please keep it safe and stay sublime!

Best and kind regards ✨

My apologies, I am not personally into Haskell currently enough to contribute in fairly near future.
Though, regarding the case, if we consider the `.gitignore` file, I would not expect a `.shellcheckignore` file but a CLI flag option (e.g. `-i`, `-I`, `--ignore`, `--gitignore`, `--git-ignore`), which would respect the `.gitignore` file in the current Git repository. There are Haskell libraries like the following, which should support the case:

> `vcs-ignore` is small Haskell library used to find, check and process files ignored by selected VCS.
> [Source](<https://hackage.haskell.org/package/vcs-ignore>)

> While `vcs-ignore` is mainly intended to be used as a library, it also comes with small executable called `ignore` that can be used standalone to verify whether given path is ignored or not.
> [Source](<https://github.com/vaclavsvejcar/vcs-ignore>)

Otherwise, if not Git but a more built-in exclusion, I would expect a similar to `GNU` `tar` CLI parameter options `--files-from` and `--exclude-from`:

> ```
> -T, --files-from=FILE
> get names to extract or create from FILE...
> -X, --exclude-from=FILE
> exclude patterns listed in FILE
> ```
> [Source](<https://linux.die.net/man/1/tar>)

And, the "pattern" would either be the standard [glob](<https://www.gnu.org/software/tar/manual/html_node/wildcards.html>), more complex [BRE/ERE](<https://www.gnu.org/software/sed/manual/html_node/BRE-vs-ERE.html>), or even the marvelous [PCRE](<https://www.pcre.org/>) but we should also consider the speed of such operations in the context of syntax validation and highlighting. Just in case, a few TypeScript references regarding the speed aspect:

> Typescript is an amazing project, but a perplexing open-source blackbox... The type checker itself is Typepscript's largest component and it’s a single 25 thousand line file.
> [Source](<https://zackoverflow.dev/writing/the-problem-with-typescript/>)

> Internally Deno uses Microsoft’s TypeScript compiler to check types and produce JavaScript. Compared to the time it takes V8 to parse JavaScript, it is very slow... We certainly think there are improvements that can be done here on top of the existing TypeScript compiler, but it’s clear to us that ultimately the type checking needs to be implemented in Rust.
> [Source](<https://deno.com/blog/v1#tsc-bottleneck>)

Related: https://github.com/microsoft/TypeScript/pull/51387 (*Convert the codebase to modules*...)
