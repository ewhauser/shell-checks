# [\#189 Issue](https://github.com/koalaman/shellcheck/issues/189) `open`: SC2034 (unused variable) when used without $ in assignment to variable declared -i

#### <img src="https://avatars.githubusercontent.com/u/129536?v=4" width="50">[dolmen](https://github.com/dolmen) opened issue at [2014-07-09T21:16:54Z](https://github.com/koalaman/shellcheck/issues/189):

Here is [some bash/zsh code that triggers a false positive](https://github.com/nojhan/liquidprompt/blob/9633ac83cad5f5702c1e853940c0ab2e166961bf/liquidprompt#L1253) for SC2034:

```
_lp_color_map() {
    # Default scale: 0..100
    # Custom scale: 0..$2
    local -i scale value
    scale=${2:-100}
    # ^^^ SC2034 triggered here
    value=100*$1/scale
    #             ^^^ but scale is used here! 
```

SC2034 is triggered for `scale`, while it is used the line just after in an expression assigned to a local variable declared as integer.

#### <img src="https://avatars.githubusercontent.com/u/255938?v=4" width="50">[ormaaj](https://github.com/ormaaj) commented at [2014-07-11T06:42:09Z](https://github.com/koalaman/shellcheck/issues/189#issuecomment-48699063):

I suppose you have to draw the line somewhere because it is impossible to guarantee `scale` was used via static checking whenever arithmetic is involved.

This fails the test as well, technically:

```
scale=5
expr=value=100*$1/scale let expr
```

shellcheck would have to actually evaluate the arithmetic to know for sure.

`value` is declared as integer, so the right side of the assignment is an arithmetic expression. In those expressions, words are variable names.

Your test is more complex as it uses one more level of evaluation.
