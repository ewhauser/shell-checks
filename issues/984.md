#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
somearray=("foo bar" baz)
for element in ${somearray[*]}; do
    echo "$element"
done
```

#### Here's what shellcheck currently says:

Nothing

#### Here's what I wanted or expected to see:

A warning to use `"${somearray[@]}"` instead.

------

When looping over the elements in an array, the safest syntax for this in bash-derivative shells is:

```
for element in "${somearray[@]}"; do
    echo "$element"
done
```

The alternative form `somearray[*]` produces a string value joining each element with `$IFS`, which is not desired when iterating over elements. ShellCheck could help notice when this form is accidentally used in the array access portion of the for-loop declaration, to point out likely mistakes.

The alternative form `somearray[@]` without quotes is likely unsafe, though I'm not exactly sure what would go wrong here.

The alternative form using `while` and manual indices can be risky if index arithmetic turns out to be wrong, but unfortunately index manipulation is often necessary, so we can't reliably lint that case.

The alternative forms using `while` loops with `shift`, or `seq`, or integer indices, are required in POSIX sh, where arrays do not exist. But the former `for element in "${somearray[@]}"` form is best for bash scripts, including bash, ash, dash, posh, ksh, ksh93, mksh, and zsh. 

Could ShellCheck include a syntax check for the above, to help users promote robust for loops?

Without quotes the array's elements by themselves are split on `$IFS`:

```bash
a=("foo bar" baz)

for e in ${a[@]}; do
  echo "$e"
done

echo

for e in "${a[@]}"; do
  echo "$e"
done
```
