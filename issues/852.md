#### Here's a snippet or screenshot that shows the problem:

```
#!/bin/bash

n="n"
y="y"
while [[ $y -ne $n ]]
do
    ...
done
```

```
$ shellcheck myscript

```


#### Here's what I wanted or expected to see:
```
line 5: [[: y: expression recursion level exceeded (error token is "y")
```

I might suggest checking whether or not a variable is actually an integer when using comparison operators such as `-ne`, `-gt`, etc. otherwise it might possibly lead to someone thinking that `-ne` is fine for comparing string values. 


#### <img src="https://avatars.githubusercontent.com/u/544596?v=4" width="50">[nihilus](https://github.com/nihilus) commented at [2017-03-21T19:27:50Z](https://github.com/koalaman/shellcheck/issues/852#issuecomment-288191718):

The major problem here is that shellscripts are not statically typed; so the comparsion when used on variables cannot easily be deduced if it is done on a string or an integer (or a float). In fact the shell cannot do this any assumptions and you are hit by this bug in run-time.

It's worth remembering that Shellcheck is a static analysis tool, not a
shell execution environment

I suggest reading up on "The Halting Problem".


This means there are cases where even in theory it cannot be determined
whether a self-reference will exceed the stack limit; for example, consider

  factorial='n>1?n--*factorial:1'

Whether or not that blows up depends on the initial value of n.
If you think that it suffices to have n statically defined, consider
instead

  ex='(n/=n&-n)==1 ? 0 : (n+=n*2+1, 1+ex)'

which, afaik, has has no proof that it must terminate; there's no
short-circuit -- the only way to tell how long it will take to run is to run
it, and if it's "forever" then you'll never know.

(As an aside, for the life of me I can't see why people insist on using
[[...]] for doing math, when that's what ((...)) is for.)
