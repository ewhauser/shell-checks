# [\#827 Issue](https://github.com/koalaman/shellcheck/issues/827) `closed`: SC2002 "Useless cat" with `wc` (cat needed to avoid printing filename)

#### <img src="https://avatars.githubusercontent.com/u/8690113?v=4" width="50">[refola](https://github.com/refola) opened issue at [2017-01-22T00:21:42Z](https://github.com/koalaman/shellcheck/issues/827):

I'm using `cat` to stop the `GNU coreutils 8.25` version of `wc` from printing the name of the file I'm counting lines of. ShellCheck calls this `cat` useless.

Rule Id: SC2002
I tried on shellcheck.net and verified that this is still a problem on the latest commit.

#### Snippet
```sh
#!/bin/sh
cat "$1" | wc -l
```

#### Here's what ShellCheck says:
```
Line 2:
cat "$1" | wc -l
    ^-- SC2002: Useless cat. Consider 'cmd < file | ..' or 'cmd file | ..' instead.
```

#### Here's what I wanted or expected to see:
I expected no output because the cat isn't useless. In particular, this use of `cat` seems to be the most direct way to prevent `wc` from echoing the name of the file.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-01-22T05:50:05Z](https://github.com/koalaman/shellcheck/issues/827#issuecomment-274310858):

The simplest and most direct way to prevent `wc` from echoing the filename is simply
```
#!/bin/sh
wc -l <"$1"
```

This requires fewer processes, no inter-process pipe, and in theory allows `wc -c` to use `fstat()` to obtain the size of the file in bytes, rather than by reading its contents (although I'm unaware of any version that is implemented this way).

So yes, the warning is correct.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2021-09-22T12:56:57Z](https://github.com/koalaman/shellcheck/issues/827#issuecomment-924903999):

@brchrisman it's not a bug to show some false positives.

Warnings should be on by default, and can and should be disabled when you know that the code is safe.

Unless you're in strict posix mode, relying on wordsplitting is already discouraged. For the suggested use case it might be clearer to write like this:
```lang=sh
list=( $pattern )
cat "${list[@]}" | wc -l
```
