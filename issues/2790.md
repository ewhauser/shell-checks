# [\#2790 Issue](https://github.com/koalaman/shellcheck/issues/2790) `open`: SC2064: arrows misead by pointing to variables

#### <img src="https://avatars.githubusercontent.com/u/84648683?v=4" width="50">[wileyhy](https://github.com/wileyhy) opened issue at [2023-06-29 15:08](https://github.com/koalaman/shellcheck/issues/2790):

#### For bugs
- Rule Id: SC2064

- My shellcheck version:  version: 0.9.0 and online

- [No, the wiki doesn't cover it.] The rule's wiki page does not already cover this.  https://www.shellcheck.net/wiki/SC2064

- [Yes, the bug persists.] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit. 


#### Here's a snippet or screenshot that shows the problem:

** See attached, four screenshots. **

Here, the outer set of quotes enclosing the entire group of arguments in `trap [-lp] [[arg] sigspec ...]` are double quoted while the inner variable is single quoted.

> `trap "rm -f '$trace_json'" EXIT`


#### Here's what shellcheck currently says:

** Note: github automatically removes white-space, and uses a quite different font than shellcheck does, so this bug report will likely render less accurately than screenshots.  **

In any event, in terminal, shellcheck's arrows point at the variables inside a trap's full argument/command list.  As you can see in the attached file, out3.txt, users have put single quotes in where the arrows indicate, around variables, which directly contradicts what the wiki page advises as "Correct code." It's easy to assume that the "this" of the warning message is intended to refer to variables.

```
In ./sorbet/test/cli/web-trace-file/test.sh line 7:
trap "rm -f '$trace_json'" EXIT
            ^-----------^ SC2064 (warning): Use single quotes, otherwise this expands now rather than when signalled.
```

#### Here's what I wanted or expected to see:

The arrows pointing to the outer quotes.

```
trap "rm -f '$trace_json'" EXIT
     ^-------------------^ SC2064 (warning): Use single quotes, otherwise this expands now rather than when signaled.
```

A possible edit to the warning message could be, "Use single quotes around _arg_ in `trap [-lp] [[arg] sigspec ...]`, otherwise this expands now rather than when signaled," or something like that.

I edited the wiki page to add four various examples of problematic code, and I removed 50-60 examples from out3.txt for security reasons, just in case. Also, "signalled" has one 'L.'
