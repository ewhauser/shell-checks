# [\#2738 Issue](https://github.com/koalaman/shellcheck/issues/2738) `open`: SC2310 doesn't take subshells into account when determining if `errexit` or `set -e` is turned on

#### <img src="https://avatars.githubusercontent.com/u/615684?u=10b3a58cd92f636bec2d46385a52273490743197&v=4" width="50">[h3xx](https://github.com/h3xx) opened issue at [2023-04-15 17:37](https://github.com/koalaman/shellcheck/issues/2738):

It's in the design of Bash that if a subshell runs `set -e`, it doesn't affect the rest of the script. SC2310 doesn't take this into account. SC2310 instead assumes that `set -e` *anywhere* in the script means the whole script is using `errexit`.

For example:
```sh
#!/bin/bash
(set -e; set -o |grep errexit) # gives "errexit on"
set -o |grep errexit           # gives "errexit off"
```

Note also that `errexit` *will* be inherited by child subshells if the parent shell sets it. The child subshell can run `set +e` and disable it without affecting the parent:
```sh
(
    set -e
    set -o |grep errexit     # gives "errexit on"
    (
        set -o |grep errexit # gives "errexit on"
        set +e
        set -o |grep errexit # gives "errexit off"
    )
    set -o |grep errexit     # gives "errexit on"
)
```

doit() {
