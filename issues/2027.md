# [\#2027 Issue](https://github.com/koalaman/shellcheck/issues/2027) `open`: False positives for both SC2034 and SC2102 with range expressions

#### <img src="https://avatars.githubusercontent.com/u/19507282?v=4" width="50">[automorphism88](https://github.com/automorphism88) opened issue at [2020-08-03 01:32](https://github.com/koalaman/shellcheck/issues/2027):

#### For bugs
- Rule Id (if any, e.g. SC1000): SC2034, SC2102
- My shellcheck version (`shellcheck --version` or "online"): 0.7.1 and online
- [x] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2086)
The wiki page for SC2102 mentions this, but not the page for SC2034
- [x] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
set -eu
shopt -s extglob
readonly AAA=abc
case ${1-} in
    x*([$AAA]))
        echo match ;;
    *)
        echo no match ;;
esac

```

#### Here's what shellcheck currently says:

```
Line 4:
readonly AAA=abc
         ^-- SC2034: AAA appears unused. Verify use (or export if used externally).
 
Line 6:
    x*([$AAA]))
       ^-- SC2102: Ranges can only match single chars (mentioned due to duplicates).
```

#### Here's what I wanted or expected to see:

No output, the script is valid and works as intended. Note that enclosing the variable expansion in double quotes will avoid this issue, but in this case, with a readonly variable that contains no whitespace, there is no need for double quotes.

#### <img src="https://avatars.githubusercontent.com/u/6044603?u=e005fc2ed0b0fbb23bf731f5fa261849ea6dc93c&v=4" width="50">[drjasonharrison](https://github.com/drjasonharrison) commented at [2025-12-01 19:53](https://github.com/koalaman/shellcheck/issues/2027#issuecomment-3598581705):

Line 6: The problem isn't the variable, it's the unquoted `[...]`

The expression `$AAA` is interpreted as a range, not a variable reference.

Maybe this will work
 
```
x*("\[$AAA\]"))
```
