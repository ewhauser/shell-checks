#### <img src="https://avatars.githubusercontent.com/u/21826?v=4" width="50">[ingydotnet](https://github.com/ingydotnet) commented at [2018-05-17T06:45:35Z](https://github.com/koalaman/shellcheck/issues/1198#issuecomment-389762337):

@Dmole, this was just an example to show the bug. The point of heredocs is to quote *any* text, without escaping anything, as long as that text does not have a line containing only the ending token.

That means you can have entire programs (even bash scripts) as heredocs, and they can contain quoted strings and even more heredocs. Or you might have documentation or XML or all sorts of things. In other words, quotes are no substitute for the elegant power of heredocs.

I've actually hit this bug in real code. It's a pretty bad UX, because shellcheck finds the wrong end and then usually gets confused by the text immediately following and gives up.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-05-25T04:51:40Z](https://github.com/koalaman/shellcheck/issues/1198#issuecomment-391940442):

This is related to #1050. ShellCheck does intend to warn for this code, because it is a relatively common error. The bug is that you can't just disable the warning, because it's parsed as stopping on that line. I.e., this perfectly valid script fails to parse because of it:

```
#!/bin/bash
cat <<FOO
  FOO
()
FOO
```

This was done intentionally at the time, but like most such "helpful" parsing choices, it turned out to be a bad idea.

This has now been reworked in 135cf5932f5ba7e29aa7df8d0c632e2b66d2110a . 

You still get the same warning, but it's purely cosmetic/informational and can be ignored in any of the normal ways. ShellCheck will continue searching for the real end token (if any) and treat the script accordingly.
