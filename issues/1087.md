#### <img src="https://avatars.githubusercontent.com/u/521538?v=4" width="50">[Earnestly](https://github.com/Earnestly) opened issue at [2018-01-05T10:50:42Z](https://github.com/koalaman/shellcheck/issues/1087):

* shellcheck version: 0.4.7
* rule id: SC1097
* [x] reproducible on shellcheck.net

```sh
#!/bin/sh --
while IFS== read -r a; do
    :
done
```

When using `shellcheck --format=gcc foo` I get the following output:

```
foo:2:10: error: Unexpected ==. For assignment, use =. For comparison, use [/[[. [SC1097]
```

A potential workaround is to quote the assignment of the `=` but that shouldn't be necessary.

  

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-01-05T18:09:29Z](https://github.com/koalaman/shellcheck/issues/1087#issuecomment-355624472):

I agree that it shouldn't be necessary to modify perfectly valid code to please tooling. 

Most of the time though, this would correctly trigger for some newbie's `while var==value` or `if ($var == value)`. Please consider helping out such users by making your intent more explicit with `IFS="="`.

You can also ignore this warning since the snippet is still being correctly parsed.
