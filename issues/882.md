# [\#882 Issue](https://github.com/koalaman/shellcheck/issues/882) `closed`: Add a command line switch equivalent to `# shellcheck source=`

#### <img src="https://avatars.githubusercontent.com/u/90273?v=4" width="50">[LukeShu](https://github.com/LukeShu) opened issue at [2017-04-20T19:40:38Z](https://github.com/koalaman/shellcheck/issues/882):

- [X] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

To borrow the example from the SC1090 wiki page:

```
. "$(find_install_dir)/lib.sh"
```

will cause shellcheck to throw SC1090.  The solution is to throw in a directive:

```
# shellcheck source=src/lib.sh
. "$(find_install_dir)/lib.sh"
```

If there are a bunch of such includes, I don't want to have to add an ugly directive before each of them.  I don't want to disable SC1090 either.

What I would like is to be able to set up a mapping of expression to file location that shellcheck uses.  Perhaps via command line arguments: `shellcheck --source='$(find_install_dir)/lib.sh=src/lib.sh' --source='$(find_install_dir)/msg.sh=src/msg.sh'`

Of the 3 directives that shellcheck supports, `source=` is the only one without a flag equivalent.

#### <img src="https://avatars.githubusercontent.com/u/1682574?v=4" width="50">[reidpr](https://github.com/reidpr) commented at [2018-07-16T23:38:05Z](https://github.com/koalaman/shellcheck/issues/882#issuecomment-405413239):

I have the following in my scripts:

```
LIBEXEC="$(cd "$(dirname "$0")" && pwd)"
. "${LIBEXEC}/base.sh"
```

One can run the scripts either from the source directory or installed to a standard location for executables. If installed, we rewrite `LIBEXEC` to a constant string containing the appropriate directory, which is not the same as the scripts.

For example, if the source code is in `/usr/local/src/foo`, the scripts as well as `base.sh` reside in `/usr/local/src/foo/bin`. If we then `make install PREFIX=/usr/local`, the scripts go in `/usr/local/bin` and `base.sh` goes in `/usr/local/libexec/foo`. (This is consistent with FHS.)

Thus, in addition to not wanting to put a directive in every script, I cannot without modifying the installation procedure to rewrite two things, and that's pretty ugly.

Currently I disable SC1090. I'd prefer to provide a command line switch saying "files you can't find are in this directory".

How about a flag `--source-from` that lets you specify search paths:

* The current behavior would be `--source-from=dir:.`
* Some absolute path would be `--source-from=dir:/usr/local/src/foo`
* Relative to the script could be `--source-from=script`
* Using the environment PATH like sh could be `--source-from=path`

And they could all stack.
