# [\#2674 Issue](https://github.com/koalaman/shellcheck/issues/2674) `open`: Warn on `echo "$var"` when `$var` might be `-e`

#### <img src="https://avatars.githubusercontent.com/u/26471?u=487b2dd94c891819f0b37be1e0f193daef31f226&v=4" width="50">[andersk](https://github.com/andersk) opened issue at [2023-01-24 20:46](https://github.com/koalaman/shellcheck/issues/2674):

#### For new checks and feature suggestions
- [x] https://www.shellcheck.net/ (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related


#### Here's a snippet or screenshot that shows the problem:

```sh
#!/usr/bin/env bash

var="-e"
echo "$var"
```

#### Here's what shellcheck currently says:

Nothing.

#### Here's what I wanted or expected to see:

ShellCheck should warn that `echo "$var"` produces non-portable results: the value might unintentionally be parsed as an `echo` option depending on the shell.

```console
$ bash test.sh

$ dash test.sh
-e
```

`printf '%s\n' "$var"` should be suggested as an alternative.

This is a sort of converse to [SC3037](https://www.shellcheck.net/wiki/SC3037), which is for code that *intentionally* tries to use a non-portable `echo` option.

(Inspired by [bash’s echo command is broken](https://www.youtube.com/watch?v=lq98MM2ogBk), which shows a real-world [example](https://github.com/madler/unzip/blob/0b82c20ac7375b522215b567174f370be89a4b12/unix/zipgrep#L72) of this bug in the `zipgrep` shell script.)
