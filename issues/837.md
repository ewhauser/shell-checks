thanks for the useful tool.

What I feel would help bigger projects like this (https://github.com/drwetter/testssl.sh) is to have the possibility to easier catch instances of a finding. E.g. if one wants to spot unnecessary sed statements (SC2001), dead variables , read without -r, and so on and so forth.

Also for unit tests/ travis Ci this would be an advantage ("does the code have dead variables?" / "doesn't it use bash internal sed/replace" ).

Cheers, Dirk





#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-02-25T23:57:12Z](https://github.com/koalaman/shellcheck/issues/837#issuecomment-282521662):

I can see how this would be useful to fish out certain high accuracy, high impact warnings from the sea of UUoC and similar. 

For anyone looking to do this until then, it's usually pretty simple to extract info from the JSON output:

    shellcheck -f json script | jq 'map(select(.code | contains(2001)))'
