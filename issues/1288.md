Rule Id: SC2094
`shellcheck -v : 0.5.0`

```
#!/bin/bash

/bin/cut -f3 -d":" < /etc/passwd | /bin/sort -n | /usr/bin/uniq -c |\
    while read x ; do
    [ -z "${x}" ] && break
    # shellcheck disable=SC2086
    set - $x    
    if [ "$1" -gt 1 ]; then
        users=$(/bin/gawk -F: '($3 == n) { print $1 }' n="$2" \
            /etc/passwd | /usr/bin/xargs)
        echo "Duplicate UID ($2): ${users}"
fi done
```

shellcheck complains:

```
/bin/cut -f3 -d":" < /etc/passwd | /bin/sort -n | /usr/bin/uniq -c |\
                     ^-- SC2094: Make sure not to read and write the same file in the same pipeline.
```
and

```
/etc/passwd | /usr/bin/xargs)
            ^-- SC2094: Make sure not to read and write the same file in the same pipeline.
```

Do `sort` & `uniq` act on the file again or they act on the result of `cut` ?

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-08-16T01:50:46Z](https://github.com/koalaman/shellcheck/issues/1288#issuecomment-413396820):

You're entirely right, and your code is correct. ShellCheck is just really bad at determining whether a mentioned file is read, written or neither. Please just exclude it. Sorry it took a while. 

It's generally a tricky problem: compare the role of `a_file` in `cat a_file`, `tee a_file` and `echo a_file`. Then try `sed '' a_file`,  `sed -i '' a_file` and `sed a_file`. 

SC2094 just triggers whenever the target of a redirection also appears in a command in the pipeline. This aims to catch things like `cat file > file` , but results in a false positive for your command, and a false negative for things like `cat foo | tee foo`.
