# [\#2679 Issue](https://github.com/koalaman/shellcheck/issues/2679) `open`: Warn about the use of unvalidated input in arithmetic contexts.

#### <img src="https://avatars.githubusercontent.com/u/11823649?u=c30712f231d62d75594f83bc310694210fae8f43&v=4" width="50">[nepella](https://github.com/nepella) opened issue at [2023-02-01 22:23](https://github.com/koalaman/shellcheck/issues/2679):

Shellcheck (the Web version) doesn't warn about the use of unvalidated values in arithmetic evaluation, which [is always vulnerable](https://mywiki.wooledge.org/BashProgramming/05#Arithmetic_Expansion). Issue #1322 reports this for array indexes, but this is broader. Here's a script showing common pitfalls, for which Shellcheck reports no issues:

```sh
#!/bin/bash
# Invoke as:
#   vulnerable.sh -c 'a[$(echo Gotcha! >&2)]'

declare count=0
while getopts c: opt; do
    case $opt in
    c) count=$OPTARG ;;
    *) exit ;;
    esac
done
shift $((OPTIND - 1))

# Gotcha!
[[ $count -lt 5 ]]

# Gotcha!
echo $((count + 1))

# Gotcha!
((count < 5))

# Gotcha!
for ((i = 0; i < count; i++)); do
    :
done
```

Even assignment is vulnerable when the assigned variable is an integer:

```sh
#!/bin/bash
declare -i count=0    # Futile attempt at input sanitization.
while getopts c: opt; do
    case $opt in
    # Gotcha!
    c) count=$OPTARG ;;
    esac
done
```

I don't know how difficult it would be to check for this, but I thought it ought at least to be reported.
