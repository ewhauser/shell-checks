# [\#692 Issue](https://github.com/koalaman/shellcheck/issues/692) `closed`: Possible 'Bourneism' info notification for using ! and /bin/sh.

#### <img src="https://avatars.githubusercontent.com/u/1708071?v=4" width="50">[rawiriblundell](https://github.com/rawiriblundell) opened issue at [2016-06-17T08:19:41Z](https://github.com/koalaman/shellcheck/issues/692):

I've found an odd behaviour while coding a mostly portable script: sometimes on Solaris 10 and older, "!" doesn't work.

For example `if [ ! -d /some/path ]; then` will break.  And sometimes it won't.  And sometimes again it will.  I can't track down which factors actually determine the failure.

On a Solaris 10 host, for example, we can see that '!' is something `/bin/sh` isn't happy with:

```
$ type [ !
[ is a shell builtin
! not found
```

On Solaris 11, /bin/sh links to ksh93, so no problem:

```
$ type [ !
[ is a shell builtin  
! is a shell keyword
```

The interesting thing is that `type` may say it's not found, but as I said, sometimes it will still work e.g.

```
$ if [ ! -d /blargh ]; then echo "not found"; fi
not found
```

The fix I settled on to make it work reliably was to use `:` as a dummy/no-op action:

```
if [ -d /blargh ]; then
  :
else
  echo "not found"
fi
```

`true` may also be acceptable under POSIX, and more readable to boot, but IIRC, `:` predates `true`, so is probably the better option to recommend.  If we're assuming a reversion towards The Bourne Way (tm)

So that's perhaps something to throw an info link on if the shebang is `#!/bin/sh`?

You can see an example of this behaviour (that I've just found to confirm that I'm not going crazy) here:

https://github.com/jordansissel/fpm/issues/129

#### <img src="https://avatars.githubusercontent.com/u/1923199?v=4" width="50">[geirha](https://github.com/geirha) commented at [2016-07-15T21:31:35Z](https://github.com/koalaman/shellcheck/issues/692#issuecomment-233074922):

`/bin/sh` on Solaris 10 is the bourne shell. On more or less every other UNIX and UNIX-like system, including Solaris 11, `/bin/sh` is a POSIX shell, and it's POSIX shell features that shellcheck checks for when the shebang is `#!/bin/sh`. POSIX defines that sh should have a `!` to negate a command's exit status, and that the `test` command should have a `!` operator to negate the following expression's truth value.

In bourne you have to use the `else` part of if instead:

```
# POSIX:
if ! cmd; then
    printf 'Command failed\n'
fi

# Bourne:
if cmd
then
    :
else
    echo "Command failed"
fi
```

So the issue is really: should Shellcheck support the bourne shell in addition to the shells it already support?

Bourne is a dying shell. It's mostly just older solaris systems that keep it alive outside a museum these days. Sounds like a waste of time to me to write support for a dying shell.

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2018-08-02T23:44:32Z](https://github.com/koalaman/shellcheck/issues/692#issuecomment-410101245):

If you might meet a shell that lacks a built-in `!` it may suffice to define it as a function:
```
type ! > /dev/null 2>&1 || eval '
! () {
  if "$@"
  then return 1
  else return 0
  fi
}'
```
Of course this assumes that said shell has functions and isn't too strict about their names. And that you don't want to use `!` on compound statements, though you might get around that by applying de Morgan's transformations.
