# [\#2279 Issue](https://github.com/koalaman/shellcheck/issues/2279) `closed`: SC2283 ("spaces around = to assign") doesn't trigger with readonly or declare

#### <img src="https://avatars.githubusercontent.com/u/69007229?u=88e8fd44699808509b7975aeb48b7d872448bb93&v=4" width="50">[wimax-grapl](https://github.com/wimax-grapl) opened issue at [2021-07-09 05:43](https://github.com/koalaman/shellcheck/issues/2279):

#### For bugs
- Rule Id (if any, e.g. SC1000): SC2283
- My shellcheck version (`shellcheck --version` or "online"): online
- [x] The rule's wiki page does not already cover this (e.g. https://shellcheck.net/wiki/SC2086)
- [x] I tried on https://www.shellcheck.net/ and verified that this is still a problem on the latest commit

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash

# This should be a shellcheck error
readonly A = 1
echo "$A"

# Counterexample (it works without readonly)
B = 2
echo "$B"

# UPDATE: You should probably do this check for declare as well!
declare -r C = 3
echo "$C"
```

#### Here's what shellcheck currently says:
```
$ shellcheck myscript
 
Line 8:
B = 2
  ^-- SC2283: Remove spaces around = to assign (or use [ ] to compare, or quote '=' if literal).

```


#### Here's what I wanted or expected to see:
```
Line 4:
readonly A = 1
           ^-- SC2283: Remove spaces around = to assign (or use [ ] to compare, or quote '=' if literal).
Line 8:
B = 2
  ^-- SC2283: Remove spaces around = to assign (or use [ ] to compare, or quote '=' if literal).

and the same for C
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-07-24 20:34](https://github.com/koalaman/shellcheck/issues/2279#issuecomment-886108327):

These warnings were a victim of fbb14d6b384a65ee16781966d9d240db9ed7b644, which improved the robustness and flexibility of warnings about e.g. `${var} = value` by moving the logic from the parser to the analyzer. However, this rewrite did not account for the other places where the assignment parser was used: after `declare`, `export`, `local`, `readonly` and `typeset`.

This fix re-adds a simple check for words starting with `=` or `+=` for these commands. Good catch, thanks!

I investigated whether it could also benefit from other assignment bugs like `export $var=$value`, but all the cases I found were using this correctly.
