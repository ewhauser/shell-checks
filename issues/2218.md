# [\#2218 Issue](https://github.com/koalaman/shellcheck/issues/2218) `closed`: SC2048: Don't warn on unquoted element number get `${#arr[*]}`

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash


bar() {
    test ${#Foo[*]} -gt 0 && printf ' %q' "${Foo[@]}"
}

main() {
    local Foo
    Foo+=("$1")
}

main "$@"
```

#### Here's what shellcheck currently says:

    / # shellcheck sc.sh 

    In sc.sh line 5:
        test ${#Foo[*]} -gt 0 && printf ' %q' "${Foo[@]}"
             ^--------^ SC2048: Use "${array[@]}" (with quotes) to prevent whitespace problems.

    For more information:
      https://www.shellcheck.net/wiki/SC2048 -- Use "${array[@]}" (with quotes) t...


#### Here's what I wanted or expected to see:

No warning.

I believe this is an Exception to SC2048:  having `#` there means I'm getting *number of elements*, therefore, while `${foo[*]}` is dangerous, `${#foo[*]}` is perfectly safe -- it is guarranteed to return a single integer.


### Notes

So far, this specific syntax hasn't been raising warning.  I always reckoned the lack of warning intentional for the above reason, but it looks like it might have been "fixed" by fe25a2b from yesterday.

Note that in my codebase, I learned to actually prefer using`*` and unquoted when getting number of elements, because it's more distinctive than the usual array extrapolation:

    "${foo[@]}"     # (A) actual array extrapolation
    "${#foo[@]}"    # (B) length get (safe, no SC now)
    ${#foo[@]}      # (C) length get (still safe)
    ${#foo[*]}      # (D) length get (still safe, more distinctive)

The form (B) is more "safeguarded"  -- and would not raise warning now.

Howeveer, it's pretty similar to the ubiquitous array extrapolation (A), hence it IMHO make sense to always use the form (D) -- "no quotes, asterisk" -- in order to highlight the difference.

You're right, this was a straight forward bug. The test for `$*` checked that the expansion started with `*` which precluded `!` and `#`. The check for `[*]` looked at the modifiers after the variable name, and therefore didn't consider any leading modifiers. Thanks!
