#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-05-20T13:20:31Z](https://github.com/koalaman/shellcheck/issues/380#issuecomment-103884903):

Putting command arguments in a string is an anti-pattern. It fails to work in many cases. You don't want to do it. Use an array. (See http://mywiki.wooledge.org/BashFAQ/050 for discussion of this topic.)

A simpler (and safer) way to do what you want in the first snippet (assuming bash and that you want local as well as exported variables unset) would be to use `unset -v "${!FSTAB_*}"`. Which uses a [bash parameter expansion](http://www.gnu.org/software/bash/manual/bashref.html#Shell-Parameter-Expansion) for "Expands to the names of variables whose names begin with prefix".

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-05-20T16:20:29Z](https://github.com/koalaman/shellcheck/issues/380#issuecomment-103945949):

The [wiki for SC2086](https://github.com/koalaman/shellcheck/wiki/Sc2086) discusses this exact case as an exception to the rule, and suggests using a function instead. 

Directives are scoped to only the immediately following command, so it's generally safe. I.e. if you do

```
#shellcheck disable=SC2046
unset $(env | grep ^FSTAB_ | sed 's,=.*,,')
touch $(date +%F)
```

you'll still get a SC2046 warning for the `$(date +%F)`.

I don't entirely see the utility in manually verifying and marking a variable as word splitting safe vs marking the command word splitting safe (by disabling SC2086 on that command). Is the same code called multiple times with just a few arguments changed? Use a function, that's what they're for.

Another point of interest is that since unquoting variables as the command name is usually intentional (very much unlike arguments), ShellCheck doesn't warn for e.g.

```
make="make -j 4"
$make install 
```

The use case for variable-specific disabling is that it means you can mark the _one_ variable on the complex line that needs to be word-split and not worry about someone later adding, incorrectly, another variable without quotes and not seeing an error about it (or an erroneous edit getting rid of necessary quotes on an existing variable).

On a complex line with multiple expansions that's a useful safety net.

```

    local CMD="$1"      ; shift
    local DIR="$1"      ; shift
    "$CMD" "$DIR" $@
}

```

It would be nice to be able to silence the warning about `$@` without allowing someone to accidentally remove the quotes from `"$DIR"` without getting a warning.

Note the use of double-quotes inside and outside of `${ ... }`.
