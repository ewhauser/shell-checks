# [\#108 Issue](https://github.com/koalaman/shellcheck/issues/108) `closed`: Heredocs are parsed incorrectly

#### <img src="https://avatars.githubusercontent.com/u/442919?v=4" width="50">[dualbus](https://github.com/dualbus) opened issue at [2014-03-07T17:50:09Z](https://github.com/koalaman/shellcheck/issues/108):

```
dualbus@debian:~$ cat heredoc.bash 
unquoted=unquoted
cat <<eof0; echo $unquoted; cat <<eof1
0
eof0
1
eof1
dualbus@debian:~$ bash heredoc.bash 
0
unquoted
1
dualbus@debian:~$ shellcheck heredoc.bash 
dualbus@debian:~$
```

Shellcheck should complain about the unquoted expansion, but it doesn't. I assume that it's because it incorrectly assumes that nothing can follow the heredoc starting token, but according to POSIX the heredoc starts after the end of line, so it should parse it like this:

```
<<token; here goes more code, heredoc has not  started\n
heredoc starts here (after the \n)\n
token\n
```

and not like this:

```
<<token; shellcheck ignores this\n
heredoc starts here (after the \n)\n
token\n
```

Related to this is another bug:

```
dualbus@debian:~$ cat heredoc-1.bash 
unquoted=unquoted
cat <<eof\\
0
eof\
echo $unquoted
dualbus@debian:~$ bash heredoc-1.bash 
0
unquoted
dualbus@debian:~$ shellcheck heredoc-1.bash 
dualbus@debian:~$
```

The shell performs quote removal on the starting part of the token, but not on the ending part.

```
<<remove-quotes(token)
...
token
```

If that notation even makes sense :)

#### <img src="https://avatars.githubusercontent.com/u/1693206?v=4" width="50">[knirch](https://github.com/knirch) commented at [2014-03-07T21:28:51Z](https://github.com/koalaman/shellcheck/issues/108#issuecomment-37069612):

Worth noting about your examples is that shellcheck will not complain about unquoted variables that it knows contains no spaces.

``` bash
unquoted="u n q u o t e d"
cat <<eof\\
0
eof\
echo $unquoted
```

```
$ shellcheck heredoc-1.1.bash 

In heredoc-1.1.bash line 5:
echo $unquoted
     ^-- SC2086: Double quote to prevent globbing and word splitting.
```
