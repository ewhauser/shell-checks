The real one:

```sh
[_version() {
        local v1 op v2 cmp
        v1=$1; shift
        case "$1" in
                =|==|-eq|-ne|-lt|-gt|-le|-ge) op=$1; shift;;
                *) echo "$1 unknown operator">&2; exit 1 ;;
        esac
        v2=$1;shift
        cmp=($(
                sort -V <<-EOF
                        $v1
                        $v2
                EOF
        ))
        local t1=0 t2=0
        if [ "${cmp[0]}" = "$v1" ]; then
                t2=1
        fi
        if [ "${cmp[0]}" = "$v2" ]; then
                t1=1
        fi
        [ $t1 $op $t2 ]
}
MYAPP_VERSION=4.3
if [_version $MYAPP_VERSION -gt 4 ] && [_version $MYAPP_VERSION -lt 4.4.12.3 ]; then echo "MYAPP_VERSION ($MYAPP_VERSION) is in the middle 1"; fi
```

#### <img src="https://avatars.githubusercontent.com/u/836788?v=4" width="50">[luizluca](https://github.com/luizluca) commented at [2018-07-06T16:58:53Z](https://github.com/koalaman/shellcheck/issues/1280#issuecomment-403090926):

"[" is just the same as "test". The only difference I know is that "[" requires "]" as the last command line argument. In bash, it is implemented as builtin command but remember that '/usr/bin/[' does exist. 
Even on a shell that does not accept function names starting with '[', you can:

```sh
$ [xxx() { a; }
sh: syntax error: bad function name
$ echo '[ "$@"' > /usr/bin/[xxx
$ chmod +x /usr/bin/[xxx
$ if [xxx 1 = 1 ]; then echo yes; else echo no; fi
yes
```

Bash "parser" (or any other shell) does not really check anything related to "[". It only cares that everything after an "if" or "while" is a valid command (with valid arguments). It is "[" command that demand "]", not bash parser. It is not different from:

```sh
_version() {
    (...)
}
if _version $MYAPP_VERSION -gt 4 ] && _version $MYAPP_VERSION -lt 4.4.12.3 ]; then echo "MYAPP_VERSION ($MYAPP_VERSION) is in the middle 1"; fi
```

The simple question is: would [_xxx be an acceptable function name for shellcheck (as it is for bash)? 

BTW, the "underscore" is just any character. "[xxx" is also accepted. No need to test as a special case.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-07-08T07:24:56Z](https://github.com/koalaman/shellcheck/issues/1280#issuecomment-403268435):

You're right, of course. ShellCheck doesn't accept all function names that Bash does, and it also purposefully treats a leading `[` as a keyword. This is a conscious tradeoff. The intention is to e.g. allow warning about `if [-x foo ]` even though it's a completely valid way of calling a program/function named `[-x`.

This is a clever use of function names, but I do unfortunately think that allowing ShellCheck to parse it would lead to poorer diagnostics for people with a less solid grasp of the language. I wish there was a workaround to at least not ruin the rest of the script, but there currently isn't. Sorry about that.
