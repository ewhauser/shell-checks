# [\#197 Issue](https://github.com/koalaman/shellcheck/issues/197) `closed`: SC2032 and SC2033 false positives when using screen -X inside a same-named function

#### <img src="https://avatars.githubusercontent.com/u/46915?v=4" width="50">[ssokolow](https://github.com/ssokolow) opened issue at [2014-07-20T08:46:12Z](https://github.com/koalaman/shellcheck/issues/197):

Because of my familiarity with screen and the late hour, It took me a while to figure out why I was getting confusing SC2032 and SC2033 errors from this code.

``` sh
title() {
    screen -X title "$@"
    echo -ne "\033_$@\033\\";
}
```

Turns out, SC2032 and SC2033 don't recognize that `-X <command>` refers to _internal_ screen commands rather than subprocesses and were only complaining because the function _happens_ to share the same name as the internal screen command.

I've worked around this using disable pragmas (I can't rename the internal command and refuse to rename my function) but, given how confused I got and how the SC2032-disabling pragma could theoretically mask real problems by being too broadly scoped, it should probably be fixed in shellcheck.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2014-08-16T18:06:47Z](https://github.com/koalaman/shellcheck/issues/197#issuecomment-52401369):

That is indeed confusing, and it's not something you can easily rewrite. 

f9c346c helps with the latter, so that you can simply quote the word if you're passing it literally: `screen -X "title" "$@"`. 

In a perfect world, ShellCheck would know which arguments will and won't be executed as external commands, but maintaining this kind of flag parsing across commands and implementations is not fun.
