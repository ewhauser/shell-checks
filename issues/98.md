When I think of it, maybe a better course of action would be to for any variable that has been defined before a source, eval, or similar evil that can alter the variable without shellcheck being able to determine its new value, to mark them as unknowns and trigger on unquoted expansions.

``` bash
foo="bar"
source /something
echo $foo # would trigger as it's now unknown
bar="foo"
echo $bar # would not trigger as no source, eval has been executed
```

A different version of SC2086 could potentially be of use to highlight the fact that shellcheck can't determine if it's safe or not.


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2014-02-19T22:47:22Z](https://github.com/koalaman/shellcheck/issues/98#issuecomment-35559639):

This is much harder and less useful than it sounds. The user could set a `trap`, override builtins (either in or outside the script), or use various forms of indirection to set a variable in non-obvious ways.

Shell semantics are not well defined, and can't be reasoned about. ShellCheck just tries to provide useful feedback for common, realistic cases.
