# [\#1007 Issue](https://github.com/koalaman/shellcheck/issues/1007) `closed`: SC1088 should be disabled but still comes up as error.

#### <img src="https://avatars.githubusercontent.com/u/5364273?v=4" width="50">[rsanting](https://github.com/rsanting) opened issue at [2017-09-15T13:21:47Z](https://github.com/koalaman/shellcheck/issues/1007):

#### For bugs
- Rule Id (if any, e.g. SC1000): SC1088
- My shellcheck version (`shellcheck --version` or "online"): 
- [x] I tried on shellcheck.net and verified that this is still a problem on the latest commit
- [ ] It's not reproducible on shellcheck.net, but I think that's because it's an OS, configuration or encoding issue

#### Here's a snippet or screenshot that shows the problem:

```#!/bin/bash
export ALICE=1.0
# shellcheck disable=1036,1083,1088
export BOB={{ salt['pillar.get']('test:test') }}
```

#### Here's what shellcheck currently says:
```In salt/keyhub/files/.test.settings line 4:
export BOB={{ salt['pillar.get']('test:test') }}
                                ^-- SC1088: Parsing stopped here. Invalid use of parentheses?
```


#### Here's what I wanted or expected to see:
I expected shellcheck to ignore SC1088. It might be that it's not possible to ignore this error because of the parentheses.

My editor doesn't give an option to set the file type for this file to Jinja (it's jinja templating that is used here) because of the bash shebang and I also couldn't find a directive for shellcheck to ignore a file (or line) completely. If there is any other way to solve this I would be happy to hear it.


#### <img src="https://avatars.githubusercontent.com/u/5364273?v=4" width="50">[rsanting](https://github.com/rsanting) commented at [2017-09-15T13:51:40Z](https://github.com/koalaman/shellcheck/issues/1007#issuecomment-329788814):

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-09-15T17:15:08Z](https://github.com/koalaman/shellcheck/issues/1007#issuecomment-329843416):

If the script doesn't parse successfully, none of the SC2xxx checks will run so the usefulness of running ShellCheck on it is extremely reduced.

There's currently no workaround for parsing scripts that are not valid shell scripts due to required macro expansions.
