# [\#961 Issue](https://github.com/koalaman/shellcheck/issues/961) `open`: Feature: Detect when something that looks like a variable is used

#### <img src="https://avatars.githubusercontent.com/u/204609?v=4" width="50">[adiroiban](https://github.com/adiroiban) opened issue at [2017-08-03T11:02:36Z](https://github.com/koalaman/shellcheck/issues/961):

You might define a variable like `some_name=something` and if you forget to use it as `$some_name` there should be a warning.


#### For new checks and feature suggestions

- [ x] shellcheck.net (i.e. the latest commit) currently gives no useful warnings about this
- [ x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related


#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/sh
some_var=my_value
echo Here we are $some_var
echo This should trigger an error: some_var
```

#### Here's what shellcheck currently says:

Nothing :)


#### Here's what I wanted or expected to see:

I am expecting to complain about line 3 where `some_var` is used as a string, but most of the time you want to call it as `$some_var` to reference its value

#### <img src="https://avatars.githubusercontent.com/u/8651654?v=4" width="50">[jleffler](https://github.com/jleffler) commented at [2017-11-21T22:11:14Z](https://github.com/koalaman/shellcheck/issues/961#issuecomment-346177855):

One issue could be that the variable is an environment variable, inherited but overridden by the script, and used by programs that the script runs.  There's no way for shell check to know whether the variable would be inherited from the calling environment, or whether it will be used by the programs run by the script.  Thus, detecting this could generate too many false positives.  I'm not entirely sure how plausible this is.  Setting but not using known environment variables could be ignored, but it would be tricky to white-list environment variables relevant to a particular context.
