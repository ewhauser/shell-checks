# [\#468 Issue](https://github.com/koalaman/shellcheck/issues/468) `open`: use `local` or `local -r` when possible

#### <img src="https://avatars.githubusercontent.com/u/1312785?v=4" width="50">[gdbtek](https://github.com/gdbtek) opened issue at [2015-09-21T08:06:26Z](https://github.com/koalaman/shellcheck/issues/468):

I saw many scripts using global variables in functions so I think it's the best practice to detect such issue

```
function myfunc()
{
    myvar='abc'    // is now global var
}
```

avoid creating global var, so the use `local -r` or `local` when possible.

```
function myfunc()
{
    local -r myvar='abc'     // is now local read-only var.
}
```

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2015-09-23T23:27:11Z](https://github.com/koalaman/shellcheck/issues/468#issuecomment-142758986):

> this is normal style for a local var, uppercase = global var

That is _a_ recognized style, but it's not the most _recommended_ style, which is that all-upper-case = system-defined or internal to Bash.

Local or private is usually indicated by a prefix (minimally, an underscore, but preferably longer).

There is a strong case for passing the names of any output vars as parameters, so that such a function starts with `local -n _results=$1`, or ends with something along the lines of `printf -v $1 '%s' "$_results"`.
