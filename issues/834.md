As with many other cases, it should recognize that my inclusion of a `#!/bin/sh` shebang means that this file is intended for Bourne shells like `dash` and `mksh` ([MirBSD Korn Shell](https://www.mirbsd.org/mksh.htm)) which don't recognize bash features like `\[` and `\]` and, instead, pass them through as literal strings.

In this case, the file is the `/etc/profile.d/colors.sh` which I deploy on resource-constrained VPSes where I use `mksh` as the lightest Debian-provided alternative to bash that still has basic tab-completion and command history. (`mksh` takes about 1/3rd the memory of bash when sourcing the same Debian default rcfiles.)

At the very least, I'd expect the incompatibility with `dash`, and `mksh` to be mentioned under Exceptions at https://github.com/koalaman/shellcheck/wiki/SC2025

(Though I can confirm that `busybox ash` has had support for `\[` and `\]` since at least the 1.13.2 release on one of my ARM devices.)

#### <img src="https://avatars.githubusercontent.com/u/46915?v=4" width="50">[ssokolow](https://github.com/ssokolow) commented at [2017-02-24T15:21:16Z](https://github.com/koalaman/shellcheck/issues/834#issuecomment-282317501):

Well, according to the experimentation I did while writing [this script](https://gist.github.com/ssokolow/3759156) (tested in dash, bash, pdksh, ash, zsh, a then-current busybox, and the then-outdated busybox on my OpenPandora ARM handheld), if you want to do it portably, the trick is to use `\033` with the `printf` builtin.


As far as using the `\[` and `\]` escapes, they're hints for cursor movement in the presence of line-wrapping. When I feel like looking up how to detect bash, I'll add them.

(Even if it supported them, `mksh` probably doesn't need them because it scrolls the command-line rather than wrapping it.)

**EDIT:** Supporting `zsh` as well is more complicated, though, because, for all of its similarity to bash, it uses a completely different mechanism for deferred resolution of variables like `$PWD` in the prompt. (You'll just get `$USER@$(hostname) $PWD $` as your prompt if you try to use that script on it.) 

However, since you reminded me, I decided to rewrite the bits of that file which I *can* do from memory (and, as a bonus, it complicated things enough for SC2025 to not realize I was using colour escapes without `\[` and `\]`):

```sh
#!/bin/sh

# Default ls and grep output to colored and make less pass colors through
alias ls='ls --color'
alias grep='grep --color=always'
alias less='less -R'

# Be as portable as possible
# (`tput` is from ncurses-bin. `type` is a builtin in every shell I know.)
if type "tput" 1>/dev/null 2>&1; then
  # If we have ncurses-bin, support esoteric terminals
  # TODO: Figure out setaf vs. setf
  icolor() { tput setaf "$1"; tput bold; }
  creset() { tput sgr0; }
else
  # A dash/mksh-compatible way to use non-literal escape characters
  icolor() { printf '\033[1;3%dm' "$1"; }
  creset() { printf '\033[0m'; }
fi

# Set up a nice, colorized prompt
# (Use colors recognizably different from my local terminal with a mix of
#  immediate and deferred evaluation for efficiency)
if [ "$(id -u)" -eq 0 ]; then
  PS1="$(icolor 5)"'$USER@$(hostname) '"$(icolor 4)"'$PWD #'"$(creset) "
else
  PS1="$(icolor 3)"'$USER@$(hostname) '"$(icolor 4)"'$PWD \$'"$(creset) "
fi

# Ensure we get colors even if a .bashrc overrules us
export color_prompt=yes
```

What the heck. I decided to just bite the bullet and do it. While I only tested it on bash, dash, busybox sh, and mksh, this should be pretty portable:

```sh
#!/bin/sh

# Default ls and grep output to colored and make less pass colors through
alias ls='ls --color'
alias grep='grep --color=always'
alias less='less -R'

# Support bash character-printability hints
if [ -n "$BASH" ]; then
  cstart='\['
  cend='\]'
else
  cstart=''
  cend=''
fi

# Be as portable as possible
# (`tput` is from ncurses-bin. `type` is a builtin in every shell I know.)
if type "tput" 1>/dev/null 2>&1; then
  # If we have ncurses-bin, support esoteric terminals
  # TODO: Figure out setaf vs. setf
  icolor() { printf '%s%s%s' "$cstart" "$(tput setaf "$1"; tput bold)" "$cend"; }
  creset() { printf '%s%s%s' "$cstart" "$(tput sgr0)" "$cend"; }
else
  # A dash/mksh-compatible way to use non-literal escape characters
  icolor() { printf '%s\033[1;3%dm%s' "$cstart" "$1" "$cend"; }
  creset() { printf '%s\033[0m%s' "$cstart" "$cend"; }
fi

# Set up a nice, colorized prompt
# (Use colors recognizably different from my local terminal with a mix of
#  immediate and deferred evaluation for efficiency)
if [ "$(id -u)" -eq 0 ]; then
  PS1="$(icolor 5)"'$USER@$(hostname) '"$(icolor 4)"'$PWD #'"$(creset) "
else
  PS1="$(icolor 3)"'$USER@$(hostname) '"$(icolor 4)"'$PWD \$'"$(creset) "
fi

# Ensure we get colors even if a .bashrc overrules us
export color_prompt=yes
```

#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-02-25T09:36:44Z](https://github.com/koalaman/shellcheck/issues/834#issuecomment-282472841):


> alias grep='grep --color=always'

Just for the record, `--color=always` is not such a good idea, especially if you're redefining an existing command, because it means that

    grep foo <somefile | grep -v foobar

won't behave as expected (because `foo\e[mbar` doesn't match `foobar`).

Recommendation: stick with `--color=auto`; let the user override it on a case-by-case basis.
