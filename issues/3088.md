#### Here's a snippet or screenshot that shows the problem:

* test.sh
```sh
#!/usr/bin/env bash
echo "integer comparison with quoted: ${1}"

if [[ "${1}" -eq 42 ]]
then
  echo "Correct"
else
  echo "Wrong"
fi
```

```sh
echo 'a[$(head -1 /etc/passwd)] + 42'
./test.sh 'a[$(head -1 /etc/passwd)] + 42'
```

> The shell evaluates values in an arithmetic context in several syntax constructs where the shell expects an integer. This includes: `$((here))`, `((here))`, `${var:here:here}`, `${var[here]}`, `var[here]=`.. and on either side of any `[[` numerical comparator like `-eq`, `-gt`, `-le` and friends.

Also affected

* `foo="${untrusted_input}"; (( foo > 10 ))`

#### Here's what shellcheck currently says:

* `"$?"` = 0 aka no issues


#### Here's what I wanted or expected to see:

* `[[ ]]` with Arithmetic Expressions is insecure and can yield to code execution, use `[ ]` instead.




#### <img src="https://avatars.githubusercontent.com/u/21310755?u=9ddd310621ecec4b69eb02fba3a5d8f902a8f001&v=4" width="50">[vimpostor](https://github.com/vimpostor) commented at [2024-11-21 13:40](https://github.com/koalaman/shellcheck/issues/3088#issuecomment-2491196375):

This is also (part of the) bullet point 7 from the famous [BashPitfalls](https://mywiki.wooledge.org/BashPitfalls), which confirms that the only safe alternative is using `[ ]`:

> If the contents of $foo are not sanitised and are out of your control (if for instance they're coming from an external source), then all but [ "$foo" -gt 7 ] constitute an arbitrary command injection vulnerability as the contents of $foo are interpreted as an arithmetic expression

There also is a short [Bash mailing list thread from 2014](https://lists.defectivebydesign.org/archive/html/bug-bash/2014-06/msg00003.html) where this was discussed, but ultimately not fixed.

Here is an [example incident from 2020](https://www.nccgroup.com/us/research-blog/shell-arithmetic-expansion-and-evaluation-abuse/) where this was used to run arbitrary shell commands and I expect more undiscovered CVEs out there, as practically noone knows about this problem.

I think this issue is also more a duplicate of #2679.
