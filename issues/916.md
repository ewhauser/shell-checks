#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash

declare -A indexes
indexes[$((RANDOM%2))]=true
indexes[$((RANDOM%2))]=true
indexes[$((RANDOM%2))]=true

# shellcheck disable=2207
indexes=($(echo "${!indexes[@]}" | tr ' ' '\n' | sort -n))

# now to show that shellcheck should be able to discover a new *regular* array being declared
indexes=()
unset indexes
declare -A indexes
indexes[$((RANDOM%2))]=true
indexes[$((RANDOM%2))]=true
indexes[$((RANDOM%2))]=true
# shellcheck disable=2116,2207
indexes=($(echo '[0]="hiworld"'))
# bash: indexes: $(echo '[0]="hiworld"'): must use subscript when assigning associative array
```

#### Here's what shellcheck currently says:
```
$ shellcheck myscript
 
Line 9:
indexes=($(echo "${!indexes[@]}" | tr ' ' '\n' | sort -n))
         ^-- SC2190: Elements in associative arrays need index, e.g. array=( [index]=value ) .
 
Line 18:
indexes=($(echo '[0]="hiworld"'))
         ^-- SC2190: Elements in associative arrays need index, e.g. array=( [index]=value ) .
```
#### Here's what I wanted or expected to see:

no error since I am redeclaring `indexes` as a normal array. shellcheck should be able to distinguish this due to the use of `$()` inside the declaration.

function fnB() {
