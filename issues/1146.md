#### <img src="https://avatars.githubusercontent.com/u/3129206?v=4" width="50">[matthewpersico](https://github.com/matthewpersico) commented at [2018-05-15T13:49:51Z](https://github.com/koalaman/shellcheck/issues/1146#issuecomment-389173486):

Furthermore, I say the existing solution to the issue is bad advice - consider:

```
#!/bin/bash

set -e

cd /tmp
mkdir -p blah bloh

avar='before all'
echo "value before all - $avar"
cd /tmp
( cd blah
  avar='after blah, with cd -'
  ls
  cd -

  cd bloh
  ls
  echo "avar after first bloh is '$avar'"
)

cd /tmp
( ( cd blah
  avar='after blah in subshell'
  ls
  )

  cd bloh
  echo "avar after second bloh is '$avar'"
  ls
)
```

When you run it:

```
$ /tmp/x
value before all - before all
/tmp
avar after first bloh is 'after blah, with cd -'
avar after second bloh is ''
```

Do you see the scoping problem? By adding a subshell, the variable that you had set between your cd commands is lost to commands after the subshell. Imagine this code being a lot longer than a few lines. Go and follow this advice by adding a subshell, and you'll spend an hour or two trying to figure out why the variable's value is messed up 50 lines later in the code.

My suggestion would be to close this issue and retire this check. Advice to change code should not potentially change the functionality of the script.
