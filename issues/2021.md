#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2020-07-25 22:03](https://github.com/koalaman/shellcheck/issues/2021#issuecomment-663910588):

This is unrelated to loops and associative arrays. It happens when `eval`ing a string without escaping the parentheses. You can see this described on the wiki page for the parenthesis error: https://github.com/koalaman/shellcheck/wiki/SC1036

As described, you can work around it by quoting the eval'd string, e.g. `eval "MyList2+=($X)"`. This does not change the behavior in bash.

The bash parser has some odd quirks due to trying to allow this. For example, you can do `name=foo; eval $name=val` and `eval foo=(42)`, but you can't do `eval $name=(42)`. Similarly, `export() { echo "$1"; }; export foo=(bar)` is syntactically valid (though you can't guess what it does), while `othername() { echo $1; }; othername foo=(bar)` is invalid. The same behavior is why `export var=$foo` does not require quotes in bash, but does on other shells.

ShellCheck copies many of these, but unfortunately trying to mimic them all tends to lead to more unhelpful messages in less advanced cases. Obviously it's never great to have to change working code to please some other tool, but hopefully it's an acceptable price for offering better pinpointed errors to users who are less familiar with the shell.
