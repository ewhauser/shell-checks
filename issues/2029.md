# [\#2029 Issue](https://github.com/koalaman/shellcheck/issues/2029) `open`: Recommend using nullglob in loops

- [x] https://www.shellcheck.net/ (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/sh
PATH=~/.local/bin:"$PATH"
for i in ~/.local/opt/*/bin; do PATH="$i:$PATH"; done
export PATH

```

#### Here's what shellcheck currently says:

No guidance

#### Here's what I wanted or expected to see:

This will iterate once, even if the glob fails to match. Use `shopt -s nullglob` on bash, or explicitly test for file existence on POSIX shell.

(in fact, the default behaviour for globs is problematic in general, but I don't know enough to say whether a general nullglob advice is appropriate)

#### <img src="https://avatars.githubusercontent.com/u/1260405?v=4" width="50">[hseg](https://github.com/hseg) commented at [2020-08-03 19:00](https://github.com/koalaman/shellcheck/issues/2029#issuecomment-668188955):

Judging by <https://unix.stackexchange.com/a/204944>, the appropriate advice seems to be `nullglob` for loops and `failglob` for nonloops. Except that `failglob` kills the command if _any_ glob failed, rather than if no files were passed, which seems to be the more relevant check. Note that that is harder to check for for e.g. `cp` or `mv`, where you have multiple sets of files, and want to fail when any of them are empty (i.e. `mv SOURCES TARGET` where one of `SOURCES` or `TARGET` fails to match).

Indeed, this is the behaviour implemented by [fish](https://fishshell.com/docs/current/index.html#parameter-expansion-globbing)
