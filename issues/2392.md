#### Here's a snippet or screenshot that shows the problem:

```sh
# shellcheck enable=check-extra-masked-returns
x="$(foo)/$(foo)"
```

#### Here's what shellcheck currently says:

```
     ^-- SC2312 (info): Consider invoking this command separately to avoid masking its return value (or use '|| true' to ignore).
```

#### Here's what I wanted or expected to see:

No SC2312. It appears that the second command substitution provokes this false positive, with just `x="$(foo)"` it is not emitted (as it should not be, as this is an assignment that does get terminated in `-e` mode, not an argument to a command which the Wiki page mentions).

#### <img src="https://avatars.githubusercontent.com/u/109152?u=782addb4cabd3acc673e2595ba43377f88414b6b&v=4" width="50">[scop](https://github.com/scop) commented at [2021-11-29 21:20](https://github.com/koalaman/shellcheck/issues/2392#issuecomment-982033686):

Hmm, seems this is not a false positive after all, the second command substitution does indeed mask the return value of the first (assignment doesn't error out in `-e` mode if the latter substitution succeeds), e.g. `x="$(this_will_fail)/$(true)"`

Would be good to make the wiki page cover this case as well as other possible ones and not talk just about command substitution in argument of another command, leaving open for someone who knows the specifics to address.
