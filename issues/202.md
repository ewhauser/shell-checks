# !/bin/bash

VAR1=1
VAR2=2
echo $((${VAR1}+${VAR2}))

-> This will report SC2004, however removing the $ sign does just lead to invalid code ->
echo $(({VAR1}+{VAR2}))

#### <img src="https://avatars.githubusercontent.com/u/255938?v=4" width="50">[ormaaj](https://github.com/ormaaj) commented at [2014-08-08T10:39:32Z](https://github.com/koalaman/shellcheck/issues/202#issuecomment-51586637):

I agree the warning is unclear. The recommendation should be to allow the arithmetic evaluator to reference variables directly except when there's a good reason to do otherwise.

@GaborLukacs is correct in that parameter expansion into an arithmetic string is sometimes the right thing to do. A good example is swapping two variable values:

```
$ (( a = 1, b = 2 )); (( a = b, b = $a )); typeset -p a b
    a=2
    b=1
```

Also note `$((array[$i]))` is always wrong. It should always be written as `$((array[\$i]))` for associative arrays, and `$((array[i]))` for indexed arrays. Unfortunately understanding the reason requires knowing about the undocumented way variable names are looked up in this context.

#### <img src="https://avatars.githubusercontent.com/u/255938?v=4" width="50">[ormaaj](https://github.com/ormaaj) commented at [2014-08-10T00:57:22Z](https://github.com/koalaman/shellcheck/issues/202#issuecomment-51702863):

Problem is it being impossible to have a rule to know whether it's actually necessary. There are a lot of exceptions.

ksh93 has always given this warning:

```
$ ksh -nc '(($i))'
 ksh: warning: line 1: variable expansion makes arithmetic evaluation less efficient
```

But the difference if any depends on whether the shell takes advantage of potential optimizations. Bash tends not to.

The particular problem with #183 was discussed [here](https://lists.gnu.org/archive/html/bug-bash/2014-05/msg00103.html) for reference.
