# [\#250 Issue](https://github.com/koalaman/shellcheck/issues/250) `open`: SC2068: Empty arrays, set -u, and double-quoting

#### <img src="https://avatars.githubusercontent.com/u/166207?v=4" width="50">[mietek](https://github.com/mietek) opened issue at [2014-11-03T16:59:55Z](https://github.com/koalaman/shellcheck/issues/250):

This is, apparently, the right way to pass `"${array[@]}"` to a function, if it can be empty and `-u` is set.

```
In src/main.sh line 381:
                halcyon_deploy ${args[@]:+"${args[@]}"} || return 1
                               ^-- SC2068: Double quote array expansions, otherwise they're like $* and break on spaces.
```

Loosely related to https://github.com/koalaman/shellcheck/issues/242.

#### <img src="https://avatars.githubusercontent.com/u/255938?v=4" width="50">[ormaaj](https://github.com/ormaaj) commented at [2014-11-07T06:05:36Z](https://github.com/koalaman/shellcheck/issues/250#issuecomment-62101669):

The right way to pass an array to a function is to either use a nameref (`typeset -n`), the indirection expansion, or `eval`. To unpack the arguments into parameters, use `f "${args[@]}"`.

`${args[@]:+"${args[@]}"}` has no purpose, and in fact the behavior of `${@:+word}` is unspecified by POSIX, and behaves differently across different shells, even for those with arrays.

#### <img src="https://avatars.githubusercontent.com/u/255938?v=4" width="50">[ormaaj](https://github.com/ormaaj) commented at [2017-01-05T08:12:51Z](https://github.com/koalaman/shellcheck/issues/250#issuecomment-270586987):

@zx2c4 The main points:
* It's not a suitable replacement for a "more proper" exception handling mechanism you might find in a "more proper language"
* Its rules are weird enough to render it mostly useless - it's there for backwards compatibility and posix follows all the broken legacy behaviors to support ancient shells.
* Fundamentally all commands in shell return a status that is essential to many control flow constructs in the language. `set -e` conflates the use of a command for its boolean value and an exception handling scheme. It can't work.

Frankly if I have some real need to test thousands of command statuses in a loop I'd probably resort to some `eval`-based scheme before using `set -e`.

See the bashfaq: http://mywiki.wooledge.org/BashFAQ/105
