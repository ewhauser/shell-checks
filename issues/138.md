# [\#138 Issue](https://github.com/koalaman/shellcheck/issues/138) `closed`: False positive for SC2034 (unused variable)

#### <img src="https://avatars.githubusercontent.com/u/243351?v=4" width="50">[brother](https://github.com/brother) opened issue at [2014-04-16T11:23:46Z](https://github.com/koalaman/shellcheck/issues/138):

I found this warning today, the foo is used to add the default option to the read command in the example, it's not a variable name.

``` shell
read -i "foo" -e -p "Input: " newvalue
        ^-- SC2034: foo appears unused. Verify it or export it.
```

From the help text:
      -e        use Readline to obtain the line in an interactive shell
      -i text   Use TEXT as the initial text for Readline
      -p prompt output the string PROMPT without a trailing newline before
            attempting to read

#### <img src="https://avatars.githubusercontent.com/u/243351?v=4" width="50">[brother](https://github.com/brother) commented at [2014-04-16T11:24:48Z](https://github.com/koalaman/shellcheck/issues/138#issuecomment-40588086):

Adding 

``` shell
foo=1
echo $foo
```

will silence the warning.

Just adding 

``` shell
echo $foo
```

Triggers the word splitting warning SC2086.


#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2014-04-19T16:07:57Z](https://github.com/koalaman/shellcheck/issues/138#issuecomment-40873232):

This has been alleviated in 2c1e414. It's not entirely fixed -- to avoid having to get into which options do or don't take parameters, ShellCheck now just stops looking after the first option. It fixes this particular case, but not .e.g `read -i foo bar` (this appears to be very rare).
