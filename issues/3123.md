# [\#3123 Issue](https://github.com/koalaman/shellcheck/issues/3123) `open`: Does ShellCheck not fully support `env -S`?

#### <img src="https://avatars.githubusercontent.com/u/163816?u=77afc0c27b34e33450514a519f6a28f1ceba4c55&v=4" width="50">[scalisi](https://github.com/scalisi) opened issue at [2025-01-16 12:33](https://github.com/koalaman/shellcheck/issues/3123):

On [2021-09-18](https://github.com/koalaman/shellcheck/issues/833#issuecomment-922361300), @koalaman wrote: 
> ShellCheck has started supporting `env -S`

I tested ShellCheck's support for `env -S` at [ShellCheck.net](https://www.shellcheck.net/) by entering the following code, which produced an [SC2164](https://www.shellcheck.net/wiki/SC2164) warning as expected:

```bash
#!/usr/bin/env -S bash
cd foo
```

As expected, the following code **does not** produce an SC2164 warning:

```bash
#!/usr/bin/env -S bash -e
cd foo
```

Unexpectedly, the following code also **does not** produce an SC2164 warning:

```bash
#!/usr/bin/env -S bash -o foobared
cd foo
```

It seems that the letter "e" in `-o foobared` (as well as any lowercase "e" after `env -S bash`) disables the SC2164 warning. When the letter "e" is removed from `-o foobared`, the code **does** produce an SC2164 warning:

```bash
#!/usr/bin/env -S bash -o foobard
cd foo
```

Does ShellCheck not fully support `env -S`?
