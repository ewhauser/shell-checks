# [\#750 Issue](https://github.com/koalaman/shellcheck/issues/750) `open`: SC2070 (test -n without quoting) needs to be generalized

#### <img src="https://avatars.githubusercontent.com/u/7582443?v=4" width="50">[patrickdepinguin](https://github.com/patrickdepinguin) opened issue at [2016-10-20T08:17:45Z](https://github.com/koalaman/shellcheck/issues/750):

SC2070 detects problems of this type: [ -n $var ] which should be changed to [ -n "$var" ], because the former will always be true, even if var is empty, or only contains whitespace.
See https://github.com/koalaman/shellcheck/wiki/SC2070 for full rationale.

However, shellcheck only complains about -n and [ ], , while the problem is more general and extra checks should be added.

Firstly, [ ] is the same as 'test', and 'test -n $var' is equally wrong.

Secondly, there are other unary checks in 'test' that exhibit problems when the tested variable is unquoted and is empty or whitespace-only, for example -f (file), -d (directory), ...
In each of these cases, an unquoted variable will result in 'true' if the variable is empty or whitespace-only.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-11-25T05:27:06Z](https://github.com/koalaman/shellcheck/issues/750#issuecomment-441416584):

Your reasoning is obviously correct. ShellCheck has historically treated `[` and `test` differently for two reasons:

1. `[` is metasyntactic so people make different mistakes. `[-f file]` is really common but `test-f file` is not. `[` therefore has its own parser that tries to figure out common mistakes rather than parsing the command as a typical shell "simple command"
1. `test` has actually been left alone as a concession. `op='-f'; [ $op file ]` is totally valid, but shellcheck did and still does refuse to parse it due to the special handling above. To allow for such usage, `test` syntax was not strictly enforced so that this could be rewritten as `test $op file`.

Now that ShellCheck is more mature, it may be that the better solution is to add support for `[` statements that don't parse without failing entirely, and start treating `test` more rigidly.
