#### Here's a snippet or screenshot that shows the problem:

```bash

#!/usr/bin/env bash
var=1
([ -v var ]; :)
echo "$var"

```

#### Here's what shellcheck currently says:

```
Line 2:
([ -v var ]; :)
      ^-- SC2030: Modification of var is local (to subshell caused by (..) group).
 
Line 3:
echo "$var"
      ^-- SC2031: var was modified in a subshell. That change might be lost.
```

#### Here's what I wanted or expected to see:

No errors, as `-v` (introduced in bash 4.2) only tests if a variable is set and doesn't modify it as far as I know. Strangely enough, though, it behaves as expected when `test -v` is run explicitly.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-01-14T00:59:00Z](https://github.com/koalaman/shellcheck/issues/1409#issuecomment-453881928):

The intention was a quick hack to avoid a `http_proxy is referenced but not assigned` in cases like this:

    if [ -v http_proxy ]
    then
      echo "Proxy set"
    fi

This subshell warning was an unintended side effect, and is definitely a bug. Nice catch!
