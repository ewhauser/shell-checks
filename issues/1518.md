# [\#1518 Issue](https://github.com/koalaman/shellcheck/issues/1518) `closed`: End-of-options signal to `source` inadvertently triggers SC1091 

#### <img src="https://avatars.githubusercontent.com/u/7399805?v=4" width="50">[cueedee](https://github.com/cueedee) opened issue at [2019-03-14T08:23:07Z](https://github.com/koalaman/shellcheck/issues/1518):

#### For bugs
- Rule id: `SC1091`;
- My `shellcheck` version: `0.6.0`;
- [‚úì] [The rule's wiki page](https://shellcheck.net/wiki/SC1091) does not already cover this;
- [‚úì] I tried on shellcheck.net and verified that this is still a problem on the latest commit;

#### Here's a snippet or screenshot that shows the problem:

```bash
#!/usr/bin/bash

i="-c"             ## This file's name is indeed "-c";
source -- "${i}"   ## The `--` argument is needed to make `source` grok `-c` as a filename argument and not as an (invalid) option to `source`;
                   ## This is just a demo; in reality the `"${i}"` argument would be the loop variable feeded from a glob;
```

#### Here's what shellcheck currently says:

> ```
> source -- "${i}"
>        ^-- SC1091: Not following: -- was not specified as input (see > shellcheck -x).
> ```

#### Here's what I wanted or expected to see:

Nothing since `--` is not an input at all.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-03-18T02:46:52Z](https://github.com/koalaman/shellcheck/issues/1518#issuecomment-473750167):

It looks like ksh/bash/ash/dash all accept `--` even though they accept no options. This has now been fixed in a simplistic way by simply skipping a literal `--` if it exists. Thanks!
