#### Here's a snippet or screenshot that shows the problem:

```
#!/usr/bin/env bash

set -o pipefail -o errtrace -o errexit -o nounset
trap 'echo >&2 trap on error \(rc=${PIPESTATUS[@]}\) near line $LINENO' ERR


test_func() {
  local timeout
  readonly timeout="$(./script_rc_200.sh)"
  echo "Should not see this"
}

test_func
```

         ^-- SC2155: Declare and assign separately to avoid masking return values.



#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-09-16T17:29:06Z](https://github.com/koalaman/shellcheck/issues/1336#issuecomment-421800329):

There's currently no warning because you can't split this into `readonly timeout; timeout=foo` like with the others, but there should arguably be a separate warning explicitly suggesting declare+assign+readonly, e.g. `local timeout; timeout="foo"; readonly timeout`.

#### <img src="https://avatars.githubusercontent.com/u/283886?v=4" width="50">[keith](https://github.com/keith) commented at [2020-10-24T00:32:08Z](https://github.com/koalaman/shellcheck/issues/1336#issuecomment-715643385):

I'm not sure if there was more of a reason this wasn't fixed over the past few years but I submitted one here: https://github.com/koalaman/shellcheck/pull/2077

Note that both of these work as a workaround for this issue:

```sh
foo=$(exit 1)
readonly foo
```

```sh
foo=$(exit 1)
declare -r foo
```

Even if there isn't a workaround as a user I would prefer to get hit by this warning and have to explicitly disable it than be surprised by a runtime failure
