Note ALSO: For POSIX `-nt` is detected by SC2039 as expected, but `-ot` is not.

#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/bash

if [ lex.yy.c -ot program.l ] ; then echo 'lex.yy.c out of date' ; fi
```

#### Here's what shellcheck currently says:
```
$ shellcheck myscript
 
Line 3:
if [ lex.yy.c -ot program.l ] ; then echo 'lex.yy.c out of date' ; fi
     ^-- SC1014: Use 'if cmd; then ..' to check exit code, or 'if [[ $(cmd) == .. ]]' to check output.

$
```


#### Here's what I wanted or expected to see:
```
$ shellcheck myscript
No issues detected!

$
```

Interesting:

```
#!/bin/bash

if [ lex.yy.c -ot program.l ] ; then echo 'lex.yy.c out of date' ; fi
if [[ lex.yy.c -ot program.l ]] ; then echo 'lex.yy.c out of date' ; fi
if [ 'lex.yy.c' -ot 'program.l' ] ; then echo 'lex.yy.c out of date' ; fi
```

```
$ shellcheck myscript
 
Line 3:
if [ lex.yy.c -ot program.l ] ; then echo 'lex.yy.c out of date' ; fi
     ^-- SC1014: Use 'if cmd; then ..' to check exit code, or 'if [[ $(cmd) == .. ]]' to check output.
 
Line 4:
if [[ lex.yy.c -ot program.l ]] ; then echo 'lex.yy.c out of date' ; fi
      ^-- SC1014: Use 'if cmd; then ..' to check exit code, or 'if [[ $(cmd) == .. ]]' to check output.

$
```

So if the filenames are quoted, then there seems to be no issue for shellcheck. All three syntaxes are evaluated properly in bash. I guess that the parser does not see a variable or a quoted string for the first token and so assumes the rest of the tokens are arguments to a command. I would propose the fix is to check ahead and if there are only three tokens and one of them is a comparison operator, then accept as written.



#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2019-11-03T21:49:04Z](https://github.com/koalaman/shellcheck/issues/1737#issuecomment-549182488):

Oops! ShellCheck looks ahead for the classic mistake of doing `[ grep -E foo bar.txt ]` before trying to parse the expression as a test expression, since this will quickly go awry. To do this, it simply looks ahead for a variable name's worth of data (a simple, unquoted alphanumeric string) and compares it to a list of common commands. This avoids the complexity of parsing and comparing a real shell token.

In this case, that variable name's worth of information stopped at the period, and `lex` is a common command name, triggering the warning.

This has now been fixed by simply requiring the command name to be followed by a space. Thanks!
