# [\#5 Issue](https://github.com/koalaman/shellcheck/issues/5) `closed`: False positive warning for ls

#### <img src="https://avatars.githubusercontent.com/u/1823914?v=4" width="50">[arth1](https://github.com/arth1) opened issue at [2013-07-24T18:44:57Z](https://github.com/koalaman/shellcheck/issues/5):

This warning is incorrect:

ls -1N | cat
^-- Don't parse ls output; it mangles filenames.

When used with -N and a pipe, ls does not mangle file names, but passes them on raw.  What you have after the pipe might mangle them, of course, but that's a different issue.

#### <img src="https://avatars.githubusercontent.com/u/1823914?v=4" width="50">[arth1](https://github.com/arth1) commented at [2014-03-10T19:42:16Z](https://github.com/koalaman/shellcheck/issues/5#issuecomment-37225189):

Because ls is an external command (unless you use tcsh), it is non-portable by nature.  Sure, it could be BusyBox ls or a legacy Unix system, but if warning against that, surely one would have to warn about hundreds of commands and non-portable options, including (but not limited to) "ps -m", and "find -print0" | "xargs -0" and even "echo -n".  One could, I suppose, check for POSIXLY_CORRECT or "set -p" and limit the well-known commands when that's in effect.

I'm of the opinion that when external programs are called, the caller is responsible for the arguments being supported where the script is run.  It seems ... easier.
