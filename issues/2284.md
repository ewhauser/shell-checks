`shellcheck` is wonderful, and I greatly appreciate all the work you've put in, @koalaman!

I use it with `zsh` scripts with pretty good success, despite many features not being supported.

## Background

One thing that would make this easier is the ability to skip lines entirely from the parser, which is especially important for constructs that `shellcheck` does not understand (e.g. [`=(...)` process substitution in `zsh`, see the end of the section](https://zsh.sourceforge.io/Doc/Release/Expansion.html#Process-Substitution), in addition to `<(...)` as in bash).

Any chance for something like this making it in? I expect the AST has to parse things before the directives are parsed, but it would be nice to have something like:

```
#!/usr/bin/env zsh
# shellcheck shell=bash
foo() {
  ls -la "$1"
  cat "$1"
  echo goodbye > "$1"
}

# shellcheck skip
foo =(echo hello)
```

Without the ability to skip lines like this, we end up with fatal syntax error messages like:

```
$ shellcheck myscript

Line 9:
foo =(echo hello)
    ^-- SC2283: Remove spaces around = to assign (or use [ ] to compare, or quote '=' if literal).
     ^-- SC1036: '(' is invalid here. Did you forget to escape it?
     ^-- SC1088: Parsing stopped here. Invalid use of parentheses?
```
#### Details

In this case, the `=(echo hello)` construct creates a scoped temporary with the contents of the subshell (in this case, "hello") and the temporary file is automatically deleted after the function returns.  It would be nice to get `shellcheck` to be able to skip lines entirely to permit this.

You can do lots of clever things with this, and create a few brain-busters, e.g. `(){<<<$1} =(:)` prints the name of the auto-deleted temporary file.

## Workaround

I've worked around this locally by modifying `src/ShellCheck/Parser.hs` to modify:
```
x <- oneOf "<>"
```

... to be ...

```
x <- oneOf "<>="
```
... and things work fine.  I know you don't support `zsh`, and this is a very unusual construct, but it would be nice to either get this integrated (unlikely, as it's not valid bash/ksh) or have a way to instruct `shellcheck` to not parse a given line.

## Proposed Mechanisms

It would be nice to have "skip this line" and "skip this region of lines" that `shellcheck` would ignore, e.g.

*Single Line*
```
# shellcheck skip
aksl ^#&@ ()*#$&dfja sdl faskjdhfl akjsf
```

*Multi-Line*
```
# shellcheck skip-start
asdf
afaw8ua9sfh9h8  *^ 
jdhf98y9O*u342
# shellcheck skip-stop
```
