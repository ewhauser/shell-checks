# [\#2573 Issue](https://github.com/koalaman/shellcheck/issues/2573) `open`: New grep versions and warnings about stray /

#### <img src="https://avatars.githubusercontent.com/u/6011536?v=4" width="50">[matteocorti](https://github.com/matteocorti) opened issue at [2022-09-13 13:33](https://github.com/koalaman/shellcheck/issues/2573):

#### For new checks and feature suggestions

According to POSIX regex standard:

```
"An ordinary character is a BRE that matches itself: any character in the
supported character set, except for the BRE special characters listed in
BRE Special Characters.
The interpretation of an ordinary character preceded by an unescaped
( '\' ) is undefined, except for: [...]"
```

grep is giving a warning since 3.8:

```
$ grep -V
grep (GNU grep) 3.8
Copyright (C) 2022 Free Software Foundation, Inc.
License GPLv3+: GNU GPL version 3 or later <https://gnu.org/licenses/gpl.html>.
This is free software: you are free to change and redistribute it.
There is NO WARRANTY, to the extent permitted by law.

Written by Mike Haertel and others; see
<https://git.sv.gnu.org/cgit/grep.git/tree/AUTHORS>.

$ grep ' ' somefile > /dev/null 
$ grep '\ ' somefile > /dev/null 
grep: warning: stray \ before white space
$ grep '[ ]' somefile > /dev/null 
```

#### Here's what shellcheck currently says:

Nothing

#### Here's what I wanted or expected to see:

Should warn about the usage of stray \

OP's code:

  <sh>
  echo ' href="123" ' \
    | grep -o -e ' href=\"[^"]*\" '
  </sh>

I don't know enough about Haskell to know whether this level of granularity
is possible, but whether it's POSIX-2024 `grep`, or GNU `grep`, the operand
to '-e' appears to me to be a malformed BRE.

What happens when a script tries to `eval` a `grep` command with this kind
of RE syntax, or when word splitting is intended, etc. -- when a level of
quoting is intended to be reduced?
