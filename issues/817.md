# [\#817 Issue](https://github.com/koalaman/shellcheck/issues/817) `open`: SC2034 doesn't consider namerefs

#### <img src="https://avatars.githubusercontent.com/u/10643?v=4" width="50">[zx2c4](https://github.com/zx2c4) opened issue at [2017-01-05T01:28:52Z](https://github.com/koalaman/shellcheck/issues/817):

Shellcheck doesn't account for namerefs properly.

#### For bugs
- Version: 4.5 and online
- I tried on shellcheck.net and verified that this is still a problem on the latest commit
#### Here's a snippet or screenshot that shows the problem:

```bash
#!/bin/bash
read_bool() {
        local -n out="$1"
        case "$2" in
        true) out=1 ;;
        false) out=0 ;;
        *) die "\`$2' is neither true nor false"
        esac
}
```

#### Here's what shellcheck currently says:

```
        false) out=0 ;;
               ^-- SC2034: out appears unused. Verify it or export it.
```


#### Here's what I wanted or expected to see:

Nothing.

7.5 year old issue now. Constant annoyance in our workflow. SC2034 is very helpful when it works correctly in identifying stray vars that need to be cleaned up, but cannot recognize any kind of nameref usage; not with a `declare/local -n`, not when adding entries to that ref, and not when using the ref in another function:
```
myfunc() {
    local -n out="${1}" str="${2}"
    str="${out[*]}"
}
array=(1 2 3)
declare -n ref="array"
ref+=(4)
myfunc ref string
echo "${string}"
```

#### <img src="https://avatars.githubusercontent.com/u/46978960?v=4" width="50">[marc-hb](https://github.com/marc-hb) commented at [2025-08-14T23:04:58Z](https://github.com/koalaman/shellcheck/issues/817#issuecomment-3190122829):

https://www.shellcheck.net/wiki/SC2034 says

> This is an intentional design decision and not a bug. If you have variables that will not have direct references, consider using an associative array in bash, or just [ignore](https://www.shellcheck.net/wiki/ignore) the warning.

> Tracking indirect references is a common problem for compilers and static analysis tools, and it is known to be unsolvable in the most general case.

On the other hand, all the warning examples are about the "inside" name, not the outside name. They're also all reading, never writing. So, does this design choice extend to the example in the description above? I don't know.

#### <img src="https://avatars.githubusercontent.com/u/46978960?v=4" width="50">[marc-hb](https://github.com/marc-hb) commented at [2025-08-14T23:23:09Z](https://github.com/koalaman/shellcheck/issues/817#issuecomment-3190177849):

A reminder that you do not have to disable the warning on the entire file. For instance this is shellcheck clean:

```shell
#!/bin/bash
read_bool() {
        local -n out="$1"
        case "$2" in
        true)
            out=1 ;;
        false)
            # shellcheck disable=SC2034
            out=0 ;;
        *) die "\`$2' is neither true nor false"
        esac
}

```
