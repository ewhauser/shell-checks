# [\#954 Issue](https://github.com/koalaman/shellcheck/issues/954) `open`: No-op line continuation preceding comment.

#### <img src="https://avatars.githubusercontent.com/u/1366303?v=4" width="50">[cheako](https://github.com/cheako) opened issue at [2017-07-14T14:30:44Z](https://github.com/koalaman/shellcheck/issues/954):

#### For new checks and feature suggestions
- [ ] shellcheck.net (i.e. the latest commit) currently gives no useful warnings about this
- [ ] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related


#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/sh

echo yes \
# comment
```

#### Here's what shellcheck currently says:

```
$ shellcheck myscript
No issues detected!
```

#### Here's what I wanted or expected to see:

```
$ shellcheck myscript
Line 3:
echo yes \
         ^-- SC####: Remove \ at end of line, it does nothing.
```

#### <img src="https://avatars.githubusercontent.com/u/151337?v=4" width="50">[nicknovitski](https://github.com/nicknovitski) commented at [2018-01-29T17:42:07Z](https://github.com/koalaman/shellcheck/issues/954#issuecomment-361325421):

I had a related problem (really just from my own ignorance):

```bash
#!/usr/bin/env bash
pssh --hosts "$(host-inventory)" \
         --askpass \ # TODO: fix key authentication
         "$@"
```

ShellCheck reports no problem with this.  But of course, it doesn't work at all.  I thought `\` indicated a line continuation by itself, but it's actually _escaping the line break_, so if it isn't followed by a line break, but say, a space and a comment, then nothing works and everything is terrible.

I imagine something like:
```
$ shellcheck myscript
Line 2:
         --askpass \ # TODO: fix key authentication
                     ^-- SC####: Comment preceded by \, did you mean to escape a line break? 📎
```

I think that's just a generalization of this issue across the one-line and two-line cases. 

#### <img src="https://avatars.githubusercontent.com/u/336433?v=4" width="50">[kevinoid](https://github.com/kevinoid) commented at [2019-03-21T23:50:12Z](https://github.com/koalaman/shellcheck/issues/954#issuecomment-475445456):

I'd suggest also warning about a no-op line continuation following a comment.  I've run across code similar to the following:

```sh
echo \
        # Comment about option/argument \
        true
```

The backslash is discarded due to [shell tokenization rule 9](https://pubs.opengroup.org/onlinepubs/9699919799/utilities/V3_chap02.html#tag_18_03), making the code equivalent to:

```sh
echo
        # Comment about option/argument
        true
```

Which is not likely to be what the author expected (whether they expected `true` to be part of the comment or an argument to `echo`).
