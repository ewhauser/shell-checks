#### Here's a snippet or screenshot that shows the problem:

```sh
#!/bin/sh

while read -r a; do
        read -r b < file.txt
done < file.txt

echo "$a" "$b"
```

#### Here's what shellcheck currently says:
```
Line 4	SC2094: Make sure not to read and write the same file in the same pipeline.
Line 5	SC2094: Make sure not to read and write the same file in the same pipeline.
```
#### Here's what I wanted or expected to see:
No warning.

The warning should only show up if there is a mixture of reads/writes. Multiple reads should always be safe. Multiple writes *might* be safe, I'm not sure about how the flushing behaviors are defined (I doubt they are well defined, so this is probably unsafe).



#### <img src="https://avatars.githubusercontent.com/u/2818758?u=40d36e49d938fda80abd46d6347b9bace1f73dd4&v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2021-09-19 01:16](https://github.com/koalaman/shellcheck/issues/2325#issuecomment-922397450):

This warning was already suppressed for multiple redirections (e.g. `>>`), and is now similarly suppressed for two inputs like here. Thanks! 

You still have similar false positives of two reads with something like `cat foo - < foo`, but this is harder for ShellCheck to determine. You can also safely do concurrent appends as they are atomic under a certain size except on NFS etc, but really the check is primarily intended to catch the very common `cat foo | sed 's/a/b/' > foo` type pitfalls.
