#### For bugs
- Rule Id (if any, e.g. SC1000): SC2039
- My shellcheck version (`shellcheck --version` or "online"): 0.4.6

#### Here's a snippet or screenshot that shows the problem:

```sh
$ head foo.sh saywhat.sh 
==> foo.sh <==
#!/bin/sh
source ./saywhat.sh

==> saywhat.sh <==
echo output from saywhat.sh
```

```
In foo.sh line 2:
source ./saywhat.sh
^-- SC2039: In POSIX sh, 'source' in place of '.' is undefined.
$ shellcheck foo.sh

In foo.sh line 2:
source ./saywhat.sh
^-- SC2039: In POSIX sh, 'source' in place of '.' is undefined.
^-- SC1091: Not following: ./saywhat.sh was not specified as input (see shellcheck -x).
```

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2018-05-13T22:43:12Z](https://github.com/koalaman/shellcheck/issues/1181#issuecomment-388661422):

This is because ShellCheck normally doesn't output warning for included files without `-a`.

The problem here is that the AST ends up being `T_Include _ (source command) (included script)` so that the `source` command is considered part of the included script instead of the original script.

One possible fix would be to add a second node type, e.g. `T_SourceCommand _ (source command) (T_Include $ originalScript)` so that the source command is no longer under the `T_Include`
