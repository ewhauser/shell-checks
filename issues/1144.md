#### <img src="https://avatars.githubusercontent.com/u/19507282?v=4" width="50">[automorphism88](https://github.com/automorphism88) commented at [2018-03-26T01:46:49Z](https://github.com/koalaman/shellcheck/issues/1144#issuecomment-376023758):

I can see this being useful in some cases, but for your script, why not put the usage message in a heredoc, and avoid the call to sed, and extra complication? You can either do something like this:

```
usage() {
    cat - <<'EOF'
Usage message goes here
EOF
}
```
or, you can go further and avoid using any external processes if you add a function to replace cat, something like the following:
```
cat_stdin() {
    local tmp
    read -r -d '' tmp && true
    # the '&& true' will avoid triggering set -e
    printf '%s' "$tmp"
}
```
in which case you'd just replace `cat - <<'EOF'` with `cat_stdin <<'EOF'` in the `usage()` function. (You can also combine them into one function if you want.) See the [bash manual](https://www.gnu.org/software/bash/manual/html_node/Redirections.html#Here-Documents) for more information on heredocs.

Also your shebang `#!bash` is invalid, unless bash is located in the current directory, since `PATH` lookups aren't done for shebangs. You should specify the absolute path, e.g. `#!/bin/bash`, or if you need the `PATH` lookup, you can use the env trick: `#!/usr/bin/env bash` which will use the first bash in your `PATH` (but has other side effects you may not want). [This Wikipedia article](https://en.wikipedia.org/wiki/Shebang_(Unix)#Portability) explains how it works.

What the more elaborate code would have shown is that I am not only doing this for scripts - I am also providing the same functionality for functions. Since functions are loaded into the environment, I can't have a `usage` function in each function script - as I source them in, they would clobber each other. And having a `foobar_usage` function for each function foobar would clutter the environment.

Even though I am in bash, I use the ksh convention of keeping all my functions in directories that I delineate in an `FPATH` envvar. In this way, finding the source for any function (because I keep each function in a public interface in it's own similarly-named file) is as simple as 

    find $(echo $FPATH | tr ':' ' ') --name $func_name

I don't exactly do that, however - that `find` is buried in a personally modified version of the `autoload` demo script that comes with the bash source code.

My ultimate goal is making it incredibly easy to write usage/help for all shell stuff - functions and scripts. All I want the developer to have to do is write the help text with as little plumbing as possible in their part. Use a template - fill it in. And I don't want to have to "install" these things to get man pages generated and put somewhere else. And I want to keep usage text with functions to make it more likely that they'll be updated in lock step and the usage text won't fall behind.

```
## -*- sh -*-

# <Function Justification: env access>

func-usage ()
{
    ##@@ none||none||func-usage||Reads '${FUNCNAME}' or first arg for a __[EO]USAGE__ tag pair and prints everything between.
    if [ "$1" = '-h' ]
    then
        func-usage ## Nice how it works on itself, no?
    else
        ## You might pass the function name if you are in a wrapper function
        ## and the help is in the wrapped function OR if you want to call this
        ## at the command line, as a 'man' substitute.
        local funcname="$1" 
        
        ## If function name not passed in, assume the calling function.
        [ -z "$funcname" ] && funcname="${FUNCNAME[1]}"
 
        tgt=$(autoload -w "$funcname") ## A personally modified version of the
                                       ## autoload demo script that comes with
                                       ## bash. -w is "where" and it locates
                                       ## the file in which the function was
                                       ## defined (effectively a find command
                                       ## on FPATH).
 
        [ -z "$tgt" ] && return "$NOT_OK"

        ## And once we have a filename, we just call script-usage on it.
        script-usage "$tgt"
    fi
}

return 0

And here is `script-usage:`
```
# -*- sh -*-

# <Function Class: util>
# <Function Justification: assumes $0>

script-usage ()
{
    ##@@ none||none||script-usage||Reads '$0' or first arg for a __[EO]USAGE__ tag pair and prints everything between.
    if [ "$1" = '-h' ]
    then
        func-usage ## Can we get any more meta? :-)
    else
        ## You might pass the script name if you are in a wrapper script
        ## and the help is in the wrapped script OR if you want to call this
        ## at the command line, as a 'man' substitute.
        declare tgt="$1"
        [ -z "$tgt" ] && tgt="$0"
        [ ! -r "$tgt" ] && tgt=$(which $tgt)
        [ -z "$tgt" ] && return "$NOT_OK" ## We get an error message from the which command 
                                          ## if not found
        ## TODO: Look for marker: if __USAGE__ then printf, if __POD__ then pod2text
        printf '%s\n' "$(sed -e '0,/^__USAGE__$/d' -e '/^__EOUSAGE__/,$d' "$tgt")"

        ## If this script is a wrapper around another script, and you don't
        ## want to repeat all of the wrapped help text in the wrapper help,
        ## just put a script-usage command in your help between these chain
        ## markers and we'll execute that, sucking in that help too. We assume
        ## that the chained help comes after your help. TODO: Allow for
        ## embedded help in the above text.
        declare chain
        chain="$(sed -e '0,/^__USAGE_CHAIN__$/d' -e '/^__EOUSAGE_CHAIN__/,$d' "$tgt")"
        [ -n "$chain" ] && $chain
    fi
}

return 0

/dev/null<<'EOSTRINGX'
__USAGE__
script-usage - search the script being run for usage text and print it

The text is identified by placing the word __USAGE__ on its own line at the end
of your function script, followed it by usage text and the __EOUSAGE__
marker. Any time in your script that you want to display the text (as part of
-help processing perhaps?) just call the 'script-usage' function. It will read

  Your usage text here and below...
  ...
  The last line to print.
  __EOUSAGE__
  EOSTRING

text when the script is executed. ( If you don't already explicitly exit in
your script, you do now. You're welcome. :-). )
