#### <img src="https://avatars.githubusercontent.com/u/197253?v=4" width="50">[kurahaupo](https://github.com/kurahaupo) commented at [2017-04-01T00:32:26Z](https://github.com/koalaman/shellcheck/issues/867#issuecomment-290878763):


Even though that syntax is accepted by Bash version 4.3, Shellcheck should
NOT say "No issues detected!"

Firstly, this syntax is specific to Bash (for a long time -- since version
2, apparently). For Bourne, ksh, mksh & others, Shellcheck is correct:
brackets are indeed not allowed there. Regardless, it's always been
recommended that the parameters to "let" be quoted if they contain shell
meta-characters, the same as any normal command.

Secondly, even for current Bash, that enhancement does not extend to
disabling globbing; before running your example, type this:

    touch 'x=(3+5)+22'

and then try your unquoted example:

    let x=(3+5)*2
    echo $x

which prints "30" instead of "16".

So you think you're safe because you don't have any "weird" filenames lying
around?

Then try this

    rm x=*
    shopt -s nullglob

and then try your unquoted example:

    let x=(3+5)*2

which now says

    bash: let: expression expected

(and doesn't change the value of x)

Even if you get around both of those, consider how much more slowly your
script is going to run if every arithmetic operation results in an attempted
(but failing) glob expansion.

Thirdly, please use $((...)) or ((...)) instead of let "..."
