# [\#524 Issue](https://github.com/koalaman/shellcheck/issues/524) `closed`: Feature: command-line option to filter warnings by severity

#### <img src="https://avatars.githubusercontent.com/u/908960?v=4" width="50">[JIghtuse](https://github.com/JIghtuse) opened issue at [2015-10-27T07:45:01Z](https://github.com/koalaman/shellcheck/issues/524):

First of all, thanks for the great tool! I love it.

Now, question. Can we have some command-line option to display warnings of specific severity level? Or at least some document about severity of different types of warnings. I would like to implement git pre-commit hook to check scripts going to commit, and I need severity for this. pre-commit should tell about most awful issues in code, but be silent about minor ones.

For example, one time I discovered that SC2050 can be dangerous when we do some check and forgot to add $ symbol:

```
if [ "err" = "0"]; then
```

But things like SC2086 are less interesting. Currently I need to `--exclude` lots of warning types to get only needed ones for my script (and exclude new warning types when they will appear). Some option would be very helpful.

You can see a nice implementation of this behavior in cppcheck. Without arguments, it displays only the most critical warnings. You can use option `--enable={style,warning,performance}` to get additional warnings you are interested in.

#### <img src="https://avatars.githubusercontent.com/u/302055?v=4" width="50">[deryni](https://github.com/deryni) commented at [2015-10-27T14:50:06Z](https://github.com/koalaman/shellcheck/issues/524#issuecomment-151527113):

SC2086 is not less interesting. It is also quite important. Fail to quote and your arguments will be wrong, you will get filenames instead of the string you expected, etc.

Try this for example:

```
$ mkdir test
$ cd test
$ touch file{1,3}
$ echo file[1] file[2]
```

If you were expecting to get the square brackets in your output or your argument you would get something quite different there and that could cause any number of problems.

As to the question specifically [this comment](https://github.com/koalaman/shellcheck/issues/470#issuecomment-147223322) by koalaman has a workaround of sorts that can be used for now.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-10-27T17:43:07Z](https://github.com/koalaman/shellcheck/issues/524#issuecomment-151583292):

The reason why this was not immediately implemented is because it's not obvious what makes an error important. 

The error level is currently some kind of sum of:
- average severity (what's likely to happen ~95% of the time)
- worst-case severity (what's likely to happen ~5% of the time)
- certainty (how likely it is that that these bad things will happen)
- confidence (true positive rate)
- smell (probability that this should be rewritten regardless). 

This means that SC2086 (unquoted variables) and SC2045 (for loops over ls) are ranked as info and error respectively, even though they break in similar ways under similar circumstances. They both have relatively low severity and certainty, but the latter has high confidence and smell. 

A programmer would agree that code smell is bad and increases priority of an issue pre-commit. A build server would strongly object that it's important in marking the build as broken.

A philosophical debate on this would be useful, but it's likely that there will be some kind of "build server" vs "precommit" vs "troubleshooting" vs "nitpicking" option that just weights things differently.
