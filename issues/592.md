# [\#592 Issue](https://github.com/koalaman/shellcheck/issues/592) `open`: Checking 'cd', and 'exit' vs 'return' in functions

#### <img src="https://avatars.githubusercontent.com/u/16140513?v=4" width="50">[drosehn](https://github.com/drosehn) opened issue at [2016-02-03T21:16:18Z](https://github.com/koalaman/shellcheck/issues/592):

Consider the case where someone wants to write a function which behaves like a specialized version of the  `cd` command.  Here's a simple (and stupid) example of :

```
#!/bin/bash
function cdalt () {
   if [ "$1" ] ; then
      cd "$1"
      touch "Kilory_IsHere"
      echo  "Kilroy has moved to: $(pwd)"
   else
      cd
      echo  "Kilroy has returned home"
   fi
}

```

If you paste that into ShellCheck, it makes the recommendation:
`SC2164 Use cd ... || exit in case cd fails.`

So you look at the code, and realize that this is a **very** prudent recommendation! If the real `cd` fails, you certainly don't want to be touching files in whatever directory you happened to be in when the error occurred.  However ...

Do you really want to do an **exit** there?  If the function does an `exit`, then it will exit the **shell** which called that function, and not the just function.  And that shell might be your login session, and thus `cdalt /tmp/NoSuchDir` will log you out.  When it comes to a specialized version of `cd`, I doubt anyone wants a `cd` which logs them out when they make a typo.

This also makes me wonder about `exit` vs `return` inside functions vs inside a shell script.  Might it also be prudent to warn the user **anytime** they use `exit` in a function?  I am sure that I have functions where I use `exit` because `exit` is exactly what I want.  But I do know there have been several times where I've used `exit` when I really wanted `return`, and then I was very unhappy when the function executed that `exit`.

FWIW, the version I ended up with was:

```
#!/bin/bash
function cdalt () {
   if [ "$1" ] ; then
      cd "$1"    || return $?
      touch "Kilory_IsHere"
      echo  "Kilroy has moved to: $(pwd)"
   else
      cd         || return $?
      echo  "Kilroy has returned home"
   fi
}

```

and as near as I can tell in my testing, that behaves the way I want it to behave.  (my script is much more elaborate and useful than this example, of course!)

I like how SheckCheck notices the bare `cd`, and that it is smart enough to **not** warn if the shell code is already checking the return code of the `cd`.  I have no issue with the way it checks `cd`.

However, when ShellCheck recommends that the user should add `|| exit`, that suggestion may result in a worse situation **->if<-** the code is inside the definition of the function.  The issue is what should ShellCheck recommend for code inside a script vs the exact same (incorrect) code when it is inside a function.  My earlier `altcd` example was probably too silly.  Consider:

```
#/bin/bash
function hcd {
  cd ~"/WorkProjects/$1/head"
}
```

This is a function you might add to ~/.bashrc because it is a function that the **user** will be typing simply to save them a few keystrokes.

For this specific example, it is better to leave the function as it is than to add `|| exit`.  If the user types in some function and that function does `exit`, it's the user's shell which is exited.  I have made this mistake myself, and then some minor error when using that function causes my ssh session to be terminated.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2016-02-04T20:12:51Z](https://github.com/koalaman/shellcheck/issues/592#issuecomment-180028411):

@drosehn It's great to hear that even extremely experienced shell scripters can use ShellCheck to discover new and interesting aspects of their scripts!

In this case, you're right of course -- you don't want to be logged out when cd'ing to a directory that doesn't exist.

It's not entirely straight forward though. How can you tell that your script is intended to be sourced by an interactive shell? What if continued like this:

```
#!/bin/bash
function cdalt () {
   if [ "$1" ] ; then
      cd "$1"    || return $?
      touch "Kilory_IsHere"
      echo  "Kilroy has moved to: $(pwd)"
   else
      cd         || return $?
      echo  "Kilroy has returned home"
   fi
}

cdalt ~/data
tar cv * | ssh host 'dd bs=1M of=data_backup.tar'
```

Optimally ShellCheck should perhaps be tracing the flow of `cd` through functions, so that any point in a function call chain that ends up changing the directory will also get a warning similar to `cd`. 

As for `exit` inside a function, I'd tend to strongly agree -- a function that decides to exit the script on error is not a great design decision. Then again, you have valid and useful idioms like `die() { echo >&2 "$*"; exit 1; };  make || die "Build failed";`. Further investigation is required.

The issue might be phrased as:  How does ShellCheck know if a snippet of code is part of a full-blown shell script, or is it for use in ~/.bashrc ?  I'm not saying that is an easy thing to solve, especially since people might not paste in the entire source file that they are working on.  (certainly I never pasted in my **entire** .bashrc.  I just pasted in a few dozen lines at a time).

Maybe ShellCheck could try to do what you did with your follow-on example.  If ShellCheck sees that `cdalt` is referenced, then it might "prefer to assume" the code is part of a script file.  I'm not saying that it would try to figure out exactly what the code is doing with `cdalt`, but only if that function is referenced.  And if ShellCheck sees zero references to `cdalt`, then it might prefer to assume that the code is part of .bashrc.

But the more I think about it, there probably is no reliable way for ShellCheck to determine that distinction on it's own.  The user would have to provide an explicit hint.  How about this:  When I was testing bits-and-pieces of my .bashrc file, I always started with the `#!/bin/bash` line, because ShellCheck needed to know my code was for bash and not for standard `sh`.  How about if it recognized `#!bashrc`?  Or maybe `# shellcheck bashrc`?  Or does it already do that?  I must admit I read very little of the documentation!
