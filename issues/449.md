# Parameters: none
# Global
#   uses: COLUMNS, pglen, idx, prompt{1,x,2}, p{1,x,2}len, ftype
#   sets: files, menu, extracase, ptn, cols, nwidth, page, pwidth, prompt

  while IFS= read -rd '' fraw; do

  ((cols+=menulen>pglen&&cols<2))
  ((ipp=cols*pglen))
  ((lastpage=menulen/ipp+!!(menulen%ipp) ))
  page=1

  if ((lastpage==1))
    then pwidth=. extra= extralen=0 extracase=

    #   on a standard 24-lines terminal 99 menu pages translate into at least

      [tT]*) page=1;;
      [bB]*) page='$lastpage';;
      [nN]*) ((page+='$lastpage'>page));;
      [pP]*) ((page-=page>1));;
    '

#### <img src="https://avatars.githubusercontent.com/u/4294278?v=4" width="50">[schragge](https://github.com/schragge) commented at [2015-08-30T13:11:07Z](https://github.com/koalaman/shellcheck/issues/449#issuecomment-136135624):

> ``` bash
> eval r=({1..$menulen}); IFS=\|$IFS ptn="${r[*]}" IFS=${IFS#|}
> ```

After giving it some more thought, now I think that instead of this monstrosity I should just have gone with _seq_ like what the comment in the code suggests:

``` bash
ptn=`seq -s'|'  $menulen`
```

Besides conveying my intent much more clearly, it also scales better. When doing this, I obviously was trying to avoid unnecessarily forking another process. While most packages come with only a few  files installed under /usr/share/doc, there're some like glibc-doc or texlive-doc with hundreds of directory entries. Brace expanding such a range, then saving it temporarily in an array just to interpolate it into a string with | as separator is probably less efficient than constructing the string with _seq_. Or, like GrayCat puts it in his excellent [BashPitfalls](http://mywiki.wooledge.org/BashPitfalls#for_i_in_.7B1...24n.7D):

> brace expansion pre-expands every argument which can be slower
> and unnecessarily consumes memory

Restricting it to only small values of $menulen like this

``` bash
if ((menulen<20)); then
  local -a r
  eval r=({1..$menulen}); IFS=\|$IFS ptn="${r[*]}" IFS=${IFS#|}
  unset r
else
  ptn=`seq -s'|'  $menulen`
fi
```

seems like just complicating things for no apparent gain.

Nevertheless, evaluating brace expanded range to interpolate variable values seems like a legitimate bash idiom that should not give parser errors.

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2015-09-03T04:19:15Z](https://github.com/koalaman/shellcheck/issues/449#issuecomment-137326593):

These are some interesting cases!

`(( !!a ))` failing is a straight up parser bug: it has been fixed in 01c27dc. 

`eval` is weirder. By standard parsing rules, `eval foo=(bar)` would not be allowed. `bash` treats it specially and allows what would normally be valid array assignments. `ksh` additionally allows things like like `eval $var=(bar)`, which bash does not. 

Here's how different magic values affect parsing in `bash`:

```
$ foo() { echo "$@"; }; eval() { echo "$@"; }; export() { echo "$@"; }
$ foo a=(b)
bash: syntax error near unexpected token `('
$ eval a=(b)
a=(b)
$ export a=(b)
a
```

7f547cc adds a warning saying to escape special characters to avoid relying on this fragile behavior.
