# [\#536 Issue](https://github.com/koalaman/shellcheck/issues/536) `closed`: Recommend "command -v" over "if which X" or "if type X" to see if there's a command

#### <img src="https://avatars.githubusercontent.com/u/813150?v=4" width="50">[david-a-wheeler](https://github.com/david-a-wheeler) opened issue at [2015-11-03T13:47:58Z](https://github.com/koalaman/shellcheck/issues/536):

Many people use "if which X" or "if type X" (or similar) to see if a command exists, but these aren't really portable.

The POSIX specification defines "command -v X", which is the portable way to do this - but a lot of people aren't aware of it.  Please point out the non-portable constructs, and explain that "command -v" is the portable way to determine it.

#### <img src="https://avatars.githubusercontent.com/u/6459309?v=4" width="50">[Artoria2e5](https://github.com/Artoria2e5) commented at [2015-11-03T14:05:49Z](https://github.com/koalaman/shellcheck/issues/536#issuecomment-153365234):

`type` is portable, at least XSI defines it: http://pubs.opengroup.org/onlinepubs/9699919799/utilities/type.html

Using `command -v` suffers from the same problem as `type` over `which`, that it tries non-PATH things which would not be useful when you try to call other programs. `type -P`, a bashism, doesn't have the problem though.

#### <img src="https://avatars.githubusercontent.com/u/6459309?v=4" width="50">[Artoria2e5](https://github.com/Artoria2e5) commented at [2015-11-04T17:12:32Z](https://github.com/koalaman/shellcheck/issues/536#issuecomment-153795558):

For `command -v` there may still be a way through, basically:

``` sh
whichable()( unalias -a; case "$(command -v "$1")" in (alias*) false;; (*/*) true;; (*) false;; esac )
```

The problem here is when the command name is things like `[`, `test`, or even `cd`…
