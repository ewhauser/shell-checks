# [\#1823 Issue](https://github.com/koalaman/shellcheck/issues/1823) `open`: Feature request: Directive to source file w/o subsequent `source` or `.` line

#### <img src="https://avatars.githubusercontent.com/u/646121?u=aa4d9efea9d7883e20365c1145f3bc9c3e9162f7&v=4" width="50">[lucianposton](https://github.com/lucianposton) opened issue at [2020-02-06 10:29](https://github.com/koalaman/shellcheck/issues/1823):

- [x] shellcheck.net (i.e. the latest commit) currently gives no useful warnings about this
- [x] I searched through https://github.com/koalaman/shellcheck/issues and didn't find anything related

The source directive requires a following `source` (or `.`) line in order for the specified file to be recognized by shellcheck e.g.
```sh
# shellcheck source=src/examples/config.sh
. "$(locate_config)"
```

I encountered two cases where a script was sourced through other means such that there was no such line above which to put a `# shellcheck source=` directive. As a consequence, I had no way to direct shellcheck to find the source required to analyse the rest of the file.

The bash testing framework, BATS, is an example of this. It has a `load` function (it is defined prior to your test script being sourced) that replaces `source`.

Would it be possible to introduce a directive that causes shellcheck to source a file w/o the need for a subsequent `source` or `.`?
