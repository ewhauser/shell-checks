- SC1117

#### Here's a snippet or screenshot that shows the problem:

```sh

#!/bin/bash
echo -e "\e[92m Color Text! \e[0m"
```

#### Here's what shellcheck currently says:
```sh
echo -e "\e[92m text here \e[0m"
^-- SC1117: Backslash is literal in "\e". Prefer explicit escaping: "\\e".
^-- SC1117: Backslash is literal in "\e". Prefer explicit escaping: "\\e".
```

#### Here's what I wanted or expected to see:
It should pass, recognizing that the -e enables escape chars and that `\e[XXm` is color coding

I think the warning is correct, see the explanation here. https://github.com/koalaman/shellcheck/issues/1000#issuecomment-329943333

#### <img src="https://avatars.githubusercontent.com/u/2818758?v=4" width="50">[koalaman](https://github.com/koalaman) commented at [2017-11-20T03:52:09Z](https://github.com/koalaman/shellcheck/issues/1055#issuecomment-345584795):

This is indeed intentional. `\\e` makes it clear that the shell should pass a literal backslash to `echo`, so that `echo` can interpret it. This is what's already happening with `\e`, just in an obscured way.
